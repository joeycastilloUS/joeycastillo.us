nous.api — the model gateway, and where Tickets + determinism fit

"Should we use nous.api? Is this where AI Pantheon + Model Garden go behind it? Does nous.api need the determinism/infra dance?" Recon says: yes to the gateway, no to determinism — and it clarifies the whole stack.
The answer in one line: nous.api is your probabilistic model gateway (live, pure C, the 38-model Pantheon behind it). Determinism stays in the prime (IDS + recall + safeguard, over the wire). The edge composes both. Fold GCP Model Garden in as provider rows and nous.api becomes the single BYO-BIG3 gateway — exactly what you asked.

What nous.api IS (found + externally verified live)

Not a separate repo — it lives in the nous substrate as a skill + a server:

  • The skill intelligence/skill/api/ — "deliberately thin — it routes, it does not reason." Public surface = four verbs (ask · judge · summarize · reconsider) via api_dispatch(); anything else = API_ERR_VERB. Composes fanout/judge/summarize/reconsider.
  • The server intelligence/tool/nousapi-server/nousapi_server.c (~112KB, pure C, static-musl FROM scratch, own TLS 1.3 — no OpenSSL, no SDKs). Routes: open GET / (freemium console) + GET /v1/models, bearer-gated POST /v1/{ask,judge,summarize,reconsider} + composed /v1/factcheck + /v1/research + /v1/quota.
  • Live: Cloud Run nous-api in acmedev-platform-hq/us-west4, rev 00013 (2026-06-18), min-instances 1. Probed: /v1/models → 200 JSON, /v1/ask no-token → 401. Health check via /v1/models (the GFE intercepts /healthz).
  • Auth: app-level Authorization: Bearer + BYO keys per-request (keys:{"ANTHROPIC_API_KEY":"..."}), RAM-only, resolution request-key → sealed vault → store triple → env.

What it fronts: models only — the AI Pantheon, literally. memory/skill/providers/providers.c is a 38-row catalogue of direct vendor APIs (Anthropic, OpenAI, Gemini, xAI, DeepSeek, Mistral, Perplexity, Cohere, Reka, Groq/Llama…). It does NOT front the substrate — no understand/learn/recall, no IDS, no wire to the prime. Its wire/relay integration is spec-ahead-of-code (aspirational).

Your three questions, answered

A. Route Tickets through nous.api instead of a bespoke edge? No — keep the edge; the edge calls nous.api for the verdict. nous.api has no recall verb, no wire client, no path to the sealed IDS — and its contract forbids adding one (API_ERR_VERB). The Hollow Edge isn't a deviation — it's the pattern nous.api's own header names: "the C# edge is the blind REST front; it never sees keys or plaintext." The edge composes two calls: (1) recall over relay→wire→prime (deterministic), (2) verdict via nous-api /v1/ask|/v1/judge (probabilistic).

B. Put the Pantheon + Model Garden behind nous.api? Yes — the Pantheon already is; fold Vertex in as provider rows. Model Garden is the gap: gcp-bridge (C# jacket) fronts Vertex via ADC/service-account — a different auth model than the vendor keys in providers.c. Don't teach pure-C nous.api Google OAuth (that's the jacket's job — language law). Add provider rows vertex-claude/vertex-gemini whose endpoint = the gcp-bridge URL and key_env = BRIDGE_TOKEN. Result: nous.api = the single model gateway, gcp-bridge demotes to a jacket behind it, and Tickets' /vertex/analyze call migrates to nous-api /v1/ask with providers:["vertex-gemini"].

C. Does nous.api do the determinism/infra dance? No. The split is already in the code: nous.api is stateless per request, holds no store, self-describes as routing-only. Determinism — sealed IDS, recall, the atom-diff safeguard — is the GCE prime's job over wire/TLS. Putting recall in nous.api would expose the substrate over REST (kernel violation), drag the sealed store into an autoscaling container, and break the trust story: a sealed deterministic answer must come from the thing that holds the seal.

Hollow Edge — C#, Cloud Run, blind REST front (the designed edge role)
   |-- recall  (DETERMINISTIC):  NTRP triples -> relay -> wire/TLS -> GCE prime -> sealed IDS
   +-- verdict (PROBABILISTIC):  HTTPS bearer -> nous-api /v1/ask | /v1/judge
                                   |-- fanout -> 38-provider AI Pantheon (direct vendor APIs, BYO keys)
                                   +-- vertex-* rows -> gcp-bridge (C# jacket) -> Vertex Model Garden

Three roles, cleanly split: prime = deterministic (IDS, recall, understand/learn, wire only) · nous.api = probabilistic (model fanout, judge, BYO keys, quota) · edge = blind translation + composition (no logic, no keys, no plaintext).

Three moves (in order)

  1. Tickets keeps the Hollow Edge — swap its direct gcp-bridge verdict call for a nous-api /v1/{ask,judge} call. The recall half is unchanged (relay→wire→prime).
  2. Add the vertex-* provider rows to providers.c (endpoint = gcp-bridge, key_env = BRIDGE_TOKEN) so Model Garden joins the Pantheon behind the one gateway. gcp-bridge demotes to a jacket.
  3. Leave nous.api's wire/relay integration as its later phase — the path by which nous.api itself becomes wire-addressable. Nothing in Tickets waits on it.

Honesty: nous.api is real + live as a REST model gateway (shipped 2026-06-18); the relay→edge→wire→api_dispatch topology in its docs is not built yet. The above leans only on what exists + was verified today.
