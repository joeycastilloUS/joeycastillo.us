Design — Recovery

A new Argus tab that puts the whole NPS mission on screen: the unhappy-customer queue, the AI's read, the trust verdict, the model panel, and the gated make-it-right fire control. Months of substrate, finally visible.

1. Need

Months of work — perception layer, atom-diff oracle, nous.api gateway, raw IDS, gated-write lane — and the Argus app looks exactly the same. Every deliverable landed behind the glass: bridges, boards, C, telemetry. The operator cannot point at the app and show anyone (including himself) what changed. Meanwhile the NPS mission (boards NPS0–NPS6) — resolve the unhappy customer, read the complaint, judge it, propose the make-it-right, fire it through the gate — is 100% board+backend with no surface that houses it. The need under the subject: a screen where the machine's judgment is watchable and the operator's hand is on the only trigger. Not a report about the work — the work, on screen, in the app.

2. Raw User Stories

  • As the operator, I want to open Argus and see the queue of unhappy customers, most urgent first, so the mission stops being invisible.
  • As the operator, I want each customer's card to tell me who they are, how mad they are, and what actually went wrong in plain language, so I don't re-read the whole ticket thread.
  • As the operator, I want a visible verdict that the AI's read is grounded in what the ticket literally says (green) or isn't (amber), so I never act on a hallucination.
  • As the operator, I want to see which models judged it and whether they agree, so "the AI decided" becomes named models with a live gateway behind them.
  • As the operator, I want the proposed apology + coupon previewed byte-for-byte before anything fires, and I want firing to require my explicit, typed, armed action — nothing auto-sends, ever.
  • As the GR team, I want food-illness tickets to visibly exit the lane (escalated, no draft, no coupon), so the never-auto rule is on screen, not just in the code.

5. Shape

Minimum viable structure — five zones on one card, one queue above them:

  • The queue — open detractors (low scores pulled read-only from Zendesk via zendesk-bridge), most-urgent first (score asc, then age desc). Each row: who, store, score, one-line why, age, verdict dot, status. Food-illness rows render ESCALATED and never enter the fire lane.
  • Identity + score — who hurts: name, store, NPS score chip, ticket id (deep-links to Zendesk), age, severity.
  • The AI's read — plain-language account of what went wrong + recovery-worthiness, from the perception judge. Cost + latency line under it (the honesty line: what this read cost).
  • The trust verdict + judge panel — the trust layer, ON SCREEN: green GROUNDED / amber UNVERIFIED from the deterministic atom-diff (nous understand() oracle — the digest is only the words in the thread; it cannot hallucinate). Beside it, the panel of models that judged (via nous.api at api.3-nous.net) with per-model verdicts + agreement count. UNVERIFIED hard-disables the fire control — the gate obeys the oracle.
  • Make-it-right + the gate — proposed action (apology draft in Zora's voice + coupon code peeked read-only from the comp sheet — the sheet is never written; a human enters the code at checkout). Below it the gate: write-mode OFF by default, literal REST call previewed byte-for-byte, capture-before noted, typed confirm phrase, Fire button dead until armed + typed + grounded. Server (zendesk-bridge WriteGate) is the authority; the UI gate is UX.

Out of scope (YAGNI):

  • Wisdom-loop UI (NPS6) — corrections land in Channel A invisibly for now; a "teach it" affordance is a later board.
  • Tone-bakeoff A/B UI (NPS3) — the backend picks the draft; showing three drafts is later.
  • Writing the coupon sheet, auto-sending anything, batch-fire — never in this design.
  • Any placement outside the Zendesk app; any new bridge endpoint invention beyond what NPS0–NPS5 already board.

Escape Routes earned — the gated write is irreversible-ish

Load-bearing choiceReversal path
Firing a recovery reply (real email to a real customer) Zendesk fields (tags, assignment, comment record) are capture-before + rollback-able via the proven remediation lane. ⚠ The outbound email itself has NO reversal — which is exactly why the lane is write-off-by-default, preview-byte-for-byte, typed-confirm, oracle-gated (UNVERIFIED cannot fire), and food-illness never enters it.
Coupon code exposed in a draft Cheap: the code is worthless until a human enters it at checkout; a leaked code is voided on the sheet by a human. The sheet is never written by Argus, so there is nothing to roll back.
Building the UI on sample data first Cheap by construction: RECOVERY_SAMPLE is shaped as the real payload contract; going live = deleting the constant, not reworking the pane. If the contract drifts, the sample is the test fixture that catches it.
The tab name "Recovery" Rename before anything external links to it = a label + id swap, minutes. After cap-URLs/briefs cite it, it's a find-replace plus stale-link risk — so the name is settled NOW, in this design.
