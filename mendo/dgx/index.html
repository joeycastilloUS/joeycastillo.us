  • Sparkfly widget on the Checkout page itself, captured during the evening re-run:

Both surfaces read directly from Mendocino's loyalty backend. Both surfaces, in two distinct contexts (account page after an order; Sparkfly widget inside the checkout flow), are returning empty for jcastillo@mendocinofarms.com — the test identity Laney mapped to the HubSpot workflow on 2026-05-11. This is not a UI rendering miss. The system itself has nothing to show this new member.

Three Places To Look

→ First check

1. Did the HubSpot workflow fire on account create?

The chain Mendo described starts with HubSpot — account-created event triggers the workflow that enrolls the new member in Sparkfly. Laney has visibility into whether that workflow actually fired for jcastillo@mendocinofarms.com. If it didn't fire, that's the root cause and nothing downstream matters until it's fixed. If it did fire, the chain moved to step 2.

→ Second check

2. Did Sparkfly enroll the new member, and into which segment?

Assuming HubSpot did its part, Sparkfly should have received the enrollment and placed the new member into a segment. If the enrollment didn't happen, the bridge between HubSpot and Sparkfly needs attention. If the enrollment happened but the member landed in a segment with no eligible offers, that's a configuration question (step 3).

→ Third check

3. Is there a current offer eligible for new members at Lab Toast Demo Vendor?

If HubSpot fired and Sparkfly enrolled, the last possibility is the offer side: there is no active wallet offer targeted at new members for the Lab Toast Demo Vendor location at the time of the test. This is a content/config question rather than a system question — verifiable in the Sparkfly admin in a couple of minutes.
