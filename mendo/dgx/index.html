The Short Version

Tonight's run automated the Mendocino Farms ordering test all the way from a fresh email through Olo's brand-new-customer signup, with a working Cloudflare bypass and a working SMS-to-email fallback. We were one step from completing an end-to-end test order — that step being "read the 6-digit verification code from the test mailbox" — when Google Workspace's automated-browser detection blocked the inbox login. The full test script is ready, the workaround is identified, and we have three viable paths to clear the block.

What Was Delivered Tonight

✓ Delivered

Olo signup automation — working

The Olo "Sign Up for an Olo Account" form is now driven end-to-end: first name, last name, phone number, terms acceptance, and submit. We discovered along the way that Olo's phone field is required despite being marked optional in the page code, and that Olo wraps every input inside a custom web component — both findings now baked into the driver.

✓ Delivered

Cloudflare bypass — holding

Cloudflare blocks straight automation at the front door of order.mendocinofarms.com. We launch a real Chrome window, you (the human) solve the Cloudflare challenge once, and the test framework attaches to that same Chrome session over the developer protocol. Tonight's session ran for four hours through the same Chrome window without a single Cloudflare re-challenge. This is the foundation everything else stands on.

✓ Delivered

Olo defaults to texting the verification code to the phone on file. With our test phone being a fictional number (industry-standard 555 prefix), no text ever arrives. The driver now detects this and clicks "Send code by email instead" automatically. Olo confirmed the email send: "Sign in code sent to mendo1000_sparkfly@mendocinofarms.com".

A new browser-driver script can run any single step of the flow on demand — fill the signup form, switch delivery channel, paste a verification code, dump the page state for diagnosis. This is the production tooling for both automated runs and manual-assist runs. It covers Olo, Gmail, and the Mendocino menu page.

The One Thing That Got In The Way

⚠ Blocker

Google Workspace blocks automated logins to the test mailbox

Olo emails the 6-digit verification code to the test account's mailbox at @mendocinofarms.com, which is a Google Workspace domain. To finish the test, the test framework needs to read that email automatically. When we tried to sign into Gmail through the same Chrome window the framework drives, Google's risk system flagged it as automation and rejected the login with "Couldn't sign you in. Google couldn't verify this account belongs to you."

This is a known Google Workspace protection — it isn't a bug we can patch around in our code. It needs a Workspace-side answer.

Three Ways To Clear The Block

→ Recommended

1. Gmail API service account (Workspace admin, ~30 min)

The cleanest, most production-ready answer. Mendocino's Workspace administrator creates a service account with read-only access to the test mailbox. The driver fetches the latest Olo email through Google's official API — no UI login, no risk-system rejection, fully headless. This is the standard pattern for automated test rigs that need email codes. Rotates cleanly, audits cleanly, scales to as many test accounts as we want.

→ Fast

2. Pre-warmed Chrome profile

You sign into Gmail manually one time, complete two-factor authentication, and we save that Chrome profile. Future test runs reuse the saved session, skipping login entirely. Simpler than option 1, but cookies expire periodically and need re-auth every few weeks.

→ Manual

3. Human-in-the-loop (immediate)

Tonight's path. Joey reads the code from the test mailbox by hand, types it into the driver, and the rest of the flow runs unattended. Works today with zero infrastructure changes, but requires a person available during each test run. Good for one-offs, not a fit for nightly automated runs.

Recommendation

Start with option 3 to validate the rest of the flow end-to-end this week (one supervised run, full Flow 1 + Flow 2 + Flow 3). In parallel, request the Gmail API service account from Workspace IT — once that's live, the entire suite goes nightly with no human in the loop, results posted here at the same URL the next morning.
