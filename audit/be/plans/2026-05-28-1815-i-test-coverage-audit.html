~i Test Coverage Audit — nous

Every test file in kastil-systems/nous, counted and characterized. The honest state of "are we tested" — feeds the ~p professional-grade test strategy (same datetime).
Headline: 70 test files exist — real investment — but they run on at least three different harness styles with no unified runner, ~480 assertions are concentrated in ~16 files (comms + security + mesh-trust), and ~50 files register zero assertions under a CHECK/assert sweep — they're run-and-eyeball integration smoke tests, not pass/fail-gated. There is no single make test, no CI gate, no coverage measurement. Strong pockets, no system.

§3 — The gaps (ranked)

Gap #1 — No system: no unified runner, no CI, no coverage Tests are built ad-hoc by security/Makefile (make test/test-deep), intelligence/build_only.bat (builds ~15 test exes), and standalone gcc lines in file headers. There is no root make test that builds + runs everything and returns a pass/fail count, and no CI that runs on every commit. A regression in an untested-by-CI path ships silently. This is the single biggest gap — not any one missing test, but the absence of a test system.

§4 — Verdict

For "best of the best," the problem isn't effort — 70 files, 480+ assertions, real fuzz, NIST vectors — it's consistency and automation. The spine (crypto, wire, mesh trust) is genuinely strong. To reach professional grade we need to turn a pile of good tests into a system: one harness, one runner, a CI gate, measured coverage, and the adversarial/negative tests that a security product lives or dies on. That's the ~p (same datetime).
