§1 — The origin story (why was audit specced?)

From the a-d-d: nous's Intel Object model includes a provenance field defined as "hash-chained log — every edit, share, access — cryptographically chained." The audit skill spec frames it as:

  • Immutable record — complete operational transparency
  • Court-admissible — evidence chain of custody (photos/video/audio hashed at capture with device, timestamp, GPS)
  • Tamper-evident — detect manipulation of any operational record, including the history itself

This is forensic-grade design for legal/intelligence operations — law enforcement, emergency response, regulated data handling. The spec is sound but oriented to a regulated future, not today's 5-node prime/relay mesh.

§4 — Honest go/no-go: DEFER

Why not BUILD:

  • No current operational need — Joey runs a 5-node field-intel mesh, not a regulated/evidence system
  • Incomplete without offline witness — audit on a compromised system can be re-written by the attacker; useful audit requires periodic external signing/storage (offline USB, third-party witness) which is out of current scope
  • Provenance can be solved simpler — provenance-as-triples integrates cleanly with store and is query-friendly
  • Journal already does half the job — write-ahead log records every mutation (just not hash-chained); for "what changed" forensics, journal replay is sufficient today
  • Low current ROI — building ~400 LOC + integration to address a scenario Joey hasn't encountered yields negative ROI

Why not DROP:

  • Real future value — when nous scales to multi-org federation, law enforcement use, or formal supply-chain roles, audit becomes mandatory
  • The spec is solid — well-reasoned, aligns with Rekor/auditd/Git patterns, not over-engineered
  • Low cost to hold — spec is frozen; revisiting it next year when context changes is fast

Trigger conditions for revisit:

  • Joey integrates with a law enforcement / government agency (compliance demand)
  • nous federated sharing crosses organizational boundaries with formal agreements
  • Joey needs "prove this data was never tampered with" as part of a contract or legal discovery
  • An incident occurs where lacking audit would have prevented disambiguating a dispute

§5 — If/when BUILD happens, the implementation sketch (for the file)

nous/memory/skill/audit/
  audit.c    ~250 LOC  — append-only log, SHA-256 hash chain, mmap'd file
  audit.h    ~80 LOC   — interface (audit_append, audit_verify, audit_walk) + types
  CLAUDE.md  + DIAGRAM.md + TRIPLES.md + skill.json

Entry shape (128 bytes fixed):
  action     uint8_t       write | read | share | delete | access | refactor
  obj_id     uint8_t[32]   SHA-256 of intel object
  actor_id   uint8_t[32]   node or user ID
  timestamp  uint64_t      ns UTC
  prev_hash  uint8_t[32]   SHA-256 link to previous entry
  _padding   uint8_t[11]   to 128 bytes

Storage:  audit.bin (mmap'd, hash-chained)
Deps:     hash (already in nous), journal (for crash safety)
Perf:     spec target 39K ops/sec (hash + disk write per append)
Hooks:    3-4 audit_append() calls in store.c on write/delete/share
Effort:   ~1 week dev + 1 week test/integration. Not critical path.
