~p Test Strategy — Professional Grade

Turns the 70-file pile (from the ~i audit, same datetime) into a test system: one harness, one runner, a CI gate, measured coverage, and the adversarial tests a security product lives or dies on. "Best of the best" is a discipline, not a count.

Intent

Make every commit to nous provably-tested: one command runs everything, returns a pass/fail count + exit code, a CI gate blocks regressions, coverage is measured (not guessed), and every trust-bearing parser is fuzzed + attacked. Each MH hardening board ships paired tests (Rule 7).

Scope

In: a unified CHECK harness + root make test; CI workflow; gcov/llvm-cov coverage; fill the trust-root unit gaps; adversarial/negative + fuzz expansion; per-MH-board tests; on-VM e2e smoke automation. Out: rewriting the ~50 run-and-eyeball intelligence smoke tests wholesale (triage, don't carpet-bomb); GUI/desktop tests.

Approach

Don't boil the ocean. Standardize the harness once, wire a runner + CI, then raise coverage where risk is highest (security + mesh trust first, intelligence later). Adopt the existing security/Makefile test/test-deep pattern as the template and lift it to the repo root. Negative-test every parser that eats attacker-controlled bytes. Pair MH-board tests with the MH boards so test and feature ship together.

The test boards

TEST-1 — Unified harness + runner + CI · 18 pts · the #1 gap

One shared check.h (CHECK macro + pass/fail tally + exit code). A root make test (or test.sh) that builds + runs every assertion-gated test and prints N passed, M failed with a non-zero exit on failure. A CI workflow (GitHub Actions on kastil-systems/nous) running it on every push — the gate that makes "tested" automatic.

Totals

5 boards · 84 pts. Critical path: TEST-1 first (it makes everything else automatic), then TEST-2/3 in parallel, TEST-4 rides alongside the MH hardening boards, TEST-5 last. If you do one board: TEST-1 — a runner + CI converts the existing 480 assertions from "ran once on someone's box" into "green on every commit."

Constraints

  • Pure C, no test framework dependency — the harness is a header (check.h), not a vendored library. Stays house-style.
  • Sub-5-second default make test (fast feedback); slow/fuzz/on-VM in test-deep/test-cov/nightly.
  • Tests are committed source + run in CI — "ran on my box" doesn't count (Rule 8: verifiable by a third party).

Unknowns (named)

  • How many of the ~50 zero-assert intelligence tests are worth converting vs. deleting vs. leaving as smoke — needs a triage pass (TEST-2 item is a guess at ~10).
  • CI runner availability for nous (private repo) — GitHub Actions minutes / self-hosted? affects TEST-1 + TEST-5.
  • Coverage floor number (85% is a starting target) — may need tuning once first measured.
  • On-VM smoke against a scratch node vs a live prime — must not perturb production (TEST-5 design risk).

When to revisit (v2 signal)

  • First CI red that a human had to notice manually → the gate isn't wired tight enough; fix TEST-1.
  • First security regression that tests missed → add the adversarial case that would've caught it (grow TEST-4).
  • Coverage floor blocks a legitimate change repeatedly → re-tune the floor or refactor the untestable code.
  • PQC / crypto-agility lands → whole new test axis (algorithm negotiation, downgrade) — re-plan.
Connects to the mesh hardening: TEST-4 is intentionally the same work as the per-board tests inside MH-2..MH-5 — this plan just names the test discipline explicitly so "professional grade" means measured + automated + adversarial, not "we wrote some tests."
