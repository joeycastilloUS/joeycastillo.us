§1 — What pantheon-dispatch actually is

A single Cloudflare Worker (src/index.ts, 245 LOC TypeScript) that accepts jobs and fans them to GitHub Actions via repository_dispatch. Created 2026-02-19, last pushed 2026-02-22 — dormant ~3 months. Private, not archived. Pantheon-branded throughout.

EndpointPurpose
POST /jobsSubmit a job → round-robin pick a worker repo → repository_dispatch (event_type: execute-job). Returns a job id.
GET /jobs/:idJob status — stub ("status lookup not yet implemented").
POST /jobs/:id/{started,complete,failed}Worker callbacks, API-key gated. Each is a TODO ("Update database") — logs only.
GET /workersLists the configured worker repos as "available" (hard-coded, no real status).
GET /healthReturns "OK" — but the worker is not deployed (live check: 404).

Fan-out targets (from wrangler.toml WORKER_REPOS): ai-pantheon/pantheon-worker-{alpha,beta,gamma} — each is a GitHub-Actions runner repo that would receive the dispatched job. Job types envisaged: dotnet | cli | python | nodejs; providers: github-actions | gcp | cloud-run.

§3 — clauderun verdict (move?)

The repo's purpose (orchestrate distributed job execution) is clauderun-shaped, so it was worth checking as a move candidate. It is not one:

  • Non-functional. No persistence, status lookups unimplemented, callbacks are logging stubs, callback URL is a placeholder. There is no working orchestrator to relocate — only a scaffold.
  • Pantheon-platform era. The "anonymized requests from Privacy Layer → route to Cloud Run / GCP workers" model is the same multi-tenant-cloud vision being retired (cf. AP-13 sentinel ARCHIVE).
  • Better as a fresh design. If clauderun needs a dispatcher, the repository_dispatch fan-out pattern is captured here for reference; that's cheaper to re-derive than to carry a dead CF-Worker stub.
