PLAN — AWAITING APPROVAL

Get from 127 misorganized repos to loadable-from-metal.

A plan, not a list. Sequenced so each phase makes the next one cheaper. v1 hypothesis — revision is the default, not the exception.

Source: ~i audit (2026-05-20 01:32) · 9 orgs · 127 repos · 10 findings

Intent

Joey said it: "I can't plan giant workloads when everything is kind of all over the place." That's a structural problem, not a tooling problem. The audit named the structure. This plan fixes it.

Concrete end-state: when Joey types ~l <name> in metal console, the result lands where his head expected. Today, half the names are ambiguous (which metal-console? which a-d-d?) and the other half live in the wrong org (Philz repos in Acme, Pantheon repos in Acme).

The win condition is measurable: a re-run of ~i at the end of this plan finds zero red findings. Amber and informational findings are fine — they're the next plan's problem. Red is "you can't even trust the structure."

Scope

In scope

  • Repo transfers across orgs (using gh repo edit --visibility-or-transfer; history preserved)
  • Repo renames (collision fixes — primarily the two metal-consoles)
  • Repo archives (isArchived=true) for dead scaffold
  • One-line org charter READMEs (so each org's purpose is visible from the org page)
  • One-line descriptions added to every active repo
  • Declaration of canonical methodology source (kastil-systems/a-d-d)
  • Redirect-only READMEs in non-canonical a-d-d copies
  • Re-running ~i after each phase to verify drift drops

Out of scope

  • Refactoring code inside any repo (organizational, not implementation, work)
  • Building anything new (lifewater.io product, audit's /audit public surface) — those are downstream
  • Pruning the ai-pantheon Feb-22 scaffold beyond Joey's per-repo decision (audit recommends; Joey calls)
  • Defining Province-Light-Foundation's charter (one decision, not a plan)
  • Deciding ai-pantheon-internal's fate beyond a single fork-in-the-road
  • Touching the healthy orgs: philzcoffee, Mendocino-Farms (leave them alone)

Approach

Intent → Purpose → What

Intent. Make the org list self-describing. The first time someone (including Joey six months from now) opens GitHub and looks at the org list, they should know what each org is for without reading any repo.

Purpose. Org = audience contract. One question per org: who is this for? Repos that don't fit the audience get moved. Once the audience contract holds, ~l becomes trivially correct because the name + org pair is unambiguous.

What. Four phases. Each one re-runs the audit at the end and proves it dropped a finding. Stop-the-bleeding first, then move, then prune, then document. The order matters — moving before declaring canonical creates ambiguity at higher stakes.

The audit is the discipline. Every phase ends with a green re-run or the phase isn't done.

Phases

Phase 1 — Stop The Bleeding

Goal: declare what's true. Before anything moves, the canonical truth has to be named.
  • Declare kastil-systems/a-d-d as the canonical methodology source. Add a one-line README header: "Canonical. The five other copies are mirrors or archives — see audit-of-all for the list."
  • Replace the README in the 5 non-canonical a-d-d copies with a redirect: "This is a stale mirror. Source of truth: kastil-systems/a-d-d." No code changes yet — just the README so anyone landing there sees the redirect.
  • Rename one of the two metal-console repos. Decision: probe the contents of both first (Phase 0 of this phase — a 5-minute read). Likely outcome: kastil-systems/metal-console → kastil-systems/metal-runtime (since it's C), joeycastilloUS/metal-console keeps the name (it's the operator surface where this kernel lives).
  • Write a one-line org charter README in each of the 9 orgs. Format: " — <audience>. <What lives here.>" Example for kastil-systems: "The systems org. Kernel, methodology, runtimes. Source of truth for a-d-d."

Re-run ~i. Finding #1 (a-d-d six places) and Finding #2 (metal-console twice) drop from red to green. Findings about org charters drop entirely.

Constraints

No destructive deletes. Archives only (isArchived=true) for any repo Joey isn't 100% sure about. Archive is reversible; delete isn't. gh repo delete is forbidden during this plan unless Joey explicitly types the command himself.
Canonical declaration precedes movement. Phase 1 must complete before Phase 2 begins. Moving work while the source of truth is ambiguous makes the ambiguity portable.
Use gh transfer, not delete-and-recreate. Transfers preserve stars, issues, PRs, and the GitHub history. Delete-and-recreate loses all of it.
Healthy orgs untouched. philzcoffee and Mendocino-Farms have working structures. Don't fix what isn't broken just because other orgs need work.
Each phase re-runs the audit. The next phase doesn't start until the prior ~i shows the expected drop. Discipline holds the plan.
Joey decides per-move. The plan sequences. Joey approves each transfer/archive before it executes. Be never deletes or transfers a repo autonomously.
v1 is a hypothesis. Per kernel rule 6: this plan is wrong in ways we don't know yet. v2 ships when v1 meets evidence it didn't anticipate (see When To Revisit).

Unknowns

  1. Which metal-console codebase is which? joeycastilloUS one is HTML (operator surface, almost certainly). kastil-systems one is C (runtime?). Until both are opened, the rename direction is a hypothesis. Phase 1 Step 1 is "open both and read for 5 minutes."
  2. Will the acmedevUS → ai-pantheon transfers cause name collisions? acmedevUS/ai-pantheon can't transfer as ai-pantheon/ai-pantheon — needs a rename mid-transfer. Don't know the right target name until inspecting the repo's contents.
  3. Is ai-pantheon-internal intentional? The plan recommends collapsing it. But if it exists for a reason (e.g. private staging before public exposure, or a per-team isolation policy), collapsing breaks something the audit can't see.
  4. What is joeycastilloUS/norman? No description. Could be a person-archive, a project named for Norman Campbell, or abandoned scratch. Affects Phase 3 archive vs keep.
  5. Was Province-Light-Foundation created by Joey or by someone else? If someone else added Joey as a member, deletion isn't Joey's call. If Joey created it for a planned initiative, that initiative needs a charter, not a deletion.
  6. Are the cross-org duplicates (overwatch_app, playwright_reports) intentional client copies or stale forks? common_csharp existing in 3 client orgs is almost certainly intentional (per-client shared libs); the other two might or might not be. Need a one-minute look at each pair.
  7. How many of the ai-pantheon Feb-22 repos have ANY content vs. empty placeholders? The triage in Phase 3 is cheap if they're empty (archive-all), expensive if they have real architecture sketches worth preserving in a different form.
  8. Does anything downstream link to a soon-to-be-moved repo URL? Joey's site (joeycastillo.us), the kernel CLAUDE.md, or other repos' READMEs may hard-link to repo URLs that change. Grep needed in Phase 4.

When To Revisit (signals that trigger v2)

  • Phase 1 reveals the rename direction is wrong. If reading the two metal-console codebases shows the C repo is actually the operator surface and the HTML one is the runtime artifact, the rename flips — v2.
  • Phase 2 transfer fails or breaks something downstream. deploy.json references, CI/CD secrets scoped to acmedev, or a Cloudflare Pages binding tied to the old org URL — any of these failing means the transfer process needs a per-repo preflight step v1 doesn't have.
  • Phase 3 triage finds a NEW kind of drift. If the ai-pantheon Feb-22 scaffold contains a class of repo (e.g. inter-repo dependency packages) the audit didn't model, v2's pruning section needs a new policy.
  • A 10th org appears. A new org membership (Joey gets added to a client's GitHub) changes the org-charter set. The plan's "9 orgs" assumption breaks.
  • Joey decides the audience contract is wrong. If, mid-plan, Joey says "actually ai-pantheon should hold all customer-facing work AND acme should hold sales-pipeline tools" — the org→audience table changes and Phase 2 reshapes. v2.
  • Final audit still shows red findings after Phase 4. If ~i finds drift v1 didn't anticipate, v2 names the new pattern and adds a phase.
  • 3 months pass without re-running the audit. Drift accumulates silently. Re-run is its own signal — if the audit hurts to read, the structure has slipped.
