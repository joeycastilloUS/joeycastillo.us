M1Patch nous/memory/skill/journal/CLAUDE.md — add WAL struct + operations

From: api-add/purpose/memory/skill/journal/CLAUDE.md (47L) → into ks-add/nous/memory/skill/journal/CLAUDE.md (27L)

ks-add has the narrative ("write-ahead log for crash safety...") but lacks the concrete C type and the operations table. Append this block:

### WAL entry format

```c
typedef struct {
    uint32_t type;         // WAL_INSERT, WAL_GROW, WAL_REHASH
    uint32_t status;       // WAL_PENDING (0), WAL_COMMITTED (1)
    uint64_t seq;          // monotonic sequence number
    uint64_t payload_size; // bytes following this header
} WalEntry;
```

### Operations logged

- **INSERT** — the Fact being added (24 bytes). Replay = re-insert via `triple_store_insert_ids`.
- **GROW** and **REHASH** — logged but self-heal on reopen (capacity checks + index rebuild).

### Behavior

- WAL is append-only during normal operation
- On `triple_store_open`, call `journal_recover` after data files are open
- Truncate WAL after successful recovery
- Clean close truncates WAL to zero

M2Patch nous/memory/skill/scope/CLAUDE.md — add Scope C struct

From: api-add/purpose/memory/skill/scope/CLAUDE.md (43L) → into ks-add/nous/memory/skill/scope/CLAUDE.md (24L)

ks-add has the conceptual model ("iterator interface... reason engine walks facts through scope"). It lacks the explicit struct. Append:

### Scope struct

```c
typedef struct {
    TripleStore* store;
    uint64_t     position;
    uint64_t     filter_key;
    IndexField   field;
    Index*       index;       // NULL = sequential scan
    bool         exhausted;
} Scope;
```

### Behavior notes

- `scope_field` uses index if available, falls back to sequential scan
- For bulk reads, direct mmap access via `ts_facts()` is faster than scope
- Scope is for filtered views, not raw throughput
- Zero allocation — stack struct, no malloc

M3Add design note to nous/memory/skill/store/CLAUDE.md — cross-purpose binary compat

From: api-add/purpose/learning/skill/store/CLAUDE.md → annotation into ks-add/nous/memory/skill/store/CLAUDE.md

api's learning/store documented an architectural decision: "Binary-compatible with memory's store so both purposes can share the same memory space at the intent level." ks-add doesn't preserve this constraint. Add a short section:

## Cross-purpose binary compatibility

The store binary format (FileHeader + facts.bin + dict.bin + strings.bin)
is intentionally shareable across purposes. Any other purpose that needs a
triple store reuses the same FNV-1a hash, the same 24-byte Fact layout, the
same 16-byte DictEntry layout. Two purposes can map the same store files
and see the same world. Constraint, not coincidence.
