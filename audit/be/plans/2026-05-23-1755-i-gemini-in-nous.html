The 3-tier key source (priority order)

  1. Per-user runtime key (highest priority) — nous_userkeys_load("users.dat") reads INI-style file next to the binary: 
    [user:joey]
GEMINI_API_KEY=AIza...
ANTHROPIC_API_KEY=sk-ant-...

[user:scotty]
GEMINI_API_KEY=AIza...
  2. Env var (middle) — GEMINI_API_KEY=... in the launching shell
  3. Compiled-in default (lowest fallback) — NOUS_KEY_GEMINI macro in src/nous_keys_builtin.h (gitignored, .example template provided). For shipping pre-keyed binaries.

Why migrate the key to kastil-systems

  1. Brand alignment — kastil-systems is the new home of the platform. Gemini quota should flow through it, not through a forgotten Google-named default project.
  2. Frees a billing slot — Acme Dev Billing Acct is at the 5-project ceiling. The orphan is one of those 5. Detach it after key migration → kastil-systems can be attached → unblocks the GCP VM migration that's currently stalled.
  3. Single source of truth — quota dashboards, alerts, and cost attribution all align to kastil-systems = NOUS-related work.

Open items for the plan (~p)

  1. Where does users.dat live in production? Next to the binary works for desktop use; for a server-class nous-desktop, may want a system path (/var/nous/users.dat) similar to /var/nous/cf.token.
  2. Key value — paste it into users.dat directly, or use the env-var path on launch (cleaner — no key on disk)? Trade-off: persistence vs. memory-only.
  3. Should the gemini key ALSO go into nous_keys_builtin.h as the compiled-in fallback? Useful if Joey ships nous-desktop binaries to others and wants a "free tier" key shared. Otherwise leave it user-key only.
  4. What's the lifespan of the AI Studio key — does Google rotate it, expire it? May want to add a key-rotation note.
  5. Multi-user case — currently users.dat supports [user:scotty], etc. If Scotty's joining the NOUS cockpit, he gets his own section with his own key.
