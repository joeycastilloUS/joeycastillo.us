The 3 phases

Phase 1 — Pre-flight (one-time setup, ~30 min)

Has to happen before any copy work. After this, Phase 2 just runs.
  • Enable Compute API on kastil-systems: gcloud services enable compute.googleapis.com --project=kastil-systems
  • Confirm billing attached to kastil-systems
  • VPC + firewall: verify auto-mode default VPC exists; replicate the firewall rules from acmedev-platform-hq (ports 22, 8080, 8443)
  • Cross-project IAM: kastil-systems default Compute SA needs roles/compute.imageUser on snapshots
  • Capture current DNS state: dig +short TXT _prime.3-nous.net + dig +short TXT _relay.3-nous.net — save the output for diff after cutover
  • Sanity-check identity files exist on old VMs: SSH into one prime + one relay, confirm /var/nous/cf.token exists and node.key/.pub/.cert live where the binary expects. Snapshot will carry these.

Compute API live on kastil-systems, IAM clear, current DNS state captured, identity files confirmed on old VMs.

Phase 2 — Copy them over (the work)

Joey's "one phase." 5 VMs × the same recipe. Order matters for safety, not architecture.

The recipe (run per VM):

1. SNAPSHOT the old disk         (in acmedev-platform-hq)
2. CREATE IMAGE                  (cross-project into kastil-systems)
3. BOOT NEW VM                   (in kastil-systems, new name, HW upgrades for elite)
4. VERIFY                        (SSH, check identity files, check service)
5. WAIT FOR DNS                  (new TXT record propagates in < 5 min;
                                  consumer-side relays pick it up at next refresh)
6. STOP OLD VM                   (don't delete — keep for rollback window)

Why not parallel? Could be — but doing 2.1 first lets you catch any image / NIC / cross-project IAM issue once instead of five times. After 2.1 succeeds, 2.2-2.5 can run with confidence (and even in parallel if you want).

Why pair-aware sequencing for relays + primes? The protocol handles multi-instance discovery natively (per relay/CLAUDE.md: "Primes come and go dynamically"). But if ONE side of a pair has a problem during migration, the other still serves. Belt-and-suspenders.

All 5 VMs running in kastil-systems with new names. Old 5 VMs in acmedev-platform-hq are STOPPED (not deleted). Cloudflare DNS has both old + new TXT records co-existing temporarily.

Phase 3 — Cleanup (after ~1 week rollback window)

Trailing work. Old VMs stay stopped until you're sure nothing broke.
  • Delete the 5 old TXT records from Cloudflare DNS (the ones pointing at acmedev-platform-hq IPs). After this, only the new TXT records remain.
  • Delete the 5 stopped VMs in acmedev-platform-hq
  • Delete the migration snapshots (or keep one set as cold backup if you want)
  • Verify dig TXT _prime.3-nous.net + dig TXT _relay.3-nous.net show only the new IPs
  • Update audit-of-all reports with final state
  • Consider: if acmedev-platform-hq has nothing else important, schedule the project for shutdown

Old fleet deleted. DNS clean. Migration complete.

What this still depends on (the "more than one phase" bits)

Joey's framing was right — at the architecture level, it's one operation (copy). The complications:

  1. Cross-project image sharing requires the IAM grant (Phase 1). Without it, Phase 2 fails on step 3.
  2. NIC + disk upgrades for elite mean the boot command for that one VM has extra flags (--network-interface=nic-type=GVNIC + --create-disk=type=pd-ssd). The other 4 use defaults.
  3. DNS overlap is automatic (the protocol handles it) — but cleanup of stale Cloudflare TXT records is manual (Phase 3).
  4. Identity files travel with the snapshot — node.key, node.pub, node.cert, /var/nous/cf.token all carry over. No re-signing needed.
  5. External IPs change — new VMs get new IPs unless static IPs are explicitly reserved in kastil-systems. The DNS protocol re-announces on boot, so this is fine for relay/prime discovery. If any DNS A records or external services point at the OLD IPs directly, those need updates separately.

That's the gap between "one phase" (the copy) and "three phases" (one-time setup, the copy, cleanup). The middle phase IS one phase — repeated 5 times for tactical safety.

Phase 2.1 — execution-ready commands (the recipe for elite)

# Snapshot
gcloud config set project acmedev-platform-hq
gcloud compute disks snapshot nous-elite-us \
    --zone=us-east4-a \
    --snapshot-names=elite-snap-2026-05-23

# Cross-project image
gcloud compute images create elite-img-2026-05-23 \
    --project=kastil-systems \
    --source-snapshot=projects/acmedev-platform-hq/global/snapshots/elite-snap-2026-05-23 \
    --storage-location=us

# Boot new VM (elite gets gVNIC + pd-ssd upgrades)
gcloud compute instances create elite-c2-standard-8-gvnic-us-east4-a-ashburn \
    --project=kastil-systems \
    --zone=us-east4-a \
    --machine-type=c2-standard-8 \
    --network-interface=nic-type=GVNIC,subnet=default \
    --create-disk=image=elite-img-2026-05-23,image-project=kastil-systems,boot=yes,type=pd-ssd,size=50

# Verify
gcloud compute ssh elite-c2-standard-8-gvnic-us-east4-a-ashburn \
    --project=kastil-systems --zone=us-east4-a \
    --command='hostname; uname -a; ip link show; ls -la /var/nous/'

For Phase 2.2-2.5 (the four e2-micros): same recipe minus the gVNIC + pd-ssd flags. Default NIC + pd-standard preserved.
