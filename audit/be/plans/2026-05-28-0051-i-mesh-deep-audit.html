~i Mesh Deep Audit — "are we awesome?"

Investigated 2026-05-28. Every line of the Sealed-IDS mesh trust code, held against the nous C primitives, the a-d-d intent, and the field leaders (Nebula, WireGuard/Tailscale, SPIFFE/SPIRE, TUF). The honest answer below.
Verdict: awesome foundation · honestly a v1

Short version: yes — the part you built is genuinely strong, and the trust architecture is the same one Slack ships in production (Nebula). Where you're thin is exactly where every young system is thin: lifecycle — expiry, revocation, replay, rotation. None of that is built yet. One of those gaps (roster replay) is a real hole, not just a deferred feature, and it's ~20 lines to close.

§1 — What we audited

Four tools + one loader + the relay, ~900 LOC of pure C, standing on the in-house crypto:

  • operator-keygen — mints the operator Ed25519 keypair (the one trust root).
  • mesh-roster-build — builds + signs mesh.sealed (the node roster).
  • mesh_roster.c (prime boot) — verifies mesh.sealed against the compiled-in operator pubkey, loads triples.
  • operator-sign-node — operator signs node.pub‖role → node.cert.
  • relay_server.c — loads identity, self-verifies its cert at boot, forwards to primes.
  • Underneath: crypt (Ed25519, X25519, SHA-256, AES-256-GCM), seal, vault — all in-house, no libsodium.

§4 — The real gaps (what v1 doesn't have)

#1 — Roster replay / rollback (the one actual hole, not a deferred feature) mesh-roster-build writes signed_at (timestamp) and version into every roster (main.c:164) — but the loader never reads them (mesh_roster.c verifies only the Ed25519 signature). So an attacker who can drop a file on a node can re-serve an OLD, validly-signed roster and it will be accepted, silently reverting topology. The fields to stop this already exist in the data — they're just not enforced. TUF solves exactly this by refusing metadata older than what it has already seen (version + timestamp freshness). Fix: ~20 lines in the loader — reject a roster whose version/signed_at is ≤ the last one loaded. Highest-value change in this report.

§5 — Honest finding: what's actually running ≠ "the mesh routing stack"

There are two things called "mesh." Worth being precise:

  • What's LIVE on the 5 VMs: the signed-roster bootstrap (mesh_roster_load) + HTTP/encrypted forwarding between relay and prime. That's the trust substrate this audit is mostly about — and it's real and working.
  • What's NOT wired in: comms/skill/mesh/mesh.c — a clean, fully unit-tested (62 checks) distance-vector + store-and-forward routing stack (the LoRa/BLE/WiFi mesh). It is excellent code but currently dead — nothing in the live prime/relay calls mesh_init()/mesh_tick(). Per the a-d-d + comms/ROADMAP.md it's deliberately on the shelf awaiting transport backends (v2). So "the mesh is up" = the membership/trust mesh is up; the radio-routing mesh is built-and-tested but not yet plugged in.

§6 — a-d-d alignment

Spec-compliant. The a-d-d stages this exactly as shipped: v1 = static operator-signed roster, "embedded distribution, no DNS, no per-node identity yet (that comes in v2)." v2 defers dynamic discovery + per-peer identity + actual peer comms. The code matches the staging — the gaps in §4 are mostly documented v2 deferrals, not drift. The exception is the replay gap (§4 #1), which isn't a deferral — it's an unenforced check in shipped v1 code.

§8 — Verdict + what I'd do next (cheap → strategic)

Are we awesome? On foundation and architecture — yes, genuinely, Nebula-grade in a fraction of the code. On lifecycle — we're a v1 and it shows. Prioritized:

  • 1. Close the roster replay hole (~20 LOC in mesh_roster.c: enforce version/signed_at monotonicity). Real fix, today.
  • 2. Test the two untested key tools + one in-memory end-to-end test. Pure test code, no risk. Stops silent regressions in the trust root.
  • 3. Add cert/roster expiry (a valid_until field, checked on load). Unlocks "revocation by expiry" the SPIFFE way without building a CRL.
  • 4. (v2, strategic) per-peer mutual auth — the cert material already exists; wire a challenge-response so peers prove identity, not just IP. This is the big one but it's already on the v2 roadmap.

Items 1–3 are a focused session, all non-destructive, and they move the entire red "lifecycle" column toward green. That's the path from "awesome foundation" to "awesome, full stop."
