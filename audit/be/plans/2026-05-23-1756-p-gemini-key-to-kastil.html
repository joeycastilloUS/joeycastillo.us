PLAN — AWAITING APPROVAL

~p Move the Gemini key from the orphan to kastil-systems.

Two birds: get Gemini working as a real NOUS provider, AND free the billing slot that's blocking the VM migration. Same plan covers both.

Trail: ~i Gemini in NOUS · ~p VM migration v4 (blocked on billing)

Intent

Joey wants Gemini available in NOUS, billed under kastil-systems instead of an orphan default project he never knew existed. Today: the orphan project (gen-lang-client-0031500671) holds his old API key, has Generative Language API enabled, and consumes one of 5 billing slots. The 5-slot ceiling is what's blocking the entire VM migration plan v4.

This plan does both: provisions Gemini correctly in kastil-systems AND clears the slot. End state: kastil-systems hosts the API key, nous-desktop reads it from users.dat, the orphan project's billing is detached, kastil-systems can be attached to billing → VM migration unblocks.

Scope

In scope

  • Enable Generative Language API on kastil-systems
  • Create a fresh Gemini API key scoped to kastil-systems (via AI Studio web UI)
  • Install the key in C:\kastil-tools\nous-desktop\users.dat as [user:joey] GEMINI_API_KEY=...
  • Restart nous-desktop, verify "1 paid (user key)" in the banner
  • Smoke-test at least one Gemini variant via the cockpit
  • Revoke the orphan project's old AI Studio key
  • Detach billing from the orphan project (frees the slot)
  • Attach billing to kastil-systems (the unblock)

Out of scope

  • Code changes to nous-desktop — already wired, just needs the key
  • Adding Gemini to nous_keys_builtin.h as the compiled-in fallback (separate decision; default plan = users.dat only)
  • Multi-user setup (Scotty's key, etc.) — same pattern, add later
  • Vertex AI migration (kept on AI Studio endpoint per existing serve.c config)
  • Deleting the orphan project entirely — just detaching billing is enough for this plan; full deletion is a separate decision
  • VM migration (v4) — this plan UNBLOCKS that one, doesn't execute it

Approach

Sequential. Each step depends on the previous. No parallelism opportunity. Total time start-to-finish: ~30 minutes if everything cooperates (most of it is browser clicks at aistudio.google.com).

Phases

Phase 1 — Create the new Gemini API key in kastil-systems (web UI)

Has to happen first. The whole plan revolves around this key existing.
  1. Open https://aistudio.google.com/app/apikey
  2. Log in as the operator (joey.castillo.acmedev@gmail.com per gcloud auth)
  3. Click "Create API key"
  4. In the project dropdown, select kastil-systems. Google enables Generative Language API automatically on the chosen project (or, if it can't enable due to no billing, this step fails — see Phase 0).
  5. Copy the generated key (format: AIzaSy..., ~40 chars)
  6. Don't paste it anywhere yet — keep on clipboard for Phase 2

A new Gemini API key exists, scoped to kastil-systems. (If this fails because of no billing on kastil-systems, see Phase 0 immediately below — this is the chicken-and-egg case.)

Phase 0 — IF Phase 1 fails due to billing (the chicken-and-egg case)

AI Studio key creation requires the destination project to have Generative Language API enabled, which requires billing. kastil-systems has neither. The orphan project is using one of the 5 slots. Need to detach orphan billing FIRST → then attach to kastil-systems → then Phase 1 succeeds.

Ordering becomes:

  1. Detach billing from orphan: gcloud billing projects unlink gen-lang-client-0031500671
  2. Attach kastil-systems to billing: gcloud billing projects link kastil-systems --billing-account=018FA8-39FA2B-18CB6E
  3. Enable Generative Language API: gcloud services enable generativelanguage.googleapis.com --project=kastil-systems
  4. Now retry Phase 1 — AI Studio will let you create the key under kastil-systems

Side effect (good): the orphan's existing API key keeps working temporarily because it lives at the Google-side API, not the local project. But its quota may drop to free-tier limits without billing. Plan accordingly — if you have active Gemini calls right now, hold off until the new key is wired in Phase 2.

Billing landed on kastil-systems, API enabled, Phase 1 can succeed.

Phase 2 — Install the key in users.dat

nous-desktop reads users.dat next to the binary on launch. INI format.
  1. Open C:\kastil-tools\nous-desktop\users.dat in a text editor (create if missing)
  2. Add: 
    [user:joey]
GEMINI_API_KEY=AIzaSy...(paste here)...
  3. Save. Permissions: tighten to user-only read if on a shared machine.

users.dat exists with Joey's Gemini key in the [user:joey] section.

Phase 3 — Restart nous-desktop and verify

Banner change confirms the key is loaded.
  1. Stop the running nous-desktop: Get-Process nous-desktop | Stop-Process -Force
  2. Restart: Set-Location C:\kastil-tools\nous-desktop; .\nous-desktop.exe (or in background as before with stdout/stderr capture)
  3. Read the new stderr banner — should now show providers: 0 free (builtin), 1 paid (user key)
  4. If still shows "0 paid": something's wrong with users.dat. Common: wrong file path, wrong section name, no newline at end.

Banner confirms 1 paid key loaded. Gemini is now reachable.

Phase 4 — Smoke test via the cockpit

Don't trust the banner. Make an actual call.
  1. Open http://localhost:5050 in browser
  2. From the AI provider dropdown, select gemini-flash (cheapest variant, $0.15/$0.60 per Mtok)
  3. Send a one-line test prompt: "Reply with the word OK and nothing else."
  4. Confirm response. Check Cloudflare/AI Studio dashboard later to confirm the call hit kastil-systems' quota, not the orphan's.

Gemini responds. The Director can now use Gemini as an available AI.

Phase 5 — Revoke the old key + complete the billing flip

Old key revoked → orphan project's API has nothing to bill. Detach billing → frees a slot. (If Phase 0 was taken, billing is already detached — this phase just revokes the key.)
  1. Go back to aistudio.google.com/app/apikey
  2. Find the OLD key (the one tied to gen-lang-client-0031500671) — delete it
  3. If Phase 0 was NOT taken (i.e., billing was already free for kastil-systems somehow), execute the detach now: 
    gcloud billing projects unlink gen-lang-client-0031500671
gcloud billing projects link kastil-systems --billing-account=018FA8-39FA2B-18CB6E
  4. Verify: gcloud billing projects list --billing-account=018FA8-39FA2B-18CB6E should now show kastil-systems among the 5 billed projects, and gen-lang-client-0031500671 should be absent.

Old key gone. Orphan billing detached. kastil-systems on billing. The VM migration plan (v4) is now unblocked.

Phase 6 — (optional) close the orphan project

Cleanup. Orphan is now empty (zero billable resources). Closing it removes a dead container from the project list.
  1. Decide: keep it (in case a forgotten dependency surfaces) or delete it
  2. If delete: gcloud projects delete gen-lang-client-0031500671 (Google holds the project for 30 days before permanent deletion — fully recoverable in that window)

No more orphan in the project list (or it's queued for 30-day deletion).

Constraints

AI Studio key creation is web-only. No gcloud CLI for it. The browser step is mandatory in Phase 1.
Phase 0 is conditional. If you DON'T want to detach orphan billing FIRST, Phase 1 may fail (depends on whether AI Studio can auto-enable the API without billing on the new project). If it fails, fall back to Phase 0 ordering.
users.dat lives next to the binary. Cross-platform. If nous-desktop ever moves to a system-wide install path, users.dat may need to follow.
Key is plaintext in users.dat. No encryption at rest. Protect via filesystem permissions or move to a secret manager later.
One key, four variants. All 4 Gemini models share GEMINI_API_KEY — you don't need 4 keys.
v1 hypothesis. Per kernel rule 6 — if any phase reveals something unexpected (AI Studio behavior change, Generative Language API quirks), v2 of this plan corrects.

Unknowns

  1. Does AI Studio enforce billing at key-creation time, or only at API-call time? Determines whether Phase 0 is mandatory or just a fallback.
  2. Quota inheritance — does the new key under kastil-systems start with default Gemini free-tier quota, or does it pick up paid-tier limits immediately?
  3. If Phase 0 detaches orphan billing, does the orphan's old key immediately revert to free-tier limits, or is there a grace period?
  4. Is anything else using the orphan project's key? (Joey forgot it existed, but worth a grep across other tools/repos.)
  5. Should users.dat be checked into git or gitignored? Default: gitignored — it holds secrets.

When To Revisit (v2 triggers)

  • Phase 1 fails for a reason other than billing — AI Studio UI changed, project picker missing, etc.
  • Banner doesn't show "1 paid (user key)" after Phase 3 — users.dat path or format wrong
  • Gemini call returns 401/403 — key wasn't actually scoped right, or kastil-systems doesn't have the API enabled, or paid tier not activated
  • Detaching orphan billing breaks something Joey forgot about (a script, an old project) — rollback by reattaching
  • Multi-user case — Scotty joins NOUS, needs his own [user:scotty] section in users.dat
