Phase 2 — Archive absorbed precursors · 5 pts

Items: gh repo archive on 9 repos already absorbed into kastil-systems canonical:

  • ai-pantheon/{cc, pantheon-memory, pantheon-brain, pantheon-mind, pantheon-runtime-security, nous-desktop}
  • joeycastilloUS/{metal.Be, metal-proposal}

Each archive is reversible. Risk: low if we've confirmed all valuable history is already in canonical (Phase 3 verifies before archive).

Phase 3 — Active moves + merges · 13 pts

The heavy work — repos with live code that goes INTO kastil-systems. Two patterns:

  • gh repo transfer (whole-repo move, preserves history) — for: assembly, pantheon-auth, pantheon-platform, workers, pantheon-governance, pantheon-patents, knowledge-archive, conversation-archive, a-d-d-template, airbyte-dispatch, kASTIL, metal-desktop, a-d-d-guidance
  • git subtree merge (fold INTO an existing kastil-systems repo, preserves blame) — for: integrity, safety, identity, pantheon-seal, pantheon-sentinel, pantheon-gateway, pantheon-dispatch, pantheon-tools, code-hints

For each, the destination is named in §1. Subtree merges require small reorg of folder structure inside the target repo (e.g. nous/security/skill/audit/ as a new skill dir).

Tilde: per-item ~u~t (update kastil-systems repo with new subtree). Risk: medium — must verify builds + a-d-d docs match after each merge.

Phase 4 — Cleanup + verification · 5 pts

Items: verify all kastil-systems builds pass after consolidations; remove any newly-orphaned cross-refs introduced during Phase 3; update gh repo descriptions; verify ~drift (the planned spec-sync tool — separate thread) sees clean state.

Tilde: ~t. Risk: discovery work; surfaces anything Phase 3 missed.

Board total: 5 + 3 + 5 + 13 + 5 = 31 pts · 2 boards (16 + 15) or 1 stretched board.
