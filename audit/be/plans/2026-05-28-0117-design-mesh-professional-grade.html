~d Mesh → Professional Grade

The target-state architecture that closes the lifecycle gaps from the 2026-05-28 deep audit and takes the Sealed-IDS mesh from "honest v1" to production-hardened — without losing what makes it good (pure-C, tiny TCB, offline-capable, FQDN-keyed, no third-party identity).

§1 — Intent (what "professional grade" means HERE)

Not "copy Nebula." We already have Nebula's trust model. Professional grade for THIS system means the trust model gains a lifecycle — freshness, expiry, revocation, mutual proof, rotation — while keeping the four things that make it ours:

  • Pure C, no malloc, tiny TCB — every addition is static-buffer C linking libsecurity, no new deps.
  • Offline-capable — a field node may be dark for days; the design cannot assume always-online renewal.
  • FQDN/operator-key as identity — the operator key stays the one root; no third-party CA, no cloud IdP.
  • Verifiable from outside — every property provable with dig/curl/a signature check, not trust-us.

§2 — The shape of the gap (from the audit)

Today the mesh authenticates origin (operator signed it) and integrity (untampered). It does not yet reason about time (freshness/expiry), life (revocation/rotation), or the peer on the other end of the wire (mutual auth). Professional grade = adding the time/life/peer axes.

The central design tension — name it before deciding SPIFFE makes revocation unnecessary with minutes-long certs auto-rotated online. That is wrong for us: a field node offline for 3 days with a 1-hour cert is bricked. WireGuard/Nebula assume always-reachable peers; we don't. So we cannot just copy any one model. The design must choose a TTL long enough to survive realistic offline windows AND add explicit revocation for the compromise case (because long TTL means a leaked key is live for a while). This is the load-bearing decision; everything else follows from it.

§4 — Target trust model (after D1–D7)

PROVISION  operator-keygen -> operator.key (root, offline)        [versioned: D6]
           operator-sign-node -> node.cert = sign(node.pub‖role‖valid_until)  [D2]
           mesh-roster-build -> mesh.sealed = sign(version‖signed_at‖valid_until‖
                                                   triples) then seal-encrypt   [D1,D2,D5]
           + revocation.sealed = sign(version‖revoked_pubkeys[])               [D3]
BOOT       node verifies cert (sig + valid_until) ............................ [D2]
           node loads roster: verify sig, decrypt, reject if version<=seen
                              or now>valid_until ............................. [D1,D2,D5]
           node loads revocation list (version-checked) ..................... [D3]
CONNECT    peer A <-> peer B: each presents node.cert; each sends a nonce;
           each signs the other's nonce with node.key; both verify against
           operator pubkey AND check peer not in revocation list ............ [D4,D3]
           -> only then is the AES-GCM channel trusted (not IP+PSK)
ROTATE     operator signs rollover(old_pub -> new_pub); nodes accept both
           during grace window; old retired after all nodes re-provisioned .. [D6]

§5 — Consequences (trace them honestly)

  • Roster + cert formats gain fields (valid_until, explicit version as signed header not just a triple). The loader (mesh_roster.c) and relay_verify_self must read them. Backward-compat: bump magic to MESHROSTER2 or version-gate; old mesh.sealed v1 rejected after cutover.
  • Mutual auth (D4) is the big lift — it adds a handshake round-trip to every peer connection and needs a small state machine in the relay/prime wire path. Everything else is localized to parsing/verification.
  • Operator workflow changes: re-issuing rosters/certs becomes a periodic act (TTL expiry), not one-and-done. Needs a tiny re-issue script/cron. This is the cost of revocation-by-expiry — accept it.
  • The operator key is now even more critical (it can revoke + rotate). Hardening its storage (Windows ACL, off shared drives) graduates from "nice" to "required."

§6 — Alternatives considered + rejected

  • SPIFFE-style minutes-TTL + no revocation — rejected: assumes online renewal, bricks offline field nodes (the §2 tension).
  • Adopt Nebula/WireGuard wholesale — rejected: drags a Go runtime + large TCB; abandons pure-C + FQDN-key identity. We keep the model, not the implementation.
  • mTLS (TLS 1.3) for peer auth — viable (tls13.c exists) but heavier than a node-cert challenge-response; defer as an option for D4 if the hand-rolled handshake proves fiddly.

  • TTL: is 30 days the right offline-tolerance window, or do field nodes need 90+?
  • Roster encryption (D5): worth it for a private fleet, or skip and keep signed-plaintext for debuggability?
  • D4 handshake: hand-rolled node-cert challenge-response, or reuse tls13.c mTLS?
