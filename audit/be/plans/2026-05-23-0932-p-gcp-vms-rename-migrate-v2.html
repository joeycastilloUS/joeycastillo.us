1. Intent

Joey is building bare metal. He thinks at the hardware layer. The names of the machines should expose that layer — not hide it behind cute abbreviations. When he reads nous-elite-c2-standard-8-gvnic-us-east4-a-ashburn he knows: it's the elite role, it's an 8-vCPU compute-optimized node, it has a gVNIC for high-throughput networking, it lives in us-east4-a (Ashburn, Virginia). No lookup needed. That's the win. v1 traded all that for 9-char airport codes; v1 was wrong about the trade.

End state: 5 VMs in kastil-systems project. Names carry their hardware spec. Source project (acmedev-platform-hq) shut down or emptied of these workloads. Cutover safe — paired roles (2× prime, 2× relay) protect against single-VM downtime during migration.

2. The naming convention

Final pattern

<role>-<machine-class>-<nic-type>-<gcp-zone>-<city>

Where:
  <role>          = nous-elite · prime · relay (functional purpose)
  <machine-class> = e2-micro · c2-standard-8 · n2-standard-4 · ...
  <nic-type>      = virtio · gvnic · idpf  (or whatever GCP supports)
  <gcp-zone>      = us-west4-a · asia-southeast1-a · ... (full GCP zone)
  <city>          = singapore · las-vegas · hong-kong · ... (hyphenated, lowercase)

Hyphens are the only punctuation GCP allows in VM names. Max length 63 chars. All proposed names below: 42-49 chars. Plenty of room.

4. Hardware upgrades discovered during inspection

Upgrade 1 — nous-elite gets gVNIC (the NIC type matters for compute-optimized)

Today the c2-standard-8 runs default VIRTIO_NET. gVNIC is Google's higher-throughput NIC, recommended for c2/n2/c3 tiers, required to hit max network throughput. On rebuild: --network-interface=nic-type=GVNIC. The name reflects this — ...-gvnic-....

This is why v2's name format includes NIC type. The hardware choice is invisible from the v1 name; visible from the v2 name.

Upgrade 2 — nous-elite gets pd-ssd disk (currently HDD-backed)

Today nous-elite runs on pd-standard — HDD-backed persistent disk, slowest tier. A compute-optimized 8-vCPU box on HDD is silly; the disk is the bottleneck for anything I/O-heavy. On rebuild: --boot-disk-type=pd-ssd. Disk type doesn't go in the name (it's mutable and would explode length), but the rebuild captures it.

Cost note: pd-ssd is ~3-4× pd-standard per GB. For 50 GB that's ~$10/mo vs ~$3/mo. Cheap to upgrade.

Upgrade 3 — optional: prime/relay also move to gVNIC

e2-micro does support gVNIC. Default is virtio for backward compat. If Joey wants uniformity (everything on gvnic) AND/OR wants future-proofing for traffic growth, switch the 4 small nodes too. Names would shift: prime-e2-micro-gvnic-us-west4-a-las-vegas.

Open decision: uniform gvnic everywhere, or only on nous-elite where it actually matters?

5. Scope

In scope

  • Snapshot each VM's boot disk in acmedev-platform-hq
  • Create cross-project images / disks in kastil-systems
  • Boot 5 new VMs in kastil-systems with the new names + upgraded NIC/disk where applicable
  • Verify each new VM serves before retiring the old
  • DNS / config cutover for any hostname pointing at the old IPs
  • Stop + delete old VMs in acmedev-platform-hq (with rollback window)

Out of scope

  • Code changes to the nous prime/relay binaries (those stay as-is — same OS, same software, just moved)
  • Architectural changes to the prime/relay protocol (NTRP0001 wire stays as-is)
  • Acme Platform project shutdown (separate decision; this plan only empties the VMs)
  • Adding new VMs in other regions (future; this plan moves existing fleet only)

6. Approach

Snapshot + rebuild, not lift-and-shift (GCP doesn't support cross-project VM moves directly). For each VM: snapshot disk → create image cross-project → boot new VM in destination → verify → cut over DNS → retire old.

The new VMs are NOT identical clones — they get the renaming AND any HW upgrades selected. So this isn't a forklift; it's a controlled rebuild with upgrades.

7. Phases

Phase 0 — Pre-flight (one-time)

  1. Enable Compute Engine API on kastil-systems: gcloud services enable compute.googleapis.com --project=kastil-systems
  2. Verify kastil-systems has a billing account attached
  3. Confirm VPC / subnet setup in kastil-systems (default auto-mode VPC should work; if Joey wants custom VPC, create first)
  4. Set up cross-project IAM: kastil-systems default Compute SA needs roles/compute.imageUser on acmedev-platform-hq images
  5. Inventory: capture exact hostnames pointing at current external IPs (if any). DNS will need updates.
  6. Identify any GCS buckets / Secret Manager secrets / IAM bindings the prime/relay code reads from in acmedev-platform-hq. Grant cross-project access OR plan to mirror those into kastil-systems.

Compute API enabled, billing attached, IAM clear, dependencies inventoried.

8. Constraints

GCP VMs are project-scoped. Cannot lift-and-shift between projects. Snapshot + rebuild is the only path. Plan reflects this.
VM names: 63-char max, lowercase alphanumeric + hyphens, must start with a letter. All 5 proposed names fit (42-49 chars).
Paired roles protect uptime. 2× prime, 2× relay. Migrate one at a time within each pair — the other carries traffic.
nous-elite first. TERMINATED = no traffic. Use to validate the recipe before touching live nodes.
Keep originals stopped (not deleted) for ~1 week after each cutover. Rollback is "start the old, update DNS back."
Hardware upgrades happen on rebuild only. Don't modify the in-place VMs in acmedev-platform-hq during migration; the original stays as a known-good rollback target.
External IPs change. Unless static IPs are explicitly reserved and re-attached, each new VM gets a new ephemeral external IP. DNS must update.
v1 hypothesis. Per kernel rule 6 — if any phase reveals something we didn't anticipate (e.g. boot fails after rebuild, NIC upgrade breaks something), v2 of this plan corrects.

Unknowns (the things we should answer before Phase 0)

  1. Why is nous-elite TERMINATED? Cost-save between jobs? Workload retired? Failure? Affects whether to migrate-and-start, migrate-and-leave-terminated, or skip entirely.
  2. What DNS / config currently points at these VMs? relay.3-nous.net? prime.3-nous.net? Static external IPs reserved? Need this map before Phase 0.
  3. Are there GCS buckets / Secret Manager secrets the VMs read from in acmedev-platform-hq? Cross-project IAM grants needed if they stay; mirroring needed if they move.
  4. VPC choice on kastil-systems. Default auto-mode VPC (one subnet per region, /20)? Or custom VPC with explicit subnets matching the old setup (10.148/16, 10.150/16, etc.)?
  5. Firewall rules. What ports do prime / relay / nous-elite serve on? SSH (22) plus NTRP0001 wire-protocol port, presumably. Replicate from acmedev-platform-hq.
  6. gVNIC on all VMs, or only nous-elite? Decision point — uniformity vs. don't-fix-what-isn't-broken on the e2-micros.
  7. Boot timing for migrated nous-elite. Should it stay TERMINATED post-migration (matching today's state) or start fresh in kastil-systems?

When To Revisit (v2 triggers)

  • Phase 1 fails to boot — image incompatibility, IAM gap, NIC upgrade problem. v2 narrows the recipe.
  • The recipe works on nous-elite but breaks on a live VM (e2-micros differ in some subtle way). v2 adds per-VM tweaks.
  • DNS cutover causes user-visible disruption longer than expected. v2 adds blue-green pattern (run both old + new in parallel briefly).
  • Any GCP API quirk discovered (image sharing perms, snapshot replication regions, etc.) that wasn't anticipated.
  • Joey decides during execution: "I want all gVNIC" → Phase 2-5 pivot. Or "actually I want different machine classes for some." → re-plan.

9. The execution-ready command set (for Phase 1)

# 0. Pre-flight
gcloud services enable compute.googleapis.com --project=kastil-systems
gcloud compute networks list --project=kastil-systems  # confirm VPC exists

# 1. Snapshot in source project
gcloud config set project acmedev-platform-hq
gcloud compute disks snapshot nous-elite-us \
    --zone=us-east4-a \
    --snapshot-names=nous-elite-snap-2026-05-23

# 2. Create image cross-project (image lives in kastil-systems)
gcloud compute images create nous-elite-img-2026-05-23 \
    --project=kastil-systems \
    --source-snapshot=projects/acmedev-platform-hq/global/snapshots/nous-elite-snap-2026-05-23 \
    --storage-location=us

# 3. Boot the new VM in kastil-systems with new name + HW upgrades
gcloud compute instances create nous-elite-c2-standard-8-gvnic-us-east4-a-ashburn \
    --project=kastil-systems \
    --zone=us-east4-a \
    --machine-type=c2-standard-8 \
    --network-interface=nic-type=GVNIC,subnet=default \
    --create-disk=image=nous-elite-img-2026-05-23,image-project=kastil-systems,boot=yes,type=pd-ssd,size=50

# 4. SSH verify
gcloud compute ssh nous-elite-c2-standard-8-gvnic-us-east4-a-ashburn \
    --project=kastil-systems --zone=us-east4-a --command='hostname; uname -a; ip link show'

# 5. (Optional, after smoke test) start any nous workload that should run there

# Rollback if needed: delete the new VM + image, the old one is still in acmedev-platform-hq
gcloud compute instances delete nous-elite-c2-standard-8-gvnic-us-east4-a-ashburn \
    --project=kastil-systems --zone=us-east4-a --quiet
gcloud compute images delete nous-elite-img-2026-05-23 \
    --project=kastil-systems --quiet
