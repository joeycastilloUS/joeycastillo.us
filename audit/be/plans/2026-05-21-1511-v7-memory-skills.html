### Triple-store composition

When the store is used as a relational layer (subject/predicate/object), it composes:

- **Fact** — 24 bytes: three uint64 IDs (subject, predicate, object)
- **DictEntry** — 16 bytes: maps hash ID → string offset/length
- **TripleStore** — owns facts, dict, strings (all MmapFile), plus up to 16 indexes

Default lifecycle:
- Creates 3 default indexes on open (subject, predicate, object)
- Opens WAL on open, recovers uncommitted entries
- Auto-grows files 2x when full
- Auto-rehashes indexes at 0.7 load factor
- All inserts logged to WAL before write, committed after

Triple-store API (in addition to blob-level put/get/exists):
- `triple_store_open/close/flush` — lifecycle
- `triple_store_insert/insert_ids` — add facts (string or ID form)
- `triple_store_query/query_strings` — find facts (0/NULL = wildcard)
- `triple_store_get_id/get_string/has_id` — dictionary lookup
- `triple_store_create_index/drop_index` — index management
- `triple_store_scope/scope_field` — scoped iteration

Deps when used as triple store: mmap, hash, index, scope, journal.
