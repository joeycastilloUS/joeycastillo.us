What this is
The Pantheon's architecture record. Joey decides → consoles capture → AI tools translate → the IO bus reads/writes → triples persist in the IDS → the cycle repeats with everyone reading from the same source.

The flow, in plain words

DOWN — Joey's intent becomes triples

  Joey types something into a console
   → console hands it to an AI tool
    → AI tool figures out what triples need to be written
     → IO bus inserts them into the IDS
      → triples are now permanent, queryable, shared

UP — triples become any AI's context

  Some other (or same) AI session starts
   → it asks IO bus "what do I need for scope X?"
    → IO bus queries the IDS for matching triples
     → returns them as { new_facts, evicted_facts }
      → IO bus renders to the AI's preferred format
       (Claude Code → CLAUDE.md  ·  Perplexity → prose  ·  …)
        → AI consumes, suggests next moves
         → Joey decides what's next  ← the loop closes

SIDEWAYS — a-d-d governs both directions

  Every write must conform to a-d-d's structure (intent / purpose / skill)
  Every read renders through a-d-d's conventions
  The methodology lives in kastil-systems/a-d-d (canonical)
  The IDS is what the methodology becomes when stored
  The IO bus is what the IDS becomes when accessed
  The consoles are what the IO bus becomes when used

Glossary — every label, said straight

JOEYThe human owner/operator. Makes every decision. Drives every intent. The system exists to serve and amplify this layer.
CONSOLESOperator-facing surfaces. metal, metal-console, metal-desktop, future ones. Each is whatever shape the operator needs (C binary, web UI, CLI, voice). Many can exist; they all hit the same bus.
AI TOOLSTranslators between operator intent and structured triples. Claude Code, Perplexity, future LLMs. Each one reads the form it prefers; the IO bus renders accordingly.
IO BUSThe single read/write substrate every tool uses to reach the IDS. Library (libio_bus.a) for C tools, CLI (io_bus) for everything else. Hosts bootstrap, scope, renderers. The bus topology — many tools, one backbone, one data set.
PANTHEON IDSThe Intelligent Data Set. The Pantheon's architecture record. Stores every decision, every dependency, every intent as triples. Composes the nous memory skills (store, journal, scope, etc.) already vendored in metal.
TRIPLES{ subject, predicate, object }. The universal data shape per the kernel's "Everything is Triple" principle. Architecture, decisions, telemetry, prompts, conversations, code — all expressed as triples.
a-d-dArchitecture-Driven Development. The methodology that defines the structure (intent → purpose → skill) and the rules. Canonical at kastil-systems/a-d-d. Not a runtime — a discipline. The IDS is what happens when a-d-d is stored. The IO bus is what happens when the IDS is accessed.

The one-line definition of each layer

  JOEY         = the source of intent
  CONSOLES     = surfaces that capture intent
  AI TOOLS     = translators between intent and triples
  IO BUS       = the substrate that reads and writes triples
  PANTHEON IDS = the place triples live
  TRIPLES      = the only shape data takes
  a-d-d        = the rules everything follows
