serve.c — what it does (12 sections, 2903 lines)

Per the CLAUDE.md spec in the repo:

  1. Globals — server state
  2. Provider loading — read API keys + provider configs
  3. WebSocket framing — RFC 6455 frame encode/decode
  4. HTTP server — request parsing + routing
  5. WS helpers — broadcast, per-connection send
  6. Relay auth/register — TOTP + registration flow with the backend
  7. Store handler — wire requests to the nous store via relay
  8. AI provider fan-out — parallel HTTPS calls to Claude/GPT/Gemini/etc.
  9. Judge/summarize — meta-AI consolidates provider responses
  10. Hints — intent-understanding helper
  11. Message router — dispatches between user, AIs, and store
  12. Event loop — accept/select/dispatch

This is the Director pattern fully realized in C. Multi-provider, with judging, with intent hints, with router, with backend-store integration.

The gaps (what's NOT done yet)

Gap 1 — Two-org split

nous-desktop lives in ai-pantheon/. Its backend lives in kastil-systems/. Per the audit-of-all reorg, ai-pantheon is being cleaned up. nous-desktop should either move to kastil-systems/nous-desktop or be folded into kastil-systems/nous as a subdirectory. Decision needed.

Gap 2 — Operational status unknown

Board 8 is in flight (CF token + prime deploy). Board 10 is blocked on Board 8. Need to confirm: does nous-desktop currently build, run locally on Windows + Linux, and successfully register-and-verify against the deployed prime relay? Until that round-trip works, the Director isn't usable end-to-end.

Gap 3 — metal-console tilde workflow doesn't flow through it (yet)

Today, metal-console runs tilde commands via Claude Code reading kernel CLAUDE.md + commands.md directly. nous-desktop is a separate cockpit with its own UI. They don't talk. For the Director vision to be coherent, the tilde workflow should EVENTUALLY route through nous-desktop's Message Router — so the same intent-understanding + multi-AI fan-out + judging applies to every tilde command. That's integration work, not redesign work.

Gap 4 — IDS surface from nous-desktop

nous-desktop talks to the backend nous via wire protocol for store queries (relay_store_query). But the audit's IDS / triple-store substrate (the "Pantheon IDS" we've been designing toward) overlaps heavily with what nous already stores. Need to confirm: does kastil-systems/nous already implement the triple store? Or does the metal-console nous/memory/skill/store substrate need to be the IDS, with nous-desktop pointing at it instead of at the older nous backend?

Gap 5 — Documentation drift

The nous-desktop CLAUDE.md cites serve.c as 3053 lines; the actual file is 2903 lines (refactored down by 150 lines since). README references kastil-systems/nous as the backend correctly. Minor — but a sign that docs lag the code.
