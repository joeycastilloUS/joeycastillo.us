Not an SDK other things consume. The whole Pantheon, organized as one nous tree. The "IO Bus" responsibilities distributed into nous tools where they actually belong.

Correction The IO Bus design (v1 and v2) treated this as an SDK other tools consume — library + CLI, link from outside. Joey's reframe: "I actually don't want an SDK I want an end to end system that makes sense library organization wise." The system already has a library taxonomy — nous/<purpose>/skill/<skill>/ and nous/<purpose>/tool/<tool>/. The "IO Bus" responsibilities don't need their own corner; they fit into the existing nous structure as new tools under existing purposes. No new layer, no SDK boundary, no library to publish — just extending nous appropriately.
What the system actually is
One nous tree. The intent is "serve Joey." The purposes are memory, intelligence, user, security, comms, action — all already in the taxonomy. The "IO Bus" responsibilities split across nous/memory/tool/, nous/intelligence/tool/, and nous/user/tool/ as new tools alongside existing skills. Operator binaries (metal, metal-console, metal-desktop) compose tools and skills from this tree. AI integrations are themselves tools INSIDE nous. Nothing outside the tree.

Each binary = main() + a Makefile + a list of skill/tool source files to compile in.
Same pattern as tlt and test_self in metal/Makefile today (lines 52-59, 119-127).
No SDK published. No library to import. Just shared source compiled per binary.

The library organization, said straight

Library organization
There's one source tree: nous/. It already exists in kastil-systems/a-d-d (canonical) and is vendored into metal/src/sdk/nous/. The tree is organized as nous/<purpose>/<skill OR tool>/<name>/. Skills are reusable C libraries; tools are small binaries built from skills. Operator binaries (metal, metal-console, metal-desktop) compose this source per-binary via their Makefile. That's the library organization. No SDK, no external API surface, no "io_bus." Just nous, extended where it needs to be.

Smallest concrete next move (replaces v2's First Move)

1. Pick ONE of the new nous tools to scaffold first.
   Candidate: nous/memory/tool/shell/
   Why: it's the operator's CLI into the IDS. Used by every console.
   Without it, no other tool is testable end-to-end.

2. Create the folder structure in kastil-systems/a-d-d/nous/memory/tool/shell/
   Files: CLAUDE.md (architecture spec) · skill.json (manifest) · tests/

3. The CODE (the .c/.h source) lives in metal/src/sdk/nous/memory/tool/shell/
   following the existing vendoring pattern (per kernel: a-d-d holds architecture,
   code repos hold implementation).

4. metal/Makefile gets a new target: shell
   Source list: src/sdk/nous/memory/tool/shell/main.c
              + src/sdk/nous/memory/skill/{store,index,hash,journal,mmap,scope}/*.c
   Same pattern as tlt target (Makefile L52-59).

5. Build: make shell. Test: ./shell assert "test" "is" "working".
   Then: ./shell query "test" "*" "*".

6. After shell works: scaffold nous/memory/tool/mirror/
   (file-mirror sync for telemetry.json, per the v2 design's Path F).

7. After mirror works: scaffold nous/intelligence/tool/claude-code-bridge/
   This is the renderer that turns triples into CLAUDE.md output.

Each step = one tool, one Makefile target, one test. Compose, don't pile.
