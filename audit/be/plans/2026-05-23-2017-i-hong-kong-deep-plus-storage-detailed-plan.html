Someone (the installer? a different deploy script?) installed nous-catalog.service on it instead of nous-relay.service. The labels never got updated, and the missing relay service never got added.

Operationally, the intent was clearly Asia-East relay coverage (paired with relay-lax for Asia↔US coverage). That intent was never realized because the install was wrong. The mistake has been quietly sitting for 16 days.

Part 3 — Archive class cost reality (read before you migrate)

Critical pricing — Archive class Cross-project copy of Archive-class data is NOT free. GCP charges:
  • Retrieval (Class B operations + per-GB): $0.05 / GB read from Archive. If a bucket has 100 GB, reading it costs $5.
  • Early deletion fee: Archive has a 365-day minimum storage duration. Deleting objects before 365 days = pay full storage cost for the remaining days.
  • Per-object retrieval: Each object retrieval is a Class B operation (~$0.05 per 10,000 objects).
Before pulling the trigger on any bucket migration, run gsutil du -sh gs://<bucket> + check the bucket creation time. If a bucket was created < 365 days ago AND contains significant data, the early-deletion fee could be substantial.

Three migration strategies for Archive class

StrategyHowCost profileBest when
A. Native rsync (the standard) gsutil -m rsync -r gs://source gs://dest Pay retrieval fee ($0.05/GB) + cross-region egress if different region. Source still incurs early-deletion if < 365 days old when deleted. Buckets with low GB but high object counts; you're OK paying retrieval to maintain control.
B. Storage Transfer Service (STS) gcloud transfer jobs create with source + dest. STS handles the parallel pull. Same per-GB retrieval cost as A. STS has no extra fee BUT operations are charged at Class B rate. Large buckets where you want managed parallelism and reporting.
C. Move via "rewrite" (in-place class change first) Step 1: gsutil -m rewrite -s standard gs://source/** (changes class to Standard in-place). Step 2: rsync as A. Step 3: gcloud projects move the bucket itself. Avoids Archive read fees (Standard reads are free). But: rewriting incurs Class A operation fees + Standard storage costs during the migration window. Very large buckets where retrieval fees would dominate.

For Joey's nous-kb-* buckets, the right strategy depends on size. Until we know GBs per bucket, Strategy A is the default conservative choice — simple, predictable, manageable.

Bucket name conflict (can't have same name in two projects)

GCS bucket names are globally unique. To copy nous-kb-us-east4 to kastil-systems, the destination bucket can't also be named nous-kb-us-east4 — that name is already taken by the Acme bucket. Two paths:

  • Rename pattern (recommended for parallel run): destination becomes kastil-kb-us-east4 (or nous-kb-us-east4-ks). Both buckets exist during the cutover window. Old gets deleted after verification.
  • Delete-first pattern: empty + delete source bucket, then create with same name in destination. NO rollback once source is gone. Bigger risk.

Recommend rename pattern. The kastil- prefix matches Joey's "kastil-systems is the new home" framing.

Storage Phase 2 — COLD buckets (11: all nous-kb-*)

Archive class. Pay retrieval fee. Run in parallel to amortize time. Per-bucket template:

# Per cold bucket:
SRC=nous-kb-us-east4         # change per iteration
DEST=kastil-kb-us-east4
REGION=us-east4

# Create destination as Archive (matches source class)
gcloud storage buckets create gs://$DEST \
    --project=kastil-systems --location=$REGION --default-storage-class=archive

# Copy (this incurs retrieval fees on source)
gsutil -m rsync -r gs://$SRC gs://$DEST

# Verify byte parity
gsutil du -s gs://$SRC gs://$DEST

Loop over all 11 cold buckets (8 "warm" named + 5 "cold" named — but actually the count is 11 in the move list: 6 without "cold" prefix in NOUS region set, plus 5 with "cold" prefix; the "warm" set has us-east4 and us-west1 in addition to the 4 international ones = 6 total non-cold-named + 5 cold-named = 11):

declare -A BUCKETS=(
  [nous-kb-asia-east1]=kastil-kb-asia-east1
  [nous-kb-australia-southeast1]=kastil-kb-au-se1
  [nous-kb-europe-west1]=kastil-kb-eu-west1
  [nous-kb-southamerica-east1]=kastil-kb-sa-east1
  [nous-kb-us-east4]=kastil-kb-us-east4
  [nous-kb-us-west1]=kastil-kb-us-west1
  [nous-kb-cold-asia-east1]=kastil-kb-cold-asia-east1
  [nous-kb-cold-au-se1]=kastil-kb-cold-au-se1
  [nous-kb-cold-eu-west1]=kastil-kb-cold-eu-west1
  [nous-kb-cold-sa-east1]=kastil-kb-cold-sa-east1
  [nous-kb-cold-us-east1]=kastil-kb-cold-us-east1
)
for src in "${!BUCKETS[@]}"; do
  dest=${BUCKETS[$src]}
  region=$(gcloud storage buckets describe gs://$src --format='value(location)' | tr A-Z a-z)
  (
    gcloud storage buckets create gs://$dest --project=kastil-systems --location=$region --default-storage-class=archive
    gsutil -m rsync -r gs://$src gs://$dest
    echo "$src → $dest done. Sizes:"
    gsutil du -s gs://$src gs://$dest
  ) &
done
wait

Parallel execution — 11 buckets copying simultaneously. Each takes minutes-to-hours depending on size.

All 11 cold buckets mirrored in kastil-systems. Byte-parity per bucket verified.

Storage Phase 3 — Naming reconciliation (decide later, after parallel-run window)

Once kastil-systems has its own kastil-kb-* set, you have a choice for the original nous-kb-* names (after you delete the originals in Acme):

  • Recreate them in kastil-systems with the original names (transient downtime on the second creation)
  • Keep the kastil- prefix permanently (matches the new brand stack)

Recommend: keep kastil- prefix — it's honest about which org owns the buckets. The bucket-name change is invisible to readers (they reference buckets by URL anyway).

Naming finalized.

Storage Phase 4 — Source bucket cleanup (after rollback window)

Per the rollback window pattern (1 week), keep source buckets intact in Acme. After window closes:

# For each bucket — delete source AFTER verifying destination
gsutil -m rm -r gs://nous-kb-us-east4/**
gcloud storage buckets delete gs://nous-kb-us-east4 --project=acmedev-platform-hq

Important: deleting Archive-class objects < 365 days old incurs early-deletion fees. Check creation time first:

for b in nous-kb-asia-east1 nous-kb-australia-southeast1 ...; do
  created=$(gcloud storage buckets describe gs://$b --format='value(metageneration)' 2>/dev/null)
  echo "$b: created=$created"
done

If a bucket has objects older than 365 days, deletion is free. If newer, you pay the prorated remainder.

Source buckets deleted (or retained-with-decision per bucket if early-deletion fees are punitive). Migration complete.

Open question — the "cold" naming duplication

Why does the fleet have TWO sets of nous-kb-* in the same regions — one with "cold" prefix, one without — both at Archive class? Possibilities:

  1. The "cold" set was planned to be a different (lower) tier (e.g. Coldline) but ended up Archive too — drift from the naming convention
  2. The "cold" set is a backup/archive of the "warm" set — same data duplicated for disaster recovery
  3. The two sets hold different content — e.g. "warm" = recent month, "cold" = older archive
  4. Historical drift — created in two phases by different scripts

Need to spot-check content of one warm/cold pair to determine. Suggested probe:

gsutil ls gs://nous-kb-us-east4/ | head -10
gsutil ls gs://nous-kb-cold-us-east1/ | head -10
# Compare object naming patterns + sizes — if mostly identical, it's a backup; if different, it's tiered

This affects the migration plan only if the "cold" set turns out to be redundant (then we can skip migrating 5 of 13 buckets and just delete them in Acme).
