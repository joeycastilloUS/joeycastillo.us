~p Mesh Hardening v2 — the build plan

Executes the design (2026-05-28 ~d mesh professional grade). Ordered so the one real hole ships first and every phase leaves the mesh shippable.

Intent

Add the lifecycle layer (freshness, expiry, revocation, mutual auth, rotation) to the live Sealed-IDS mesh, in pure C, without a flag day — each phase is independently deployable and leaves the fleet working.

Scope

In: mesh_roster.c, mesh-roster-build, operator-sign-node, operator-keygen, relay_server.c + prime_server.c wire/boot paths, operator_pub.h, security tests. New: revocation-list tool + format, peer-handshake module.
Out: wiring the transport mesh.c routing stack (D8 defer); BLE/LoRa backends; any non-mesh repo.

Approach

Spec-first (Rule 11): land the a-d-d v2 spec, then code flows from it. Format changes bump the roster magic to MESHROSTER2 so v1/v2 can't be confused; cutover re-issues mesh.sealed + certs once per node (same scp path already proven). Every change ships with its test (Rule 7). Each phase = one board.

Constraints

  • Pure C99, no malloc, static buffers, link libsecurity, constant-time on secrets, zero-after-use — non-negotiable house style.
  • No flag day: every phase deploys node-by-node (Ashburn-first, like the live cutover) and degrades gracefully if a peer is still on the old format during rollout.
  • Offline-tolerant: nothing may assume a node can reach the operator or a peer to stay valid (drives the 30-day TTL, not minutes).
  • Done = production (Rule 8): each phase verified on a real VM (self-verify OK / replay-rejected / handshake-OK in the journal), not just unit-green.

Unknowns (what I don't know yet — named)

  • TTL value. 30 days is a guess at the offline window. Real field behavior may need 90+. Unknown until we know the longest expected dark period.
  • Handshake cost. Whether the extra round-trip per peer connection matters at the relay's connection rate — unmeasured.
  • D4 build-vs-reuse. Hand-rolled node-cert challenge-response vs reusing tls13.c mTLS — won't know which is cleaner until P4 spike.
  • Clock trust. Expiry (D2) assumes nodes have a roughly-correct clock; field nodes without NTP could false-reject. May need a signed time anchor.

When to revisit (the signal that triggers v3)

  • First time a field node is dark longer than the TTL → revisit D2 (TTL or offline-grace).
  • First measured handshake latency hit at scale → revisit D4 (session resumption / caching).
  • First real key compromise or near-miss → pressure-test D3/D6 against an actual incident, not a hypothetical.
  • The moment BLE/LoRa transport lands → D8 un-defers; wire mesh.c and re-plan peer auth over a non-IP transport.
Sequencing logic: P0 first because it's a live hole and costs ~nothing. P1 before format changes so the tests guard P2+. P2→P3 because revocation only matters once TTL is long. P4 last of the security work because it's the largest and depends on nothing above it. P5 is operational polish. Stop after any phase and the mesh is still strictly better than today.
