~i Vault Anonymity — Design Violation + Fix

Joey flagged: "from inside the vault, no idea what belongs to who." I checked the a-d-d / kernel / master plan — the requirement is there, and my MH-VAULT V1 violates it. This report confirms, locates, and proposes the surgical fix.

§1 — The requirement (it IS in your stack, repeatedly)

WhereQuote

The principle is consistent across architecture, spec, and kernel: the mesh substrate must mask both the content AND the identity behind the data it stores. Quote #32 names it explicitly for the data shape; it applies equally to the identity shape. My V1 design respects content masking (ciphertext) but breaks identity masking — that's the gap.

§2 — The violation (what I shipped)

MH-VAULT V1 key shape leaks operator identity Current shape: <operator-fqdn>:<sha256-content> (e.g. joeycastillo.us:a61b...). The FQDN is in cleartext, visible to anyone with relay disk access OR anyone watching wire traffic (URL path). A peer or attacker can:
  • Link blobs to an operator — every key starting with joeycastillo.us: belongs to Joey. Operator inventory is enumerable per FQDN.
  • Profile activity — count of blobs, write rates, size patterns per operator FQDN.
  • Single out individuals — relay logs (request paths) name the operator on every request.
This contradicts the spec's "opaque" property and the kernel's "pseudonym masks substrate" rule.

§3 — The fix (small, surgical, restores the property)

New key shape — both components opaque 64-hex 
<key> = <namespace-hex> ":" <content-hex>

namespace-hex = 64 lowercase hex chars (32-byte HMAC-SHA256 over op privkey)
content-hex   = 64 lowercase hex chars (32-byte SHA-256 of plaintext)
Both components are indistinguishable random-looking hex. Nothing in the key reveals the operator; mesh, peers, and disk-readers see two opaque 32-byte values per blob.

Namespace derivation (client-side, deterministic, secret)

namespace = HMAC-SHA256(operator_privkey, "vault-namespace-v1")[:32 bytes]
            → hex-encode → 64 chars
  • Deterministic — same operator key → same namespace, every time, on every machine. Operator can always find their own blobs.
  • Unforgeable — only someone with the operator private key can compute it. Mesh peers (who have only the public key, if even that) cannot reverse-engineer.
  • Per-operator scoping preserved — the relay still does per-namespace quota; the namespace is just opaque hex, not a recognizable FQDN.
  • Stronger than pubkey-derived (e.g. sha256(op_pub || "vault")) which would still leak to anyone who has the pubkey. HMAC-with-priv ensures only the operator can compute it.

Storage path (server-side, opaque)

/var/nous/vault/<namespace-hex>/<content-hex>
                    (64-hex subdir, opaque)

Why this satisfies Joey's brief verbatim

  • "from inside the vault, no idea what belongs to who" — relay disk shows directories named by 64 hex chars; no FQDN, no domain, no recognizable identifier.
  • "not knowable from inside the vault" — even with the operator pubkey in hand, you can't compute the namespace (requires the private key).
  • Quote #32 "pseudonym masks substrate" — the namespace IS the pseudonym for the operator; substrate (FQDN) is masked.

§6 — Honest note on what this does and does NOT give you

  • Does hide which operator owns which blob from any relay-disk reader, wire snooper, or peer relay.
  • Does hide blob content (already ciphertext under seal v2).
  • Does NOT hide that some operator wrote some blob at time T (timing/traffic analysis); that's a tor-class property requiring mix-networking — out of scope for v1.
  • Does NOT hide the number of blobs per namespace (quota visible by directory listing) — but the namespace is unlinkable to the operator, so this is mostly meaningless to an attacker.
  • The operator's identity remains anchored at the Pantheon trust root (operator pub baked into nodes) — anonymity here is for vault contents only, not for mesh participation.
