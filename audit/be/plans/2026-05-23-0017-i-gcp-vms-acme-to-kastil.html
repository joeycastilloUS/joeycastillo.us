Migration approach — snapshot + rebuild (NOT lift-and-shift)

Constraint GCP does NOT support moving Compute Engine instances directly between projects. You can move resources via images / snapshots / disks, but the VM itself is project-scoped. So the migration is: snapshot the disk → create image in the new project → boot a new VM from that image with the new name → cut over → delete the old.

Per-VM migration recipe

# 0. Pre-flight (one-time, kastil-systems project)
gcloud config set project kastil-systems
gcloud services enable compute.googleapis.com
# Verify VPC + firewall rules exist (or create default)
gcloud compute networks list

# 1. Snapshot the source disk (acmedev-platform-hq side)
gcloud config set project acmedev-platform-hq
gcloud compute disks snapshot <old-vm-name> \
    --zone=<zone> \
    --snapshot-names=<new-name>-snap-<date>

# 2. Create an image from the snapshot, shared with kastil-systems
gcloud compute images create <new-name>-img-<date> \
    --source-snapshot=<new-name>-snap-<date> \
    --storage-location=<multi-region matching zone>

# Grant kastil-systems read access to the image
gcloud compute images add-iam-policy-binding <new-name>-img-<date> \
    --member="serviceAccount:<kastil-systems-default-SA>@kastil-systems.iam.gserviceaccount.com" \
    --role="roles/compute.imageUser"

# 3. Create the new VM in kastil-systems with the new name
gcloud config set project kastil-systems
gcloud compute instances create <new-name> \
    --zone=<same-zone> \
    --machine-type=<same-type> \
    --image=<new-name>-img-<date> \
    --image-project=acmedev-platform-hq

# 4. Verify (boots, reaches dependencies, serves)
gcloud compute instances describe <new-name> --zone=<zone>
# Check service health via the appropriate port (relay/prime/nous)

# 5. Cut over (DNS / config update points at new IP)
# Verify nothing else is talking to the OLD IP before proceeding

# 6. Stop the old VM (don't delete yet — keep for rollback window)
gcloud config set project acmedev-platform-hq
gcloud compute instances stop <old-vm-name> --zone=<zone>

# 7. After 24-72h verification: delete old VM + snapshot
gcloud compute instances delete <old-vm-name> --zone=<zone>
gcloud compute snapshots delete <new-name>-snap-<date>

Pre-flight checklist

  1. Enable Compute Engine API on kastil-systems — currently disabled. gcloud services enable compute.googleapis.com --project=kastil-systems
  2. Verify billing on kastil-systems — Compute requires a billing account attached
  3. VPC / subnet plan — auto-mode default works (one subnet per region, /20 each). Confirm or use a custom VPC for tighter control.
  4. Firewall rules — replicate the rules from acmedev-platform-hq that allow the prime/relay/nous traffic (likely TCP 22 SSH + the NTRP0001 wire-protocol port + HTTPS)
  5. Service account — what SA do the VMs run as today? Replicate in kastil-systems with same roles, or use the default Compute SA
  6. External IPs — do any of these VMs have static external IPs that DNS points at? Those need to be reserved in kastil-systems too (or expect new IPs + DNS update)
  7. Cloud Storage / Secret Manager dependencies — does the prime/relay code read from buckets or secrets in acmedev-platform-hq? Those need to be accessible from kastil-systems too (cross-project IAM grants).

Open questions before executing

  1. nous-elite-iad — c2-standard-8 is expensive. Why's it terminated? Cost optimization (start on demand)? Unused? Different decision changes whether we keep the c2-standard-8 sizing or downsize during migration.
  2. External IPs / DNS — what hostnames currently point at these VMs? relay.3-nous.net? prime.3-nous.net? The cutover step depends on this.
  3. nous-elite role — singleton or part of a fleet? "elite" suggests a special compute node. If singleton, no redundancy during migration — schedule downtime.
  4. Same-zone constraint — should the new VMs stay in the same zone as the old (us-east4-a, us-west4-a, etc.) for minimal disruption? Or take this opportunity to consolidate / rebalance?
  5. Snapshot vs. clean rebuild — does each VM contain stateful data, or can it be rebuilt from a base image + config? Clean rebuild is safer architecturally (no carryover cruft) but requires having the build automation handy.
  6. Acme Platform project fate — after all 5 VMs move, what's left in acmedev-platform-hq? If nothing important, the project can be marked for shutdown.

What I'd do first (the smallest concrete move)

  1. Enable Compute Engine API on kastil-systems (one command, takes a few minutes to propagate)
  2. Snapshot nous-elite-us in acmedev-platform-hq (it's TERMINATED, zero impact)
  3. Create an image in kastil-systems from the snapshot
  4. Boot it as nous-elite-iad in kastil-systems, us-east4-a
  5. Don't start workload yet — just confirm the boot + SSH access works
  6. That's the migration recipe validated. Steps 2-5 for the others follow the same shape, just with care around live traffic.
