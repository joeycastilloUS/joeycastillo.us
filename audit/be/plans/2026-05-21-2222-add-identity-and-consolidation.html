What counts as an a-d-d file? Is all a-d-d in one place yet?

Two related questions. File-level identity (what makes a file part of a-d-d vs ambient project doc). Repo-level consolidation (where a-d-d content actually lives across all your orgs today).

Trail: final scan · merge investigation · ~i sweep
Answer 1 — file identity
An a-d-d file is anything with a skill.json next to it, or any file inside a path matching <repo>/<intent>/<purpose>/skill/<skill>/. Plus the top-level methodology files: GLOBAL.md, NOTICE, and the root CLAUDE.md in a repo whose own description says "Architecture Driven Development."
Answer 2 — consolidation status
Not yet. kastil-systems/a-d-d is the canonical home (10 merge commits today consolidated the methodology + nous architecture + Pantheon NOUS content into it). But 5 other a-d-d-shaped repos still exist across orgs — 1 archived, 4 active. Concrete cleanup list below.

Part 1 — How to tell if a file is "a-d-d"

The kernel CLAUDE.md (just absorbed via commit 308ea1f) declares the structure: kASTIL/<intent>/<purpose>/skill/<skill>/ with each skill being a .c + .h + skill.json trio. From that the identification rules fall out:

YES — a-d-d artifacts (the strong signatures)

skill.json + sibling CLAUDE.md + sibling <name>.h + sibling <name>.c
If a folder has skill.json, it IS an a-d-d skill folder. This is the most distinctive marker — skill.json exists nowhere else in your stack. Every CLAUDE.md sitting next to one is part of the a-d-d architecture spec.

YES — a-d-d artifacts (structural)

CLAUDE.md inside a path matching <intent>/<purpose>/skill/<skill>/ OR at any parent level (intent/, purpose/) inside an a-d-d repo
A CLAUDE.md at nous/memory/skill/journal/CLAUDE.md is an a-d-d skill spec. A CLAUDE.md at nous/memory/CLAUDE.md is an a-d-d purpose spec. The hierarchical position tells you it's architecture, not documentation.

YES — a-d-d methodology files (top of repo)

GLOBAL.md + NOTICE + root CLAUDE.md (when the repo is named *-a-d-d or *-ADD)
GLOBAL.md = the methodology source ("Global Rules" — kASTIL SYSTEMS INC. branded version is canonical). NOTICE = trade-secret declaration. Root CLAUDE.md = kernel code-organization rules for the repo. All three together = "this repo IS an a-d-d repo."

AMBIGUOUS — could go either way

CLAUDE.md at a project root that doesn't have skill.json or hierarchical structure
A CLAUDE.md at the root of a code repo (e.g. metal-console/CLAUDE.md) is a Be kernel doc OR a project README, not an a-d-d architecture artifact. The presence of skill.json anywhere in the tree disambiguates — without it, the CLAUDE.md is project-level docs.

NO — not a-d-d (even if it looks similar)

README.md, PRIMER.md, CHANGELOG.md, docs.html, any .md file outside the intent/purpose/skill structure
These are docs or teaching content. joeycastilloUS/a-d-d-guidance's entire payload is this category — narrative explanation of ADD, not ADD itself. Useful for newcomers, but not architecture.

NO — code, not spec

.c / .h / .py / .go / .ts / etc.
Implementation files. Per the kernel rule we just merged ("When syncing from a-d-d/, always overwrite code-tree copies — a-d-d/ wins"), code repos may have a copy of the a-d-d folder structure synced down INTO them — but the spec is the a-d-d/ source, not the per-repo copy.

The decision tree

  1. Does the file have a skill.json sibling? → YES = a-d-d skill spec.
  2. Is the file path <repo>/(nous|purpose|intent)/.../CLAUDE.md? → YES = a-d-d hierarchy doc.
  3. Is the file GLOBAL.md at a repo root? → YES = a-d-d methodology source.
  4. Is the file NOTICE at the root of a repo named *a-d-d* or *ADD*? → YES = a-d-d trade-secret declaration.
  5. Is the file CLAUDE.md at a repo root that isn't named *a-d-d*? → It's a Be kernel doc or project README, NOT an a-d-d artifact.
  6. Anything else? → Not a-d-d.

Plus: a-d-d structure inside code repos

Per the kernel rule absorbed today ("When syncing from a-d-d/, always overwrite code-tree copies — a-d-d/ wins"), code repositories can hold a-d-d/ folders that are scaffolded copies — synced from the canonical, regenerated on sync. These are NOT sources of truth — they're build-time artifacts. Currently this pattern is implicit; whenever a project starts ingesting a-d-d/ spec content into its own tree, that copy is downstream, not authoritative.
