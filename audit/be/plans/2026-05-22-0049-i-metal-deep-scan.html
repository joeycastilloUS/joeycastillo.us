Headline

metal-console is not just a "tools" repo — it's a 5-layer operating substrate. The C substrate for IO Bus is already vendored in at src/sdk/nous/, and src/tlt/ is the closest existing analog (pure C binary that already links the memory skills + has its own protocol). The IO Bus design has 6 different ways to ship — picking one matters.

How metal uses a-d-d (three distinct ways)

WayWhereRole
1. As methodology CLAUDE.md (kernel) — references kastil-systems/a-d-d as the canonical methodology source. The kernel doc absorbed Foundational Principles, Scope & Workflow, Trade Secret Notices from there today (commit 308ea1f). CONSUMES
2. As runtime spec be/docs/commands.md + be/docs/conventions.md. Every tilde command, every protocol. The kernel ROUTES to these on-demand; Claude Code reads the relevant section as it executes. RUNS
3. As code substrate (vendored) src/sdk/nous/ — full C source for all the nous skills. memory/skill/store, security/skill/crypt (with TLS 1.3 + cf_pin), comms/skill/{relay,waf,mesh,pulse,wire,trace}, user/skill/self, etc. Compiles into the metal binary + tlt binary + test_self harness. EMBEDS

metal does NOT hold a-d-d methodology CONTENT (no GLOBAL.md here, no NOTICE declaring metal as a-d-d). metal is the operator's runtime — it consumes the methodology, runs the tilde-system, and ships compiled binaries that use the SDK.

What tlt already does that IO Bus would also do:

  • Pure C binary linking nous/memory/skill/{store,index,hash,journal,mmap,scope} + security/skill/crypt
  • Has its own JSON serializer (no external deps)
  • Runs a server (HTTP on POSIX via pthread, Win via ws2_32)
  • Has a bridge to "workers" (LLMs) — Playwright proxy for Google scout already implemented
  • Persists state via the store skill
  • Has a documented wire protocol (PROTOCOL.md)

What tlt does NOT yet do that IO Bus would add:

  • Multi-tool API surface (tlt is its own self-contained binary, not a library)
  • Per-AI-convention renderers (no Claude-format / Perplexity-format output adapters)
  • The newborn↔mother bootstrap protocol
  • Scope manager with user alerts
  • Public io_bus_* C API for other tools to link

Many ways to skin this cat — 6 paths for the IO Bus

Path A — Pure new library (libio_bus.a) — what v1 design proposed

Build libio_bus.a from scratch. Wraps the nous memory skills. Every tool (metal, metal-console, metal-desktop, tlt) links it. New header io_bus.h. New _store/io_bus/ tree in kastil-systems/a-d-d.
Pros: clean abstraction, ABI-stable, exactly what was designed.
Cons: tlt already does ~70% of this. Duplication risk. Two C runtimes that read the same store binary format.

Path B — Extend tlt with public API

Add tlt.h public API. Refactor tlt's existing main.c into a library + small CLI shim. Other tools link libtlt.a. The Playwright worker stays. The HTTP server can be opt-in.
Pros: zero duplication. The Playwright proxy already exists. PROTOCOL.md is the starting wire spec.
Cons: tlt name is tied to "The Last Test" — overloading it as the IO bus dilutes naming. Refactor risk in shipped C code.

Path C — CLI binary (ids-cli or io-bus exe)

Ship a single CLI binary. Tools shell out via execve. No library linking. Stdin/stdout JSON protocol.
Pros: simplest integration. Claude Code can call it via Bash without linking. Every language can use it. Crash-isolated per call.
Cons: process-launch overhead per call. State management trickier (PID, sockets, file locks). Less ergonomic for C tools.

Path D — Daemon + IPC (single-process backbone)

One long-running process owns the IDS. Tools connect via Unix sockets / named pipes / HTTP. tlt already has a server, this builds on that.
Pros: shared cache, single writer, real-time alerts via push. Matches the "bus" metaphor literally.
Cons: daemon lifecycle to manage. Crash recovery. Per kernel: "no daemons" preference. Adds an attack surface.

Path E — Library + CLI hybrid

Ship both: libio_bus.a for C tools to link, io-bus CLI wrapping the same library for shell + AI use. Same engine, two interfaces.
Pros: best of A and C. C tools get fast in-process calls; Claude Code shells out via Bash. The CLI is a thin wrapper, low maintenance cost.
Cons: two surfaces to test. Slight maintenance overhead.

Path F — File-mirror migration (gradual)

Keep board.md, board-status.json, telemetry.json, etc. as the surface tools touch. Add a sync daemon that mirrors them into the triple store. Tools migrate to direct triple reads when convenient. Files become "compatibility renderings" eventually phased out.
Pros: zero disruption to existing tools. render-board-view.py (6162 L!) doesn't need rewriting. Triples become source-of-truth without breaking anything.
Cons: dual-source-of-truth during transition (drift risk). Sync daemon is itself an extra component. Slower path to full triple-store benefit.

What the right answer is starting to look like

From this inspection, three observations dominate:

  1. The C substrate is done. src/sdk/nous/memory/skill/store + index + hash + journal + mmap + scope are all vendored and compiling. The IO Bus library doesn't need to ship the IDS — it composes existing source.
  2. tlt is the proof. src/tlt/ is a working pattern of "pure C binary that links nous skills + serves a protocol." It's not a library yet, but the parts are there. Path B (extend tlt) is the cheapest engineering-wise but requires a naming decision.
  3. The tools are file-driven. 23 tools, mostly Python, read JSON + markdown state files. They don't link C libraries today. For Claude Code to use the IO Bus, the CLI path (C or E) is essential. Path A alone leaves Claude shelling out to nothing.

Recommendation that the v2 design will articulate: Path E (library + CLI hybrid) as the shape, with Path F (file-mirror) as the migration ramp. The library lives at kastil-systems/a-d-d/_store/io_bus/. The CLI lives next to it. Existing tools keep using state files until they choose to migrate. The watch.py daemon becomes the file→triple mirror during transition.

Open questions surfaced by this inspection

  1. Does tlt get refactored or stays as-is? If we extend tlt (Path B), we save engineering but blur naming. If we keep tlt standalone, we duplicate ~30% of code with libio_bus.
  2. The 6162-line render-board-view.py — does IO Bus replace it eventually? If triples become source of truth, the renderer becomes ~500 lines that read triples instead of 6162 lines that parse markdown/JSON.
  3. Where do the 11 state files (board, telemetry, h-cache, prompts, tags, users, ...) migrate to? Path F says gradually. But the order matters — telemetry first? Prompts first? Board first?
  4. Does the Playwright worker get folded in or stay separate? tlt/worker/ has Playwright as C#. The IO Bus could either reuse it (Path B) or treat it as out-of-scope (Paths A/E).
  5. How does Claude Code talk to IO Bus? If library only: it can't (Claude can't link C). If CLI: easy (Bash tool). Path E says both, so Claude uses the CLI; metal-desktop links the library.
  6. Single canonical CLI binary location? kastil-systems/a-d-d/_store/io_bus/io_bus built locally? Or installed system-wide via Go.Fe.bat?
  7. How do hooks integrate? capture_prompt.ps1 currently writes prompts.jsonl. Path F would have it ALSO write a triple via the CLI. Eventually it stops writing JSON.
  8. What's the minimum viable First Move now? Possibly: refactor a SINGLE state file (say telemetry.json) to be backed by triples via a thin CLI. Prove the concept end-to-end on one file before scaling.
