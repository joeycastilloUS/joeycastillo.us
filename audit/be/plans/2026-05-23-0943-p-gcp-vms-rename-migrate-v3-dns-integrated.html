1. What the DNS a-d-d says (found and read)

Location: kastil-systems/a-d-d/nous/comms/skill/relay/ — three files:

  • CLAUDE.md — relay skill spec (Trust Model, Boot Sequence, Gate Port :8443, Protocol, Session, DNS Records)
  • DNS_DISCOVERY.md — the implementation diagram (boot flow + 5-min refresh + Cloudflare API + SPKI pin + signature verify)
  • TRIPLES.md — the triple-store records this skill writes

The architecture in 9 facts

  1. Domain: 3-nous.net (Cloudflare-authoritative DNS, not GCP DNS)
  2. Trust root: ONE operator Ed25519 public key, compiled into binaries. Nothing else trusted by default.
  3. Roles: exactly two in the protocol — "prime" and "relay" (ASCII strings signed into node.cert).
  4. Discovery records: _prime.3-nous.net and _relay.3-nous.net — TXT records.
  5. TXT format: "p=<ip>:<port> k=<pubkey-64hex> s=<sig-128hex>" (prime). "r=..." (relay).
  6. Per-node identity: three files — node.key (private Ed25519), node.pub (public), node.cert (operator's signature over node.pub || role).
  7. Self-announce: on boot, the node hits POST /zones/<id>/dns_records on Cloudflare API. Auth = bearer from /var/nous/cf.token (chmod 600). Pinned via cf_pin.h (GTS WE1 intermediate SHA-256 SPKI).
  8. Refresh: every 5 minutes — re-query DNS, re-verify sigs, refresh peer triples. Stale peers expire naturally.
  9. Multi-instance friendly: multiple TXT records under the same name are allowed. Read side filters by signature validity. Old + new can coexist during cutover.

2. The renaming — node + relay only

Joey: "name it relay or node." The protocol uses "prime" and "relay" as the cert role strings. There's a decision to make below; v3's working assumption is that VM NAMES use "node"/"relay" while the PROTOCOL ROLES stay "prime"/"relay" (no protocol change — just operational naming).

Open decision — "node" in VM names vs "prime" in protocol

Two options:

  • (A) Operational rename only. VM names use node-* but the node.cert role string stays "prime", DNS records stay _prime.3-nous.net. Zero protocol change. Recommended for v3 execution. The VM name is a GCP/OS identity; the protocol role is separate.
  • (B) Full rename — protocol too. Update relay/CLAUDE.md, relay code, DNS records, certs, and rebuild all binaries to use "node" instead of "prime". Bigger change. New DNS record: _node.3-nous.net. Requires re-signing every cert.

Recommendation: (A) for this migration. If you want (B), it's a separate plan — it's a protocol change, not a VM rename.

4. The DNS migration story (this is the GOOD news)

Because 3-nous.net is Cloudflare-hosted (not GCP-bound) and the protocol uses self-announce with signature verification, the migration has a clean overlap pattern:

BEFORE MIGRATION:
  _prime.3-nous.net  TXT  "p=<old-ip-singapore>:8080 k=... s=..."
  _prime.3-nous.net  TXT  "p=<old-ip-las-vegas>:8080 k=... s=..."
  _relay.3-nous.net  TXT  "r=<old-ip-hong-kong>:8080 k=... s=..."
  _relay.3-nous.net  TXT  "r=<old-ip-los-angeles>:8080 k=... s=..."

DURING CUTOVER (new VM boots, announces self, old still running):
  _prime.3-nous.net  TXT  "p=<old-ip-singapore>:8080 k=... s=..."   ← old
  _prime.3-nous.net  TXT  "p=<NEW-ip-singapore>:8080 k=... s=..."   ← new
  ...
  Discovery on the consumer side finds BOTH, verifies BOTH (same operator key
  signed both certs), tries both, uses whichever responds. Multi-instance.

AFTER CUTOVER (old VM stopped, manually delete old TXT record):
  _prime.3-nous.net  TXT  "p=<NEW-ip-singapore>:8080 k=... s=..."
  ...

No DNS downtime. No "switch the DNS at midnight" panic. The protocol was DESIGNED for this — primes come and go dynamically per the relay/CLAUDE.md spec (boot sequence step 6 — "Primes come and go dynamically").

5. Per-VM materials (what each new VM needs)

This is the part v2 missed. Each new VM needs the following on first boot:

MaterialWhere it livesHow to provision
node.key (32 B Ed25519 private)On the VM (file path per code)Generate fresh OR carry forward from the old VM (snapshot includes it if previously installed)
node.pub (32 B Ed25519 public)On the VMDerived from node.key — generated when key is generated
node.cert (64 B signature)On the VMOperator signs: ed25519_sign(operator_privkey, node.pub || role) where role is "prime" or "relay". This is a one-time provisioning step per VM.
/var/nous/cf.token (chmod 600)On the VMCloudflare API token, scoped: Zone → DNS → Edit on 3-nous.net. Can be same token across all VMs (or per-VM if Joey wants tighter isolation).
NOUS_EXTERNAL_IP env varVM systemd env / shell profileIf unset, code falls back to 169.254 metadata server (works on GCE). Recommended: leave unset, let metadata work.
NOUS_CF_ZONE_ID env varVM systemd env / shell profileIf unset, code does GET /zones?name=3-nous.net. Recommended: set it explicitly to avoid a startup HTTP call.
Operator Ed25519 PUBLIC keyCompiled into the binaryAlready baked into existing builds. No per-VM action needed.

Two paths for node.key/node.pub/node.cert:

  1. Carry forward (easier): Snapshot the old VM disk → boot new VM → existing node.* files are already present. No re-provisioning needed. Same identity continues, just at a new IP.
  2. Fresh provision (cleaner): Boot new VM blank → generate fresh node.key/node.pub on the VM → operator signs node.cert offline → push back to the VM. Cleaner audit trail, but requires the operator-signing step. Use this if you suspect compromise of an old node.key.

Default for this migration: carry forward. Snapshot-based migration brings the identity files with the disk. No re-signing needed unless we want it for hygiene.

6. Approach

Snapshot the source disk in acmedev-platform-hq → cross-project image into kastil-systems → boot new VM with new name + HW upgrades + same identity files → new VM self-announces to DNS → both old and new appear under _prime.3-nous.net (or _relay.3-nous.net) for ~5 minutes → stop old VM → manually delete old TXT record from Cloudflare → done.

8. Constraints

Snapshot-based migration is forced. GCP cannot lift-and-shift VMs across projects.
DNS coexistence is by design. Multiple TXT records per subname (_prime.3-nous.net) are EXPECTED — relay reads BOTH old and new and tries both during cutover.
Cloudflare manages 3-nous.net. Not GCP DNS. The DNS migration is independent of the GCP project move. The cf.token is what binds VMs to DNS write access.
node.cert is operator-signed. If we regenerate keys (instead of carrying forward), the operator must sign new certs. This requires the operator's Ed25519 PRIVATE key — which should be air-gapped per the Trust Model. Carry-forward path avoids this.
SPKI pin matters. cf_pin.h compiled into the binary pins the Cloudflare API cert. If Cloudflare rotates their intermediate cert, ALL nodes need rebuilt binaries. Not a migration concern but worth noting.
Manual DNS cleanup. Cloudflare doesn't auto-expire stale TXT records on cutover — must delete via API after the old VM stops. This is the only manual coordination step.
v1 hypothesis. Per kernel rule 6 — if any phase reveals something we didn't anticipate, v4 of this plan corrects.

Unknowns to resolve before Phase 0

  1. Operational rename or protocol rename? Decision (A) vs (B). Recommend (A).
  2. Carry-forward identities or fresh-generate? Default carry-forward. Switch to fresh-generate if any node.key compromise is suspected.
  3. Why is nous-elite TERMINATED? If it's been off for weeks, is the workload retired? Or is it cost-saved between runs?
  4. Cloudflare API token rotation. Same token on all VMs, or per-VM? Carry-forward keeps whatever's currently on the old disks.
  5. NOUS_CF_ZONE_ID currently set as env var on the VMs, or letting code do the GET /zones lookup? Either works.
  6. VPC choice. Auto-mode default vs custom VPC with explicit subnets matching old IPs (10.148/16 etc.).
  7. External IPs. Currently ephemeral on the old VMs? Or static reserved? If static, those need to be released from acmedev-platform-hq and reserved in kastil-systems if you want to keep them.
  8. Does 3-nous.net have other records besides _prime and _relay? e.g. A records for users, MX records for mail, root TXT? Capture full zone state before migration.

When To Revisit (v4 triggers)

  • Phase 1 (nous-elite) fails to boot — image issue, NIC upgrade incompatibility, missing carry-forward files
  • DNS self-announce on the new VM fails (cf.token expired? SPKI pin mismatch?)
  • Multi-record TXT confuses the relay code (unlikely — spec says supported)
  • Joey decides Decision (B) — protocol rename. Triggers a v4 protocol-change plan.
  • Any cross-project GCP gotcha not in the recipe (image storage location, IAM propagation delay, etc.)
