What about the relays?

OLD relay SSH failed on both relay-lax and relay-hkg post-migration (likely SSH metadata key rotation during the gcloud snapshot operation). I couldn't directly audit OLD relay disks. But this doesn't matter for parity:

  • Per kastil-systems/a-d-d/nous/comms/skill/relay/CLAUDE.md § State Model: "The relay uses the SAME triple store the prime uses, but in-memory only. No disk backing. No journal. No encryption at rest. Restart = blank slate = re-discover, re-announce, re-auth."
  • Relay disk holds: identity files (node.key, node.pub, node.cert), config (systemd unit), binary, OS. All snapshot-captured.
  • Relay state (sessions, peer cache, etc.) lives in RAM and dies on restart anyway. Migration is functionally a restart — same loss profile as a normal reboot.

NEW relay-lax verified working: nous-relay.service active, talking to NEW prime-las (HTTP 400 = server responding to bad GET, which is correct behavior — proves the wire protocol path is up). NEW relay-hkg has no nous-relay.service, but that's pre-existing drift (OLD relay-hkg also had no service).

The 5 methods for "are we the same on both sides" (add to plan as Phase 2.5)

Method 1 — Byte-count parity (cheapest, do first)

du -sb /var/nous on both. Same number = first-pass match. If different by even 1 byte, drill deeper.

Method 2 — File-count parity

find /var/nous -type f | wc -l on both. Same count = no missing files. If new count < old count, files lost. If new > old, you have unexplained additions.

Method 3 — Drift detection (the time-window check)

find /var/nous -type f -newermt "<snapshot-time>" on OLD. ANY output = OLD has files newer than snapshot. Those files are NOT on NEW (snapshot is point-in-time). For each drifted file: decide if it matters, manually copy to NEW or accept the loss.

Method 4 — Bit-for-bit hash audit (definitive)

find /var/nous -type f -exec sha256sum {} + | sort -k 2 on both. Diff the outputs. Identical lines = identical files. This is the proof. Slightly expensive on large disks but cheap on small ones (NOUS prime: ~380KB — milliseconds).

Method 5 — Functional parity (proves the SERVICE works, not just the disk)

Call the same endpoint on both: curl -s http://<new-ip>:8080/status vs curl -s http://<old-ip>:8080/status. Diff responses. Should be identical except for VM-specific data (hostname, uptime). Catches "files match but service is broken" scenarios.

Parallel-run viability

Both meshes ARE running right now. They CAN coexist indefinitely:

  • Different GCP projects = different VPCs = no internal-IP collision
  • Different external IPs = direct-to-IP calls hit whichever you choose
  • Same DNS records = currently consumers see OLD IPs only (need DNS update for them to see NEW)
  • Two independent fleets = changes to one don't affect the other

This means you can:

  1. Take your time on cutover. No rush. Both serve identical data.
  2. Compare behaviorally — point a test client at NEW external IPs, point another at OLD, compare. Method 5 above.
  3. Cut over progressively — update DNS for one relay first (or use weighted DNS if Cloudflare supports it), watch errors, proceed.
  4. Roll back instantly — revert DNS = consumers go back to OLD. No data loss possible (NEW only got reads, no writes during parallel run unless you specifically routed write traffic).
