Risk assessment

  • Low-risk: The new binary is a strict superset of the deployed one — it adds enforcement, doesn't change format. Existing roster (today's 6-node mesh.sealed) will load cleanly because MH-2 valid_until=1782759766 (June 29) is in the future and MH-1 signed_at=1780167766 is greater than any prior seen.
  • Mitigated: The .seen sidecar doesn't exist on any prime yet — first boot writes it, no rollback rejection.
  • Watch-item: If the prime aborts on boot, the old .bak is right there for instant restore. Each prime is independently rolled, so worst-case is one prime down 60 seconds while we restore.

Prerequisites (must be true before Phase B)

  • TEST-5 (on-VM smoke harness) exists and is green against a scratch VM — proves the binary works in a production-like environment without touching production.
  • Prime binary is built from a known-good commit hash (current nous HEAD d64beae is the TEST-2 green commit — could use that or wait for HEAD to absorb any pending fixes).
  • A 30-minute monitoring window scheduled (the soak after Ashburn).

My recommended Gap 2 sequencing

  1. Do TEST-5 first (Gap 1 work) — gives us the smoke harness we need for Phase A. Sequence is TEST-5 → Gap 2 Phase A → Phase B → Phase C. ~half a day end-to-end.
  2. OR skip TEST-5 and do Phase A by hand — provision a fresh GCE micro VM, install + smoke manually, then proceed. Faster (~2 hours) but doesn't leave a reusable harness behind.
  3. Either way: Phase B (Ashburn canary) waits for your explicit ~go — production touch.
