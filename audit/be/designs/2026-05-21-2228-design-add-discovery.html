Design — ADD Anchor + Registry

Make a-d-d discoverable to any AI or system that runs ~design, ~inspect, or any tilde — without guessing.

Trail: ~i identity + consolidation · ~i final scan

1. Subject + Intent

Subject: A two-part mechanism (a per-repo Anchor file plus a centralized Registry) that lets any AI or system reliably answer "Is this a-d-d? Where does all a-d-d for this org live?"

Intent: Joey's framing — "the 'a' in a-d-d is the HOW for the driver for the intent of the owner/operator". The HOW must be reachable. Today an AI running ~design or ~inspect has to infer what's a-d-d via file signatures and out-of-band knowledge. That's how V1, V5, and V13 verdicts went wrong — surface-level guesses. If a-d-d is the architectural ground truth, every agent should be able to find it the same way git finds .git — by walking up the tree to a marker, then following the marker to the canonical source of truth, then reading the registry that lists every other a-d-d-shaped artifact across the org. No inference. No drift between what one AI thinks is canonical and what another AI thinks. The owner/operator's intent travels with the architecture, and the architecture is one query away.

2. Discovery

Landscape today. 6 a-d-d-shaped repos exist across 4 orgs (see V18 ~i report). Identification works via file signatures (skill.json next to CLAUDE.md, hierarchical intent/purpose/skill paths, root GLOBAL.md+NOTICE pair). But these are conventions, not declarations. An AI reading a fresh repo cold can derive "this looks a-d-d-shaped" — it can't confirm "this IS the canonical / this is a scaffold / this is historical." No machine-readable assertion exists.

Existing shape. Kernel CLAUDE.md (post-M4 merge) declares kastil-systems/a-d-d/GLOBAL.md as the methodology source. That's one pointer at one file. It doesn't enumerate the rest of the footprint (the 5 other a-d-d-shaped repos), doesn't distinguish scaffolded copies from sources, doesn't track historical archives. The audit work just done filled this gap manually — but it'll drift again the moment a new a-d-d-shaped folder appears in a code repo and no one updates the kernel.

Joey's gut lean. Something declarative + indexable. Like .git for git, .npmrc for npm — a sentinel that any tool can find by walking parents, plus an org-level index of all the sentinels so a top-down sweep is possible. The owner/operator's intent should be the root, the registry should make the HOW discoverable, and any AI should land at "I know what's a-d-d and where it lives" within seconds of starting work.

What's missing in plain words.

  1. A per-repo declaration: this repo is a-d-d (and what role it plays).
  2. A per-org inventory: here are all the a-d-d declarations across the org footprint.
  3. A discovery trail: any AI starting work knows to look at X, which points to Y, which lists Z.

4. Shape

Two artifacts. One trail.

Artifact 1 — The Anchor

A small JSON file named .add.json at the root of every a-d-d-shaped repo. Schema:

{
  "schema": 1,
  "role": "canonical" | "scaffold" | "historical" | "teaching" | "code-tree-copy",
  "canonical": "kastil-systems/a-d-d",
  "intent": "<one-line statement of what this repo's a-d-d serves>",
  "owner": "<person or org responsible>",
  "upstream_commit": "<sha> or null",
  "notes": "<free text>"
}

One required field that does the heavy lifting: role. Tells any AI immediately what this repo IS in the ADD hierarchy.

  • canonical — the source of truth. Exactly ONE repo across all orgs has this. (Today: kastil-systems/a-d-d.)
  • scaffold — starter/template variant. Spawns new project a-d-d folders.
  • historical — archived design content preserved for reference. Never edited.
  • teaching — explanatory/docs surface. Newcomer-facing.
  • code-tree-copy — scaffolded into a product code repo, synced from canonical. Per the kernel rule, canonical wins on conflict.

Artifact 2 — The Registry

A single markdown file at kastil-systems/a-d-d/REGISTRY.md listing every known a-d-d repo across all orgs the operator manages. Schema (markdown table, human-readable, machine-greppable):

| Repo                              | Role        | Status     | Intent                                           | Notes              |
|-----------------------------------|-------------|------------|--------------------------------------------------|--------------------|
| kastil-systems/a-d-d              | canonical   | active     | kASTIL methodology + nous architecture           | THE source         |
| acmedevUS/acme-architecture       | scaffold    | active     | Acme business architecture (renamed from a-d-d)  | per cleanup C2     |
| ai-pantheon-internal/ADD          | historical  | archived   | Pre-merge Pantheon NOUS design                   | merged 2026-05-21  |
| joeycastilloUS/a-d-d-guidance     | teaching    | active     | Newcomer docs + PRIMER + CHANGELOG               | per cleanup C4     |
| Life-Water/a-d-d                  | scaffold    | empty      | LifeWater product architecture (not started)     | per cleanup C3     |

This file is the org-wide truth. gh api-fetchable. Generated initially by hand, kept current by either: a manual edit when a new a-d-d repo appears, OR a periodic script that runs gh search repos --topic=add and reconciles.

The Discovery Trail (how an AI reaches both artifacts)

  1. Entry point. Any tilde command starts by reading the kernel CLAUDE.md (already happens). Kernel CLAUDE.md gets one new section: ## Canonical ADD declaring the canonical repo path and the registry URL.
  2. From kernel to canonical. Agent fetches kastil-systems/a-d-d/REGISTRY.md via gh api repos/kastil-systems/a-d-d/contents/REGISTRY.md --jq .content | base64 -d. One API call. Cached for the session.
  3. From canonical to local repos. If the agent is operating IN a repo, walks up the tree looking for .add.json. Found = this is an a-d-d-shaped repo, role declared. Not found = this is a regular project/code repo, treat as such.
  4. Cross-org discovery. When an agent needs to know "where else is this concept defined?", it consults the in-memory registry from step 2. No new searches.

Out of scope

  • Not a runtime daemon. No live service. Static files only.
  • Not enforcement. Doesn't prevent drift; surfaces it. If someone creates a misnamed a-d-d-shaped repo without an Anchor, it stays invisible to the system until the Registry is updated.
  • Not auto-migration. Scaffold and code-tree-copy roles describe state; they don't sync themselves. Sync is the existing manual go.bat + "canonical wins" rule from the kernel.
  • Not a replacement for skill.json. The Anchor is repo-level; skill.json is file-level. Both stay.

6. Escape Routes

Load-bearing choiceWhat if it's wrong?How to back outReversible?
.add.json at repo root (Anchor format) JSON parsing turns out brittle in some edge case (BOM, encoding, etc.); OR a different format becomes more useful. Schema version 1 means we can ship schema: 2 with a new format and dual-read for a period. Tools fall back gracefully on unknown schema. Yes — versioning baked in.
Registry inside canonical repo Canonical repo grows too large (currently 467KB, no near-term concern); OR registry needs faster updates than the methodology repo supports. Spin kastil-systems/a-d-d-registry as a separate repo. Old REGISTRY.md becomes a one-line redirect. Anchors don't change (they point at canonical = repo slug, not at registry). Yes — easy split.
GitHub topic add GitHub deprecates topics, or they get used by an unrelated community for something else. Rename to kastil-add or add-architecture. Update all repos' topics + the auto-populate script. Naming convention *-a-d-d still works as the fallback identifier. Yes — rename is bulk-scriptable.
Hybrid auto-populate The auto-populate script breaks or produces noise (false positives). Disable the script; registry stays as the last good manual version. No agent depends on the script — they depend on the file. Yes — script is additive, never load-bearing.
Kernel CLAUDE.md as discovery entry point Kernel doc grows so long that the canonical pointer gets lost in noise. Split out a be/docs/discovery.md referenced from kernel. Agents read both. Mildly painful because every agent's startup sequence has to learn the new path. Yes, with friction.

No load-bearing choice is unreversible. Every component is a static file in git. Worst case: rip out, redo. The cost of being wrong is one short cleanup, not a system rewrite.

Dependencies and their character:

Verdict. Zero new infrastructure. The whole design is static files in git plus existing gh tooling. Pure metal. Survives loss of any external service (GitHub topics, the auto-populate script, registry-as-separate-repo) by gracefully degrading to "just read the canonical Anchor + walk REGISTRY.md by hand."

8. First Move

The smallest concrete step that proves this design is real:

  1. Write .add.json at kastil-systems/a-d-d/ root, declaring it canonical: 
    {
  "schema": 1,
  "role": "canonical",
  "canonical": "kastil-systems/a-d-d",
  "intent": "kASTIL SYSTEMS Architecture-Driven Development — methodology + NOUS architecture + target hardware specs",
  "owner": "Joseph M. Castillo",
  "upstream_commit": null,
  "notes": "Source of truth for ADD across all orgs. See REGISTRY.md for footprint."
}
  2. Write REGISTRY.md at kastil-systems/a-d-d/ root, seeded with the 6 known a-d-d repos discovered in the audit (V18 ~i report).
  3. Tag kastil-systems/a-d-d with the GitHub topic add via gh repo edit kastil-systems/a-d-d --add-topic add.
  4. Add a 4-line ## Canonical ADD section to the kernel CLAUDE.md (in metal-console) declaring: canonical repo = kastil-systems/a-d-d, registry path = REGISTRY.md, anchor file convention = .add.json.
  5. Verify: from a fresh shell, run gh search repos --topic=add --json fullName,description. kastil-systems/a-d-d appears. Then gh api repos/kastil-systems/a-d-d/contents/.add.json --jq '.content' | base64 -d | jq. Should print the Anchor as parsed JSON. If both work: the design is real.

Total work: ~30 minutes. Three new files (.add.json, REGISTRY.md, kernel patch). One gh repo edit. One verification fetch. After this First Move lands, scaling to the other 5 a-d-d repos is mechanical — each gets its own .add.json with the appropriate role, then gets registered.
