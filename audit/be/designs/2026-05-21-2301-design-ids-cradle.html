Design — IDS IO Bus

The pure-C library every metal-* tool links against. Reads from and writes to the IDS. Hosts the bootstrap protocol, scope manager, and pluggable renderers. Multiple tools and multiple AIs attach to one shared backbone. Triples in, AI-convention text out.

Renames applied 2026-05-21 Two corrections in sequence. (1) Original draft called this "IDS Cradle." Joey: "IO layer not cradle layer. I-O Input/Output." Layer named for its function, not the newborn-mother metaphor. (2) Then refactored from io to io_bus — avoids platform header collision (<io.h> exists on Windows and Linux) AND captures the bus topology (multi-tool, multi-AI, one backbone). Final: library libio_bus.a, header io_bus.h, all symbols io_bus_*, types IOBus*. The newborn↔mother bootstrap is still the bootstrap mechanism — it's a function on the IO bus, not the bus's name.

1. Subject + Intent

Subject: A pure-C shared library (libio_bus.a, header io_bus.h) that every metal-* tool links against. The IO bus between the tools and the IDS. Wraps the existing IDS triple store (nous/memory/skill/store). Implements the newborn↔mother bootstrap protocol, scope management with user alerts, and a pluggable renderer slot for emitting per-AI-convention output (Claude Code + Perplexity at v1).

Intent: Joey's framing: "this thing needs to work for all of my tools." The tools (metal, metal-console, metal-desktop, and the consoles that don't exist yet) all need the same behavior — read the Pantheon IDS, feed a fresh AI session the architecture-and-intent it needs, alert the operator when scope shifts, render whatever the AI wants to consume. Today, none of this exists; the consoles invent their own conversations from raw files. The IO bus is the substrate that makes every console transparent to the IDS underneath. The owner/operator's intent lives in IDS once; every tool reads it the same way; the AI gets seeded the same way; the operator sees the same load/unload alerts no matter which surface they're on. One library, every surface, zero drift.

2. Discovery

Landscape today. Each metal-* tool reads files (CLAUDE.md, board.md, plans/*) and talks to Claude Code directly. No shared substrate. No common bootstrap. metal-console (HTML/operator surface) and metal-desktop (C/cockpit on localhost:5050) already diverge in how they hand context to Claude. As more consoles appear, this fans out — and as Perplexity joins as a second AI target, the divergence doubles.

What the nous spec already gives us.

  • nous/memory/skill/store — the triple store. Mmap, FNV-1a, 24-byte Fact, 16-byte DictEntry, WAL crash safety, 3 default S/P/O indexes. Documented in detail today (M6 commit e47fe79).
  • nous/memory/skill/kts — the volatile working-set pattern. "If it dies, it reboots from ROM." The mother↔newborn dynamic in code form already.
  • nous/intelligence/skill/acquire — "Layer 1 — The Seed: Six engines the system is born knowing." Hardcoded trust + bootstrap pattern, named.
  • nous/memory/skill/scope — iterator over filtered facts. Already the right primitive for "give me triples matching this focus."
  • The whole nous/ tree treats triples as foundational. The IO bus doesn't invent — it composes.

Joey's gut lean. Pure C. Library, not service. Every tool links it. Works on Windows + Linux + Mac because the nous mmap skill already handles platform divergence. Single header, single archive, dead-simple link line. No framework, no daemon, no IPC. Bare metal.

What's missing in plain words.

  1. A tool-facing API: simple C functions any console calls (io_bus_open, io_bus_query, io_bus_scope, io_bus_render, io_bus_close).
  2. A bootstrap protocol: the newborn↔mother handshake — newborn declares what it has, mother answers with the diff for current scope.
  3. A scope manager: tracks working-set membership; alerts the operator on load/unload.
  4. A renderer slot: function-pointer table of per-AI-convention emitters. v1 ships Claude Code + Perplexity; future renderers register at link time.

Winner: IO Bus. The middle layer is a bus — multiple tools attach, multiple AIs attach via renderers, data flows both ways, one IDS at the backbone. It IS a bus topology, not metaphor. Public API — io_bus.h

// ───── Lifecycle ─────
typedef struct IOBus IOBus;  // opaque
int  io_bus_open(IOBus **out, const char *store_path);
int  io_bus_close(IOBus *c);
int  io_bus_flush(IOBus *c);

// ───── Bootstrap (newborn ↔ mother) ─────
typedef struct {
    const char  *session_id;        // unique per-session
    const char  *scope_name;        // what the newborn is focused on
    uint64_t    *known_fact_hashes; // what the newborn already has (or NULL if empty)
    size_t       known_count;
} IOBusBootstrap;

typedef struct {
    uint64_t *new_facts;     // triples to load into the newborn
    size_t    new_count;
    uint64_t *evicted_facts; // triples no longer in scope (newborn should forget)
    size_t    evicted_count;
} IOBusSeed;

int  io_bus_bootstrap(IOBus *c, const IOBusBootstrap *req, IOBusSeed *out);
int  io_bus_seed_free(IOBusSeed *seed);

// ───── Scope management ─────
typedef struct IOBusScope IOBusScope;  // opaque
int  io_bus_scope_open(IOBus *c, const char *name, IOBusScope **out);
int  io_bus_scope_add_focus(IOBusScope *s, const char *predicate, const char *object);
int  io_bus_scope_query(IOBusScope *s, uint64_t **fact_hashes_out, size_t *count_out);
int  io_bus_scope_close(IOBusScope *s);

// ───── Alerts ─────
typedef void (*IOBusAlertFn)(const char *event, const char *detail, void *user);
int  io_bus_set_alert(IOBus *c, IOBusAlertFn fn, void *user);
// Events: "load", "unload", "scope_shift", "bootstrap_complete"

// ───── Renderers ─────
typedef struct {
    const char *name;                   // "claude-code", "perplexity"
    int (*render)(const uint64_t *facts, size_t count,
                  IOBus *c, char *out, size_t out_size);
} IOBusRenderer;
int  io_bus_render_register(IOBus *c, const IOBusRenderer *r);
int  io_bus_render(IOBus *c, const char *renderer_name,
                   const uint64_t *facts, size_t count,
                   char *out, size_t out_size);

// ───── Direct triple R/W (used by consoles' write path) ─────
int  io_bus_assert(IOBus *c, const char *s, const char *p, const char *o);
int  io_bus_retract(IOBus *c, const char *s, const char *p, const char *o);
int  io_bus_query(IOBus *c, const char *s, const char *p, const char *o,
                  uint64_t **fact_hashes_out, size_t *count_out);

Bootstrap protocol — the Socratic handshake

Joey's spec: "the newborn brain asks its mother if it is completely empty of information. then the mother brain answers based on the same information the newborn brain has received." The IO bus implements this in three calls:

  1. Newborn declares. Tool calls io_bus_bootstrap(c, &req, &seed) with known_fact_hashes set to whatever facts the AI session already has (empty if truly newborn).
  2. Mother computes the diff. The IO bus queries the IDS for all triples matching the requested scope_name. Subtracts the newborn's already-known set. Returns: new_facts (load these) + evicted_facts (newborn was holding these but they're out of scope now; forget them).
  3. IO bus alerts. Fires the registered IOBusAlertFn with "bootstrap_complete" and a detail string ("loaded N, evicted M, scope=X"). The operator's console surfaces it.

No data dump. The mother only hands over what's missing for the current focus. Same pattern works for re-bootstrap on scope shift — the newborn isn't reborn, just rehoused.

Scope shifts

When the operator's intent changes (e.g. from "auditing orgs" to "designing IO bus"), the tool calls io_bus_scope_open with the new scope, then io_bus_bootstrap again with the newborn's current fact set. Mother computes the new diff. Alert fires: "scope_shift" with detail "prev=X loaded=Y new=Z evicted=W". Operator sees the brain just changed shape.

Renderers

Each renderer is a struct with a name and a function pointer. v1 ships with two compiled-in renderers:

  • claude-code — emits CLAUDE.md-flavored markdown. Headers, fenced code blocks, the trade-secret notice, the kASTIL conventions. Matches what Claude Code expects to read.
  • perplexity — emits a tighter prose form. Perplexity reads system prompts and user messages, not file conventions. Output: structured prose with citation markers pointing back at triple IDs.

Future renderers register at link time via io_bus_render_register. Adding GPT/Gemini/RDF/JSON-LD is one function — no other code changes.

Out of scope

  • Not a daemon. Library, not service. Process lifecycle tied to caller.
  • Not network-aware. Single-host. Cross-host sync happens at nous/comms/skill/sync, not here.
  • Not a UI. Consoles render UI. The IO bus just emits text per renderer.
  • Not GC. Caller frees seeds (io_bus_seed_free). No background sweep.
  • Not multi-tenant. One IDS per IOBus instance. Multi-org reads = multiple IOBus instances.

5. Choices + Consequences

ForkOption AOption B2nd-order consequence3rd-order consequencePick
API style Opaque handles (IOBus*, IOBusScope*) Direct struct exposure Opaque: ABI-stable across versions; struct changes don't break callers. Direct: faster (no indirection) but ABI-fragile. Opaque keeps every tool's binary working when the IO bus evolves. Direct forces recompile-all on any internal change. With many tools (metal, metal-console, metal-desktop, more), recompile-all becomes punishing. A. Opaque. ABI stability outweighs the tiny indirection cost.
Renderer plug-in mechanism Function pointer table, compiled-in at link time Dynamic loading via dlopen / LoadLibrary Compiled-in: zero runtime dep, every tool ships its own renderer set. Dynamic: smaller binary, but requires a plug-in directory + DLL search path on Windows. Compiled-in is pure metal (no dynamic linker). Dynamic adds a deployment surface every tool has to manage. Per kernel rule "No dynamic linking. Everything compiles statically into one binary." A. Compiled-in. Kernel rule says so. Adding a renderer means recompile + relink, but that's already mechanical.
Scope state location In-IDS triples (scope membership is itself triples) Separate in-memory side table In-IDS: scope is queryable like any other knowledge — "what's in the operator's working set right now?" answered by an IDS query. Side table: faster ops but invisible to anyone reading the IDS. In-IDS means the audit trail covers scope shifts automatically (temporal provenance triples land naturally). Side table loses the audit. Eat-own-cooking principle says scope is part of what the brain knows. A. In-IDS. Self-describing system stays self-describing.
Alert transport Callback function pointer (IOBusAlertFn) Write to stderr / log file directly Callback: each tool decides how to surface (GUI toast, terminal print, log entry). Direct write: simple but tools can't filter/redirect. Different consoles will display alerts differently (metal-console = HTML overlay, metal-desktop = browser cockpit, future = voice). Callback lets each tool own its surface. A. Callback. Tools own presentation.
Bootstrap detection Caller declares with known_fact_hashes (the IO bus trusts the caller) IO bus computes from session_id history Caller-declared: simple, no bus-side state. Bus-computed: more magic, more failure modes (session-id tracking, mid-session crashes). Caller-declared keeps the IO bus stateless between calls. Sessions can crash, restart, fork — caller manages, bus reacts. Recovery is the caller's domain. A. Caller declares. The IO bus stays stateless, deterministic.
How tools find their _store Caller passes store_path to io_bus_open IO bus walks parent dirs looking for .a-d-d/_store (like git finds .git) Caller-passed: explicit, no surprises. Auto-walked: ergonomic — tools don't need to know the path, just io_bus_open(NULL) from anywhere in a repo. Walked: matches git's affordance, less for each tool to remember. Caller-passed: works in test harnesses and embedded contexts where there's no filesystem tree to walk. Both. If store_path == NULL, walk parent dirs. Else use the explicit path. One function, two ergonomics.

6. Escape Routes

Load-bearing choiceWhat if it's wrong?How to back outReversible?
Opaque handle API Performance turns out to matter (1000s of ops/sec become a bottleneck) Add a parallel direct-access struct API (io_bus_unsafe_*) without removing the opaque one. Tools opt in to the fast path; ABI for opaque stays stable. Yes — additive.
Compiled-in renderers The renderer count grows past a comfortable static-link size (unlikely — each renderer is small) Switch select renderers to dlopen-loaded plug-ins. Public API doesn't change; only the registration mechanism does. Yes — internal change.
In-IDS scope state Scope ops thrash the IDS with churn (insert + retract cycle on every scope shift) Add a side-table cache; flush dirty scope triples to IDS on commit boundaries. Behavior preserved; throughput improved. Yes — performance fix.
Caller-declared known-facts Tools forget to track what they've handed the AI; bootstrap returns wrong diffs Add an optional bus-side session log (file-backed). Tools opt in. Default stays stateless. Yes — opt-in.
Renderer table at link time Different tools want very different renderer sets (e.g. metal-desktop wants Claude only, metal-console wants all of them) Already works — each tool picks which renderer object files to link. io_bus_render_register means "available," not "always loaded." Yes — already addressed.
Path-walk auto-discovery Path-walk collides with a non-IOBus .a-d-d/ folder somewhere (false positive) Require a sentinel file inside _store/ (e.g. FORMAT.PTS with magic 0x50545300) before treating the directory as valid. Mismatch = walk past it. Yes — guard added at open.
Pure C constraint A renderer needs something C can't trivially do (HTTP fetch, JSON parsing, image bytes) Renderers may shell out to small helper binaries via execve if needed. The IO bus itself stays pure C; helpers are explicit subprocess calls, not library deps. Yes, with discipline — helpers must be tiny + explicit.

No load-bearing choice is unreversible. Every component is a static C file plus the existing nous/memory infrastructure. Worst case: rip out one piece, swap in the alternative.

Verdict. The whole IO bus is C + libc + the nous memory skills that already exist on paper. Zero new dependencies. Pure metal. Survives loss of any non-load-bearing component by graceful degradation (read-only mode without journal, single-renderer mode without others, explicit-path mode without auto-discovery).

8. First Move

The smallest concrete step that proves the design is real:

  1. Write io_bus.h in kastil-systems/a-d-d/_store/io_bus/. Just the public API declarations from §4. Compiles clean with gcc -std=c11 -Wall -Wextra -Wpedantic -c -x c /dev/null -include io_bus.h. No bodies yet.
  2. Write io_bus_open.c + io_bus_close.c as minimal stubs that just open and close the underlying triple store (forwarding to triple_store_open / triple_store_close). No bootstrap logic, no renderers — just lifecycle.
  3. Write a 30-line C program (test/hello_io_bus.c) that calls io_bus_open(&c, "_store"), then io_bus_close(c). Compiles, links, runs, returns 0.
  4. Build it: gcc -std=c11 -I . io_bus_open.c io_bus_close.c test/hello_io_bus.c -o hello_io_bus. Run. Should print nothing, return 0.
  5. Write io_bus_assert.c — wraps triple_store_insert. Now hello_io_bus.c can insert one fact. Read it back via io_bus_query (also wraps existing). Done — full round-trip on three triples.

Total work for First Move: maybe 200 lines of C. One header, four implementation files, one test. Compiles, runs, demonstrates the API shape. After this lands, scaling to bootstrap + scope + renderers is mechanical — each is its own implementation file added to the archive build.
