Design — IDS IO Bus v2

Multi-tool substrate. Library + CLI hybrid. File-mirror migration ramp. Built on the C SDK that's already vendored at src/sdk/nous/. Composes, not duplicates.

Supersedes v1 (2026-05-21-2301-design-ids-cradle.html) called this Cradle → IO Layer → IO Bus and proposed a pure new library built from scratch. v2 corrects course after the ~i of C:\metal (2026-05-22-0049-i-metal-deep-scan.html) found that (a) the C SDK substrate is already vendored at src/sdk/nous/memory/skill/*, (b) src/tlt/ is the closest working analog (already links the memory skills + has a wire protocol), and (c) Claude Code can only call CLIs via Bash — a library-only approach leaves it stranded. v2 picks Path E (library + CLI hybrid) as shape and Path F (file-mirror migration) as on-ramp. v1's API design is preserved — just the packaging changes.
Trail: ~i C:\metal deep scan · ~d v1 (superseded) · ~i triples reframe

1. Subject + Intent

Subject: A pure-C library (libio_bus.a, header io_bus.h) AND a thin CLI wrapper (io_bus binary) that share one engine. Every metal-* tool that can link C does so; Claude Code and Python tools shell out to the CLI. Both surfaces compose the existing src/sdk/nous/memory/skill/* code already vendored in C:\metal — IO Bus doesn't ship the IDS substrate, it ships the multi-tool surface around it. A file-mirror layer ramps existing state files (board.md, telemetry.json, prompts.jsonl, h-cache.json) onto triples without breaking any of the 23 current tools.

Intent: The kernel doc and the tilde specs are already a triple-shaped knowledge graph wearing markdown. The 23 tools in be/tools/ read JSON state files. src/tlt/ is a pure-C binary that already uses the IDS substrate but doesn't expose it. The IO Bus closes the gap: makes the same engine reachable from C tools (link) and from non-C tools or AI conversation (CLI). It does this WITHOUT a daemon, WITHOUT shipping new substrate code (compose existing), and WITHOUT breaking the 6162-line render-board-view.py until that tool chooses to migrate. The owner/operator's intent travels as triples; every surface — every metal-* tool, every AI convention — reads the same source through the surface it prefers.

2. Discovery

Landscape today (from the metal deep scan).

  • Layer 1 — Hooks: settings.json declares UserPromptSubmit → capture_prompt.ps1 and SessionStart → commit_leftover_prompts.ps1. Every prompt is already captured to be/prompts.jsonl.
  • Layer 2 — Kernel: CLAUDE.md (758 L). The Be system declared.
  • Layer 3 — Specs: commands.md (1764 L, 28 tilde specs) + conventions.md (1067 L, 8 runtime protocols).
  • Layer 4 — State (today: JSON + markdown): 11 state files + 7 artifact folders.
  • Layer 5 — C substrate (already vendored): src/sdk/nous/{action,comms,memory,security,user}/skill/* with the full memory store, journal, scope, index, hash, mmap, pack, plus security (crypt with TLS 1.3, vault, seal, auth), plus user/interface/{console,desktop}/. Plus src/tlt/ as a working pure-C binary using these.

Joey's gut lean. Pure C. Library, not service. Every tool links it where possible; everything else shells out. Works for all his tools. No daemons, no new infrastructure, no duplicate substrate. "Many ways to skin this cat" was the framing — the deep scan narrowed to one.

What v1 got right. The API shape (io_bus_open, io_bus_bootstrap, scope mgr, alerts, renderer table). The newborn↔mother protocol. The three-layer architecture. The naming progression (Cradle → IO Layer → IO Bus). The escape routes.

What v1 got wrong.

  1. Treated IO Bus as if it had to ship the IDS substrate. It doesn't — src/sdk/nous/memory/skill/* is already vendored. IO Bus composes; doesn't duplicate.
  2. Ignored src/tlt/ entirely. tlt already proves the pattern (C binary, links memory skills, runs a protocol). It's not the IO Bus, but it's the closest working precedent.
  3. Library-only. Claude Code can't link C. Half the tools in be/tools/ are Python. The CLI surface isn't optional — it's the only way most consumers can reach the IO Bus.
  4. "First Move" was 200 lines of header stubs and a hello world. After the deep scan, the right First Move is migrating ONE existing state file end-to-end as the proof. Smaller AND more valuable.

Public C API — io_bus.h (unchanged from v1)

v1's API design holds. IOBus* opaque handle. io_bus_open/close/flush. io_bus_bootstrap with IOBusBootstrap + IOBusSeed. io_bus_scope_*. io_bus_set_alert with IOBusAlertFn. io_bus_render_register + io_bus_render. io_bus_assert/retract/query. See v1 §4 for full signatures.

CLI verbs — io_bus <verb> [args]

The CLI wraps the same engine. Verbs mirror the C functions. Stdout = result; stderr = alerts + errors. JSON for structured output (so Python/PowerShell parse easily).

# Lifecycle (no-op for CLI; each invocation opens-closes)
io_bus init [path]                          # create _store/ if missing
io_bus status                                # show open file count, last write, etc.

# Direct triple ops
io_bus assert  <subject> <predicate> <object>
io_bus retract <subject> <predicate> <object>
io_bus query   [<s>|*] [<p>|*] [<o>|*]   # 0 / * = wildcard

# Bootstrap (newborn ↔ mother)
io_bus bootstrap --session=<id> --scope=<name> --known=<hash-list-file>
# Returns JSON: { new_facts: [...], evicted_facts: [...] }

# Scope
io_bus scope create <name> --focus <pred>=<obj> [--focus ...]
io_bus scope query  <name>
io_bus scope close  <name>

# Render
io_bus render <renderer-name> --facts=<hash-list-file> [--out=<path>]
# renderer-name: claude-code | perplexity | json | markdown | ...

# File-mirror (Path F — see below)
io_bus mirror enable  <file-path> --as=<scope-name>
io_bus mirror sync    <file-path>            # one-shot sync
io_bus mirror watch                           # daemonless poll (like watch.py)
io_bus mirror status                          # which files are mirrored

Path F — file-mirror sync (the migration ramp)

The 11 state files in be/ don't get rewritten. Instead, io_bus mirror watches them and reflects each change into triples. Reverse direction too: triple writes can regenerate the state file. This is the bidirectional bridge during migration.

be/board.md            ←→  scope:board       triples
be/board-status.json   ←→  scope:active      triples
be/telemetry.json      ←→  scope:telemetry   triples  ← MIGRATE FIRST
be/prompts.jsonl       ←→  scope:prompts     triples
be/h-cache.json        ←→  scope:h-help      triples (derived; one-way)
be/tags.json           ←→  scope:tags        triples
be/users.json          ←→  scope:users       triples
be/inbox.json          ←→  scope:inbox       triples
be/config.json         ←→  scope:config      triples (rarely changes)
be/board-recency.json  ←→  scope:recency     triples (derived; one-way)

Order of migration matters: telemetry.json is the safest first target (append-only, well-schemafied). Once it's mirrored, render-board-view.py can optionally read from triples for its telemetry section without losing JSON compatibility. Each subsequent file follows the same pattern.

The bootstrap protocol (unchanged from v1)

Newborn declares → mother computes diff → alert fires. Implementation: io_bus_bootstrap() in the library; io_bus bootstrap in the CLI. Both call the same engine.

Renderers (unchanged from v1, plus two new for migration)

  • claude-code — emits CLAUDE.md-flavored markdown
  • perplexity — emits Perplexity prose with triple-ID citations
  • json — emits raw JSON (for tools that consume programmatically)
  • markdown — generic markdown (for board.md regeneration during Path F migration)

Out of scope (sharper than v1)

  • Not a daemon. No long-running process. Library + CLI only.
  • Not a tlt refactor. tlt stays standalone. IO Bus links the same nous skills tlt does; both binaries coexist.
  • Not a replacement for any of the 23 tools. Tools migrate to IO Bus when convenient. Nothing forced.
  • Not a network protocol. Single-host. Cross-host sync = future via nous/comms/skill/sync.
  • Not a UI. Consoles render UI. IO Bus emits text per renderer.
  • Not a 6162-line render-board-view.py rewrite (yet). Migration is opt-in per-tool.

5. Choices + Consequences (the path picks)

ForkPickedWhy this, not the others
Packaging shape Path E — Library + CLI hybrid v1 was library-only. After deep scan: Claude Code can't link C, half the tools are Python. Path C (CLI only) was rejected because C tools deserve the in-process fast path. Path E gives both with one engine — minimal extra surface (CLI is a thin main() over the library). Path B (extend tlt) rejected to keep names clean: tlt is The Last Test, not the IO bus.
IDS substrate Compose existing src/sdk/nous/ v1 implied IO Bus would carry the IDS substrate. Wrong — it's already there. libio_bus.a #includes from src/sdk/nous/memory/skill/* via the same Makefile pattern tlt uses (see Makefile L52-59). Zero new C code in the substrate layer.
Migration approach Path F — File-mirror sync The 23 tools work today. render-board-view.py is 6162 lines that parse markdown/JSON. Path F lets triples become source-of-truth WITHOUT rewriting any of that. Mirror enables bidirectional sync; tools migrate when convenient. Big-bang migration was rejected — too disruptive to working infrastructure.
First migration target be/telemetry.json Append-only. Well-schemafied (8 fields per entry). Already mirrored to be/otel.jsonl by otel_collector.py — adding a triple mirror is the same shape of work. Lowest risk: no tool depends on telemetry.json being editable. Worst case rollback: turn off mirror, JSON keeps working.
Scope state location In-IDS triples v1 picked this. v2 keeps it. Self-describing system stays self-describing.
Alert transport Callback (library) + stderr (CLI) C tools register IOBusAlertFn; CLI consumers parse stderr. Same engine fires both. Each tool owns presentation.
Store path discovery Both — explicit OR auto-walk parents Same as v1. Auto-walk for ergonomics (like .git); explicit for tests + embedded contexts.

The rejected paths (with one-line reasons)

A — pure new library only

Leaves Claude Code stranded (can't link C). Path E supersedes.

B — extend tlt

Name overload (tlt = The Last Test, not the bus). tlt stays its own binary, IO Bus links the same nous skills tlt uses.

C — CLI only

Process-launch overhead per call hurts C tools. Library half of Path E is for them.

D — daemon + IPC

Kernel rule prefers no daemons. Adds lifecycle + crash recovery + attack surface for no clear gain.

F-alone — file-mirror without library/CLI

The mirror has nothing to mirror to without the engine. F is the ON-RAMP for E, not a path alone.

6. Escape Routes

Load-bearing choiceWhat if it's wrong?How to back outReversible?
Library + CLI hybrid Library or CLI surface turns out unused Drop the unused half. The library is the real engine; the CLI is a thin main() over it. Either can go without the other. Yes.
Compose src/sdk/nous/ via #include The vendored skills evolve in incompatible ways and break IO Bus builds Vendor a frozen copy of the skill source into _store/io_bus/sdk-snapshot/. Sync forward deliberately. Adds maintenance but isolates. Yes — versioned snapshot fallback.
File-mirror with telemetry.json first The first migration corrupts telemetry.json or breaks render-board-view.py Mirror is bidirectional and append-only on the JSON side. Worst case: disable the mirror (io_bus mirror disable), tool keeps working on JSON. The triple store side is independent and can be reset. Yes — both sides survive.
Pure C constraint A renderer needs HTTP/JSON/etc. that C can't trivially do Renderers may shell out to small helpers via execve. The CLI itself can also help — a Python script can call io_bus query and post-process freely. Yes — helpers explicit.
CLI as JSON-over-stdio JSON overhead matters for high-throughput CLI calls Add a binary protocol mode (io_bus --binary). Default stays JSON. C tools mostly use the library anyway; CLI binary mode is for niche speed. Yes — opt-in.
Path F before full triple-only Dual-source-of-truth drift between JSON and triples Per-file mirror has a "canonical side" config — start with JSON canonical, flip to triple canonical when ready. Drift detection is a single query (io_bus mirror diff <file>). Yes — flag controls direction.

No load-bearing choice is unreversible. Every path-pick has a graceful fallback.

8. First Move (revised — smaller AND more valuable)

v1's First Move: write io_bus.h + open/close stubs + hello world (~200 lines, proves the API shape compiles).

v2's First Move: migrate ONE state file end-to-end. Picks be/telemetry.json because it's append-only, well-schemafied, and already mirrored to otel.jsonl (so adding a third mirror is familiar shape).

The 8 steps

  1. Create kastil-systems/a-d-d/_store/io_bus/. Three files: io_bus.h (signatures), io_bus.c (single-file impl wrapping the nous skills via #include), Makefile (copy tlt's pattern). Skill.json for the a-d-d marker.
  2. Implement the minimal CLI verbs needed for telemetry mirror: io_bus init, io_bus assert, io_bus query, io_bus mirror enable, io_bus mirror sync.
  3. Build: make -C kastil-systems/a-d-d/_store/io_bus/. Produces libio_bus.a + io_bus binary.
  4. Test the engine alone: io_bus init /tmp/test-store; io_bus assert "test" "is" "working"; io_bus query "test" "*" "*" — get the triple back. Proves the C path end-to-end.
  5. Mirror enable: io_bus mirror enable C:/metal/be/telemetry.json --as=scope:telemetry. The CLI parses telemetry.json's known schema (entry array of 8-field objects) and inserts triples for each entry.
  6. Verify mirror: count triples vs JSON entries. Same count = pass. Append a new entry to telemetry.json directly (as a tilde would today); run io_bus mirror sync; verify the new triples appear.
  7. Reverse direction: insert a triple via io_bus assert; verify it appears in telemetry.json after io_bus mirror sync (or wait for io_bus mirror watch).
  8. Document + commit: add the IO Bus skill folder to ks-add canonical. Commit. Push.

Total work for First Move: probably 400-500 lines of C (single-file impl + CLI main() + minimal JSON parser for telemetry schema) + Makefile (copy from tlt). Maybe an evening of work, not a weekend.

What the First Move proves:

  • The library + CLI hybrid pattern works end-to-end.
  • The vendored src/sdk/nous/ composes correctly into a new binary.
  • The file-mirror migration ramp is viable (one file works → the rest are mechanical).
  • The trade-secret + signing conventions hold for the new tree.
