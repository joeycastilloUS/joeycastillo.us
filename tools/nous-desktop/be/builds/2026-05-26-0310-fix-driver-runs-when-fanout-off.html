~f: driver runs when fan-out is OFF (master toggle bug)

Joey's report: "off button for AI still uses all AI to handle my chat requests. instead I would think the driver or engine would deal with it when I turn off the AI sections and all providers." Root-caused, fixed, verified. Driver now always answers; fan-out tiles obey the off button.

The Bug

Symptom: when the master "AI OFF" toggle is clicked (or every per-tile toggle is OFF), Joey expected the DRIVER (operator-picked Claude-Opus-4-7) to keep answering — only the fan-out tiles should go dark. Instead the THINKING column also died with "driver 'claude-opus' has no key or is disabled" while every fan-out tile was correctly muted. Net: no AI responded.

Root cause: the master toggle in loop.js (toggleAllProviders()) sends set_disabled for every provider, including the one selected as driver. handle_task's driver-lookup at serve.c:1834-1841 then refused to use it:

// BEFORE
for (int i = 0; i < g_provider_count; i++) {
    if (strcmp(g_providers[i].name, g_active_driver) != 0) continue;
    if (g_providers[i].disabled || !g_providers[i].api_key[0]) {
        // emits error -- THINKING column dies
        break;
    }
    driver_idx = i; break;
}

The disabled check applied universally — including to the operator's explicit dropdown choice. That's wrong: the dropdown is supposed to be the authoritative override.

The Fix

The operator-picked driver bypasses the disabled flag. Only the missing-key check still refuses (a provider with no key literally cannot run). When the driver is "on despite master-off", emit an honest trace line so the operator sees what's happening.

// AFTER (serve.c handle_task, around line 1834)
for (int i = 0; i < g_provider_count; i++) {
    if (strcmp(g_providers[i].name, g_active_driver) != 0) continue;
    if (!g_providers[i].api_key[0]) {
        // refuses only when there's no key
        send_event(...,"driver '%s' has no key");
        break;
    }
    if (g_providers[i].disabled) {
        // honest trace: driver runs anyway because operator picked it
        send_event(..., "search_step", "sense",
            "driver '%s' runs despite master-off (operator pick wins)");
    }
    driver_idx = i;
    break;
}

NOUS-decide path (when driver is "Let NOUS decide") still respects the disabled flag — there's no operator override to honor, so the first-available-non-disabled selection is correct.

What Joey will see in the cockpit

  1. Click master AI OFF button (top-right of DATA ACCESS column). Every fan-out tile goes dark.
  2. The DRIVER badge in the THINKING header still shows claude-opus-4-7 (green).
  3. Type a query. The trace fires:
    • SENSE: "driver 'claude-opus' runs despite master-off (operator pick wins)" — honest disclosure

    • REQUEST → RESPONSE → DONE — Claude answers normally
  4. The fan-out tiles (DATA ACCESS column) stay dark — they're properly muted.

To switch the driver while master is OFF: use the DRIVER dropdown as usual. The dropdown selection always wins over disabled state.
