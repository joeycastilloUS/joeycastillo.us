~f: claude dark fix — Windows env staleness + permanent launcher

All 3 Claude tiles were dark in the cockpit despite the API key being installed via setx and live-probing OK at the Anthropic API. Root cause: Windows env-block staleness. Fix: explicit env injection at every launch via a new launcher script + bump openai-responses token budget so reasoners actually produce visible output.

Root Cause

Windows setx writes to the registry but does not propagate to processes that started before the setx call. This includes the PowerShell process that Claude Code spawned to launch the server. The PowerShell got a stale copy of the env block at its own start; setx updated the registry but not the live block; child processes (including nous-desktop.exe) inherited the stale block via CreateProcess.

Verified with a one-line probe in a fresh PowerShell process:

$env:ANTHROPIC_API_KEY  →  $null   (process scope, stale block)
[Environment]::GetEnvironmentVariable('ANTHROPIC_API_KEY','User') → 108-char key   (registry, fresh)

So getenv("ANTHROPIC_API_KEY") inside nous-desktop.exe returned NULL. In load_providers():

const char *key = resolve_key(NULL, d->key_env, d->builtin_key);
if (key) strncpy(p->api_key, key, sizeof(p->api_key) - 1);
/* key was NULL → p->api_key stayed empty → fan-out skipped this provider */

The fan-out filter at serve.c:1740 is:

if (g_providers[i].disabled || !g_providers[i].api_key[0]) continue;

Empty key → provider silently dropped. Cockpit tiles for Claude showed nothing because the server never even attempted to call Anthropic.

The Fix

1. launch.ps1 — permanent launcher with explicit env injection

New file C:\tools\nous-desktop\launch.ps1. Reads every known provider key from User-scope env via [Environment]::GetEnvironmentVariable($name, 'User') (which goes through the registry, not the stale block), assigns each to $env: on the running PowerShell process, then launches nous-desktop.exe as a child — which inherits the freshly-populated env. Future launches no longer depend on the PowerShell's startup-time env block.

Output on a successful run:

[launch] injected 7 / 7 provider keys into process env
[launch] backgrounded PID=18236 on port 5050 (--no-auth)

Params: -Port <n>, -NoAuth $true|$false, -Background. Default: port 5050, --no-auth, foreground.

2. build_openai_responses_body — bump max_output_tokens 4096 → 16384

While verifying the fix end-to-end via the WS task message, observed that the three new openai-responses entries (gpt-pro, gpt-pro-prev, gpt-codex) returned empty content. Root cause: GPT 5.5-pro / 5.4-pro / 5.3-codex are reasoning models that reserve a portion of the max_output_tokens budget for hidden chain-of-thought. With 4096 tokens, the reasoning chain consumes most of the budget on even tiny prompts, leaving zero for visible output. Bumped to 16384 (8K typical reasoning + 8K visible) which is the OpenAI-recommended floor for these models. Code change at src/serve.c:

/* 16384 not 4096 — reasoning models (5.5-pro, 5.4-pro, codex)
 * reserve hidden chain-of-thought tokens out of this budget.
 * 4096 left tiny prompts with empty visible content. */
return snprintf(out, (size_t)maxlen,
    "{\"model\":\"%s\",\"input\":\"%s\",\"max_output_tokens\":16384}",
    esc_model, esc_input);

Why this happens (Windows env model in one paragraph)

Windows stores user env vars in HKCU\Environment. When a process starts, the OS reads the registry and merges Machine + User scopes into a process-private env block. setx writes to the registry only — it does NOT touch any live process's env block, and Windows' broadcast of WM_SETTINGCHANGE is honored by some apps (Explorer, cmd.exe spawned via shell shortcut) but ignored by long-lived dev tools like Claude Code's PowerShell wrapper. So in this session: setx wrote the key → registry updated → existing PowerShells still saw $env:ANTHROPIC_API_KEY as null → any child they spawned inherited the same null. Only a brand-new PowerShell that goes through the Windows shell-launch path would have picked up the change. The fix bypasses all this by reading the registry-fresh value at every launch via [Environment]::GetEnvironmentVariable(.., 'User') and writing it to the launcher's $env: before spawning the child.

What Changed in Operator Workflow

Before: launch via & 'C:\tools\nous-desktop\nous-desktop.exe' --no-auth 5050 — worked only if PowerShell was started AFTER all setx calls had landed. Fragile.

After: launch via C:\tools\nous-desktop\launch.ps1 -Background — works regardless of when keys were set, regardless of PowerShell session age. Self-documenting (prints which keys were found).

SETUP.md update (recommended follow-up, not in this fix): replace any "to run, do ./nous-desktop.exe --no-auth 5050" instructions with "to run, do ./launch.ps1 -Background".
