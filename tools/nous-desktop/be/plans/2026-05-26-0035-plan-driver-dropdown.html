~p: DRIVER dropdown implementation

Plan to ship the design from ~d 2026-05-26 00:30. Single phase, ~6 small edits across 4 files, no rebuild-blocker dependencies, ~30 min wall time. Chained as ~d~p~e~u~t~s do not stop.

Plan — running execution (no approval gate)

1. Intent

Make the dropdown the source of truth for the THINKING-column orchestrator. Default to claude-opus-4-7. Show the active driver visibly in the header. Honor "Let NOUS decide" honestly (fail silent when mesh is down — no surprise fallback to gemini). Keep the SENSE/SELECT trace visible but make it reflect reality.

2. Scope

In

  • HTML: add second <select id="driver-select"> with labeled wrapper, add #driver-active status span, restructure header for two-row controls.
  • JS: buildEngineSelectors() populates both selects with provider list + "Let NOUS decide"; both default to "claude-opus"; on-change sends set_driver/set_engine WS msg; updates #driver-active label.
  • JS: rename existing selectedEngine var stays for /summarize; new selectedDriver for the orchestrator.
  • C globals: g_active_driver[64] = "claude-opus", g_active_engine[64] = "claude-opus".
  • C WS handlers: set_driver + set_engine route to the globals.
  • C handle_task: lookup g_active_driver in g_providers; if "" → check mesh, fail-silent if down; if specific → use that provider for guided_ai_thread.
  • C guided_ai_thread: skip SENSE step when driver was operator-picked; emit SELECT with actual provider.
  • CSS: small additions for .driver-controls row + .driver-active styling.
  • Rebuild + relaunch via launch.ps1.
  • Smoke via WS task probe + visual browser check.

Out

  • Real domain classifier (SENSE remains placeholder for future).
  • Failover on driver error (Joey explicit: fail and do nothing).
  • Per-connection driver state (one global for v1; multi-user is v2+).
  • Persisting driver across restarts (config-file read for default value is v2).

3. Approach

Single phase (P3-a). Backward-compatible at every step. Edits are isolated per file — no cross-file refactors. Rebuild required (provider-table + handle_task touched). Server restart required (in-memory globals).

4. Phases (one phase, see ~e for itemized board)

  1. HTML — add driver-select, driver-active, restructure header into two rows.
  2. JS — buildEngineSelectors() handles both; defaults set; on-change WS + label update.
  3. C globals + WS — add set_driver/set_engine handlers; init g_active_driver/g_active_engine to "claude-opus".
  4. C handle_task — replace first-available loop with lookup of g_active_driver; honor empty-string sentinel for NOUS-decide with mesh-availability check + fail-silent path.
  5. C guided_ai_thread — conditional SENSE-line emission (skip when driver was operator-picked).
  6. CSS — minimal additions for the new layout.
  7. Rebuild + smoke — gcc, launch.ps1, WS task probe with each dropdown state.

5. Constraints

  • No new external dependencies (Pure Metal Test passed in ~d).
  • Backward compat: if g_active_driver is somehow corrupted, fall through to first-available so the cockpit doesn't hard-fail.
  • Server restart required after rebuild (no hot-reload path for C state).
  • launch.ps1 stays as launch entrypoint (env injection still good for non-vault path).
  • SENSE/SELECT visibility preserved per Joey's explicit ask.

6. Unknowns

  1. How does the cockpit handle a stale driver value (e.g. operator picked claude-opus but then the key got removed)? Need an "api_key present" check after the lookup; if missing, fail-silent with a "selected driver has no key" message.
  2. Does the existing selectedEngine client-side state need migration? Probably default both to "claude-opus" on cold start; if localStorage saved the old "Let NOUS decide" default, the user will see it persist. Acceptable.
  3. Should the dropdown options exclude providers that have disabled=1? Probably yes — operator can't pick a disabled provider as driver. Easy to add to buildEngineSelectors.
