3. Tool names vs model names — the two-name reality

In the C struct ProviderDefault (serve.c:340-359), every entry has BOTH:

{
  const char *name;         // operator-facing alias  →  "claude-haiku"
  const char *endpoint;     // API URL                →  "https://api.anthropic.com/v1/messages"
  const char *format;       // request format type    →  "anthropic"
  const char *model;        // exact API identifier   →  "claude-haiku-4-5-20251001"
  const char *key_env;      // env var holding key    →  "ANTHROPIC_API_KEY"
  const char *builtin_key;  // compile-time fallback  →  NULL
  double input_cost;
  double output_cost;
}

name is the short, operator-friendly alias. It's what appears in:

  • The DRIVER and ENGINE dropdowns in the cockpit
  • The fan-out tile labels (DATA ACCESS column)
  • The SELECT trace line ("expert: name (model)")
  • WS message payloads ({type:"set_driver", driver:"claude-haiku"})
  • Anywhere we reference the tool in code or config

model is the exact string the vendor's API expects. It's what goes:

  • In the JSON body of the actual HTTP request
  • Into the request-builder functions (build_anthropic_body etc.)
  • Into the dropdown's option label (for operator clarity)

Why two names? Stability. When OpenAI ships gpt-5.6, we change the model field from "gpt-5.5" to "gpt-5.6" — but the name "gpt" stays the same. Operators don't have to relearn aliases on every vendor release. Scripts, board items, transcripts, telemetry, audit trails — all reference the stable name.

Is there a third name? No — not in our config. The underlying vendor may have additional aliases (e.g., Anthropic's claude-haiku-4-5 is a dateless alias resolving to claude-haiku-4-5-20251001); we just pin to one. If a future need arises (e.g., showing the user-friendly vendor display name like "Claude Haiku 4.5"), that's a new display field — not part of v1.

4. Naming inconsistencies worth a ~u

Issue 1 — Grok ordering is backwards

CurrentMaps toGenerationShould be
grokgrok-4.20-0309-reasoning4.20 (older)grok-prev
grok-flagshipgrok-4.34.3 (newer)grok
grok-fastgrok-4.20-0309-non-reasoning4.20 (fast variant)(keep — it's already descriptive)

The unadorned grok name should be the current flagship — it matches the gpt → 5.5 (current) / gpt-prev → 5.4 (older) pattern, and matches claude-opus / claude-sonnet being the latest of each tier.

Issue 2 — Gemini naming is scattered (defer)

"gemini" (= 2.5-pro) is the stable production line; "gemini-flash35" (= 3.5-flash) is newer but feels like an outlier in the naming. Could be cleaned up but: Gemini's preview/GA cadence is messy and rebranding "gemini" might mean changing what people are used to. Defer until Google's lineup stabilizes.

Issue 3 — Documentation gap

The two-name convention (name vs model) isn't documented anywhere. Add a section to nous-desktop/CLAUDE.md so it's clear to future readers (Joey + Claude itself) why both fields exist and how they relate.
