Phase F — Claude-as-integral-employee seam (design only, 0 pts here)

Joey relayed Rudi: "if I had an active business, my AI agent would be an integral employee. Work that into the plan too."

The cockpit today is per-task fan-out + one orchestrator. Claude IS the orchestrator; that's progress. But it isn't yet an "employee" because it lacks:

  • Persistent memory — Claude remembers what it learned across sessions (current cockpit is per-task; no cross-session state)
  • Task queue / autonomous run — Claude can pick up multi-step work and run with it (today: each query is one-shot)
  • Tool access — Claude can browse web (Firecrawl), run code, edit files, query stores
  • Decision authority + delegation — Claude decides what to do and delegates parts to fan-out tiles or specific specialists (today: fan-out hits all tiles for the same prompt)

What this plan does: name those gaps; ensure the new providers slot in without blocking the future build (e.g., MiniMax as a delegated specialist; Kimi/Qwen as alt-orchestrator options).

What this plan doesn't do: build any of those four. Each is a substantial ~d + ~p in its own right. Recommend pulling the first one (persistent memory) into a new plan after Phase E lands.

5. Constraints

  • OpenClaw stays out. Joey's explicit no. Multi-agent runners aren't our path; fan-out + driver IS.
  • OpenAI-compatible API preferred. Most new vendors expose v1/chat/completions-style endpoints; we reuse the openai format type. Cohere is the one exception (their format is similar but needs a tiny wrapper — could add cohere format type or massage payload).
  • Rule 4 — Joey pastes, Be handles. Each new key follows the existing flow: Joey signs up, pastes once in chat, Be installs via setx (or eventually users.dat per P2-a), updates users.dat when that lands, never logs the value.
  • Cost ceiling. Combined monthly new cost estimate: $10 (MiniMax) + free tiers everywhere else = ~$10-30/mo on top of existing Anthropic/OpenAI/Google. Joey can disable any tile that gets expensive.
  • MAX_PROVIDERS bump. Currently 32 in serve.c #define; bump to 64 to allow ~12 additions with headroom.
  • Honor the naming convention just documented in nous-desktop/CLAUDE.md. Each addition uses <vendor> for current flagship, -mini / -fast / etc. for variants.
  • No fan-out per-query cost explosion. 42 tiles × per-query cost = real $$$ if Joey runs a lot of tasks. The DRIVER concept already mitigates (only one tile drives THINKING). For fan-out: add a g_disabled tracking + set_disabled WS handler so Joey can mute expensive tiles when not testing them. Already partially there; verify in Phase E.

6. Unknowns

  1. MiniMax exact API shape. Their docs claim OAI-compatible but the endpoint differs (chatcompletion_v2). Need to verify the request body format at first probe. May need small wrapper.
  2. Cohere format incompatibility. Their v2/chat endpoint accepts a different message format. Either add a cohere format type or transform payload at request-build time. Decide at implementation.
  3. Free-tier limits. Each new vendor has different free-tier ceilings. Hard to estimate cost until we run real traffic.
  4. Account-tier gating. Some models may require paid tier (saw this with gpt-5.5-pro). Mitigation: probe each one before committing to the table.
  5. Latency variance. Adding 12 tiles to the fan-out adds parallel HTTP calls. With Cohere or Reka having higher latency, the overall task feels slower (last response = total time). May need a "fan-out subset" feature.
  6. Tile organization in the UI. 42 tiles in DATA ACCESS column gets crowded. May need a per-vendor grouping or scroll affordance. Out of scope for v1.
  7. Claude-as-employee specifics (Phase F). Persistent memory is the first piece. Could be: (a) NOUS mesh memory skill, (b) local SQLite, (c) JSON file. Each has implications. Defer to its own ~d.
