~i: how SENSE / SELECT actually works (and why the dropdown is theatre)

From the screenshot: dropdown says claude-opus-4-7, trace selects gemini, request 429s, "expert request failed". Three independent bugs stacked. Here's the mechanism, what's broken, and what to fix first.

Headline

The "driver/director" today is whoever is first in g_defaults[] — currently gemini. Not Claude. Not what the dropdown shows. Not the result of any actual analysis.

The dropdown does nothing for the main task flow. It only affects /summarize and /advise side-commands. Setting it to claude-opus-4-7 has zero effect on which provider drives the THINKING column.

SENSE is fake. The "detecting knowledge domain..." and "pillar: GENERAL (confidence: 0.50)" strings are hardcoded literals. No classification happens. The C code just emits those two lines on every task.

SELECT is "first available," not "best for the job." Loops g_providers[] in declaration order, picks the first with an API key, breaks. That's the entire selection algorithm.

No failover. When the picked expert errors (e.g. Gemini 429), the THINKING column dies with "expert request failed". No retry against the next provider.

1. The actual code path (serve.c)

Step 1 — Task arrives via WS

handle_task() at serve.c:1708 runs in three phases:

  1. Phase 1: emit phase: understand, the sense event (with the list of providers having keys).
  2. Phase 2: attempt relay_intelligence() — would do real sense+select on the NOUS mesh. With mesh down (no PSK loaded), this phase is skipped entirely.
  3. Phase 3: launch one guided_ai_thread + fan out to every keyed provider.

Step 2 — Guided AI thread selection

serve.c:1782-1811:

/* Launch guided AI thread — picks first available provider */
for (int i = 0; i < g_provider_count; i++) {
    Provider *p = &g_providers[i];
    if (p->disabled || !p->api_key[0]) continue;
    /* ...spawn guided_ai_thread with this provider... */
    break; /* only one guided AI thread */
}

That's the entire SELECT logic. Whichever provider is first in the table with an API key wins the THINKING column. No query analysis, no domain detection, no scoring, no operator preference honored.

Step 3 — The "SENSE" and "SELECT" trace lines are theatre

guided_ai_thread() at serve.c:1532-1545:

send_event(ctx->conn_id,
    "{\"type\":\"search_step\",\"step\":\"sense\","
    "\"detail\":\"detecting knowledge domain...\"}");

send_event(ctx->conn_id,
    "{\"type\":\"search_step\",\"step\":\"sense\","
    "\"detail\":\"pillar: GENERAL (confidence: 0.50)\"}");

/* ...then emit the "select" event with the provider that was
   ALREADY selected (passed in via ctx->provider_idx)... */
snprintf(step, sizeof(step),
    "{\"type\":\"search_step\",\"step\":\"select\","
    "\"detail\":\"expert: %s (%s)\"}",
    p->name, p->model);

The "detecting knowledge domain..." string is a hardcoded literal. The "pillar: GENERAL (confidence: 0.50)" is also a hardcoded literal — confidence 0.50 is fixed for every task. The "SELECT: expert" event just announces which provider was already chosen by the loop above.

The original design intent (per the relay_intelligence() comment at serve.c:160) was for the NOUS mesh to do real domain detection and route to the appropriate expert based on a real confidence score. The C code has the trace-emission scaffolding but no actual classifier — that's mesh-side work that doesn't exist yet locally.

2. Why the dropdown shows claude-opus-4-7 but doesn't matter

The dropdown <select id="engine-select"> in index.html:57 is wired in loop.js:

// loop.js:374
sel.addEventListener('change', function() {
    selectedEngine = sel.value;
    if (ws && ws.readyState === 1) {
        ws.send(JSON.stringify({ type: 'set_engine', engine: selectedEngine }));
    }
});

It sends a set_engine WS message to the server. The server stores this preference, but only uses it for /summarize and /advise commands:

// loop.js:715 (the /summarize command path)
ws.send(JSON.stringify({ type: 'summarize', engine: selectedEngine }));

Main task flow ignores set_engine completely. The HTML's title attribute on the dropdown even gives this away: title="default engine for advise/summarize". It was never claimed to drive the main orchestrator.

So the screenshot has two truths:

  1. Joey picked claude-opus-4-7 in the dropdown — that's now the engine for /summarize / /advise calls.
  2. Joey typed a regular query, which became a task WS message — which goes through the loop above and picks gemini (first in table) regardless of dropdown.

The dropdown UX is misleading. It looks like a primary control, but it's a side-channel preference.

3. Why gemini errored 429 and the whole chain died

The chain in the screenshot:

SENSE     detecting knowledge domain...        (hardcoded)
SENSE     pillar: GENERAL (confidence: 0.50)   (hardcoded)
SELECT    expert: gemini (gemini-2.5-pro)      (first available in table)
AUTH      GEMINI_API_KEY loaded                 (key found in users.dat or env)
PROMPT    system prompt (32 chars)              (the "Provide a clear..." hint)
REQUEST   POST https://...gemini-2.5-pro:generateContent
RESPONSE  HTTP 429 — 284ms — 2739 bytes        ◀── free-tier rate limit
ERROR     HTTP 429
error     expert request failed                ◀── chain ends here

Gemini's free tier is small. After a few requests in a short window, the API returns HTTP 429 (Too Many Requests). guided_ai_thread hits the error, emits error: expert request failed, and does not try another provider. The THINKING column is dead for that task.

Meanwhile the fan-out to the other 26+ providers still ran in parallel — they all responded normally in DATA ACCESS column. Only the THINKING-column orchestrator failed because it was pinned to gemini.

5. What "fixing sense/select first" should mean

Joey's instinct: fix this BEFORE the FQDN-vault migration. Reasonable — the sense/select bugs are user-visible every time the cockpit is used; the vault work is plumbing.

Minimal viable fix (smallest possible patch, biggest UX win):

  1. B2 — move claude-sonnet to slot 0. Now Claude is the default driver when dropdown is "Let NOUS decide".
  2. B1 — honor the dropdown. When operator picks claude-opus-4-7, the main task uses claude-opus-4-7 (not gemini, not sonnet). The dropdown UX finally tells the truth.
  3. B3 — failover on error. When Claude rate-limits (or any picked expert errors), try the next provider. THINKING column never dies on a single provider's hiccup.
  4. B4 — drop the fake SENSE strings. The trace shouldn't lie about what's happening. Replace with honest "selecting expert..." or just remove the SENSE step entirely until real classification exists.

Estimated total effort: ~60-90 LOC in serve.c, ~30 min wall time, one rebuild + smoke. Roughly the same size as P2-a but more focused on the operator's main interaction surface.

Recommendation: do B1+B2+B3 in one shot (call it P2-c "real orchestrator selection"). Defer B4 until later because honest SENSE requires actual classification work, which deserves its own design pass (could even be the first real use case for Claude-as-orchestrator: "Claude, what domain is this query in, and which other providers should weigh in?").

6. Which AI is the driver/director (definitive answer)

Today (with current g_defaults[] order): gemini-2.5-pro is the driver. First in the FREE block. The dropdown doesn't change this.

After P2-a idx:4 (claude-sonnet to slot 0): claude-sonnet becomes the default driver. Dropdown still ignored.

After this proposed fix (B1+B2+B3): dropdown determines the driver. Default is claude-sonnet. Failover walks the table if the picked driver errors.

Long-term (NOUS mesh-side classifier): a real SENSE step classifies the query into a pillar (CODING, FACTUAL, CREATIVE, MATH, etc.) with a real confidence score, then SELECT picks the best-for-that-pillar expert. The dropdown becomes an override rather than a default.
