Plan -- Awaiting Approval

1. Intent

Three coupled goals:

  1. Sovereign key storage. No more Windows registry / setx / env-block-staleness. Keys live in users.dat next to the binary, sectioned by FQDN. Per-operator, per-machine, plaintext-first then encrypted-second.
  2. GitHub vault as backup + recovery. The same key file (eventually encrypted) lives in joeycastilloUS/vault (private). Any machine can gh repo clone and have a working keychain. The FQDN↔org mapping ("joeycastillo.us → joeycastilloUS") makes the vault location deterministic for any future operator.
  3. Claude as orchestrator. The cockpit's THINKING column is driven by whichever provider is first in g_providers[]. Move claude-sonnet to slot 0. Claude becomes the guided-AI driver + the synthesis target for summarize/judge dispatch.

End state: registry-free, multi-operator-capable from day one, Claude leading the show, encryption flipping on as a v1.5 follow-on.

2. Scope

In (v1 — this plan)

  • Add FQDN→org name mapping function in nous_userkeys.c per the kernel convention just committed.
  • Change load_providers() at serve.c:409 to pass the operator FQDN as the user argument to resolve_key().
  • Seed users.dat from current Windows env-scope values (one-shot script or first-launch convenience).

  • Vault sync at startup: gh repo clone joeycastilloUS/vault (or pull if already cloned), decrypt, load.

Out (deferred to v2+)

  • Multi-operator UI (operator picker, TOTP for switching). v1 is single-operator; FQDN→org mapping is in code but only Joey is the operator today.
  • Auto-rotation reminders for keys that have hit transcripts. Manual for now per SETUP.md.
  • Vault-to-mesh push (Layer 2 → Layer 3) once mesh is up. Same crypto + same identity, so future-proof.
  • Non-.us TLD mapping rules. Per kernel: deferred until first need.

3. Approach

Two phases. P1-a is shippable on its own (local users.dat + Claude orchestrator); P1-b adds the encrypted vault + GitHub sync on top.

Why split: P1-a unlocks immediate value (registry-free local) and is verifiable in 15 minutes. P1-b is real crypto + remote sync — bigger code, more risk; we want P1-a's wins banked before we touch crypto.

Why this order matters

P1-a is reversible in seconds (one-line revert in resolve_key call site). P1-b touches crypto and remote sync — much harder to roll back cleanly. Banking P1-a first means we ship value even if P1-b reveals a surprise.

5. Constraints

  • Rule 4 (Joey pastes, Be handles). Any new secret (CF token if not already in env, passphrase if chosen) follows the paste-handle-rotate flow.
  • Rule 8 (commercial intel stays private). The committed users.dat.enc / vault repo content is sovereign; never echoed in chat, never logged unencrypted.
  • FQDN↔org kernel convention (just committed). All mapping in code MUST follow it exactly; no clever alternatives.
  • Pure C, no new dependencies. NOUS crypt skill is already vendored. gh CLI for vault sync is a runtime tool, not a build-time dep — that's OK.
  • users.dat.enc is committed, users.dat (plain) is NEVER committed. .gitignore entry mandatory.
  • Backward-compatible Phase 1-a. If users.dat doesn't exist, fall back to env vars (existing behavior). No flag-day breakage.
  • WDAC launch path preserved. launch.ps1 stays as the launch entrypoint; only its env-injection responsibility migrates away over time.

6. Unknowns

  1. Cloudflare API token availability at boot. If the binary needs CF token to derive the decryption key, but the CF token itself isn't in env at boot, we have a chicken-and-egg problem. Fallback: passphrase prompted at first launch, cached for the session. Decide between (a) CF-token-derived, (b) passphrase-derived, (c) both with fallback during P1-b implementation.
  2. gh CLI availability + auth state. Vault sync uses gh repo clone. Need gh auth login to be valid for the operator's GitHub user. Verify at startup; surface clear error if missing.
  3. Vault repo creation. First-launch needs to create joeycastilloUS/vault if it doesn't exist. gh repo create with --private handles it. Need to also seed README + .gitignore so the repo isn't barren.
  4. Concurrent writes from multiple machines. If Joey's NEO and a future laptop B both modify keys, last-push-wins with potential conflict. v1 accepts this (single-operator, single-machine usually); v2 needs proper merge or single-writer convention.
  5. Encryption key rotation. If Joey rotates his CF token, the previous users.dat.enc becomes unreadable. Need a re-encrypt utility (cheap; one extra command). Document.
  6. What's the "user" name in users.dat once we go FQDN? Sections become [user:joeycastillo.us]. Existing dead-code path uses string match on the section name; should "just work" once we pass FQDN as the lookup key.
  7. Claude latency in THINKING column. Sonnet is faster than Opus; Opus is smarter. If Sonnet feels too rushed for orchestration, swap to Opus and accept slower THINKING-column update.
