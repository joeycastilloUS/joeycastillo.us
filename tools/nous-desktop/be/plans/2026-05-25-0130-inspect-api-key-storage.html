~i: API key storage — where keys live, where they should live, what's broken

Audit of the 4-layer key resolution in serve.c. Why the Windows registry path bit us yesterday, why the existing user-store path is dead code today, and exactly what flipping a single argument unlocks.

Headline

Keys live in 4 documented layers; today only 1 actually fires. The user-store layer (which Joey would want — a sovereign local file, no registry, no env-block staleness) is built and ready in nous_userkeys.c, but unused: load_providers() passes user=NULL to resolve_key(), so the user-store lookup short-circuits to NULL on every call. The whole "no registry" pivot is essentially one argument change from working.

For Claude as orchestrator: the existing "guided AI" thread (the one that drives the THINKING column in the cockpit) picks the first available provider from g_providers[]. To make Claude the orchestrator, put a Claude entry first in g_defaults[] — no new code, just reordering the table.

1. The four documented storage layers

From serve.c:322-338:

static const char *resolve_key(const char *user, const char *env_name,
                                const char *builtin)
{
    /* 1. Per-user key from users.dat */
    if (user) {
        const char *uk = nous_userkeys_get(user, env_name);
        if (uk && uk[0]) return uk;
    }
    /* 2. Environment variable */
    const char *env = getenv(env_name);
    if (env && env[0]) return env;
    /* 3. Compiled-in builtin (free tier) */
    if (builtin && builtin[0]) return builtin;
    return NULL;
}
#LayerWhere it livesStatus today
1 Per-user store users.dat next to the binary. INI format: [user:joey] sections with KEY=VALUE lines. Loaded at startup via nous_userkeys_load("users.dat"). DEAD CODE — load_providers calls resolve_key(NULL, ...) at line 409, so the if (user) guard is always false. Lookup never fires.
2 Process env var OS env block of the running process. Populated by Windows from HKCU\Environment registry at process-start. setx writes the registry, but the propagation is fragile — yesterday's Claude-dark bug. ACTIVE but FRAGILE — only layer doing real work right now. Bug we hit: setx doesn't propagate to running PowerShell, child binary inherits stale block.
3 Compile-time built-in nous_keys_builtin.h with #define NOUS_KEY_GEMINI etc. Used for "free tier" zero-config first-run. UNDEFINED — file doesn't exist; only .example is checked in. #ifndef guards in serve.c (line 25-32) fall back to NULL.
4 NULL (no key, provider disabled) FALLBACK — works as designed.

2. Why the registry-via-env path is the wrong primary

Joey's instinct: "we should not be using registry at all for storage." Three reasons this is right:

  1. Windows env-block staleness. setx writes to the registry but doesn't broadcast to already-running processes in a way they reliably honor. Yesterday's Claude-dark bug — the server child inherited a stale block and saw getenv("ANTHROPIC_API_KEY")=NULL despite the registry having the key.
  2. Cross-OS portability. The registry doesn't exist on Linux/macOS. Any code path that assumes "setx + getenv" is Windows-only. The C code itself (getenv) is portable, but the install path isn't.
  3. Sovereignty. Per kernel CLAUDE.md / a-d-d Rule 4 (Joey pastes, Be handles): keys never leave the local machine to a vendor service; they sit in user-controlled storage. The Windows registry is technically local but it's a system-managed surface — backups, sync tools, group policy, can all touch it. A file-on-disk that only the app reads is cleaner.

3. Where keys SHOULD live

Primary: users.dat next to the binary. The code is already written; nous_userkeys.c handles load/get/set with the INI format above. Pure-C, no dependencies, ~150 LOC. The format:

[user:joey]
ANTHROPIC_API_KEY=sk-ant-api03-...
OPENAI_API_KEY=sk-...
GEMINI_API_KEY=...
DEEPSEEK_API_KEY=...
MISTRAL_API_KEY=...
XAI_API_KEY=...
PERPLEXITY_API_KEY=...

What needs to change in code:

  1. load_providers() at serve.c:409 currently passes NULL as user: resolve_key(NULL, d->key_env, d->builtin_key). Change to resolve_key(g_active_user, d->key_env, d->builtin_key) where g_active_user is the operator name (default "joey" or read from FQDN identity).
  2. Pick the operator name. Simplest: hardcode "joey" for v1 (Joey is the only operator). Cleaner: read from desktop.conf or $USERNAME env. Cleanest (v2+): from FQDN-keyed identity via NOUS user skill.
  3. Optionally a CLI flag --user <name> so multiple operators can share one binary.

One-time migration: populate users.dat by reading the current env-scope values and writing them out:

# PowerShell migration (one-time, run once, then can clear setx)
$dest = 'C:\tools\nous-desktop\users.dat'
"[user:joey]" | Out-File $dest -Encoding ASCII
foreach ($k in 'ANTHROPIC_API_KEY','OPENAI_API_KEY','GEMINI_API_KEY',
                'DEEPSEEK_API_KEY','MISTRAL_API_KEY','XAI_API_KEY',
                'PERPLEXITY_API_KEY') {
    $v = [Environment]::GetEnvironmentVariable($k,'User')
    if ($v) { "${k}=${v}" | Out-File $dest -Encoding ASCII -Append }
}

Then: remove all setx-based persistence; launch.ps1 no longer needs to inject env vars; the binary self-sufficient with users.dat. Registry-free.

Security note (v1 vs v2+)

v1 (now): users.dat is plain text. File is gitignored. Filesystem permissions are NTFS user-only by default. Acceptable for single-operator dev machine.

v2+: encrypt users.dat using NOUS's existing security/skill/crypt (AES-256-GCM, key derived from TOTP). Same path, encrypted-at-rest. nous_userkeys_load() grows a decrypt step. Not in v1 scope but the path is clear.

6. Claude as orchestrator — what makes it the main driver

Joey's framing: "Claude is the main driver/orchestrator." The cockpit already has two distinct concepts:

  • Fan-out (DATA ACCESS column) — every keyed provider gets the prompt in parallel. All 30 tiles light up. This is the "many advisors" view.
  • Guided AI (THINKING column) — a single provider drives a curated/synthesized answer. Today this is the first available provider in g_providers[] order (serve.c:1780):
/* Launch guided AI thread — picks first available provider */
for (int i = 0; i < g_provider_count; i++) {
    Provider *p = &g_providers[i];
    if (p->disabled || !p->api_key[0]) continue;
    /* ...launches guided_ai_thread with this provider... */
    break;  /* (effectively — only takes the first) */
}

"First available" today = first in g_defaults[] table order. Right now that's gemini (line 354) — because Gemini was put first in the "FREE" block. So Gemini gets to be the guided AI, not Claude.

Two ways to make Claude the orchestrator:

  1. Simple (recommended for v1): reorder g_defaults[] so claude-opus is the first entry. The existing "first available" logic picks Claude. Zero new code. One line moved.
  2. Explicit (v2): add an is_orchestrator bool to ProviderDefault. The guided-AI selection looks for is_orchestrator && api_key[0] first; falls back to first-available if no orchestrator. Lets you have a configured primary independent of table order. Useful when you want different orchestrators per workspace.

What the "orchestrator" actually does in the current code:

  • It's the model that drives guided_ai_thread. The same model gets called twice per task — once in fan-out (as a peer) and once as the guided AI (as the lead).
  • The guided AI's output goes into the THINKING column. The fan-out outputs go into the per-provider tiles.
  • The guided AI is also the model invoked for summarize, judge, and other meta-cognitive tasks (per the WS message router).

So making Claude the orchestrator gives Claude:

  • The THINKING column (primary visible voice)
  • The summarize/judge dispatch (synthesis tasks)
  • Still a tile in DATA ACCESS (peer in the fan-out)

Specifically: put claude-opus as #1 (best reasoner) OR claude-sonnet as #1 (best speed/intelligence trade for orchestration latency). For an AI LIVE FACTORY show where each query feels snappy, Sonnet is probably better; for solo deep-work, Opus.

7. Recommendation (one paragraph)

Migrate to users.dat as the primary key store: one PowerShell snippet writes the file, one C-source line change (NULL → "joey") activates the user-store layer, rebuild, relaunch. Then reorder g_defaults[] so claude-sonnet is the first entry — Claude becomes the orchestrator. Result: no more setx, no more env-staleness bugs, Claude drives the THINKING column, all 27+ keyed providers fan out as peers, Mistral revives the moment Joey pastes a fresh key, and the system is one step closer to the NOUS-vault future (v2+ encrypts users.dat with AES-256-GCM via existing crypt skill). Total wall time ~20 min including the verify.
