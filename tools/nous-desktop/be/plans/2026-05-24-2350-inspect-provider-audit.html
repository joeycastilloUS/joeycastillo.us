~i: provider audit — keys, models, staleness, free coding options

Why Claude is dark for all 3 models, why Mistral 401s, what's stale across the whole provider table, and what free coding models exist outside our current lineup. Everything verified by live API probe + official-docs read — no guessing.

Headline

Claude is dead because ANTHROPIC_API_KEY is not set. All three Claude entries (opus/sonnet/haiku) share one env var; with no key, all three fail before any model ID gets evaluated. Separately, claude-opus-4-6 is now legacy — Opus 4.7 is the GA flagship as of late 2025 / early 2026 per docs. So even after the key lands, the opus model ID still needs a bump.

Mistral is dead because the MISTRAL_API_KEY env var is rejected (401). The 57-char key in User-scope env returns "401 Unauthorized" on both /v1/models and chat-completion probes. Key is revoked, expired, or scoped to the wrong workspace. All three Mistral entries (mistral/mistral-large/magistral) share that var; fix one, fix all.

Two additional broken-or-stale entries we didn't notice yet: (1) grok-fast uses model ID grok-4-1-fast-reasoning which is not in xAI's current catalog — stale name. (2) DeepSeek's deepseek-chat and deepseek-reasoner aren't listed in /v1/models response (only deepseek-v4-flash and deepseek-v4-pro are) — aliases may still resolve at chat-completion time but worth verifying.

3. Built-in free-tier keys (the NOUS_KEY_* path)

The 4-layer key resolution in load_providers() is: user-store > env var > built-in > NULL. The built-in layer uses NOUS_KEY_GEMINI, NOUS_KEY_DEEPSEEK, NOUS_KEY_MISTRAL compiled in from src/nous_keys_builtin.h.

Reality: src/nous_keys_builtin.h does NOT exist in the repo. Only nous_keys_builtin.h.example exists with placeholder text. The serve.c gracefully falls back to NULL via #ifndef guards (lines 25-32), so the build succeeds but the built-in keys are NULL.

So: the "free" providers (gemini, deepseek, mistral, magistral) are ONLY working today because Joey's env-var keys (GEMINI_API_KEY, DEEPSEEK_API_KEY, MISTRAL_API_KEY) cover them. There is no actual free-tier built-in key path active. The whole NOUS_KEY_* free-tier story is aspirational right now.

Implication: if Joey ever wants the "zero-config, just-run" experience the comment in the provider table promises ("FREE — hardcoded, zero config"), someone has to populate nous_keys_builtin.h with actual free-tier keys and check it in via a private build pipeline. Not in scope for this audit; flagging.

5. Why Mistral specifically returns 401 (root-cause confirmed)

Per Mistral community + docs, 401 Unauthorized on a key-present request means one of:

  1. Key was revoked in console.mistral.ai → API Keys panel.
  2. Key expired (Mistral keys can have expiration dates).
  3. Key scoped to a different workspace and you're hitting the default workspace.
  4. Account behind a billing block (no payment method on file for paid model access).
  5. Double "Bearer" prefix in Authorization header (we tested with single Bearer; not our issue).

Our probe uses correct single-Bearer auth and a key of plausible length (57 chars). The 401 is definitive: Mistral's auth server rejected the key value. Fix is to generate a fresh key, paste, and setx MISTRAL_API_KEY. No code change needed on our side.

Source: docs.mistral.ai/api · mission-open-data.fr/mistral-erreur-401 · drdroid.io Mistral AI Authentication Failure diagnosis
