SSH Signing — the fleet default
automated — install Step 5b
Go.Sign wires SSH signing for you.
The metal fleet signs commits with the ed25519 key already in your ~/.ssh — no GPG, no RSA keygen, no OAuth scope. Go.Sign runs automatically during install (install.bat Step 5b), verifies the key, enables commit.gpgsign, smoke-tests a signed commit, and exits. If you curl-installed metal, signing is already done. Zero manual steps.
What Go.Sign does on the SSH path
Verify, enable, smoke-test, done
auto Detect gpg.format = ssh and resolve your ~/.ssh ed25519 signing key
auto Enable commit.gpgsign so every commit is signed
auto Smoke-test an empty signed commit in a scratch repo
auto Exit clean — no GPG keyring ever touched
On this path Go.Sign never installs gpg, never mints a 4096-bit RSA key, and never needs the write:gpg_key scope. It just confirms the SSH key works and gets out of the way.

one thing to know
Register the SSH key as a signing key for the green badge
GitHub only shows the green "Verified" badge if your SSH public key is registered as a signing key (not just an auth key). It's a distinct key type on your account. Register it once:
gh ssh-key add --type signing ~/.ssh/id_ed25519_signing.pub --title "metal signing"
Go.Sign is being updated to do this automatically. Until that ships, run it yourself — commits sign fine without it, but the badge stays grey until GitHub knows the key is for signing.
GPG Fallback (boxes not on SSH signing)
If a box has gpg.format set to something other than ssh, Go.Sign falls back to the GPG path: it finds or generates a GPG key, wires git config, registers the key with GitHub, and smoke-tests. This path needs gpg installed and the write:gpg_key OAuth scope. Preserved below for those boxes — most operators never hit it.
Step 1 — the only manual moment (GPG path)
Grant GitHub CLI the GPG scope
Run automatically by Go.Sign when it generates a new GPG key. Opens a browser — you approve it, control returns. Equivalent command:
gh auth refresh -h github.com -s write:gpg_key
Interactive — opens your browser with a device code. Approve it on GitHub. One-time per laptop.
Step 2 (GPG path)
Go.Sign continues automatically
After the browser approval, Go.Sign continues without further input:
auto Find an existing local key matching your GitHub email, OR generate a fresh 4096-bit RSA key
auto Configure git to sign all commits with the chosen key
auto Upload the public key to your GitHub account via gh gpg-key add (when generated)
auto Smoke-test with an empty signed commit in a scratch repo
Result: every commit from this machine shows the green "Verified" badge on GitHub.
GPG Path — If Something Breaks
gpg failed to sign the data
Git is pointing to a key that doesn't exist on this machine. Run C:\metal\Go.Sign.bat — it finds an existing key or generates a new one and rewires git config.
insufficient OAuth scopes
gh hasn't been granted write:gpg_key yet. Go.Sign runs the refresh automatically on first key gen, or run it yourself: gh auth refresh -h github.com -s write:gpg_key.
New machine?
Just curl-install metal. install.bat runs Go.Sign.bat at Step 5b automatically — SSH signing on the default fleet, GPG fallback only where configured.
