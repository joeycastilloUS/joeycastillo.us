Is metal.ce Pure C?

~i (Fable 5) · metal Console Edition · the operator recalls "before it was a lot of batch files" — is the logic pure C99 now?

Yes — metal.ce is pure C99 now.

Everything that compiles into the product is C. The scripts that remain are launchers and installers, not logic. The batch-heavy era was real — and it was deleted (2026-03-23), not migrated; the C tree was built fresh starting 2026-05-12.

The binary is C, top to bottom

src/main.c is the metal.exe dispatcher — "No external libraries. -std=c11. Targets Win/Mac/Linux equally." Plain strcmp routing:

if (strcmp(a, "--version") == 0 || strcmp(a, "-v") == 0) return print_version();
if (strcmp(a, "handshake") == 0)                         return dispatch_handshake(argc, argv);

Behind it: src/handshake/ (FQDN first-launch), src/sdk/security/ (~6,100 lines — crypt.c 2,030, tls13.c 1,222, gate/QR/TOTP/seal/vault), src/sdk/nous/ (~26,300 — comms/wire, memory, action, user), src/tlt/ (~3,600, header: "Pure C99… No managed code is linked into this binary"). No shell-out to batch anywhere in src/. The only system() calls are test scaffolding and tlt/llm.c shelling to curl for LLM HTTP — C calling a cross-platform tool, not batch carrying logic.

The scripts are launchers, full stop

The 9 root .bat files total ~10 meaningful lines. go.bat is one line: @powershell -File "%~dp0go.ps1" %*. The .bat/.ps1/.sh triplication is one launcher × three platforms, not 3× logic.

The C# is the sanctioned jacket

src/tlt/worker/Program.cs (336 lines) is the Language Boundary rule in action: the .NET Playwright sidecar driving a browser on behalf of the pure-C tlt runtime. It is a separate subprocess — never linked in — pumping 9-op JSON-RPC over stdin/stdout. Playwright is Microsoft's API; this is the foreign-API wrapper the law permits ("C# is for foreign-API jackets only").

The JS is all served web assets — zero in the binary

~3.4K lines are the Desktop Edition cockpit (nous/user/interface/desktop/wwwroot/ — loop.js, store.js, terminal.js, browser UI). The rest: the served auth-gate page, the report viewer, vault ops utilities. The CE tool runs with no JS dependency.

The historical arc — deleted, not migrated

  • Mar 2026 — born batch. The repo started as a Windows provisioning/dotfiles system: provision.bat, then split: go.bat thin launcher, go-runtime.bat has all logic.
  • 2026-03-23 — batch killed. v2.0: inline tools/setup/provision into Go.Fe.2; delete 6 old files (eff7ed5). Batch logic collapsed into one PowerShell/bash installer pair.
  • 2026-05-12 — first C lands. ~a~t src/main.c dispatch (e77f350).
  • 2026-05-20 — the C tree is absorbed (Pure-C Runtime canon: "a single pure-C binary… no frameworks, no cloud lock-in").
  • 2026-06-20 — Language Boundary locked (C99 for everything we own; C# only as foreign-API jackets).

On record too: ~f self.c: pure C99, zero Windows #ifdef and ~s: Tier 0 ~unload landed — pure C, $0.00, ~100 ms. The batch you remember was deleted; the C tree is fresh.

The one purist gap

tlt/llm.c shells to curl for LLM HTTP instead of using the in-tree tls13.c. That's the only place a purist would point — and it's a candidate for a future ~f, not a Language-Boundary violation (curl is a cross-platform tool, not product logic in batch).

Straight answer

metal.ce is pure C. ~92% of the actual tool is C99 by line count (42K of ~46K product lines). The remaining 8% is one-line launcher shims, the Iron toolchain installer (which must precede the compiler), build scripts, and one sanctioned 336-line C# Playwright sidecar that talks to C over stdin/stdout. No product logic lives in batch. The batch era ended in March 2026; what you have now is a pure-C tool with thin, honest, cross-platform launchers around it.
