Salesforce → Raw IDS: the OneDine seed

Land the raw. Watchable, resumable, replayable — from a tab, not a side repo. The Salesforce twin of the blessed Zendesk import.

1. Summary

Operationalizes be/designs/2026-07-09-design-salesforce-raw-ids-seed-fable5.html: give OneDine's Salesforce seed an operational home — a Raw IDS first tab in the Argus Salesforce app, backed by a small C99 import service exposing a location-blind /import/start | /import/status | /import/refresh | /import/replay?case= contract. The service drives the live salesforce-bridge for reads and writes the sealed nous TripleStore. The seed itself is already landed — 42,402 cases + 16,385 comments → 634,907 triples, sealed at gs://acmedev-onedine-cases-ids/full/. This plan wires it into the console and closes the refresh loop.

Rule 6: this is a v1 hypothesis. Revision expected — the Unknowns and When To Revisit sections name where.

2. Intent

OneDine's full case history — cases WITH comments — lands in one sealed, append-only Raw IDS with per-batch provenance and a stored cursor (done), and the operator can seed it, watch it (pulled / triples / elapsed), refresh it (get-latest since the cursor), and replay one real case's actual body from the raw inside the console — proof by eyes, not by claim. When this plan is done, "do we have the data?" is a glance at the Raw IDS tab, and the import is boring.

3. Scope

In scope

  • The v0 tools promoted into argus/src/import/salesforce/ (pull_full.py, emit_jsonl.py, sf_ids.c, sf_replay.c) — done.
  • Full seed pull (adaptive CreatedDate windows) → JSONL + sealed IDS → GCS — done.
  • The C99 import service: /import/start, /import/status, /import/refresh, /import/replay?case= — localhost first, Cloud Run promotion later.
  • The Raw IDS tab — new FIRST tab in the Salesforce app (web/index.html APPS registry): Seed / Refresh / Replay, live progress, standing-raw panel.
  • Refresh / get-latest incremental loop off the stored cursor.
  • Replay of one real case end-to-end from the landed raw, rendered in the tab.

Out of scope — cut, not forgotten (the chains that stalled Zendesk)

  • ✗ Mesh residency (content-address, manifest signal, bulk channel) — a downstream board consumes this raw.
  • ✗ Decode → .pi programs / SF4 "cases as programs" — a decoder pass over the landed raw, a later board.
  • ✗ Config-object walk beyond Case + CaseComment (User, record types, etc.) — append-only means adding later costs nothing.

4. Approach

Staged, local-first → Cloud Run promotion. The tab binds ONLY the /import/* contract — the seam. The same C99 binary serves it from the mesh today and from Cloud Run at promotion (one-per-source pattern, mirrors the bridges). The seed is already mesh-run and proven; promotion is a deploy, not a rewrite. The one divergence from Zendesk — no incremental firehose — is solved by adaptive CreatedDate windows, already built.

5. Phases

RAW1 — Land the seed DONE

Adaptive-window full pull → JSONL + sealed nous IDS → GCS. 42,402 cases + 16,385 comments → 634,907 triples, 182 calls, replay+decode clean. Tools committed to argus/src/import/salesforce/.

RAW2 — The import service NEXT

C99 /import/status (reads the manifest cursor + store stats) and /import/replay?case= (open sealed IDS, return one case's fields + atoms). Localhost first; the seam is fixed here.

RAW3 — The Raw IDS tab

First tab in the Salesforce app: standing-raw panel (cases · comments · triples · cursor · seed date), Seed / Refresh buttons, one-case Replay rendered in-console.

RAW4 — Refresh loop

/import/refresh: pull WHERE CreatedDate > cursor, append to the raw, advance the cursor. Boring, incremental, resumable.

RAW5 — Cloud Run promotion

Deploy the import service as its own Cloud Run (one-per-source). Tab points at the run.app URL; contract unchanged.

6. Constraints

  • Read-only: the bridge exposes only /read; the import never writes to Salesforce.
  • Language law: seal/replay is C99 + nous; the only C# is the bridge jacket; Python is transport only.
  • Rate: ~100k API calls/day; the full seed used 182. Refresh uses a handful. No pacing pain.
  • PII: OneDine case bodies never leave GCP unredacted; console replay is operator-gated (daily passphrase).

7. Unknowns (Rule 6 — named, not hidden)

  • Comment coverage: CaseComment (16,385) is the public thread; do we also need EmailMessage / CaseFeed for the full conversation? The 82%-email finding suggests EmailMessage may hold real body content. Unknown until a decode pass measures signal.
  • Refresh cadence: ~26 new cases/day — is a manual Refresh enough, or does the tab need a scheduled pull?
  • Wide-row page cap: the ~1000/page shrink is empirical (Salesforce auto-batching); could shift if field selection changes.

8. When to revisit

  • A decode pass shows email-transport noise still dominates after atomization → add EmailMessage to the walk (append-only, cheap).
  • The refresh backlog grows faster than manual pulls clear it → schedule RAW4.
  • OneDine crosses ~500k cases or a bulk export is mandated → swap to the bridge extract escape route (design §Escape Routes).
