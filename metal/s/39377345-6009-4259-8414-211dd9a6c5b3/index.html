Argus over the Mendo Zendesk extract

Point Argus at the right corpus — the 431K-ticket Mendo Farms Zendesk extract, not the Salesforce OneDine seed — and let the operator find things, see the underlying .pi programs, see patterns, see matches, and see the shape of support, all off the replays of the sealed base dataset. The engines already exist; this design stands up the one service that's missing.

Course-correction on design-gemini-vertex-ids-brain: that shipped the retrieve-then-read loop grounded on onedine-raw (source: salesforce:onedine) — wrong dataset. Same engine, aimed at the Zendesk corpus. Model: Fable 5.

1. Need

Argus is Argus for Zendesk, but its grounded answers read a Salesforce store — 42K OneDine cases, and a handful surfaced per query. Wrong data, and thin. Meanwhile the real thing sits sealed and untouched: 431,260 Mendo Farms Zendesk tickets / 977,029 comments / 8.2M triples (gs://acmedev-mendo-tickets-ids/full/raw-ids-full.tgz), plus a 1,000-ticket derived snapshot that already carries the .pi program library. The operator doesn't just want search — he wants to see support the way the data actually shaped it: the program under each ticket, the patterns across them, the matches between them, the shape of the whole. Why now: the corpus is sealed + hash-verified, the decode engines are written and proven, and Argus is live on Gemini — the only missing part is the door.

2. Raw user stories operator voice

  • "As the operator, I want to find Mendo Zendesk tickets by what they're about — real tickets, real IDs, this corpus."
  • "I want to open one ticket and see the underlying program — the .pi steps the conversation actually is, de-atomized from the replay."
  • "I want to see matches — 'tickets like this one' — so a new problem lands next to its precedents."

  • "And I want to ask Argus in words and have it answer from THIS corpus, citing the tickets it read."

Boundaries. Read-only over the sealed extract — never writes it, never calls the live Zendesk API (that's zendesk-bridge's job). Language law: all intelligence stays C99 (the engines + the jacket socket, same as raw2); vanilla JS for the tabs. Out of scope (v1): BM25/embeddings recall (the standing Kaizen), live incremental refresh (the corpus is a frozen snapshot; refresh is a follow), any write-back.

6. Choices + consequences the forks

Fork 1 — Which corpus tier first. (A) 1,000-ticket derived snapshot [PICK for First Move]: tiny (0.4 MB), already carries the .pi program library — proves programs/matches/patterns/shape today with zero derive step. (B) full 431K raw [PICK for v1 scale]: the real thing, but 1.45 GB unpacked (needs ~2–4 GB Cloud Run memory) and programs must be derived across it first. Consequence: A gives the whole experience on real Mendo data in one small deploy; B is the immediate scale-up once the shape is proven. Ship A, then B — not either/or.
Fork 2 — One service or reuse onedine-raw. A dedicated zendesk-raw [PICK] vs. generalizing onedine-raw to multi-source. Consequence: one-store-per-service is the proven pattern (onedine-raw already is exactly this), keeps mmap simple and the ticket:/case: prefixes clean, and lets the two corpora scale + fail independently. Generalizing later is easy; coupling now is not worth it.
Fork 3 — Derive programs: full corpus vs on-demand. Pre-derive the .pi library across all 431K (batch, deterministic, $0) vs derive per-ticket on replay. Consequence: pre-derive makes /shape and /similar corpus-wide and instant; on-demand keeps deploy light but can't answer "shape of the whole." Since "shape of support" is an explicit ask, pre-derive is required for the full tier — boarded in v1-scale, not First Move.

7. Escape routes

  • zendesk-raw is additive — a new service; if wrong, delete it, nothing else depends on it. onedine-raw untouched. Fully reversible.
  • Source-aware /ask: defaults preserved; source=zendesk is opt-in. Revert = one line. Reversible.
  • The sealed extract is never mutated — read-only mmap; nothing to roll back on the data side. ⚠ none.

8. First move smallest proof it's real

Stand up zendesk-raw over the 1,000-ticket derived snapshot (it already has the .pi programs) with two endpoints — /search and /replay?ticket= — and point /ask?source=zendesk at it. Then:

DM Argus:  "find billing complaints and show me the program"
  -> real MENDO ZENDESK ticket ids + the .pi program under each,
     answered by Gemini, grounded on THIS corpus.

Why this first: it flips Argus to the correct dataset and proves the two hardest claims — find on Zendesk data and see the underlying program — in one small deploy, reusing the raw2 pattern already in production. Matches / patterns / shape are the same jacket over engines already written; the full-431K scale-up follows once the shape is proven on the small tier.
