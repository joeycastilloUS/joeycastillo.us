Design — Sandbox NPS-Recovery Loop Simulation

Close the NPS-detractor guest-recovery loop end-to-end against the brand’s Zendesk sandbox: a sim .pi plays the guest, pirun runs the recovery card, the sandbox sends a real email, Cloudflare routes it, TLT byte-matches it. Prod stays disarmed.

“we should be able to build one or two more .pi to handle sending in the nps as the guest.. also use the mesh and purcius skills to service the email loop in this simulation.. let’s use the-last-test.com” — jc, 2026-07-14

1. Need

The recovery is modeled but nothing executes. We proved the practice against LIVE Zendesk (244 “Follow Up On Your Mendo Feedback” tickets, agent Tyler’s fixed skeleton reproduced byte-identical, guest reply-back receipt on #456251) and we proved the mechanism (the alert’s requester is the NPS system account 49995154539796 — the send lane is a new follow-up ticket, requester = guest, public comment). The send lane is even built (zendesk-bridge ReplySend FollowUpTicket mode, sandbox creds wired). What’s missing is the loop closed with a receipt: no ticket has ever been raised by us, no recovery email has ever left the sandbox, and nothing catches the email on the far side to prove the body is right. Until one detractor round-trips — sim guest in, recovery email out, captured, byte-matched — “it works” is a claim, not a receipt. jc hurts because he cannot arm prod on a claim. Why now: the sandbox exists, the skeleton is proven, pirun is real, and the-last-test.com is sitting there waiting to catch mail.

2. Raw User Stories

  • As jc, I want a .pi that submits the NPS as the guest — raising a real Guest Experience Alert detractor ticket in the sandbox — so the recovery has a real trigger without waiting on a real unhappy guest.
  • As jc, I want the recovery to run as a pirun card — hold at preview with every slot editable, stop AWAIT_FIRE at the send — so nothing emails a guest without my finger on the trigger.
  • As jc, I want the guest’s address to be one we control on the-last-test.com, so the sandbox’s real outbound email lands where we can read it — and forwarded to my inbox so I see it too.
  • As jc, I want TLT to pull the captured email and byte-match it against the fixed skeleton with a real claimed code, so “the email is correct” is an exit code, not an opinion.
  • As jc, I want feedback that signals food-borne illness to end the run with escalate-only — no draft, no code, no email — proven by a sim ticket that says “I got sick.”
  • As jc, I want every step on the run ledger (IDS triples) so I can audit who wrote what, edit a slot mid-run, and replay the whole thing later.

Proposed approach: Invent almost nothing. The email leg is pull-based — the dev box takes no inbound (Rule 13: the home router is not an edge).

Part 1 — submit-nps.pi (sim card, pirun protocol[])

name = submit-nps          # plays the guest + the NPS survey system
env  = sandbox

protocol[]:
  { unit: compose-alert,  skill: compose-alert,
    in:  [sim.nps, sim.store, sim.feedback, sim.guest_email],
    out: [alert.subject, alert.body],           gate: auto, fail: abort }
  { unit: raise-alert,    skill: zendesk-alert-create,     # /zendesk/alert-create, sandbox-armed
    in:  [alert.subject, alert.body, sim.guest_email],
    out: [alert.ticket_id],                     gate: hold, fail: abort }   # operator sees the ticket id
  # optional second act, run AFTER recovery lands:
  { unit: guest-replies,  skill: zendesk-guest-reply,      # mirrors the #456251 thank-you receipt
    in:  [followup.ticket_id],
    out: [reply.comment_id],                    gate: hold, fail: continue }

Seed data (sim.*) is written to the run ledger at pirun_start — nps=2, a real-shaped complaint, store=“Santa Barbara”, guest_email=guest@the-last-test.com. A second seed variant carries an illness phrase to prove the hard exclusion.

Part 2 — recover-guest.pi (the recovery, re-expressed as a pirun card)

The prose intent (rules, verdicts, slots, voice, vault law) stays in the file header as operator canon; the protocol[] below is what pirun_load parses. Every verdict maps to a gate.

name = recover-guest
env  = sandbox

protocol[]:
  { unit: read-alert,        skill: zendesk-read,         # bridge /read, env=sandbox
    in:  [alert.ticket_id],
    out: [alert.nps, alert.feedback, alert.store, alert.guest_email, alert.event],
    gate: auto, fail: abort }

  { unit: understand-feedback, skill: understand,          # nous C99; cannot hallucinate
    in:  [alert.feedback],
    out: [complaint.category, illness.flag],
    gate: auto, fail: abort }

  { unit: exclude-illness,   skill: gate-illness,          # writes run.exit=true on illness,
    in:  [illness.flag],                                   # records the escalation triple first
    out: [run.exit, escalate.note],
    gate: exit, fail: abort }                              # exit gate: "true" ends run — verdict escalate-only

  { unit: skip-non-detractor, skill: gate-detractor,       # nps > 6 or wrong event or already
    in:  [alert.nps, alert.event, alert.ticket_id],        # handled (SentDrafts lookup) → run.exit=true
    out: [run.exit, skip.reason],
    gate: exit, fail: abort }                              # verdict skip

  { unit: claim-code,        skill: vault-claim,           # gcp-bridge Firestore, atomic,
    in:  [alert.ticket_id],                                # idempotent per ticket
    out: [code.value, code.amount],
    gate: auto, fail: hold }                               # vault empty → HOLD, operator refills

  { unit: generate-reply,    skill: generate-reply,        # fixed skeleton + slots + 2 sense ¶s
    in:  [alert.guest_email, alert.store, alert.feedback,
          complaint.category, code.value, code.amount],
    out: [email.subject, email.body, slot.name],
    gate: hold, fail: abort }                              # verdict draft-and-hold — operator preview;
                                                           # every slot editable via pirun_verb edit
  { unit: send-followup,     skill: zendesk-followup-send, # bridge FollowUpTicket: NEW ticket,
    in:  [alert.ticket_id, alert.guest_email,              # requester=guest, PUBLIC comment,
          email.subject, email.body],                      # idempotent draft_id=recovery-sandbox-<id>,
    out: [followup.ticket_id],                             # tag argus_recovery_sent, capture-before
    gate: fire, fail: hold }                               # verdict fire — DISARMED, stops AWAIT_FIRE

Slot law rides in the skills, not the flow: generate-reply resolves name (ZD profile → email local-part → “there”, suppressing the closing name on “there”), keeps the amount identical in offer and fine print, renders _**{code}**_, appends the original alert below the * * * rule, and never paraphrases the fixed copy. The food-illness exclusion is a gate, upstream of any draft — structurally incapable of emailing an ill guest.

Part 3 — verify-loop.pi (the receipt)

name = verify-loop
env  = sandbox

protocol[]:
  { unit: pull-capture,  skill: cloudflare-kv-read,        # bridge /read → EMAIL_CAPTURE ns;
    in:  [alert.guest_email],                              # poll with backoff, mail is not instant
    out: [captured.raw],                        gate: auto, fail: hold }
  { unit: byte-match,    skill: email-match,               # C99: decode quoted-printable, strip
    in:  [captured.raw, email.body, code.value],           # transport headers, compare skeleton+slots,
    out: [match.result, match.diff],                       # assert claimed code present exactly once
    gate: hold, fail: abort }                              # operator reads the verdict + any diff

tlt email-check is the same C99 matcher exposed as a TLT check for CI use — one matcher, two callers.

Part 4 — bridge gaps to close (foreign edges, C# jackets)

  • zendesk-bridge: per-env WRITE-ARM (ZENDESK_WRITE_ARM=sandbox; write lanes hard-refuse any env not named — prod is never named) + the /zendesk/alert-create lane. FollowUpTicket send lane already exists.
  • cloudflare-bridge: /cloudflare/email-route — gated write modeled on /cloudflare/dns: allow-listed zone (the-last-test.com only), body built in-process, never free-form. Creates: Email Routing enablement (MX), destination-address entry for jc’s inbox (Cloudflare emails jc a verification link — the one human step, jc already volunteered: “i can forward email based on what you want”), and the rule guest@the-last-test.com → email-capture worker + forward to jc. Plus a KV read lane on the EMAIL_CAPTURE namespace.
  • email-capture worker: the one piece of code on the foreign edge. It does exactly one thing — message → KV.put(raw MIME) — zero logic, so it lives in the same moral category as a jacket: foreign surface, minimal, dumb.

Part 5 — argus-run (C99 pirun host)

Loads a card (pirun_load), starts a run (pirun_start), polls pirun_advance, and surfaces the operator verbs. Its PIRUN_SKILLS[] table compiles in: understand, gate-illness, gate-detractor, generate-reply, email-match (pure C99) and the bridge shims zendesk-read, zendesk-alert-create, zendesk-followup-send, vault-claim, cloudflare-kv-read (each shim shells the respective jacket — the only HTTP speakers). A skill not yet in the table slice-ends cleanly, so the cards run honestly from day one even if a shim lands late.

Out of scope:

  • Prod arming, prod writes, prod anything. Only jc fires prod, and not in this design.
  • Webhook auto-trigger on real NPS events (v2 of recover-guest.pi — still operator-approved).
  • The real NPS survey UI, Sheet-sync automation beyond seeding the sandbox vault, coupon redemption tracking.
  • An armed fire verb in pirun itself — today fire = operator runs the send explicitly against the AWAIT_FIRE run; the runtime keeps the gate visible and disarmed.

6. First Move

One ticket, all the way around: wire the email leg (/cloudflare/email-route mints guest@the-last-test.com → email-capture worker + forward to jc; jc clicks the verification link) — then run submit-nps.pi once to raise a single detractor alert in the sandbox, run recover-guest.pi to AWAIT_FIRE, jc fires, the sandbox emails the guest, and verify-loop.pi pulls the capture and byte-matches skeleton + claimed IMSORRYTEST code. One green match.result proves every leg of the design at once.

These go into the plan: WRITE-ARM flag + alert-create lane (zendesk-bridge), email-route write + KV read (cloudflare-bridge), email-capture worker, sandbox vault seed, argus-run host + the five C99 skills + five bridge shims, the three cards, tlt email-check, and the illness-exclusion sim variant as the second run.
