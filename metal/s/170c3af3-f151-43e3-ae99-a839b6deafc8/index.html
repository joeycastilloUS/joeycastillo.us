Raw IDS Explorer: drill into the triples

A cool little data explorer for the raw. Pick a thing — a case, a ticket — see its triples, follow the links. Source-agnostic, keyboard-drillable, read-only. A triple browser, not a BI dashboard.

The EXPLORE half of the Raw IDS tab. The IMPORT half (Seed / Refresh / Replay, the /import/* contract) is being built concurrently per 2026-07-09-design-salesforce-raw-ids-seed-fable5 — that design is untouched here. See the split, §5.

1. Need

The operator has 634,907 triples sealed in the Raw IDS — 42,402 OneDine cases + 16,385 comments, landed and proven — and cannot see any of them. The raw is the source of record the whole system replays from, and today it is a 50 MB opaque blob in GCS. Trust-by-replay exists; trust-by-eye does not. Why now: the seed landed, the import tab is going up in a sibling session, and the Zendesk/NPS raw follows the same a-d-d §7 contract — the moment there are two sealed raws, an explorer that serves both is the difference between 'sealed' and 'visible'. The operator's words: "make the raw more viewable, let me drill in."

2. Raw user stories operator voice

  • "As the operator, I want to open one real case and see its triples — predicate, object, grouped — so that I trust the raw by eye, not by claim."
  • "As the operator, I want to click an object that is itself a subject (a case's comment, a comment's case) and hop to it, so drilling feels like browsing a graph, not grepping a file."
  • "As the operator, I want to search for a value — a case number, a store name, a word in a body — and land on the subjects that carry it."
  • "As the operator, I want to see provenance on every triple — source, batch, time — so I know which import wrote it."
  • "As the operator, I want the same explorer to work on the Zendesk/NPS raw the day it lands, so I never build a second viewer."
  • "As the operator, I want to drive it from the keyboard — move, drill, pop, search — because fast beats pretty."

4. Name it true literal

The skill: rawq.c (C99, nous store, read-only — it queries the raw, so it's rawq). The endpoints on the bridge:

EndpointDoes
GET /raw/subjects?source=&type=&after=&limit=Paged subject list (id, type, one label triple). Cursor paging via after.
GET /raw/subject?id=&source=All triples for one subject, grouped by predicate, provenance per triple, and a links flag on objects that are themselves subjects.
GET /raw/search?q=&field=&source=&limit=Substring match across subjects / predicates / objects → list of hit subjects with the matching triple.

The views: Subjects · Subject (the drill card) · Search. Collision flagged: the import half owns /import/replay?case= — replay means decode to PI atoms, proof by eyes. The drill card is NOT replay and is never called replay: it shows triples as sealed, no decode. Two words, two meanings, kept apart on purpose — the drill card links out to replay for the decoded view.

The split, stated so two sessions converge on one tab: the import half writes the raw and owns /import/* (Seed / Refresh / Replay + counts). The explorer half reads the raw and owns /raw/* (Subjects / Subject / Search). Same tab because they're two verbs on the same sealed store — one place to trust the raw. Neither touches the other's contract; the only seam is the drill card's "view decoded" link into their replay.

Source-agnostic boundary: rawq opens a sealed store, not the Salesforce store — source= picks which sealed raw (salesforce today, zendesk the day it lands). One explorer, N raws; the a-d-d §7 shape is the contract that makes this free.

Language law: query layer is C99 (rawq.c on nous store.c), served through the existing C# bridge jacket, rendered in vanilla JS in the console. Python appears nowhere.

Out of scope, v1: no editing the raw (read-only, ever — the raw is sealed); no re-decode inside the explorer (replay owns decode); no charts, counts-over-time, or BI; no cross-source joins.

6. First move what ships first

One endpoint, one card, one hop. GET /raw/subject?id=<a real OneDine case> — rawq opens the sealed store, ids_query(subject, 0, 0), returns that case's triples grouped by predicate with provenance — rendered as the drill card in the Raw IDS tab, with one working follow-link (case → one of its comments, and back). That single vertical proves the whole thing: C99 query, jacket serving, console drill, graph hop. These go into the plan: /raw/subjects paging, /raw/search, the zendesk source, the provenance expand, full keyboard nav.

Choices + consequences earned — the fork

Fork: query the sealed IDS live server-side vs pre-index a static browse manifest. A manifest (pre-baked JSON per subject, served static) needs no runtime — but it is a second derived artifact that goes stale on every Refresh and re-plants the exact drift the sealed-raw contract kills. Chose live server-side: rawq behind the bridge, store kept warm (50 MB in memory — trivial), always current the instant an import batch seals. Consequence: the bridge must host and call a C99 binary; that seam already exists in the jacket-over-metal pattern, and it's the same seam the zendesk source reuses for free.

Placement, resolved by the same logic: explorer-in-the-Raw-IDS-tab, not a standalone tab — write and read on one store belong on one screen. If the explorer outgrows the tab (multi-source browsing as its own activity), it lifts out along the /raw/* seam untouched; that's the escape route, and the only one this design has earned.

Definition of Done — this design

  • Need grounded in landed fact: 634,907 sealed triples, zero of them visible
  • Coordinated with the concurrent import-half design (8cbac5f0) — split stated, contracts disjoint (/import/* theirs, /raw/* ours), replay-vs-drill collision named
  • Sourced from owned metal: nous store.c query, bridge jacket, tab pattern — only rawq + /raw/* + the drill UI are new
  • Language law holds: C99 query, C# jacket only, vanilla JS surface, no Python anywhere
  • Source-agnostic by contract (a-d-d §7 shape), read-only by law
  • First move is one provable vertical: one endpoint, one card, one hop
