Plan — Sign in with Zendesk

Plan — Awaiting Approval. Policy (A) agent-of-subdomain is BLESSED (operator, 2026-07-09). Grows from design 2026-07-08-2151-design-zendesk-user-login-fable5.html + inspect 2026-07-08-2151-inspect-zendesk-user-login.html.
Cost reality (verified, not assumed): bridge/providers.json literally says “Adding a door = one row.” Both live doors are idToken:true; the callback (bridge/Program.cs:231-250) hard-reads the id_token JWT today. So the entire new surface is: one provider row (idToken:false + userinfoUrl), one userinfo branch in the callback, one pure policy resolver, tests, and a button. Everything else — session, member/session persist, CSRF state, the sign-in row — already exists.

Intent

Let Mendo's Zendesk agents sign in to Argus as their Zendesk identity — being a current agent/admin in Mendo's instance is the credential. Kill per-head operator toil and the stale Crew list for this tenant, make offboarding automatic (Zendesk suspension kills the next login), and stamp every session with a verified Zendesk identity so a fired NPS reply has real attribution. Build Layer 2 (tenant-agent) without touching Layer 1 (the sovereign Purcius/overlord spine).

Scope

In: a zendesk provider row (idToken:false, per-subdomain authorize/token/userinfo URLs, scope minimized against the live API); the userinfo branch in the callback (opaque token → GET /api/v2/users/me.json → normalize, handling Zendesk's {"user":{...}} wrapper); the pure tenant-agent policy resolver ZendeskAgentAllowed(userinfo, policy) (roles agent|admin in the tenant subdomain, end-user denied, fail-closed) with bare-assert tests; the zendeskAccess field on tenants/{slug}; role + subdomain stamped on members/ + sessions/; one sign-in button + role in the member chip; denial UX; the OAuth client registered against <fqdn>/auth/zendesk/callback.

Out (v1): role-tiered capabilities inside Argus (role is stamped, grants nothing); group-scoped access (escape route, not built); storing/refreshing Zendesk tokens or ANY Zendesk write on behalf of an agent (the no-live-write law stands — the token is used once for identity and discarded); a second tenant's instance (per-tenant data makes it a doc + a client registration, but v1 wires exactly Mendo); any Purcius/overlord change; the session-cookie payload shape (stays {sid}|{identityKey}|{email} — role lives in Firestore, not the signed cookie).

Approach

  1. Register the door (operator + Be). Operator creates one OAuth client in Mendo's Zendesk Admin (redirect https://<fqdn>/auth/zendesk/callback), pastes id+secret; Be installs via setx ZENDESK_CLIENT_ID/SECRET and verifies (Rule 4). This is the one gate that needs the operator's Zendesk Admin.
  2. Add the provider row. zendesk in providers.json: per-subdomain authorizeUrl/tokenUrl/userinfoUrl, scopes:"read" (minimize), idToken:false, map:{sub:"id",name:"name",email:"email",role:"role"}.
  3. Fork the callback on idToken. When false: instead of decoding a JWT, call userinfoUrl with the access token, unwrap {"user":{...}}, map claims. The token is used once and discarded — Argus never stores it; data reads keep flowing through zendesk-bridge's own credential.
  4. Add the policy resolver. Pure ZendeskAgentAllowed(userinfo, policy) beside EmailAllowed; fail-closed; bare-assert suite (agent→in, admin→in, end-user→denied, wrong-subdomain→denied, malformed→denied). Persist role+subdomain; mint the same argus_session.
  5. Front-end. One button in the sign-in row; showMemberChip() shows “Name · role”; clean denial copy for a rejected end-user.
  6. Prove it (First Move). One real Mendo agent round-trips in; one end-user is denied; every write path still operator-gated; zero writes fired.

Constraints

  • Language law: all new code is inside the existing C# jacket — this IS the foreign-API (Zendesk OAuth) boundary. Nothing we own changes language; no new service.
  • No live writes: the Zendesk access token is used once for identity and discarded. Argus never writes to Zendesk on an agent's behalf; gated operator-approved writes are the only write lane, unchanged.
  • Auth home: callback on <fqdn>/auth/zendesk/callback via the Cloudflare Pages Function proxy (Operator-Identity canon); Cloud Run custom domains stay banned; no *.run.app in the Zendesk client registration.
  • Layer 1 untouched: zero changes to Purcius key / overlord / PURL. Session cookie shape unchanged.
  • Fail-closed everywhere: unknown provider 404s; malformed userinfo denied; end-user denied; policy default deny.

Unknowns

  • Zendesk userinfo shape — GET /api/v2/users/me.json returns {"user":{id,name,email,role,...}} (wrapped). The current map reads top-level claims; the userinfo branch must unwrap user first. Confirm the exact envelope + the role enum values (end-user/agent/admin) against the live API — read it, don't guess (Rule 6).
  • Per-subdomain authorize URL — Zendesk's authorize/token/userinfo hosts are https://<subdomain>.zendesk.com/..., not a static host like Google/Slack. Decide: bake Mendo's subdomain into the row (simplest, v1) vs template it from the tenant doc (needed at second tenant). Lean bake-in now, template when tenant #2 arrives.
  • OAuth scope minimization — smallest scope that still returns role (likely read; is there a narrower users:read?). Verify against the live client, not the docs alone.
  • Token endpoint auth — Zendesk expects client_id/client_secret in the body for the code exchange; confirm the exact grant_type=authorization_code params match the generic exchange at Program.cs:226-243 or need a per-provider tweak.
  • Chip role display — whether operators want role shown to agents or kept internal; trivial either way, but a UX call.

When To Revisit (v2 triggers)

  • Second tenant appears — templatize the per-subdomain URLs from the tenant doc; the bake-in becomes data.
  • A feature needs role tiers — the role stamp is already persisted; add the capability map then, not before (avoid inventing permissions ad hoc).
  • Contractor-role concern proves real — if agent-of-subdomain lets in someone it shouldn't, tighten allowRoles to ["admin"] or add allowGroups (escape to fork C) — data-only edits.
  • Attribution consumed downstream — once fired replies carry the Zendesk identity, revisit whether the audit trail wants group/team context too.
  • v2 auth hardening lands — align with the existing JWKS/id_token-signature work; the userinfo branch has no JWT to verify but should ride the same token-hardening review.
