Design — Sign in with Zendesk

A fourth login door for Argus: Mendo's Zendesk agents authenticate as their Zendesk identity. One new provider row, one non-OIDC branch, one access policy. Everything else is already built.

Rule 11 — the fork, resolved up front: Argus auth is TWO LAYERS, not one. The sovereign-operator spine (Purcius key, Overlord auth, one operator per FQDN) is Layer 1 and is untouched. “Zendesk users log in” is Layer 2 — tenant-agent identity, many Mendo agents, scoped and read-mostly. This design builds Layer 2 as the credential the 2026-06-29 tenant-access model already anticipated. No drift; the layering is named in §Shape.

1 · Need

Mendo's support agents are the people Argus serves — the NPS boards, the Focus tab, the fired-reply audit trail are all about their tickets — yet not one of them can get in without the operator hand-adding their email to the Crew Allow-list. That's operator toil per head, a stale list the day someone joins or leaves, and zero identity attribution when an agent-facing action fires. Being an agent in Mendo's Zendesk instance should be the credential: Zendesk already knows who is a real agent, what their role is, and suspends them the day they leave. Why now: zendesk-bridge is live, the OAuth flow is built, and NPS attribution (tying a recovery reply to a verified agent) is the next thing the Focus work needs.

2 · Raw user stories

As a Mendo support agent, I want to click “Sign in with Zendesk” and be in Argus as myself, so that I don't have to ask anyone to add my email first.
As the operator, I want any current agent/admin in Mendo's Zendesk to get read access automatically — and lose it when Zendesk suspends them — so that I never maintain a per-person list for this tenant again.
As the operator, I want writes (arming a Zora reply, changing config) to stay operator-gated regardless of who signed in, so that a new door never widens the blast radius.
As an auditor of a fired reply, I want the session that triggered it stamped with a verified Zendesk identity (id, role), so that attribution is a fact, not a guess.
As a Zendesk end-user (a customer, not an agent), I want nothing — and the design wants me hard-denied, so that customer accounts can never open Argus.

The only NEW code (all inside the existing C# jacket — this IS the foreign-API boundary, per the language law; nothing we own changes language):

  1. The userinfo branch. Zendesk returns an opaque access token, not an OIDC id_token. When a provider row says "idToken": false, the callback makes one extra call — GET /api/v2/users/me.json with the token — and normalizes identity from that response (id, name, email, role) via the same map mechanism.
  2. The tenant-agent access policy. A pure function beside EmailAllowed: given the /users/me result and the tenant's policy doc, allow role ∈ {agent, admin} in the configured subdomain; deny end-user; fail-closed on anything else.

Outside-world check: Zendesk's Authorization Code flow (/oauth/authorizations/new → POST /oauth/tokens, scope read) is standard and confirmed in the 2026-07-08 inspect. Scope minimization (users:read vs read) is verified against the live API in the plan, not guessed here.

5 · Shape

The two-layer model — the load-bearing structure of this design:

LayerWhoCredentialAccess rulePowers
1 · Sovereign operatorJoey / overlordPurcius device key (PURL board) · legacy password until retiredOverlord listEverything: config, arming writes, ceremony. Untouched by this design.
2 · Tenant agent NEWMendo Zendesk agents/adminsSign in with Zendesk (this design) · Google/Slack OIDC via Crew listtenant-agent policy: role ∈ allowRoles in the tenant's subdomain, fail-closedRead-mostly Argus. Gated writes remain operator-approved. Role stamped for attribution.

Parts and boundaries:

  • providers.json — zendesk row: authorizeUrl = https://<subdomain>.zendesk.com/oauth/authorizations/new, tokenUrl = .../oauth/tokens, scope = read (minimize in plan), idToken: false, userinfoUrl = .../api/v2/users/me.json, map = { sub:"id", name:"name", email:"email", role:"role" }.
  • Callback (C# jacket) — existing flow + userinfo branch: exchange code → opaque token → GET /users/me → normalize → tenant-agent policy → persist member (with role, subdomain) → mint the same argus_session. Token is used once for identity and discarded — Argus never stores a Zendesk access token; data reads keep flowing through zendesk-bridge's own credential.
  • Session shape unchanged — payload stays {sid}|{identityKey}|{email}. Role lives in the Firestore session/member docs, not the signed cookie (the tenant-model design flagged cookie-shape changes as the one expensive reversal; we don't take that cost).
  • Auth URL home — the callback registers as https://<fqdn>/auth/zendesk/callback, fronted by the Cloudflare Pages Function proxy per Operator-Identity canon (AUTH_REDIRECT_BASE already supports this; Cloud Run custom domains stay banned; no *.run.app in the Zendesk client registration).
  • Front-end — one button in the sign-in row; showMemberChip() gains role display (“Ana · agent”).

Explicitly out of scope (v1):

  • Role-tiered capabilities (admin vs agent inside Argus) — role stamp only.
  • Group-scoped access (specific Zendesk groups) — named escape route, not built.
  • Storing/refreshing Zendesk tokens, or any Zendesk write on behalf of the agent — never in this design; the no-live-write law stands.
  • Other tenants' Zendesk instances — the policy is per-tenant data, so a second tenant is a doc + a client registration, but v1 configures exactly one (Mendo).
  • Purcius/overlord changes of any kind.

6 · Choices + Consequences

Access-policy fork2nd order3rd order
(A) Agent-of-subdomain — any agent/admin in Mendo's instance is in; end-user deniedZero operator toil; offboarding is automatic (Zendesk suspension kills login — the next OAuth round-trip fails). New hires get in day one.Access truth delegates to Mendo's Zendesk admin hygiene: a contractor granted agent role gets Argus too. Acceptable — they see tickets in Zendesk already, Argus shows derivatives of the same data.
(B) Explicit allow-list — Zendesk login but Crew emails still gateKeeps the exact toil this design exists to remove; the door adds nothing over Google OIDC.Stale lists, day-one lockouts, operator becomes the help desk for the help desk.
(C) Group-scoped — only named Zendesk groups get inPrecise, but imports Mendo's group taxonomy as config Argus must track.Silent lockouts every time a Zendesk admin reshuffles groups; debugging lands on the operator.

Pick: (A) agent-of-subdomain, roles agent + admin, fail-closed. One line why: being a Mendo agent is exactly the population Argus serves, and Zendesk already does the joiner/leaver work — (B) re-creates the pain, (C) buys precision v1 doesn't need. The gated-write rule makes (A) safe: login grants eyes, never hands.

8 · First Move

The single smallest step that proves the design — one real round-trip, zero writes armed:

  1. Register one OAuth client in Mendo's Zendesk Admin (redirect: https://<fqdn>/auth/zendesk/callback); operator pastes id + secret, Be installs (setx, Rule 4).
  2. Add the zendesk provider row + the userinfo branch + ZendeskAgentAllowed (with bare asserts: agent→in, admin→in, end-user→denied, wrong-subdomain→denied, malformed→denied).
  3. One real Mendo agent clicks the button and lands in Argus: member chip shows their name + role, members/ doc carries zendesk:<subdomain>:<id> + role stamp, and every write path is still operator-gated. One end-user account tries and gets the clean denial.

If that round-trip lands, the design is real. These go into the plan: scope minimization against the live API (read vs narrower), the sign-in button + chip role display, denial UX copy, attribution wiring into the fired-reply audit trail, the deferred role→capability map, second-tenant generalization, and JWKS/token-hardening alignment with the existing v2 auth list.
