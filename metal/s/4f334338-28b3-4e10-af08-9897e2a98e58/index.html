What It Is

“Sign in with Purcius key” is operator-identity auth, pure‑C99, DNS-anchored, no third party. Your operator root key (the same a61bdace… that already signs the mesh) warrants a browser-minted device key into the console. The device signs a server challenge; the server checks the device sig and your root’s warrant over that device; if both hold it mints the same argus_session cookie the Slack/Google logins use. “Signing the mesh” and “signing you in” are the same ceremony — the root key certifies members, whether that member is a mesh node or your laptop.

Patterns Found

  • Membership, not PKI (PRACTICES #6) — a warrant is a signed roster entry (device_pub‖"console"), not an X.509 chain. Verifiers pin the root; they don’t walk a chain.
  • Pure-C auth, C# only as a foreign jacket (PRACTICES #12 / Language Boundary) — every crypto op is the C sidecar; the C# bridge only speaks the PIW1 wire. Red C test fails the Docker build.
  • Adopt, never mint — Registration publishes a61bdace’s pubkey; it never forks a second root. Minting is the net-new-operator branch only.
  • The domain is the identity — _key.<fqdn> makes the root fetchable + auditable by anyone; recovery floor is FQDN re-proof.
  • Fail-closed activation — no secrets ⇒ /auth/key/* returns 503 key-login-not-configured ⇒ the button’s keybar stays hidden. Nothing half-open.
