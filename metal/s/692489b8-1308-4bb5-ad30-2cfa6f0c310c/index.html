Sandbox↔Prod environment selector + Compare

The confirmation gate: see which Zendesk environment Argus is pointed at, and prove sandbox ≠ prod, before any simulated send. Registry-driven — scales horizontally to N environments. Chain: Fable5 ~d~p~e → Opus ~b~t~s.

What was built

  • Environment registry (ENVIRONMENTS[]) — the horizontal-scale seam jc asked for. Each entry is {id,label,subdomain,tone}; add one to add an environment. The selector, badge, env-aware reads, deep links, and Compare are ALL driven off this list — zero code change to scale. Today: prod + sandbox.
  • Selector + amber badge in the nav — a per-registry-entry dropdown that sticks (localStorage argus_env). Any non-prod environment lights an always-on-screen AMBER badge + a whole-frame amber ring. Prod is neutral.
  • Env-aware reads at one choke point — zdFetch appends &env=<id> from ENV.param(), so all ~15 read call-sites route to the selected environment. Prod appends nothing (bridge default) — no regression for the default case.
  • Per-env deep links — ZD_ADMIN resolves the selected environment's subdomain; the ZD_DEEP builders follow it.
  • Compare panel — fingerprints every registered environment side by side (subdomain, ticket count, newest ticket #/subject/date) and returns a verdict: ISOLATED ✓ only when every env answers AND their live fingerprints are pairwise distinct; otherwise ⚠ NOT isolated (misrouted creds / same tenant). Empirical proof, not a declared label.
