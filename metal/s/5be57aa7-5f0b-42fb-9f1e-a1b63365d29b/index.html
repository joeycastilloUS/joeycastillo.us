~d — De-atomize POC: conversation from programs

"Now that we have this massive amount of data… put it up in memory ALL and run all kinds of things on it… conversation from programs… let's de-atomize. Give me a small POC." — Joey, 2026-07-09

1. Need

The full Mendocino Farms Zendesk corpus is sealed and mesh-resident: 431,260 tickets · 977,029 comments · 8,247,494 triples · 4,610,372 unique strings · 563 MB (gs://acmedev-mendo-tickets-ids/full/, IDS handle 8963900c…). Every comment is a first-class comment:<id> subject with the public/internal flag. Every ticket-thread is a program (a-d-d §6, conversation-as-program).

We proved the reduce direction this session: understand() → 43 atoms on a real ticket. The decoder is bidirectional — restate() goes the other way — but we have never run the expand direction against this corpus. What's missing is proof that a program reconstitutes to conversation (de-atomize), and the payoff proof: one canonical conversation synthesized from a cluster of real programs. Why now: the corpus just became real, the store is mmap-mountable, and both primitives already exist. Zero new theory — one small harness.

2. Raw User Stories

  • As the operator, I want the whole 563 MB store mounted in memory so passes over all 431K programs run at memory speed, not I/O speed.
  • As the operator, I want one real ticket-thread de-atomized back into a readable conversation, side by side with the original, so I can see the program reconstitute — deterministically, no model.
  • As the operator, I want one canonical conversation synthesized from the common structure of N similar tickets — "this is how support actually handles X" — from real programs, not a summary hallucinated by a model.
  • As the operator, I want this as a small read-only C99 tool so I can run cheap corpus-wide passes next, without touching the sealed source.

5. Shape

One C99 file, deatomize_poc.c, read-only against the sealed store. Prints to stdout.

  1. Mount — open data/full/raw-ids-full the way derive.c does; the 563 MB store is now RAM-speed.
  2. Move 1 — round-trip one thread — pick one real ticket, pull its comment:<id> facts (public flag respected), call restate(facts, nfacts, store, focus=ticket, INTENT_TIMELINE, out). Print original comments vs de-atomized restatement side by side.
  3. Move 2 — conversation from programs — take N=5 tickets sharing a group/type, pull each program's facts, keep the facts common across the cluster (predicate+object intersection), call restate(common, …, INTENT_DESCRIBE, out). Print the ONE canonical conversation.

Out of scope (YAGNI)

  • ✗ Argus UI wiring — POC prints to stdout (a small HTML dump at most).
  • ✗ Model-based generation — restate() is deterministic; no model anywhere.
  • ✗ Full situation-classifier — cluster pick is "same group/type," nothing smarter.
  • ✗ Any write path — read-only; the sealed .ids is never touched. No network.

Choices — reconstruct vs synthesize

Move 1 reconstructs the actual thread (all of one program's facts, INTENT_TIMELINE — fidelity is the test). Move 2 synthesizes a canonical one (only the facts N programs share, INTENT_DESCRIBE — generality is the test). Different outputs, same primitive, one parameter apart. The POC does both, small — together they bracket what de-atomize can do.

6. First Move

Compile a ~60-line deatomize_poc.c: mount data/full/raw-ids-full, pull one ticket's comment facts, call restate(…, INTENT_TIMELINE, …), print original vs de-atomized. One ticket, one round-trip, off the store we already have. Move 2 (cluster → canonical) follows only after move 1 reads clean.

TripleStore store;              /* mmap the sealed .ids — all in RAM   */
Fact facts[MAX_FACTS]; int n;   /* one ticket's comment facts          */
char focus[1][MAX_ENTITY];      /* "ticket:<id>"                       */
char out[8192];
restate(facts, n, &store, focus, 1, INTENT_TIMELINE, out, sizeof out);
printf("--- original ---\n%s\n--- de-atomized ---\n%s\n", orig, out);
