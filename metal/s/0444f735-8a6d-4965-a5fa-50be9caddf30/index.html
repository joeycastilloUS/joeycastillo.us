Salesforce → Raw IDS: the OneDine seed

Model the blessed Zendesk raw-IDS import for the Salesforce source. Land ALL of OneDine's cases in a sealed Raw IDS — then replay and decode forever as code or intentions change. Salesforce untouched after the seed.

Mirrors 2026-07-09-1443-design-zendesk-raw-ids-import-tool-fable5. Difference that matters: the Zendesk import kept not landing — the Salesforce seed is already landed and proven (below). This design gives it the same operational home.

1. Need

The operator wants all of OneDine's Salesforce cases in a sealed Raw IDS so the system can replay from it forever — the same blessed pattern as Zendesk (ARCHITECTURE §7, operator-directed 2026-07-04: "import once to raw sealed IDS… then replay on the raw and always refresh/get-latest").

Unlike Zendesk, the Salesforce seed has landed, proven live 2026-07-09:

42,402 cases + 16,385 comments → 634,907 triples → sealed nous Raw IDS, 182 rate-safe API calls, round-trip + understand() decode clean. Raw at gs://acmedev-onedine-cases-ids/full/ (JSONL source-of-record + sealed raw-ids.tgz, 50 MB). Tools at argus/src/import/salesforce/.

What's still missing is exactly what stalled Zendesk: no operational home (it's a mesh job, not a console tab — no button, no progress, no resume surface), no refresh loop (get-latest since the cursor), and no in-console replay (proof by eyes for one real case).

2. Raw user stories operator voice

  • "Seed all of OneDine's case history into raw triples — I don't want to re-pull from Salesforce every time."
  • "When I change the decoder or the intentions, replay the raw — don't touch the live org."
  • "Refresh: pull only what's new since the last seed."
  • "Show me one real case replayed from the raw — proof by eyes, not a claim."
  • "Do it from the Argus console, watchable and resumable — not a hand-run job in a side directory."

6. First move what ships first

Already shipped: the full seed landed + proven (§1). Next First Move: the Raw IDS tab — a first tab in the Argus Salesforce app that binds only the /import/* contract: Seed / Refresh / Replay, live counts (cases · comments · triples · elapsed), and a one-case replay rendered in the console. Ship the boring, watchable thing; decode-to-.pi is a later board off the landed raw.

Choices + consequences earned — the fork

Fork: date-windows vs Bulk API 2.0. Salesforce recommends Bulk API 2.0 for 50k+ records — but Bulk is CSV-only, async/polling, and mangles rich-text bodies (newlines/commas). We chose REST + adaptive CreatedDate windows → JSON: lossless (bodies preserved), maps 1:1 to triples, no polling. Consequence: at true millions this would strain; at OneDine's 42k it lands in 182 calls. Verified the right call — the seed is landed.

Fork: JSONL vs IDS as "the raw". Kept both. JSONL = human/re-derivable source of record; sealed IDS = the queryable replay form. Consequence: 2× storage (trivial at 50 MB), but the raw survives a store-format change.

Escape routes earned

  • If the org ever outgrows date-windows → a server-side paginating extract on the bridge (nextRecordsUrl relay). The tab's /import/* seam is unchanged.
  • If Salesforce adds Change-Data-Capture / Pub-Sub → swap the refresh source; the raw contract is identical.
  • If decode changes → re-derive from JSONL; never re-pull the org.

Pure metal test earned

C99 + nous store + understand() for the seal/replay (pure metal). The one C# is the bridge jacket (foreign-API law). Python is transport/orchestration only (the pull), never the substrate. Passes.
