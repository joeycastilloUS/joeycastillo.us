Inspect — Latchkey as a Purcius C99 Skill

De-mendo the cap-URL capability and give it a canonical home. Verdict: yes — it's a latchkey skill, nous/comms, core tier, C99. (Fable 5 architecture inspection.)

What It Is

The capability-URL ("cap-URL") capability — mint an unguessable token, keep a private token↔artifact ledger, publish an artifact under /s/<token>/, carry a comment thread the session attends. It was hand-built in the mendo-konductor project as Python (cap_tokens.py), then copied into metal. The pattern is already canon (a-d-d/patterns/latchkey.md, active). What's missing is the skill — the reusable implementation — and it's currently a mendo copy that metal forked. Joey's ask: lift it, organize it, stop "talking mendo," and answer whether it belongs in kastil-systems/a-d-d + Purcius + the C99 stack.

Answer: yes. PRACTICES #11 — "substrate capability ships as a pure-C skill slotted into the mesh (Slot pattern), not a web service." Latchkey is exactly that, and it has already passed the two-consumer promotion bar (mendo built it, metal is expanding it to every artifact) that the other proposed nous/comms skills are still waiting on.

The confusion to date was treating all three as one Python file in one project. Separating them is the whole point of this inspection.

Stack — where it lands, and why C99

Language law (kernel, locked 2026-06-20): "All other code is C99." Python is legal only as dev tooling — which cap_tokens.py squeaks by as today (it runs in the render chain on the operator's box). But the pattern itself already names the endgame (latchkey.md § Related: "a future prime-side comments/attend capability seats as a conventional Slot in a C binary"). The moment the Attendant goes autonomous or the Thread moves to the mesh, this is production substrate and Python is illegal. Lift now, on schedule, not later under duress.

Name + placement

  • Name: latchkey — skill-name = canon-name (Signals canon ↔ skill/signals). capurl/captoken are the aka's, not the name.
  • Purpose: comms (not security) — it's the artifact-facing sibling of signals (Signals = outbound; the Attendant = inbound). It mints randomness but signs nothing.
  • Architecture home: a-d-d/nous/comms/skill/latchkey/ (CLAUDE.md + skill.json) + one core ★ proposed row in the nous/comms Skills table.
  • Code home (when built): kastil-systems/metal-console/src/sdk/nous/comms/skill/latchkey/ — the same home the-family-trust names for promoted comms skills.
  • Entropy: source UUID randomness from security/crypt's RNG (or OS entropy), consistent with roll-your-own.

All stdlib-able: uuid = 16 entropy bytes + version/variant bits; ledger = the json skill (already promotion-planned); harden = string ops; stage = mkdir/fopen/rename. Parameterize what the Python hardcodes — PUBLIC_PREFIX (/metal/s vs /mendo-konductor/s) and the ledger path become arguments, never constants. A tiny CLI front (latchkey mint|token|path|list|stage) preserves the Python's exact contract so metal's chain swaps binaries without respec. Seats as a Slot when the prime-side attend story lands.

Patterns Found — the de-mendo plan

  1. a-d-d first (needs Joey's approval — Rule 10; this ~i is the proposal). Add the ★ proposed row + skill/latchkey/{CLAUDE.md,skill.json} to nous/comms. "Build from a-d-d, never from context" — nothing gets built until this lands.
  2. Declare one canonical interim Python. metal's copy is already the better one (added title/rel_path for ~fu). Flip provenance: metal's be/tools/cap_tokens.py is canonical dev tooling; mendo's becomes a vendored copy pointing at it. Kills the fork today, before C99 exists.
  3. Implement C99 in metal-console/src/sdk/nous/comms/skill/latchkey/ against the a-d-d spec. Ledger JSON stays byte-compatible — existing be/cap-tokens.json files need zero migration.
  4. Cut consumers to the CLI. metal's chain + mendo's render scripts call the latchkey binary with the same verbs. The ★ proposed marker becomes a regular row.
  5. Thread stays put on the gcp-bridge with the same retirement contract as firestore; when mesh lands, the Attendant/Thread seats as the prime-side Slot and the latchkey skill doesn't change.

Open Questions / Hazards (all verified, not hypothetical)

  • ⚠ The fork is already live. metal's cap_tokens.py diverged from mendo's (added title/rel_path, changed PUBLIC_PREFIX). A third project would copy one at random. De-mendo step 2 is urgent independent of the C99 work.
  • ⚠ Bridge URL is hardcoded in two places — ~fu's spec AND baked into every published artifact's HTML (var BRIDGE='...gcp-bridge-692654781984...'). Published Doors freeze against it; a bridge/mesh move orphans their widgets unless the re-stage escape route is honored. The skill must take the bridge URL as config; a-d-d should record the canonical bridge project in one place.
  • ⚠ Ledger privacy is a comment, not code. "never copied to the public site" is convention. If a publish glob ever sweeps be/, the whole catalog leaks — the exact blast radius the pattern exists to prevent. The C skill should refuse to stage the ledger file, as code.
  • Ledgers stay per-project — centralize the skill, never the Ledger. One skill, many private ledgers.
  • Tier call is Joey's: Fable reads it core (mint/ledger/stage have no GCP dep and survive the mesh). Confirm.
  • GCP naming drift — bridge lives in acmedev-platform-hq (us-west1); the-family-trust runs us-west4; metal docs once said pantheon-platform-hq. Record the canonical bridge project once.
