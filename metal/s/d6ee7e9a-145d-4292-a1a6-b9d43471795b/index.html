Design — Every Artifact Is Born a Latchkey

The automatic, going-forward shape: every metal artifact leaves the chain reaction already minted, already published at its cap-URL, already carrying a live comment thread that ~fu can sweep. Zero extra operator steps.

1. Need

The Latchkey pattern is blessed (a-d-d/patterns/latchkey.md) and the build is planned (5 phases, approved). What's not yet designed is the automatic part: the shape that makes every artifact a Latchkey without anyone deciding to make it one. If minting is a step the operator (or the session) has to remember, it's the "I'll add tests later" lie wearing a new coat — the catalog forks into attended artifacts and dead ones, and ~fu sweeps a comment surface that only sometimes exists. The pattern's own anti-pattern list already names unwired attendance as the failure mode. The need under the subject: a placement — exactly where in the existing chain reaction the mint, the thread wiring, and the publish slot in so that "born a Latchkey" is a property of the chain, not a choice per artifact. Bonus force: metal artifacts are not web-published today (no joeycastillo.us/metal/be/ mirror exists), so there is no legacy path to defend — the cap-URL can be the first and only public shape metal artifacts ever have.

2. Raw User Stories

  • As Joey, I open any artifact metal just popped, leave a comment on the section I care about, and later type ~fu — and metal has read it, acted on it, and stamped it seen. I never asked for the thread; it was just there.
  • As Joey, I comment ~f the estimate on board 3 is wrong on a plan, and ~fu executes it as a tilde scoped to that plan.
  • As Joey, I paste an artifact's link to Scotty and nothing else — the link is the whole credential, and if it leaks, exactly one artifact is exposed.
  • As Joey, when a tilde re-renders an artifact, the link I already shared still works — same token, same thread, comments intact.
  • As the metal session running ~fu, I get a comment keyed doc=<token> and can resolve it to "the Latchkey design, section Shape" from my own ledger — without the comment store ever knowing the artifact's name.
  • As any future tilde author, I add nothing: the chain I inherit already mints, wires, and publishes.

The only genuine inventions are (1) the placement rule — mint before render, so the token exists when the HTML is born; (2) the Thread widget — a small inline vanilla script in every artifact template that speaks to the gcp-bridge with doc = the Latchkey token; and (3) three ledger fields so ~fu can attribute a token back to a human-readable artifact. Everything else is wiring existing parts into the existing chain.

5. Shape

Minimum viable structure — the chain reaction with three links inserted (bold = new):

do the work
  ▷ MINT      cap_tokens.py get_or_mint(kind, key)     kind = artifact folder (design/plan/build/...)
  |                                                      key  = the artifact's dated filename slug
  |            idempotent: a re-render reuses its token -- shared links + threads survive
  ▷ render    Thread widget baked in, [LATCHKEY] filled with the token
  ▷ save      be/<kind>/YYYY-MM-DD-HHMM-<cmd>-<slug>.html    (repo source, unchanged)
  ▷ STAGE     cap_tokens.py stage  ->  be/s/<token>/index.html    (the Door copy, hardened)
  ▷ index -> telemetry -> commit -> push     (ledger + Door copy ride the same commit)
  ▷ PUBLISH   s/<token>/ mirrors to joeycastillo.us/metal/s/<token>/    (cap-URL-first:
  |            ONLY s/ folders ever publish -- no be/ mirror, no index, robots-disallowed)
  ▷ DoD       + "born a Latchkey": ledger row exists, cap-URL 200, thread GET answers
  ▷ pop       Pop-and-link surfaces the CAP-URL as the Live link (never a predictable path)

Why this is zero extra steps: the operator's per-artifact cost is nothing and the session's per-artifact cost is two tool calls (mint, stage) plus one template placeholder — all specified once in the chain spec that every tilde already obeys. Nobody decides whether an artifact gets a Latchkey; the chain has no path that skips it.

The Thread widget ↔ ~fu alignment (doc-is-the-token)

  • Widget writes: POST /comments?doc=<token>&obj=<section-id> on the gcp-bridge, author = the commenter. Comments render in-page under the section they attach to; a "seen by metal" receipt renders when stamped.
  • Widget reads: the bridge's per-doc read (GET /comments?doc=<token>). The recent-sweep endpoint proves the store reads; if the per-doc GET isn't exposed yet, exposing it is a First-Move-adjacent bridge task — the widget contract doesn't change.
  • ~fu resolves: the sweep returns comments keyed doc=<token>; ~fu looks the token up in be/cap-tokens.json and gets {kind, key, title, rel_path, issued_at, commit} — enough to say "comment on the Latchkey design, § Shape," open the repo source, and execute a ~-prefixed comment scoped to that artifact. The comment store never learns an artifact name; attribution lives only in the private Ledger.
  • Ledger record (the three added fields in bold): 
    "<uuid4>": { "kind": "design", "key": "2026-07-02-1208-design-every-artifact-is-a-latchkey",
             "issued_at": "...", "title": "Every Artifact Is Born a Latchkey",
             "rel_path": "be/designs/2026-07-02-1208-design-every-artifact-is-a-latchkey.html",
             "commit": "<hash, backfilled like telemetry>" }
  • Widget config: the bridge base URL is a single const inside the inline widget in each template — no external config file, no extra request. Published Door copies freeze the URL they shipped with; a bridge move is handled by re-staging (see Escape Routes).

The operator's whole flow

Open the cap-URL metal popped → comment on any section (plain note or a ~ command) → whenever ready, type ~fu → metal sweeps, acts, replies in-thread as author=metal, stamps seen. No poller, no idle cost, no per-artifact setup — the Attendant is on-demand and the surface it attends is universal because birth is universal.

Out of scope:

  • Backfilling the historical artifact catalog (mint-going-forward only; batch backfill is opt-in later, per the plan).
  • The autonomous Attendant (comment-wakes-a-session) — plan Phase 5, explicitly deferred; ~fu is the Attendant for now.
  • A per-kind publish toggle — not built until "every artifact" proves too heavy (it's an escape route, not a feature).
  • Retiring the gcp-bridge to the Pantheon mesh — tracked in the pattern and nous/comms canon.

6. First Move

Make THIS artifact the first one born a Latchkey: lift cap_tokens.py into metal/be/tools/, mint (design, 2026-07-02-1208-design-every-artifact-is-a-latchkey), bake the Thread widget in with the minted token, stage the Door copy, publish it, then comment on it and run ~fu. If the design that specifies the mechanism round-trips a comment through its own mechanism, the design is correct.

These go into the plan (they slot into the approved plan's Phase 4, updated for the bridge backend): insert the mint/stage/publish links into the kernel chain spec + be/docs/conventions.md; add [LATCHKEY] + the widget to _template.html and the report skeletons; extend the ledger schema with title/rel_path/commit; update the Pop-and-link rule to surface the cap-URL as the Live link; confirm/expose the bridge's per-doc GET; add the "born a Latchkey" DoD line.

Choices + Consequences optional

ForkOptionsPick + Reason
When does an artifact get its Latchkey? (a) mint-per-artifact, at birth, in the chain — 2nd: every artifact is sweepable by ~fu, zero decisions — 3rd: every artifact is public-behind-a-token, publish cost on all kinds
(b) mint-on-share, when the operator asks — 2nd: smaller public surface — 3rd: a manual step, a forked catalog, threads that only sometimes exist		 (a). The subject is the automatic. A manual mint step is exactly the unwired-attendance anti-pattern the pattern forbids; (b) is the escape route if cost bites, not the design.
Is the thread's doc id the Latchkey token, or a separate id? (a) doc-is-the-token — 2nd: one mint, one identity, ~fu resolves via the Ledger alone — 3rd: re-minting (revocation) orphans the thread
(b) separate stable doc id per artifact — 2nd: thread survives re-mints — 3rd: two identities to keep aligned, and a name-shaped doc id leaks artifact existence through the open comment store (the pattern's named anti-pattern)		 (a). The pattern requires the Thread key on the Latchkey; revocation-orphans-the-thread is the documented price of revocation (export before re-mint, see Escape Routes).
Where does the Thread widget + its bridge URL live? (a) inline script in each template, URL as a const — 2nd: artifact fully self-contained, no extra request, no shared-file dependency — 3rd: published copies freeze their widget version
(b) sibling thread.js copied into each Door folder — 2nd: updatable per re-stage — 3rd: same freeze, plus a second file to stage
(c) one shared /metal/thread.js on the public site — 2nd: update once, all artifacts follow — 3rd: a predictable public path coupled to every private artifact, and a single point of breakage		 (a). Pure-metal and self-contained; the freeze is real but re-staging from repo source is cheap and already idempotent (that's what stage does).

Escape Routes optional

Load-bearing choiceReversal path
doc-is-the-token (one identity for address + thread) Revocation = re-mint: new token, new Door, old path retired. The old thread orphans by design — export it (one GET /comments?doc) before retiring the token. Cost: one fetch + a ledger edit.
Bridge URL frozen inside published Door copies Re-stage: cap_tokens.py stage is idempotent from repo source. When the bridge moves (Cloud Run → Pantheon mesh), update the template const and re-stage every ledger token in one loop. Cost: one batch run + one publish.
Mint-per-artifact (all kinds publish behind cap-URLs) Add a per-kind toggle (be/config.json, e.g. latchkey.kinds) and unpublish by deleting s/<token>/ folders + retiring ledger rows. The chain shape doesn't change — the mint link just consults config. Cost: small config read in one chain link.
Ledger as critical private state (lose it, lose the token→artifact map) It's committed to metal-console, so git history is the backup. ⚠ It must never ride a publish — the publish step copies s/ folders only, by construction, never be/*.json.
