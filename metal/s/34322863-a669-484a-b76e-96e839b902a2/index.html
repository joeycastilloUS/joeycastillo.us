Build — Concurrent Sessions via Worktree (First Move)

Teach the launcher go <n> to run session n in its own git worktree. Proven: two sessions land on main, zero index-lock contention.

What Was Built

The First Move of the concurrent-sessions design (be/designs/2026-07-02-0849-design-concurrent-sessions.html): the launcher now accepts a session number.

  • go (bare) — primary session, C:\metal, OTel 4318. Byte-identical behavior to before — zero regression.
  • go <n> (n ≥ 2) — the launcher creates (or reuses) a git worktree at C:\metal.<n> on branch session-<n>, sets $root to it, starts the OTel collector on port 4318+n, and launches claude there.

Each worktree shares the .git object store + hooks (installed once into the main .git/hooks) but has its own working tree and its own git index — which is what makes the index.lock race impossible by construction. Each session's be/ (board-status, telemetry) is naturally isolated; the per-session OTel port removes the collision.

Decisions Made

  • $root indirection. The launcher computes one operational root (main or worktree) and everything — hook, autoheal, board, telemetry, collector, settings, prompt — hangs off it. Session 1 is unchanged; a numbered session just re-points $root.
  • Hooks are shared, installed once into the main .git/hooks (worktrees share the common git dir) — so the ASCII guard applies to every session.
  • Land on main via rebase-retry (git push origin HEAD:main) — the same loop that already lands concurrent writers. Per-session branch keeps a named line to reason about.
  • Per-session OTel port 4318+n — clean cost attribution, no collision.

What's Next

  • Session-branch push discipline. A tilde in a session-<n> worktree pushes HEAD:main (proven here); fold that into the chain's push step so every tilde does it automatically in a numbered session.
  • Lifecycle: go <n> --end to git worktree remove + delete the branch; prune orphan worktrees from crashed sessions on launch.
  • board.md concurrency — the one shared mutable file; rebase-retry handles it; escalate to a board write-lock only if same-item conflicts get noisy.
