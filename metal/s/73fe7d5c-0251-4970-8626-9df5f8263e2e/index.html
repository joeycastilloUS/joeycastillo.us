After GO, no wave ever waits on a human. (2) Wave 1 banks the headline and the floor in ONE M3 deploy slot: MA P4 secrets + RV fail-closed-auth ship together (Slack login live in hours, never on a broken gate), remaining RV fixes and PURL P2 follow in ordered slots; STAGE4 fires the moment STAGE3 seals; NPS1→NPS2 judge lands so Wave 2's S2 starts fed; burn-rate gauges ship first, not last; MAXIMUS/CONAN revive to widen the fleet. Every serial chain (MT, DR, TSS, NPS, RAW) gets exactly one machine walking it. (4) Wave 3 converges the rails — ONE refresh pipeline, ONE gated write lane fired live by the operator's tap, identity closed end-to-end — with the mesh lane deliberately shed (REC2 code was built off-prime). (5) Wave 4 hardens, ships customer surfaces, and runs the fleet-wide Rule 9 sweep that the verifier pool has been amortizing all along. Operator-gated items are built dark/disarmed, surfaced as red OPERATOR rows in the fleet view, and the machine moves on — the fleet never idles on a human.

The GO Protocol (what “say GO and nothing stops” means)

~GO means: this plan is approved, the GO window is being answered NOW in one sitting, and the fleet runs unattended until every targeted board is production-verified — no machine stops on its own, and after the GO window no wave ever waits on a human. MECHANICS: (0) WAVE-0 EXCEPTION — the Mercury lease layer is the WAVE-0 PREREQUISITE (confirmed: no mercury_lease tool exists on disk; the fleet is dark to Mercury; fan-out without leases guarantees double-work and clobbers). Until the lease proof passes, at most 3 machines work on provably disjoint repos (NEO: lease layer; floater: reconciliation; one: STAGE3 monitor). The proof is TWO SESSIONS ON NEO — session A claims, session B's duplicate claim visibly rejected on non-fast-forward push — so no operator hardware gates fan-out; MAXIMUS/CONAN revival is Wave-1 width, not a Wave-0 gate. (1) LEASE: claim work by writing leases/<board-id>.json {board, project, host, sid, ts, ttl_min, status} to the metal-telemetry hub via pull→write→commit→push; rejected non-fast-forward push = you LOST the race — pull, pick another board. Hotspot repos (web, meraki-bridge, zendesk-bridge, gcp-bridge, mesh-SSH, metal repo, nous-main) carry RESOURCE leases whose lease file also declares the per-wave publish ORDER; you may hold a board lease but you queue a reviewed patch to the resource-lease holder in its declared order — you never write the hotspot yourself, and a patch out of declared order is bounced by the holder. A lease request naming a second Zendesk write path or a second refresh pipeline is DENIED automatically and logged by the reaper — safety stop, not preference. (2) WORK: run ~l (presence stamp + board open), execute items strictly as verify loops per Rule 7 (~a~t/~u~t/~f~t/~b~t); red = fix-and-retest in place, NEVER advance on red; heartbeat = write your per-host file every 2-3 min (piggybacked on the presence stamp, which also pushes board-status.json so Mercury sees the fleet again); lease TTL = 3× heartbeat so the reaper never races a live renew. (3) DONE: an item is Done ONLY after external verification per Rule 9 — hand the evidence request to the verifier pool (M23) or a GCP shell for wire/TLS surfaces (Rule 13: home-LAN curl 000 on the prime is a vantage failure, not an outage; check answers + recent logs, never PIDs); write the commit hash into board.md, commit board state with the work, release the lease, claim the next unblocked board. Machines are FUNGIBLE — a machine whose lane empties returns to the pool; if no lane is claimable it joins the verifier pool rather than idling or inventing scope. (4) OPERATOR-GATED items: build everything up to the gate DISARMED/dark, set lease status=operator-gated with a one-line ask (renders as a red OPERATOR row in the published fleet view — the operator queue), release, move on; when the operator completes the ask, ANY machine may reclaim and finish the last mile. The fleet never idles on a human. (5) BLOCKED (⛔ upstream): skip; when a board completes, its finisher removes the ⛔ from the next board — the signal idle machines poll. (6) COLLISION + FAILURE: leases are advisory-with-teeth (TTL expiry makes a dead machine's lease reclaimable; takeover logged who/when); push retries carry jitter; deploys batched one-per-board (TPL-K law); every M3 deploy re-smokes Slack /auth so no later deploy silently kills the headline; GCS full/ objects never overwritten — publish new handle, flip manifest. (7) FUEL: TAC4 burn-rate + CU4 Fable-weekly gauges are the fleet fuse — weekly cap >80% sheds to critical-path lanes only (M7, M11, M23 + the current gating chain); 100% resets the calendar to the cap window. (8) STOP: only two stop conditions — every targeted board Done-with-external-evidence (run closeout, ship the retro), or the operator says stop. 'Built, awaiting creds' is NOT done — the 2026-07-09 called-it-done retro is the anti-pattern this protocol exists to kill.
