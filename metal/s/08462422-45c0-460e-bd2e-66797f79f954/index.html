Item 1 — the unblock: GET /tickets/{id}/attribution

The engine was proven but offline (a CLI). Now it's a live endpoint: read a ticket's comments (+users sideload for author roles), and for each agent-authored public comment spawn the vendored nous C99 engine (/app/attribute, compiled in the Docker build stage) with the body on stdin → one JSON verdict. Language law held: the matcher stays pure C99; C# is only the Zendesk jacket. Read-only, fail-soft per comment.

Item 3 — backlinks: GET /templates/rollup + Templates tab

A bounded .pi attribution rollup over recent tickets (default 60, hard cap 200, honest capped flag — no silent truncation, 300s cache, 1000-spawn budget) tallies distinct-tickets-per-template. Item 2 — Oracle tiers: sharper, with a no-regression gate

The MVP scored Sorensen-Dice over content atoms — blind to negation and amount. The Oracle scorer subtracts penalties for contradictions (a NEGATION mismatch on a shared action) and quantity mismatches ($10 vs $25). Because a template and its faithful edit share scopes and amounts, true pairs are mathematically unpenalized — the baseline could not regress.

Item 4 — probe: no macro_id per comment (and that's the point)

Rule-6 unknown, resolved with evidence: the incremental ticket_events comment stream carries no macro_id; a comment's via is {channel, via_reference_id} with source.rel = null; ticket-audits isn't exposed. Conclusion: Zendesk won't tell you which template an agent used — so the .pi engine inferring it is the actual value, not a redundant guess. No ground-truth enrichment path exists; the .pi score stays the measure.
