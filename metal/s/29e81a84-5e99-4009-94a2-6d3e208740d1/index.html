Retro — TPL2 live template attribution

What Happened

Fable5 ~go finished Board TPL2 (4 items) across three repos in one session: the proven attribution engine went from an offline CLI to a live per-ticket endpoint in production, plus rollup backlinks on the Templates tab, an Oracle-tier precision upgrade, and the macro_id probe. Also this session: shelved Sign-in-with-Zendesk after ~i~d~p, and made the commentable Latchkey cap-URL law in argus (vendored the publisher). Shipped: nous a769443, zendesk-bridge 7b327e5 (ZBR1f rev 00018-hk7), argus web 2495212.

What Went Well

  • Offline proof → live in prod, externally verified on real Mendo tickets (Rule 9), not just unit tests.
  • No-regression gate on the Oracle upgrade held the line: 93.3% top-1 + precision 1.00 preserved, margin improved 0.26→0.31. The gate caught nothing to reject because the design (penalties only subtract) made regression impossible — but the gate was real and enforced.
  • Language law + read-only held under pressure: the matcher stayed C99 (spawned sidecar), C# was only the Zendesk jacket, zero writes to Mendo.
  • Delegation pattern worked: Fable5 wrote + locally-verified each item; the orchestrator owned deploy + live verify + git + board. Control stayed where the risk was.
  • The Latchkey law now self-serves — the build report published commentable with one call, no scramble.

What Could Be Better

  • Vendored engine drift. zendesk-bridge attrib/ holds a hand-copied snapshot of nous sources. It works and is commit-pinned, but a nous fix now needs a manual re-copy. A build-time pull of the pinned nous commit (or a real nous artifact/skill-socket) would remove the drift risk.
  • Three deploys in one session. ZBR1d/e/f each cost a ~3-4 min Cloud Build. Items that touch the same service should stage and deploy once at board end, not per item.
  • Rollup cost is linear + serial. One engine spawn per agent comment, sequential; a pathological ticket serializes. The 300s cache helps repeat loads, but a batch/parallel spawn or a persisted rollup would scale better.
  • No per-ticket ground truth. The macro_id probe came back negative, so attribution precision can only be trusted via the corpus proof, never validated against a live label. A periodic re-proof against a small hand-labeled sample would keep confidence honest as templates drift.
  • Operator/env state visibility. A live-env value differed from what was believed current, and there was no quick way to reconcile belief vs the running service. A read-only "what is the service actually configured with" surface (fingerprints, not secrets) would catch drift without exposing anything.

Tool Failures

No functional failures. Cosmetic only: a PowerShell here-string commit message parsed as separate args once (recovered with -m flags); git stderr surfaced as NativeCommandError wrapping (pushes succeeded); the Remove-Item guard blocked a __pycache__ delete (worked around with a .gitignore). No be/debug/ logs present this session.

Performance Notes

Cloud Build deploys ~3-4 min each (x3). The live rollup over 40 tickets took double-digit seconds (40 sequential comment-reads + spawns) — acceptable for a cached tab load, flagged for scale (see Kaizen). Fable5 build agents ran ~10 min each with clean local verification.

Commitments

  • Replace the vendored attrib/ snapshot with a pinned build-time pull from nous (kills drift).
  • Batch same-service board items into one deploy at board end.
  • Stand up a periodic attribution re-proof against a labeled sample.
