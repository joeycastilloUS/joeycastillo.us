~d — The generic source slot (any source, not just Zendesk)

1. Need

"~d that slot again because it should not only work for Zendesk, right?" — Joey

Right. The prior design (2026-07-09-2039-design-one-argus-source-scoped-fable5.html) locked the shape — one Argus, one bot, per-conversation source slots — but every worked example was Zendesk. A slot whose fields only make sense for a ticket system is a Zendesk class with a generic name. The moment Salesforce wants a Slack surface, that slot forks, and the fork is the exact "Argus-for-X" split the prior design exists to kill.

The ground truth says the generic case is already the real case. Six source bridges are live (a-d-d §3, one Cloud Run service each, us-west1): zendesk-bridge, salesforce-bridge, gcp-bridge, airbyte-bridge, meraki-bridge, cloudflare-bridge. Only two have sealed IDS (acmedev-mendo-tickets-ids, acmedev-onedine-cases-ids). Only some have a write lane. Only one has a persona. The sources genuinely differ — so the slot must carry those differences as data, and the shell must read what a slot can do instead of assuming what Zendesk does.

Why now: Salesforce is next in line for a surface, and the console (web/index.html) already half-models this with its APPS registry — six sources, each a record with label/tabs/source. Two half-registries (console APPS, Slack's Zendesk wiring) are one refactor away from being one registry, or one fork away from being permanent duplicates. Unify before the fork.

2. Raw User Stories

  • As the builder, I want to add a source by writing a slot record — data, not a class, not an app, not a branch in the shell.
  • As the builder, I want a read-only source (GCP cost, Meraki) to simply have no write lane, and the shell to know that from the slot — never to discover it by a failed call.
  • As the operator, I want a source with no IDS yet to still answer — ungrounded and honest about it ("no sealed store for this source; answers are unverified"), not silently pretending.
  • As the builder, I want one shell definition to serve both the console and the Slack app — same slot record, two surfaces reading it.
  • As a Salesforce-only operator, I want my rights to scope which slots exist for me — the others aren't hidden, they're not mine.
  • As the operator, when Argus drafts in a slot with a persona, I want that voice; in a slot without one, the plain default — never Zora quoting cloud invoices.

REUSE vs NEW

  • REUSE — all six bridges, both IDS buckets, the Slot pattern, §3, §8, the Operator-Identity canon, the console APPS registry, the argus-slack shell.
  • NEW — the shared slot record (the schema above), replacing per-surface half-registries.
  • NEW — the capability-driven shell behavior: the shell reads capabilities + nulls and degrades gracefully instead of assuming Zendesk's full set.

4. Naming

NameMeans exactly
source slotOne source's complete record: bridge, grounding store, persona, write lane, rights, capabilities, bindings. Data, mounted as a unit.
slot registryThe one file/config holding every slot record. Both surfaces read it.
capabilitiesThe set a source declares: search / ground / draft / act. The shell's contract for what to offer.
grounding storeThe slot's sealed IDS (ids field). Null = no grounding claims made.
write laneThe slot's single gated action path. Born disarmed. Null = read-only source.
rightsWho may resolve the slot. Operator-Identity boundary, not a UI filter.
the shellThe generic code (console + Slack) that mounts whichever slot the conversation resolves to. Vendor-blind by law.

Collision flagged: the console's APPS entry and the source slot are the same idea at two maturities. APPS is the UI sliver (label, icon, tabs); the slot is the superset it becomes. During transition, an APPS entry is the console-bindings face of a slot — one record, and APPS keys stay the slot ids so nothing renames. Also re-flagged from the prior design: there is no "Argus-for-X"; there is the X slot mounted in Argus.

5. Shape

Minimum viable:

  1. The slot record schema — the eight fields above, with the three nullable ones (ids, persona, writeLane) explicitly legal.
  2. The slot registry — one static config file, all sources as records. The console's APPS keys become slot ids; APPS UI fields fold in as console bindings. Adding a source = adding a record.
  3. The capability-driven resolver — shared logic both surfaces run: conversation/tab → binding → rights check → slot. The shell then offers exactly capabilities ∩ non-null fields: no ids → answers labeled unverified; no writeLane → action UI never rendered; no persona → default voice. Degradation is declared, never discovered.
  4. Zero vendor branching in the shell — the lint rule that keeps it honest: no vendor name in shell code paths. Vendor shape lives in the slot data and behind its §3 jacket, nowhere else.

Out of scope (YAGNI)

  • Per-message source switching — scope stays per surface (prior design's law).
  • Cross-source joins ("compare Zendesk volume to GCP cost") — a different product.
  • A rights admin UI — rights are a static config map first.
  • Auto-provisioning IDS for the four sources that lack one — ids: null is the design, not a gap to rush-fill.

Language law: each bridge stays its own C# jacket (foreign API, §3 / Language Boundary canon). The shell, the slot registry, the resolver, and the grounding check are ours. Read-only by default: every writeLane is born disarmed:true; absence of one means the shell never even offers the action.

Choices + Consequences

The core fork: how is a slot defined and discovered?

Option2nd/3rd-order consequencesVerdict
A — per-source classes in the shellFast for source #1. Second order: every shell improvement is edited N times or drifts; the "generic" shell accretes if vendor scar tissue. Third order: the classes are the fork — "Argus-for-X" ships itself, and Joey's question gets answered wrong permanently.✗
B — central slot registry (config the shell reads)One file is the whole truth: every source, every capability, every null, reviewable in one diff. Second order: adding a source is a record + a bridge deploy; a bad slot is a one-line revert; rights live where an auditor can read them. Third order: the registry is a single point of coordination — a source team can't self-serve a slot without touching the central file. Acceptable: today one operator owns all six sources.✓ PICK
C — self-describing bridges (GET /slot manifest per bridge)Most §8-pure: the source declares itself, adding one is zero-touch. Second order: every bridge grows an endpoint + version skew between manifest and reality; the shell must cache and re-fetch. Third order: trust moves outward — a bridge now asserts its own capabilities, persona, and (worst) rights; a compromised or sloppy jacket can widen its own lane. Rights self-declared by the thing being ridden is the wrong direction.✗ (for now)

Pick B. With six sources and one operator, central-and-visible beats distributed-and-clever. The decisive line: rights and writeLane must never be declared by the bridge they govern. And B doesn't foreclose C — see Escape Routes.

Escape Routes

  • Bad slot — it's data. Revert the record; no vendor code, no shell deploy.
  • B → C is a walk, not a rewrite — the registry can later hydrate from each bridge's self-description (bridge proposes capabilities/ids; registry stays the arbiter of rights/writeLane). Same record shape either way; the shell never notices.
  • Source outgrows the shell — the slot boundary is the fork seam. Lift the record and its bridge out whole into a standalone app; the other slots don't move.
  • Schema too thin — new optional field, null everywhere it doesn't apply. The null discipline is the extension mechanism.

Pure Metal Test

✓ The shell, the slot registry, the resolver, and the grounding check are ours — vanilla, no frameworks, vendor-blind. The one foreign piece per source is its bridge jacket (C#, §3), swappable by editing one slot's bridge field. Replace any vendor tomorrow and nothing we own changes shape.

6. First Move

Mount two maximally different sources through the identical slot shape, with zero vendor-named branching in the shell.

  1. Define the slot record + registry with exactly two entries: zendesk (full capabilities, acmedev-mendo-tickets-ids, Zora, disarmed /reply-send) and gcp (search+ground, ids: null, persona: null, writeLane: null).
  2. The generic resolver serves both: Zendesk answers grounded and can offer its gated draft/act; GCP answers, labels itself unverified, offers no action, speaks plain.
  3. Acceptance = a grep: no vendor name in any shell code path. Both behaviors fall out of slot data alone.

Zendesk grounding is the prior design's first move; this one proves the shape generalizes — the richest source and the barest source, one record schema, no fork. Slot #3 (Salesforce) then lands as a data entry on a proven rail.
