1. Need

"We need to maybe make a specific app for Argus-for-Zendesk as opposed to Argus-for-Salesforce — because when chatting with each concern they should be focused. But having said that it would be good to use one Argus for all. Maybe this is a rights thing." — Joey

Two pulls in opposite directions. Focus: a Zendesk invoice thread (US Foods, Mendo) and a Salesforce opportunity (OneDine) must never blur in one chat — different data, different voice, different write lane. Unity: one bot, one install, one Argus brand. Joey's instinct — "maybe this is a rights thing" — is the resolution: focus is a scope, not a separate app.

The concrete driver underneath: the DM ticket-search route ("find", "how many", "trending") sends the question to the bridge's /ask and trusts the model's cited ticket IDs — it never checks them against the real sealed IDS. Argus can confidently report tickets that don't exist. The fix (ground on the per-source sealed IDS) requires knowing which source the conversation is about. Source scope isn't just organizational hygiene — it's the precondition for grounded answers.

Why now: Salesforce is about to want a Slack surface. Decide the shape before someone forks argus-slack into a second bot and the split becomes permanent plumbing.

2. Raw User Stories

  • As the operator, when I'm in the Mendo GR channel, I want Argus to talk only Zendesk — Mendo tickets, Zora's voice, the Zendesk write lane — with zero Salesforce bleed.
  • As a Salesforce-only teammate, I want my Argus to never touch Zendesk — not see it, not search it, not mention it. Rights, not politeness.
  • As the operator, I want ONE bot named argus, one install, one place to @mention — not a zoo of per-source bots.
  • As the operator, when I add a third source, I want to register it — not build and install a new Slack app.
  • As the operator, when Argus cites ticket IDs, I want every ID checked against the real sealed store before it reaches me. No invented tickets. Ever.

REUSE vs NEW

  • REUSE — both bridges, both sealed IDS buckets, the argus-slack shell, the Slot pattern, the Operator-Identity canon.
  • NEW — the source-scope resolver: channel/DM → source binding + operator rights check.
  • NEW — the slot registry: {source → bridge URL, IDS bucket/handle, persona, write-lane}.
  • NEW — grounded ticket-search: cited IDs verified against the scoped slot's sealed IDS before the answer ships.

Everything new is glue between things that already exist. The isolation Joey wants was built at the bridge and data layers months ago — this design surfaces it in the chat layer instead of re-creating it as separate apps.

4. Naming

NameMeans exactly
source slotOne source's complete kit: bridge + sealed IDS + persona + write lane. Self-contained; mounted as a unit.
source scopeThe one source a conversation surface is bound to. Explicit, never inferred per message.
channel↔source bindingThe map entry that sets a channel's (or DM context's) source scope.
rightsThe set of sources an operator is entitled to. Gates every binding and every resolution.
Argus shellThe one Slack app (argus-slack, bot argus) that mounts whichever slot the scope selects.

Collision flagged: "Argus" is the product and the shell — never a per-source thing. There is no "Argus-for-Zendesk"; there is the Zendesk slot mounted in Argus. Saying it wrong is how the fork happens.

5. Shape

Minimum viable, in the existing argus-slack shell:

  1. Slot registry — static config: { "zendesk": { bridge, ids_bucket, ids_handle, persona: "zora", write_lane }, "salesforce": { … } }. Adding a source = adding an entry.
  2. Scope resolver — on every event: channel/DM → bound source; then operator rights check (starts as a static config map: operator → entitled sources). No binding or no right → Argus says so and stops. Nothing is inferred.
  3. Grounded ticket-search — the brain's DM search route reads the scoped slot's sealed IDS; every ticket ID the model cites is checked against that store before the answer posts. Uncheckable ID → dropped or flagged, never asserted.
  4. One app, unchanged install — same bot, same workspace install, same Approval Card flow. The write lanes stay each source's own gated/disarmed path; grounding is read-only.

Out of scope (YAGNI)

  • Per-message source switching — scope is per surface, period.
  • Cross-source queries in one thread.
  • A rights admin UI — rights start as a static config map.
  • Multi-workspace.

Choices + Consequences

Option2nd/3rd-order consequencesVerdict
A — separate app per source ("Argus-for-Zendesk", "Argus-for-Salesforce")Hard focus, yes. But: N bots, N installs, N token sets, N deploy pipelines. Second order: every shell fix (Approval Cards, brain wiring, auth) lands N times or drifts. Third order: "one Argus for all" is dead as a brand and as an architecture — the fork is the org chart forever, and source #3 costs a whole app.✗
B — one app + channel-scoped rights slotsFocus comes from the explicit binding; unity from the one shell. Second order: adding a source is a registry entry; every shell improvement reaches all sources at once. Third order: the slot boundary doubles as the app-fork seam if a source ever genuinely outgrows the shell — B keeps option A open; A forecloses B.✓ PICK
C — one app + per-message source inferenceKeeps one app but the scope is a guess. Second order: a US Foods invoice question bleeds into a OneDine opportunity on a bad classification — exactly the blur Joey named. Third order: rights become unenforceable (the model decides what you can see), and grounding can't pick an IDS deterministically — the hallucination gap stays open.✗

Pick B. It's the only option that delivers both of Joey's pulls: focus is a rights-gated, explicit scope; unity is the one shell. And it's the only option where the grounding fix is deterministic — the scope names the IDS.

Escape Routes

  • Bad binding — slots are config-mounted; reverting a channel↔source binding is a one-line change, no deploy of vendor code.
  • Slot outgrows the shell — the slot boundary is exactly the app-fork seam. If Salesforce ever needs its own app, lift the slot out whole: its bridge, IDS, persona, and write lane already travel as a unit.
  • Rights model too thin — the static operator→sources map upgrades to real entitlement (Operator-Identity, FQDN-keyed) without touching the resolver's interface.

Pure Metal Test

✓ The shell, the slot registry, the scope resolver, and the grounding check are ours. The two foreign APIs stay jacketed per-source behind zendesk-bridge and salesforce-bridge (C# jackets per the Language Boundary law) — either vendor is swappable by replacing one slot's bridge, and nothing we own knows a vendor's shape.

6. First Move

Bind the Zendesk slot explicitly on the live Mendo GR channel, and ground the DM ticket-search on it.

  1. Registry gets one entry: zendesk → { zendesk-bridge, acmedev-mendo-tickets-ids (handle 8963900c…), Zora, gated /reply-send }.
  2. The Mendo GR channel gets an explicit binding to that slot — the first channel↔source binding in existence.
  3. The DM ticket-search route checks every cited ticket ID against the slot's sealed IDS (431,260 real tickets) before answering.

One channel, one grounded source, one Argus. This proves the slot mechanism and closes the hallucination gap on the only source that's live — before the Salesforce slot exists, so slot #2 lands on a proven rail instead of a theory.
