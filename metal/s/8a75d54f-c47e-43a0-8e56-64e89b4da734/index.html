Gemini-on-Vertex + .ids grounding: the build

The build plan behind 2026-07-09-1922-design-gemini-vertex-ids-brain-fable5. Take Argus off Anthropic onto Gemini/Vertex, and give the brain a read-only door into the 431K-ticket .ids so it finds tickets and surfaces trends from real data. Two lanes, four phases, 21 pts.

Intent

Make Argus-the-voice run on Gemini in your own Vertex (ADC, no key) and answer from the corpus, not from imagination. Success is one live sentence: DM Argus "find refund-declined tickets this month" → it returns real Mendo case IDs + a grounded summary, produced by Gemini, with zero Anthropic traffic. Then we look at where it's sharp and where it whiffs — that's the "how good are we" readout you asked for.

Scope

In: the Anthropic→Vertex brain swap in argus-slack; two new read-only C99 endpoints on onedine-raw (/search, /facets); one new grounded-answer route on gcp-bridge (/ask); routing Argus DMs to the right lane. Out: vector/embedding search, Gemini function-calling, BI dashboards, any write-back, and Argus thread-memory (all fast-follows once the floor is proven). api.3-nous.net stays a v2 escalation, unused here.

Approach

Reuse-first — the recon says 5 of 8 capabilities are already live. Lane A (brain swap) is independent and trivial; land it first so "no Anthropic" is true within the hour. Lane B is the value: expose the search matcher we already wrote, add a bounded facet count, then a deterministic retrieve-then-read loop in the C# jacket. Every task is a verify loop (~a~t) and every service change is proven live (Rule 9 — external /version + a real curl), not just compiled.

Phases

Ends: you DM Argus, it replies, and the log shows a Vertex call — no api.anthropic.com. The "no Anthropic" requirement is done here.
Phase 2 — Open the door: search over the .ids. 6 pts — the First Move
  • {GET /search?q=&field=&limit= on onedine-raw — wrap the existing rawq substring matcher; JSON hits (s,p,o); passphrase-gated; deploy + curl "refund" → real case IDs, add to, onedine-raw} · 3 · ~a~t

Phase 3 — Ground the brain: retrieve-then-read. 8 pts — the value
  • {POST /ask { question } on gcp-bridge — terms → /search → top-K /replay (real threads + restated program) → Gemini answers WITH cited case IDs; no hits → honest "no matching tickets", add to, gcp-bridge} · 5 · ~a~t

Ends: DM "find refund-declined tickets" → real IDs + grounded summary, Gemini, zero Anthropic. The design, proven in one sentence.
Phase 4 — See how good we are. 2 pts
  • {run 10 real operator questions (find + trend) against /ask, record hit/miss/whiff vs eyeballed truth, write the honest scorecard — where substring wins, where it needs BM25/vector, test, argus} · 2 · ~t
Ends: a one-page readout of the floor — the evidence that decides whether Phase 5 is inverted-index, embeddings, or function-calling. This is the "When To Revisit" trigger made concrete.

Constraints

  • Language law: C99 for search/facets (onedine-raw); C# jacket only for the Vertex plumbing (gcp-bridge); vanilla for the Slack surface.
  • Read-only: no lane writes the sealed raw. Source-of-record is immutable here.
  • No key at the edge: Vertex via ADC; Argus presents only the bridge token; Anthropic key is removed, not left dormant.
  • Done = live (Rule 9): each service task proven by external /version + a real curl / real DM, not a green build.
  • Cost honesty: /ask reads top-K tickets into the prompt — cap K, log tokens/cost per call (the ledger already exists in gcp-bridge).

Unknowns named, not hidden

  • Does substring find what operators mean? "refund" won't catch "money back / comp'd / credited." Phase 4 measures the miss rate — we don't know it yet.
  • Top-K sizing. How many tickets does Gemini need in-context to answer well without blowing tokens/latency? Unknown until we watch real questions. Start K=8, tune.
  • Trend shape. /facets counts by an existing predicate; "complaints this week" needs a category + a date filter that may not be a single clean predicate on the raw. May need a small derived pass. Flagged, not solved.
  • onedine-raw is the OneDine seed, not the full Mendo NPS corpus. The 431K/977K full pull is sealed in gs://acmedev-mendo-tickets-ids; confirm which store the live onedine-raw serves before claiming corpus-wide answers. (Recon flag — verify at Phase 2.)
  • Term extraction. Literal keyword split vs a cheap Gemini pass to pull query terms — unknown which gives better recall; try the free one first.

When to revisit the v2 trigger

Phase 4's scorecard is the signal. If substring recall is ≥ ~80% on the 10 real questions → ship it, memory + more sources next. If it whiffs on paraphrase (the likely case) → v2 is an inverted index or embeddings, and the specific misses are the spec. If operators ask multi-hop questions ("compare this month to last") → that's when Gemini function-calling earns its keep and api.3-nous.net comes in for corpus-wide reach. We revise when the evidence says so — not before (Rule 6).
