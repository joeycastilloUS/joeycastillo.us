Design — Argus-for-Zendesk, Wire-Native

Re-cast the Zendesk corpus pipeline from HTTP / GCS / Cloud-Run to the nous wire — the way the Communication Law requires, and the only way the operator's event-ripple actually works.

1 · Need

The pipeline we shipped works but violates the communication law: the way to talk to Purcius is the wire protocol — never HTTP/REST into the substrate. Today every hop is a violation: console → gcp-bridge (C# HTTP) → zendesk-raw (C99 HTTP service, daily-passphrase bearer) → GCS objects → Cloud Run fold/derive jobs → env-var promote.

The violations aren't cosmetic — they're the direct cause of the operator's three complaints:

  1. Slow. The daily feed re-derives all 433K tickets (~1–2h) because GCS blobs + batch jobs can't ripple; plus HTTP round-trips and Cloud Run cold starts on every read.
  2. Stale. Ticket #455091 was solved in Zendesk and showed unhandled in Argus for days. Batch pull + batch fold = staleness by construction.
  3. No ripple. The operator's stated vision: “we receive a webhook/event, the ripple effect is the key, all data self-organizes and relates, DMA at the metal is light speed.” HTTP+GCS+batch cannot deliver that. The wire can — the corpus already is a sealed IDS store, and the mesh already speaks triples.

Need: the corpus lives on a prime, is served over NTRP0001, and is fed by an event ripple — one ticket in, O(one ticket) of work out, fresh in seconds.

2 · Raw User Stories

  • When a ticket changes in Zendesk, Argus reflects it in seconds — #455091 never happens again.
  • As the console, I query the corpus over wire and get triples back — no bearer, no HTTP, no cold start.
  • As Focus / Find / precedents, I hear a ticket signal and update myself — “one pipe in, many ears out,” no polling.
  • As the operator, the daily 1–2h re-derive disappears; ingest cost is O(one ticket).
  • As the operator, “is Argus in sync with Zendesk?” is one wire query, not a script counting HTTP GETs.
  • As the builder, I migrate incrementally — keep the C# Zendesk jacket and the sealed IDS format, no big-bang.

3 · Sourced Solution

Everything needed already exists in the substrate. Nothing new is invented; the outside world contributes nothing (no Kafka, no Pub/Sub, no GCS).

PieceSourceWhat it gives us
Transportnous/comms/skill/wire (wire.h)NTRP0001 packed triples over TCP; self-framing (triple_count + payload_bytes); wire_request(), wire_pack/unpack; atom macros W_ACTION/W_ENTITY/W_STRUCTURE. A wire message is an array of understand()-atoms — the same shape the store holds. No serialization boundary anywhere in the system.
Gatenous/comms/skill/relay (relay_server.c)Authenticated forwarder between the open internet and locked primes — auth / forward / status verbs, sessions in the triple store, AES-256-GCM transport. Forked from prime_server.c, which is exactly the skeleton the corpus prime forks again.
Storenous/memory/skill/store (store.h)The sealed IDS TripleStore: facts.bin/dict.bin/strings.bin, mmap'd, fnv1a ids, ids_insert/ids_query, WAL (Journal), AES-256-GCM at rest (ids_open_encrypted), O(1) dict hash at 4.6M strings. The 433K-ticket corpus (raw-ids-full) already IS one.
Decomposenous understand()Ticket text → atoms. Atoms are wire triples are store facts — one representation from Zendesk edge to query result. Cannot hallucinate.
EventsSignals canonEvery event = sealed IDS record {kind:"Signal", what, by, cmd, context, at, seal}, Ed25519-sealed by operator FQDN, broadcast into nous/comms. The ripple's ignition.
Foreign edgeexisting C# zendesk-bridge jacketThe ONE legitimate jacket (Language Boundary canon). Keep it. Repoint its output: instead of gzip deltas to GCS, it emits wire.
Mesh plumbingnous/comms/skill/mesh, dns (relay.3-nous.net), pulseDiscovery, liveness, addressing — already deployed for relay/prime.

The ripple mechanism (the one genuinely new part, built from store primitives): projections (BM25 postings, similar-ticket links, Focus rollups) are stored as facts, each keyed by the fnv1a content-hash of its input fact set. A delta ingest touches a known set of ids; only projections whose input set intersects those ids recompute. BM25 is per-term incremental (adjust postings + doc count for one doc); similar/precedent recomputes only the new ticket's neighborhood. The 20-shard full derive becomes O(one ticket).

5 · Shape

Zendesk (foreign REST + webhooks)
   |  their HTTPS -- foreign edge
   v
zendesk jacket (C#, the ONE jacket -- pull + webhook receipt)
   |  understand(ticket) -> atoms -> NTRP0001 frame + sealed ticket signal
   v  wire over TLS
relay  (relay_server.c: auth / forward / status)  <-- console (C99, wire native)
   |  forward triples                             <-- product edge (web->wire,
   v                                                   product boundary only)
corpus prime  (corpus_prime.c <- fork of prime_server.c)
   |-- sealed IDS: facts.bin / dict.bin / strings.bin  (AES-256-GCM at rest, WAL)
   |-- ingest -> ids_insert(delta) -> ripple -> intersecting projections recompute
   |-- search / replay / similar / version / audit -> ids_query + projections
   \-- emits "corpus updated" signal -> many ears (Focus, Find, precedents)

Ingest as signal/ripple. Webhook fires → jacket calls understand() → atoms → one NTRP frame through the relay to the corpus prime's ingest verb + one sealed ticket signal into nous/comms. The prime inserts the delta and ripples. Focus/Find/precedents are ears on the pipe — they update in near-real-time. Contrast: today a solved ticket waits for the nightly pull, a GCS delta, a 1–2h 433K-ticket re-derive, and a promote flip. Wire: the ripple is O(one ticket). That is “DMA at the metal is light speed,” realized with primitives that already exist.

Serve. Console speaks wire_request() natively (it's C99). zendesk-raw's HTTP surface and the gcp-bridge corpus lane are replaced by the prime's verbs; the daily-passphrase bearer is replaced by relay auth (sessions already live in the relay's own triple store).

Sync/freshness. Freshness becomes intrinsic — a live-rippled store IS current; there is no “promote,” just version (fact count + journal seq). The coverage audit becomes the audit verb: prime reports per-window ticket counts / max-id / id-set hash from the IDS; the jacket reports the same from Zendesk REST; the diff comes back as triples. No HTTP counting scripts.

Kill list (all law violations): zendesk-raw HTTP service, /corpus/pull, GCS gzip deltas, Cloud Run fold job, 20-shard derive job + merge, env-var promote, daily-passphrase bearer.

Constraints / Unknowns

  • WIRE_BUF_SIZE is 128KB / 1024 triples per frame — a long ticket thread may need multi-frame chunking on ingest; spec a part i/n meta-triple before build.
  • The corpus prime co-locates with existing prime infra behind relay.3-nous.net; heed Rule 13 — verify from inside GCP, wire over TLS, curl will lie.
  • Projection content-hash keying needs a reverse index (input-id → projection) — ids_create_index exists; confirm IndexField coverage fits or add one.

6 · First Move

Serve the EXISTING sealed corpus over the wire — read path only, prove the law end-to-end before touching ingest.

  1. Fork prime_server.c → corpus_prime.c; ids_open_encrypted() the existing raw-ids-full store (format unchanged — zero data migration).
  2. Implement ONE verb: search — unpack W_ACTION("search") + entity terms, run BM25 over the existing derived projection, return top-k hits as triples {ticket-id, "hit", subject} + {ticket-id, "score", n}.
  3. One C99 console client: wire_request() through the relay to the corpus prime; print hits.
  4. Prove: same query against zendesk-raw HTTP vs the wire — identical results, latency measured side by side.

HTTP stays live during the proof; nothing breaks. Then the migration runs incremental:

  • Phase 1 — cut console + gcp-bridge corpus reads to wire; add replay/similar/version.
  • Phase 2 — webhook → jacket → signal → ingest ripple, with the nightly HTTP pull demoted to audit backstop.
  • Phase 3 — retire GCS + Cloud Run fold/derive + zendesk-raw once audit runs green N consecutive days.

Kept throughout: the C# zendesk jacket, the IDS seal/format, the derive algorithms (re-hosted in the prime as incremental ripple, not batch).
