Purcius Identity — Platform Master Plan

~p · the identity layer that spans every app, not just Argus — the gap the redesign exposed. Sibling to the identity design and above the Argus roadmap. Comment any section; ~fu answers in-thread.

One substrate. Many consumers. Argus is the first app to plug in — never the shape of the plug.

Genesis mints the roots and operator keys. A pure-C Operator Identity & Key Lifecycle skill (kastil-systems) exposes identity over the wire. Every app consumes it and none reimplements it (PRACTICES #12). This plan sequences that substrate on its own timeline — so identity is a platform, not an Argus feature.

Intent

Why plan this now: the Argus master roadmap sequences Argus. Identity is bigger than Argus — the Overlord, operators, and their Warranted keys govern every app that will ever run on Purcius. Today the pieces exist unsequenced: the Genesis root is minted, the crypto substrate is KAT-tested, the identity design is done — but there is no platform-level plan that owns the identity track independent of any one app. Run app-by-app, identity would be re-decided in each product, drift apart, and violate the "auth lives in kastil-systems, apps consume" law by accretion.

The thesis: identity is a substrate with consumers, planned once. Prove the wire seam with one app (Argus), then generalize. The value is not "Argus login" — it is that the second app, and the tenth, inherit a login/onboard/recover/rotate/monitor system they never had to build.

Scope

In:

  • The Genesis-rooted identity substrate as a cross-app service: Overlord root, operator keys, device Warrants, the _key.<fqdn> anchors.
  • The pure-C Operator Identity & Key Lifecycle skill (kastil-systems) and the wire ops apps call to consume it.
  • The OIDC user layer (Slack / Discord / Google) as the USE half, and the Overlord control plane (pause / cut-off / expire / monitor).
  • Enforced rotation + recovery, as substrate capabilities every app inherits.

Out:

  • Per-app access policy — each app owns its own module grants (Argus owns "Argus for Meraki"). The substrate authenticates + carries grants; it does not define them.
  • The Argus product roadmap (lives in argus/be/plans) — this plan rides beside it, not over it.
  • UI / ceremony specifics — the identity design owns those; this is sequence + gates.
  • New crypto. The primitives are done and KAT'd; this plan is plumbing, not invention.

Approach

One substrate, many consumers — sequenced by proof, not by app.

  1. Prove the seam with one real consumer. Argus is the first app to call the skill over the wire. If the seam is right, it is app-shaped nowhere.
  2. Widen the roster of key-holders (operators) before widening what they can do.
  3. Add the control plane (pause/cut-off/expire/monitor) once there is more than one operator to control.
  4. Harden (enforced rotation, recovery drills) as a substrate guarantee, not a per-app afterthought.
  5. Generalize — a second consuming app is the real test that "substrate" was true.

Every phase is a coherent capability the whole platform gains, gated on the one before it proving out.

Operators before control-plane because you can't govern who doesn't exist. Rotation/recovery before generalizing so the second app inherits a hardened substrate, not a half-built one.

Genesis call map (the substrate this rides on)

Real, exported, KAT-tested in nous/purcius-genesis/core/genesis_core.h. No abstraction — the plan names the call:

CallWhere the plan uses it
genesis_mint(n,t,…)Overlord root (Phase 0) and every operator key (Phase 2) — same mint, their device/FQDN
genesis_sign(root_sk,…)Warrant issuance + login-nonce signing (Phases 1–2)
genesis_anchor(root_sk,…)the _key.<fqdn> TXT value; re-emitted after recovery (Phases 0, 4)
genesis_recover(t,shares,…)lost device — reassemble the key from shares (Phase 4)
genesis_resplit(root_sk,n,t,…)fresh shares, same key, after every recovery (Phase 4)

Constraints (won't bend)

  • Auth/security lives in kastil-systems, pure C99. Apps consume the skill; none reimplements it (PRACTICES #12). C# only as the foreign-OIDC jacket.
  • Wire protocol only into the substrate. Never HTTP/REST/SDK into Purcius — apps speak the wire.
  • No passphrase, no vendor auth. The root of trust is your own minted key, verified against a published anchor. Not WebAuthn, not Auth0.
  • Done = production, externally verified (Rule 9): each phase proven from a machine other than the dev box — dig the anchor, key-login from another device.
  • Flow up first (Rule 11). The a-d-d / spec is amended, with your approval, before substrate code — and this plan has no a-d-d home yet (Unknown #4).

Unknowns (named explicitly)

  • Wire client shape. In-process C# client vs. a C sidecar subprocess — collides with the bridge's "no new runtime" constraint. Decided at the first seam, not before (Rule 6).
  • Control-plane state store. Where operator status (active/paused/expired) lives: Firestore, or a pure-C store on the prime. Firestore is the fast path; pure-C is the sovereign one.
  • How the C# OIDC jacket speaks wire. The USE half is C#; it must hand identity to the substrate over the wire like any consumer.
  • This plan's a-d-d home. No Purcius-wide architecture spec exists — only two canons + the auth-skill spec. Identity may need its own a-d-d before the substrate grows further.
  • Which second app proves generality. Undefined — and until one exists, "substrate" is a claim, not a fact.
  • Multi-device per operator. One operator, several enrolled devices — how many Warrants, and how revocation of one device works.

When To Revisit (the v2 signal)

  • First Argus Overlord key-login lands. The wire seam is proven or it isn't — the cheapest place to learn the plan is wrong. Re-plan the seam shape against what actually worked.
  • A second operator (Scotty) is real. Onboarding + control-plane assumptions meet a human who isn't you.
  • A second consuming app appears. Phase 5's "substrate" claim gets tested; if the skill fights the new app, revisit the wire contract before app #3.
  • A customer's rotation/compliance requirement lands. Enforced-rotation policy meets a real deadline and may jump the queue.
  • A P1 in the substrate crypto. Safety pre-empts everything — back to Phase 0.

This is master v1 of the identity track. It is wrong somewhere; the Phase-1 seam is where we find out (Rule 6).
