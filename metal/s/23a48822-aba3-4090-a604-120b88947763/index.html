Stage the Mendo Zendesk raw: full decode + BM25 index

Turn the 431K-ticket raw into a staged working set — de-noised, fully decoded to .pi programs across every comment, and indexed for fast ranked search — while the raw stays the untouchable source-of-record you can reconstruct anything from. Operator decisions locked: full thread, C99 inverted index + BM25, Cloud Run Job (us-west1). 18 pts.

Intent

Stage once, serve fast forever. Precompute what's expensive (decode + index) so zendesk-raw answers find / program / matches / patterns / shape instantly and with real ranking — and prove the raw still reconstructs any ticket exactly. Deterministic, model-free, $0/decode.

Scope

In: a C99 stager over the full sealed raw (gs://acmedev-mendo-tickets-ids/full/raw-ids-full.tgz): de-noise (scrub) → full-thread decode (every comment) → .pi program + content-atom bag per ticket → BM25 inverted index → sealed derived-full store + index, uploaded + hash-verified; a reconstruction + search proof. Runs as a Cloud Run Job in us-west1. Out: the serving service (zendesk-raw, next), embeddings/vector (BM25 first), live incremental refresh (frozen snapshot; refresh is a follow).

Approach

Reuse the proven C99: scrub_derive.c (de-noise: HTML/CSS/URL to side-channel, clean prose to understand()), derive_prog.c (multi-unit .pi program), seal_triples.c (seal), nous understand/wire/store. New = a full-thread driver (walk has_comment for every ticket, decode each comment) + a BM25 inverted index builder (term→postings+tf, doc-length table, sealed alongside). Package as a Cloud Run Job so it runs next to the data (no 1.45 GB local pull) and re-runs on refresh.

Why this exists: it feeds zendesk-raw stage → serve, one arc

This staging pass is the substrate for the serving design (design-argus-zendesk-mendo-corpus-fable5). Staging is not a side quest — it produces the exact store zendesk-raw mounts, and each staged artifact powers a specific endpoint/view. Build the fuel here; the service just serves it.

Sequence: stage (this ~p) → serve (zendesk-raw, the design) → the five Argus views. Doing the full decode + index now is what makes the service fast, ranked, and corpus-wide instead of a thin substring scan over 5 tickets.

Constraints

  • Raw is immutable source-of-record. The stager only reads it; derived is additive, same format, subject-keyed back to raw.
  • Language law: all C99 (stager, index, seal) + nous; no model in the decode path ($0).
  • In-region: job runs in us-west1 next to the bucket; no cross-region egress, no local 1.45 GB pull.
  • Deterministic + re-runnable: same raw + same decoder version = byte-identical staged output; manifest pins decoder + sha256.
  • Done = the staged store is live in GCS, hash-verified, and a third party can pull + reconstruct a ticket (Rule 9).

Unknowns named

  • Decode wall-clock. 977K understand() calls over 1.45 GB mmap — minutes? tens of minutes? Sets the Job timeout + memory (start 4 GB / 1 hr, measure, tune).
  • Index size. Inverted index over ~4.6M unique strings — how big sealed? Affects the serving service's memory. Measure at ST2.
  • Scrub aggressiveness. How much Zendesk boilerplate (signatures, quoted replies, footers) survives scrub and pollutes programs/index — may need a second de-fluff rule set. Watch at ST4.
  • Wordnet in the container. understand() needs the wordnet dir at runtime (all tools call lang_init(wordnet)); confirm it ships in the image or mounts from GCS.
  • Nous API + header source. The mendo tools #include "../skill/store/store.h" against a nous tree NOT vendored in the repo, and use the triple_store_* API — while the proven raw2/onedine build compiled against C:\metal\src\sdk\nous using the lower-level ids_*/ts_* API. Step 0 of the build reconciles these: pick one nous tree + API layer that compiles both the scrub/derive tools AND the store walk, before writing stage.c. (zendesk-bridge already vendors a working nous under attrib/ — a candidate source.)

When to revisit v2 trigger

ST4's proof decides the next layer. If BM25 ranking lands the right tickets where substring missed → wire zendesk-raw to the staged store and ship the five views. If paraphrase still slips (BM25 is lexical) → embeddings become the justified v2, layered on the same staged atoms. If scrub leaves fluff in the programs → a de-fluff pass is the next stage before serving.
