Intent

The button exists. The crypto is built, tested, and deployed. It is dark for exactly one reason: the bridge has no operator pubkey and no MAC secret, so /auth/key/* fails closed with 503 and the login panel never shows. This plan lights it up — get the root pubkey off the prime once, feed it to the env var and the _key.<fqdn> record, set the server secret, warrant one device, and prove a real sign-in from a machine that isn’t yours. On the already-blessed adopt-a61bdace path — the Keystone evolution is v2.

Approach — one value unlocks the wall

The linchpin: operator-key.sh adopt on the prime prints k=<64-hex> — the public half of a61bdace, nothing secret. That one string is both PURCIUS_OPERATOR_PUB and the _key.<fqdn> content. Capture it once (Phase 1); everything downstream is wiring.

Phases

SSH the LV prime (34.125.36.235, where ~/.purcius/operator.key = a61bdace lives). Run bash be/tools/operator-key.sh detect then adopt. Capture k=<64-hex> and the self-sig s=. Verify: 64-byte key confirmed (seed32‖pub32), pubkey is 64 hex chars, self-sig verifies against it. Private key never leaves the box.

Phase 4 Warrant a device + live sign-in · 5 pts · ~u~t

Verify end-to-end (Rule 9): from a second machine / clean profile — challenge → device-sign → verify → argus_session minted → /auth/me returns authed. A wrong/absent warrant is refused (0x26). Fix the enroll copy: replace the genesis-cli line with the real warrant.wasm + paste paths.

Phase 5 Sealed-Core pin round-trip · 5 pts · ~a~t

Close Board REG1’s open item. Decide the Unknown first: pin as config (the env var — v1, done in Phase 2) vs compiled-in (prime rebuild). For v1 the env-var pin is the pin. Prove one live challenge → sign(offline key) → verify-vs-pin round-trip against the deployed bridge. If compiled-in is chosen, that’s a scoped follow-on, not a blocker for a working button.

Constraints

  • Adopt, never mint. One root. operator-key.sh refuses a second. The private half never leaves the prime.
  • Pure-C auth, C# only as jacket (PRACTICES #12). No crypto logic moves into the bridge. DNS write is the only vendor call that rides C#.
  • Secrets: you paste, Be handles (Rule 4). PURCIUS_ID_KEY never touches git — only its env-var name.
  • Done = a third party can verify (Rule 9). dig from elsewhere, sign-in from a second machine. Not “pushed.”
  • Verify from the right vantage (Rule 13). Don’t judge the bridge from this router; test status + dig from a real client plane.

Unknowns

  1. Keystone vs adopt. The sovereign-identity design wants a purpose-built cold Keystone above a61bdace. This plan ships on adopt (blessed); the Keystone later swaps the pubkey without touching the wire. If you’d rather mint the Keystone first, Phase 1–3 change to a cold genesis ceremony (+~13 pts, needs the air-gap/hardware decision). Your call gates the whole shape.
  2. Pin location. Config (env var) vs compiled-in on the prime. v1 answer is config; compiled-in is a hardening follow-on.
  3. Prime reachability + auth service. Phase 1 assumes SSH to 34.125.36.235 works and the key file is at the expected path. The opaque server-mediated session (REG2) needs a pure-C auth service on the prime — separate open item after the button works.
  4. genesis-cli. Referenced in copy, not built. Not needed (wasm + paste work). Build only if you want the terminal warrant path.
