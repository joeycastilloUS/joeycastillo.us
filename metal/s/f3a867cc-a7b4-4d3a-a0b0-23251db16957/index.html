How recover-guest.pi runs on Purcius

Schematic — the NPS-detractor guest-recovery intent, from operator speech to the guest's inbox, and exactly where Argus sits. Modeling only — nothing executes against Zendesk yet.

The execution trace

  1. Operator speaks the intent. jc's words distill into recover-guest.pi — a sealed punch card — and land in the IDS (Ed25519 by FQDN, no unsealed writes).
  2. Argus raises the event. Argus has its eyes on Zendesk. A Guest Experience Alert · NPS Detractor arrives; Argus hands the event to biRT.
  3. biRT reads the .pi. The punch card is data at rest, code when biRT reads it. It evaluates the rules: nps ≤ 6 → draft-and-hold; food-borne illness → escalate-only (hard exclusion — never a reply).
  4. Claim one code. biRT atomically claims a single IMSORRY code from the vault in the IDS (synced from the brand's Google Sheet). Spent once, never double-spent; empty → halt.
  5. Generate the sense. biRT dispatches the two free slots — empathy + resolution — to the AI Pantheon, then fills the byte-verbatim fixed skeleton + data slots.
  6. Operator gate. The finished email is held in preview, every slot editable. jc owns the trigger: approve → fire, or edit → rebuild. Nothing sends itself.
  7. Cross the boundary — the only HTTP. On approval, biRT calls out through the thin C# jacket (zendesk-bridge /reply-send). Everything left of the jacket is Purcius/wire; the jacket is the single foreign edge. Sandbox first.
  8. Zendesk emails the guest. The jacket opens a new follow-up ticket (requester = guest), posts a public comment — public is what makes Zendesk send the email. Ticket solved, tagged argus_recovery_sent; audit + spent code sealed back to the IDS.
