Zendesk → Raw IDS: an import tool in Argus

Give the blessed import an operational home: a first tab in the Zendesk app + a small C99 import service. Decouple landing the raw from mesh residency and NPS-derive. Ship the boring blessed thing first.

1. Need

The operator wants ALL of Zendesk in a sealed Raw IDS so the system can replay from it forever. The architecture is already blessed (ARCHITECTURE.md §7, operator-directed 2026-07-04: "import once to raw sealed IDS… then run replays on the raw and always refresh/get-latest") and already proven at v0: 1,000 Mendo tickets → 5,002 triples, round-trip clean. And yet the full import keeps not landing. Three evidenced causes:

  1. No home. The importer is loose C99 in a sibling repo — C:\mendo-zendesk-ai\be\tools\ (raw_import.c, derive.c, derive_prog.c, scrub_derive.c, replay.c, retrieve.c, ticket_json.c, program.c, qstore.c). Never wired into the Argus console. Every run is a hand-driven C invocation in a side directory — no button, no progress view, no resume surface. It dies when the session ends.
  2. Tangled + gated. The full pull got bundled into an AWAITING-APPROVAL chain (board L1→L2→L3 + the 07-08 design) alongside mesh residency (content-address + manifest signal + bulk channel) AND NPS-derive / Focus-demoware-kill. The simple blessed thing — land the full raw — can't ship on its own because it's chained behind ambitious things.
  3. Long job, nothing to watch. Zendesk incremental export runs ~10 RPM; a full-history pull is real wall-clock. With no operator surface to start / watch / resume it, it stalls every time.

The reframe: decouple landing-the-raw from the mesh + NPS ambitions, and give the import an operational home — a tab in the Argus Zendesk app driving a small C99 import service. Mesh residency and derive-nps come later, off the already-landed raw. Why now: every downstream board (NPS, Focus-real-data, audit) is blocked on the same missing artifact. Land it once and everything else becomes a decoder pass.

2. Raw User Stories

  • As the operator, I want to click Import all in the Argus console and walk away, so that the full Zendesk history lands without me babysitting a C invocation in a side repo.
  • As the operator, I want to watch the pull live — tickets/comments pulled, current cursor, triples written, elapsed, ETA at ~10 RPM — so that a long job is a progress bar, not a leap of faith.
  • As the operator, I want Refresh to resume from the stored cursor, so that a closed laptop or a killed session costs nothing.
  • As the operator, I want to pick a real ticket and Replay its actual thread from the raw, inside the console, so that I can see with my own eyes the raw is real, not a claim.
  • As the operator, I want the standing raw shown as a first-class thing — count, seal, last-refresh, cursor — so that "do we have the data?" is a glance, not an archaeology dig.
  • As the operator, I want a bad pull to be droppable and re-runnable without fear, so that importing is boring.

5. Shape

Minimum viable structure — six pieces, five of them mostly existing:

  1. The tab. rawids becomes the FIRST tab in the Zendesk app (web/index.html APPS registry, line 577). Three panels: Standing raw (triple count, seal/hash, last-refresh, stored cursor) · Live progress during a pull (tickets/comments pulled, current cursor, triples written, elapsed, ETA computed from cursor progress at ~10 RPM) · Actions: Import all (full pull from start_time=0), Refresh (incremental from stored cursor), Replay (pick a ticket → decoded real thread → trace shown inline).
  2. The import service (C99). One small service, one contract: /import/start, /import/status, /import/replay?ticket=. It drives zendesk-bridge for reads, writes the sealed raw, stores the cursor in the raw, survives the browser closing. Home decision in Choices below. Crucially: the tab talks only to this contract — where the contract is served (localhost vs Cloud Run) is swappable.
  3. The promoted C99 tools. raw_import.c, replay.c, retrieve.c, ticket_json.c, program.c, qstore.c move from mendo-zendesk-ai/be/tools/ into argus (src/import/) — they stop being a side-repo secret and become the service's body. derive*.c comes along but stays downstream (it reads the raw; it is not the import).
  4. The read path. zendesk-bridge as-is: incremental/ticket_events.json?start_time=<cursor>&include=comment_events pages the full history with comments until end_of_stream. No new bridge endpoints needed for tickets+comments.
  5. The sealed raw. One nous triple_store, one writer (the import service), append-only, per-batch provenance per §7. ticket:<id> + comment:<id> subjects.
  6. Refresh + replay-one proof. Refresh = read cursor → incremental → append → new cursor. Replay = decode one real ticket's thread from the raw and render the trace in the tab — the console-native proof the raw is real and replayable.

Invariants

  • Read-only: the import READS Zendesk and WRITES only the local sealed raw. It never writes back to Zendesk. No write path exists to disarm.
  • Language law: C# exists only in the zendesk-bridge jacket at the foreign boundary. Importer, store, replay, service — all C99, all ours.

Out of scope (YAGNI — deferred to downstream boards off the landed raw)

  • ✗ Mesh residency (content-address, manifest signal, bulk channel).
  • ✗ derive-nps / Focus real-data wiring.
  • ✗ Full config-object walk (triggers, automations, macros, SLAs, forms) beyond tickets+comments — §7 wants it eventually; the raw's append-only shape means adding it later costs nothing now.

6. First Move

Prove the thinnest path end-to-end, bounded. Promote raw_import.c (+ replay.c, retrieve.c, ticket_json.c) into argus/src/import/, wrap the /import/start | /import/status | /import/replay contract around them, run the service on localhost, and drive a bounded pull — the last 30 days of tickets WITH comment events — through zendesk-bridge into a fresh sealed raw. Then, in the new Raw IDS tab, watch the progress land and replay ONE real ticket — real comments, real thread, decoded from the raw, rendered in the console.

Not the whole corpus. The full start_time=0 pull is the second move — it starts the moment the pipe is proven watchable and cursor-resumable, and by then it's just a bigger number on the same progress bar.

Choices + Consequences earned — load-bearing fork

Where does the importer live? (the "home" decision)

A — C99 import service (own Cloud Run)B — Local runner, tab as viewer
1st order Job survives closed browser, closed laptop, ended session. Start from any machine, watch from any machine. Zero new infrastructure. Runs today with the promoted binaries.
2nd order Mirrors the one-service-per-source pattern (bridge already lives there); the raw accrues in a place other services can later read (mesh board inherits a running start). Cost: another Cloud Run service to stand up, deploy, and keep honest — plus the sealed raw now lives off-box, which changes how replay/derive tools reach it. The job dies with the session — which is cause #1 and #3 of this design's Need, re-created with a nicer UI. A multi-hour ~10 RPM pull on an operator box is exactly the thing that has stalled every time.
3rd order The import becomes an operational fact of Argus, not an operator habit — refresh can later be scheduled, and downstream boards trust the raw exists without asking whose laptop it's on. The raw's existence stays coupled to one person's uptime; every downstream board inherits that fragility.

Pick: A, staged. A long resumable job needs a home that outlives the session — that is the whole point of this design. But the honest cost of standing up another Cloud Run service before the pipe is proven is real, so the contract is designed to be location-blind: the First Move serves /import/* from localhost with the same C99 binary; deploying that binary to Cloud Run is the promotion, not a rewrite. B is not the alternative — B is the failure mode we're leaving.

Escape Routes earned

  • Bad pull / full re-import. The raw is append-only with per-batch provenance — a bad pull is a droppable batch, not a corrupted store. Worst case: delete the store file, re-run Import all; the cursor lives in the raw, so a partial pull resumes instead of restarting. Cost of total do-over: wall-clock at ~10 RPM, zero code.
  • The home choice. Because the tab binds to the /import/* contract and not to a location, backing out of Cloud Run = point the tab at localhost and run the same binary. Backing out of the tab entirely = the promoted C99 tools still run by hand, exactly as v0 did — nothing is lost, only the home.
  • The decoupling. If the mesh/NPS boards land sooner than expected, nothing here is wasted: they consume the raw this tool lands. The 07-08 design's architecture is untouched — only its sequencing moved.

Pure Metal Test earned

  • ✓ Store, importer, decoder, replay, service — C99 on nous primitives. Ours, load-bearing, no framework.
  • ✓ Console surface — vanilla HTML/JS in the existing Argus tab registry. No build step.
  • ✓ The ONE foreign dependency — Zendesk — is jacketed (zendesk-bridge, C#) at the boundary and replaceable: swap the jacket, the raw's shape and everything downstream stand.
