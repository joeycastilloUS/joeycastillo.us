Design — metal in every view

Make the metal comment system the prevailing in-app comments system in Argus — every view gets a metal thread, so jc and metal work together to continue the build from inside the build.

1. Need

jc and metal already collaborate through comment threads — but only on Be artifacts (designs, plans, reports) at their cap-URLs. Inside Argus itself, where jc actually looks at the work, there is only a homegrown comment bar (cmBar/acx, web/app/index.html:628–658) on 8 of 21 views — and it is a dead letterbox: comments land in Firestore, nobody answers. No session wakes, no tilde routes, no model/token/cost attribution, no live status. Meanwhile the metal stack shipped all of that this week (comments route straight to a live session; the exchange narrates itself). Two comment systems have diverged; the better one isn't in the app. The need: one prevailing comments system — the metal one — on every Argus view, so a comment in any view is a working conversation with metal about that view, and the build continues from where the work is seen.

The Analysis — Argus today (Zendesk console)

Argus is the read-only multi-source ops console in front of Zendesk (plus Meraki, Cloudflare, Airbyte, Salesforce, GCP). Frontend is one vanilla-JS single page (web/app/index.html) with a cached-pane view dispatch (paneInner(TAB), lines 4909–4951) — 21 views, one switch, one render path. That is the whole integration surface.

The existing acx bar posts {doc:"argus-zendesk", obj:"zd-<view>"} to gcp-bridge /comments — same backend the metal widget uses. So the store is already shared; what Argus lacks is the widget (threads, replies, live status, N-new, attribution) and the ear (a session that hears and acts). ARCHITECTURE.md §4-B already blesses this exact feature: "Argus itself will get a similar in-app comment mechanism later — that too is dialogue / app-dev, not institutional knowledge." No spec drift — this design fills a slot the architecture reserved.

2. Raw User Stories

  • As jc, I want a metal thread on every Argus view, so that I can steer the build from the exact view I'm looking at.
  • As jc, I want a ~ comment in any view to wake a metal session that executes it, so that build orders don't require leaving the app.
  • As jc, I want the session's reply in-thread with the live exchange (asking → [model] working Ns → done · tok · $cost), so that I see who did it and what it cost.
  • As a metal session, I want the view's identity and repo resolvable from the comment's doc token, so that I wake in C:/argus with the right context, not in the wrong repo.
  • As the architecture owner, I want view threads to stay Channel B (dialogue) and never feed wisdom, so that the two-channel law (§4) holds.

Proposed approach: Reuse the entire metal comment stack unchanged where possible. Invent only three pieces of glue: (1) view tokens — mint one cap-token per Argus view into the Argus Ledger (kind: argus-view, key: <tab id>) with a new repo field; (2) a mount point — one injection in Argus's render() that mounts the hosted widget under every pane with the active view's token as doc; (3) an ear ledger list — session-ear reads N Ledgers (metal + argus) and wakes the session in the token's repo. gcp-bridge and Firestore need zero changes.

5. Shape

Minimum viable structure — four parts, three of them already built:

  • View tokens (new, one-time). Mint 21 tokens via cap_tokens.get_or_mint("argus-view", <tab>, title, rel_path="web/app/index.html#<tab>") + repo:"C:/argus" into C:/argus/be/cap-tokens.json. Ship the public map VIEW_TOKENS = {tab: token} into the frontend (tokens are capabilities, not secrets to the operator using the gated app).
  • Mount point (new, ~10 lines). In render() after paneInner(TAB): append <div class="metal">, set window.ARTIFACT={doc: VIEW_TOKENS[TAB]}, and (re)init the hosted widget. Widget gains a remount(doc) entry so the SPA can switch docs on tab change — one visible view, one active doc. Script loaded once by URL (/metal/artifact-comments.js, never inline).
  • Ear ledger list + repo wake (small extension). session-ear.c reads a list of Ledger paths (metal + argus). When an order's doc resolves to an argus-view token, the ear spawns claude -p with cwd = the token's repo, prompt naming the view. Answers post back with the standard attribution trailer; the widget narrates the exchange live.
  • Store + law (unchanged). gcp-bridge /comments endpoints as-is; Firestore comments/comment_seen collections as-is. View threads are Channel B dialogue — they never write to corrections/wisdom (§4 rule holds by construction: different collections, no code path between them).

Out of scope (YAGNI):

  • Per-ticket or per-row threads (a thread per Zendesk ticket) — the view is the unit; go finer only when a real need shows.
  • Migrating historical acx comments — old doc:"argus-zendesk" threads stay readable in Firestore; no data migration.
  • Comment auth/identity beyond the existing gate + name field.
  • Any wisdom/AI-training coupling — forbidden by §4.

6. First Move

Mint ONE view token (argus-view/focus), hand-mount the metal widget on the Focus pane only, post a ~ comment from inside Argus, and watch the ear wake a session in C:/argus and the answer land in-thread with the model/tok/cost trailer. One view round-tripping end-to-end proves the whole design.

These go into the plan (not written to board.md here): mount point for all 21 views + VIEW_TOKENS map; widget remount(doc); ear ledger list + repo wake; acx retirement (remove cmBar calls + dead CSS); per-section obj anchors inside heavy views (Focus cards, Audit sections).
