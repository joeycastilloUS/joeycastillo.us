~d — Make Focus + Conversations real (off the sealed raw IDS)

Two tabs are demoware. The raw is real now. Kill the SAMPLE.

1. Need

Two Argus tabs are invented data wearing a real UI:

  • Focus (focusPane(), web/index.html ~line 2849) renders const FOCUS_SAMPLE — hardcoded nps:61, responses:418, medianEyes:"0:47", three fictional detractors (Marina del Rey #375102, Cherry Creek #375088, El Segundo #375061 — every field invented: id, store, DO, score, timestamp, read, verdict, precedent, notified list) and one fictional food-illness case (#48213, fake Zora draft, fake Zendesk PUT).
  • Conversations (programsPane(), ~line 2968) renders const PROGRAMS_SAMPLE — two hand-authored "programs" whose steps, reduce ratios, precedents, and "what worked" prose were written, not decoded.

Why now. This session removed the last excuse. The full Mendo corpus — 431,260 tickets · 977,029 comments · 8,247,494 triples · 563 MB — is sealed into a nous TripleStore at mendo-zendesk-ai/data/full/raw-ids-full (GCS gs://acmedev-mendo-tickets-ids/full/, IDS handle 8963900c…), mmap'd and mesh-resident. And the decoder round-trips in pure C99, no model: understand() atomizes (ticket → 43 atoms, proven), derive.c builds the .pi program per thread, retrieve.c finds nearest precedents (query → 6 nearest, all same situation, deterministic), and restate() de-atomizes store facts back into a readable thread (de-atomize POC, be/designs/2026-07-09-1755-design-deatomize-poc-fable5.html). Everything the two tabs pretend to show, we can now actually compute.

No existing design wires the tabs to the store — conversation-as-program v1–v3 and nps-raw-ids are architecture; they predate the store existing. This design is the wiring: delete both *_SAMPLE consts and read the real sealed raw IDS.

2. Raw User Stories

  • As the operator, I want the Focus detractor feed to be real low-CSAT tickets, not three invented stores — so I can act on one and it exists in Zendesk.
  • As the operator, I want every card in Conversations to be a real ticket decoded into its real .pi program, with real nearest precedents from the 431K corpus.
  • As the operator, I want each detractor's "read" de-atomized from the actual comments of that thread — never a written summary of a thread that doesn't exist.
  • As the operator, I want the verdict chip to be the real §5 grounding check run against that thread's ground truth — GROUNDED means the atom-diff passed, not that someone typed the word.
  • As the operator, I want nothing on screen that isn't traceable to a real ticket id I can open in Zendesk and verify with my own eyes.

4. Naming literal

  • real program — the .pi program derive emits for an actual thread. Not "decoded story," not "playbook."
  • de-atomized read — restate() output: store facts back into conversation. Not "AI summary" — no model touches it.
  • grounded verdict — the §5 atom-diff result on the real thread. The chip means the check ran.
  • detractor — a real bad CSAT rating joined to its real ticket. If we display an NPS-style number, label it as what CSAT gives (satisfaction %), not as NPS.
  • mendo-ids-read — the C99 read service over the sealed Mendo raw IDS. Per-source name, mirrors the one-service-per-source pattern.

Collision flagged: Argus already has a "Raw IDS" tab — that's the OneDine seed, a distinct source. The Mendo store is not it and does not rename it. Per-source naming (mendo-ids-read) keeps them apart.

5. Shape

Minimum viable, staged — Conversations first (pure read, lowest risk), Focus second (needs the CSAT join + grounding):

  1. Mount the seal. mendo-ids-read mmaps raw-ids-full and links derive / retrieve / restate in-process — one C99 binary holding the seal. Endpoints: /programs (real ticket list), /programs/:id → {program, precedents[], read, strip, counts}, /focus/summary, /focus/detractors.
  2. Conversations goes real. programsPane() fetches /programs the way the Tickets tab fetches the bridge. PROGRAMS_SAMPLE deleted.
  3. Focus goes real. Bridge satisfaction_ratings → real summary numbers + bad-score list; mendo-ids-read joins each to its thread, runs restate + grounding + retrieve. FOCUS_SAMPLE deleted. Food-illness = tag-matched real tickets; the gated panel keeps its disarmed shape.

Invariants: read-only end to end — the one write lane stays gated and disarmed (the Zora reply is a draft; nothing fires). Language law — C# exists only at the zendesk-bridge boundary; mendo-ids-read and every primitive it links are C99. Done = production: a tab card shows a real Mendo ticket id the operator can open and verify.

Out of scope (YAGNI)

  • Firing any Zendesk write — the Zora reply stays a disarmed draft. The read-only invariant does not move in this design.
  • The full derive-nps wisdom loop (fleet-wide mining) — this is two tabs going real, not the loop.
  • Other tabs; the OneDine Raw IDS tab; any model-generated prose in the read path.
  • The medianEyes KPI — removed until it can be measured, not simulated.

Choices + Consequences the read-path fork

Path2nd/3rd-order consequences
AC99 read service (mendo-ids-read) mmaps the seal, links the primitives in-process, serves the tabs2nd: one binary holds the seal — the deterministic read comes from the thing holding it (§9), the tab composes at the edge like it already does with the bridge. Any ticket drillable live; nothing precomputed to go stale. 3rd: the same service is the natural home for every future read off this store (derive-nps, explorer) — the read path is built once.
BPrecomputed derived JSON the tab reads statically2nd: fastest first pixel, but frozen — a 431K-ticket corpus reduced to whatever slice the pass chose; drill-in on an unprecomputed ticket dead-ends. 3rd: a rebuild pipeline to babysit, and the "live" dot on Focus becomes a lie again — demoware with better provenance.
CFold reads into an existing service2nd: into the C# bridge violates the language law (the store reader must be C99) and couples the seal to a foreign-API jacket's deploy cycle. Into another C99 service muddies one-service-per-source. 3rd: every store change now risks the unrelated service it hitched to.

PICK: A. §9 says the deterministic read comes from the thing holding the seal — so build the thing that holds the seal. The store is mmap'd precisely so reads are cheap and live; precomputing (B) throws that away, and C has no clean home. B survives as a fallback (any A response is trivially snapshottable), not as the design.

Escape Routes

  • SAMPLE behind a flag for one release. The consts move behind ?sample=1 instead of deleting on day one — a bad read falls back to the known render while the read path is fixed. Deleted next release.
  • Everything derived is rebuildable. Programs, precedents, reads are computed from the seal on demand; the seal itself re-pulls from GCS full/. Nothing here is unrecoverable state.
  • Nothing to roll back on Zendesk. The whole design is read-only; the one write lane never arms. Worst case is a wrong pixel, not a wrong ticket.

Pure Metal Test

Store: nous TripleStore — ours. Decoder: understand/derive/retrieve/restate — ours, C99, no model in the path. Read service — ours. The only foreign surface is Zendesk CSAT, and it's already jacketed at zendesk-bridge — swap the jacket, keep everything else. ✓ Pure metal.
