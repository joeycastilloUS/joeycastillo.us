Inspect — Can Zendesk users log in to Argus?

VERDICT: YES — and it's mostly already built. Adding “Sign in with Zendesk” is a third OAuth door on plumbing Argus already runs, not a new auth system. The one thing to decide is a policy question, not a technical one: does being an agent in Mendo's Zendesk grant access, and to what.

The Question

Operator ask: “add the ability for Zendesk users to log in… is this a possibility?” Read literally: let a person who has a Zendesk account (a Mendo support agent) authenticate into Argus as their Zendesk identity — “Sign in with Zendesk” — rather than being hand-added to an allow-list.

Login already exists three ways: a legacy operator password, Google/Slack OIDC (built, awaiting client secrets), and Purcius device-key (the sovereign-operator path, 90% wired on the active PURL board). They all converge on one session cookie. A Zendesk door drops into the same convergence.

What Zendesk provides

Zendesk is a first-class OAuth 2.0 provider. The Authorization-Code flow is standard and well-documented:

  1. Register an OAuth client in Zendesk Admin (Apps & integrations → APIs → OAuth Clients) — yields a client id/secret + redirect URI.
  2. Redirect the user to https://<subdomain>.zendesk.com/oauth/authorizations/new?response_type=code&client_id=…&scope=read.
  3. Exchange the returned code at POST /oauth/tokens for an access token.
  4. Fetch the authenticated identity with GET /api/v2/users/me.json → returns id, name, email, role (end-user / agent / admin) and group memberships.
The one real difference from Google/Slack: Zendesk's token endpoint returns an opaque access token, not an OIDC id_token JWT. So the callback can't decode identity from the token — it makes one extra call to /users/me. That's a small, well-contained branch in the existing callback, not a new mechanism.

Bonus that Google/Slack can't give: a Zendesk login tells you the person is a real agent in Mendo's instance, and hands you their role and groups for free. That is exactly the population Argus serves. It also ties an operator action (e.g. firing a Zora recovery reply on the NPS boards) to a verified Zendesk identity — attribution the current password/allow-list model doesn't have.

The actual delta (the 20%)

  • Register one Zendesk OAuth client for Mendo's subdomain (secret handled per Rule 4: operator pastes, Be installs).
  • Add a zendesk provider to the providers-as-data list with a non-OIDC branch: exchange code → call /users/me → build the member from that.
  • Decide the access policy (the real question, see fork): allow-list of emails, OR “any agent/admin in subdomain X,” OR role/group-scoped.
  • Add the button to the web sign-in row — one line next to Slack/Google.
  • Map Zendesk role → Argus capability if we want tiered access (admin vs agent vs end-user).

The fork to settle in ~d (flag per Rule 11)

Argus's auth spine has been building toward one sovereign operator per FQDN — the Purcius key, the 2026-06-30 Overlord-auth “you log in AS your domain” model, and the Operator-Identity canon (one login app per operator). “Zendesk users log in” is multi-user tenant identity — many Mendo agents. These are not in conflict, but they are two different layers and the design must name them or they drift:

The Zendesk login is the natural credential for the tenant-agent layer that model already anticipates.

Open questions (for ~d / Rule 6)

  • Access policy — is agent-of-subdomain sufficient, or does the operator still want an explicit allow-list on top? (Recommend: agent/admin role in Mendo's subdomain = in; end-users excluded.)
  • Scope minimization — smallest Zendesk OAuth scope that still returns role + groups (likely read); confirm against the live API, don't guess.
  • Where auth lives — Operator-Identity canon says <fqdn>/auth/* on the operator's FQDN via a Pages-Function proxy (Cloud Run custom domains banned). Zendesk callback URL must land there, not on *.run.app.
  • Role → capability map — do admins get more than agents inside Argus? Or is login just “are you real,” with the operator still gating every write?
  • Session identity shape — today the payload is {sessionId}|{identityKey}|{email}; a Zendesk member wants zendesk:<subdomain>:<userid> as the identity key + role stamped for later attribution.
