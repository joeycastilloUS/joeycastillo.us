1. Need

Joey: "where can I put this in GCP, cheapest cost… I need the mesh to use this for replays, decode to atoms, and all kinds of things… it was supposed to import to the mesh; no idea why it's local."

Why it's local — the plain answer: the pull ran in the LLM zone on Joey's box, and the seal → tar → upload → announce tail only ever ran for the Jul-4 1,000-ticket sample. The bucket exists; the steps that fill it never ran at full scale. Not a mystery — an unexecuted tail.

2. Raw User Stories

  • As the operator, I want the full raw off my drive and into GCP cheaply, so that a laptop disk is never the single copy of 431K tickets.
  • As the operator, I want the sealed IDS resident in the mesh, so that any node replays or decodes to atoms without ever re-pulling Zendesk.
  • As the operator, I want the cheapest storage tier that still lets the mesh mount it fast — cold for the archive, hot for the working set.
  • As the operator, I want to see residency status in Argus — is it in GCS, which handle, which regions, when refreshed.
  • As the operator, I want regional copies where the compute is, so that no node pays egress to read its own truth.

5. Shape

Minimum viable — six steps, four of them one-command:

  1. Raw → Coldline (in flight now): the 7.6 GB gzips to ~1.3 GB and lands in Coldline, us-west1. The cold source-of-truth — re-read only on re-shape.
  2. Full seal: the v2 pipeline (C99 shaper + raw_import.c) runs over all 431,260 tickets → the full sealed .ids.
  3. Tar + hash → handle + manifest: tar the sealed store, SHA-256 it — that hash is the IDS handle. Write the full manifest.json in the existing shape (sha256, count, cursor, decoder version, gs:// locator).
  4. Upload to the hot tier: nous-kb-us-west1 (Standard) + manifest to gs://acmedev-mendo-tickets-ids. Egress to every bridge: free.
  5. Manifest signal → prime: NTRP0001 frame over the wire — handle, cursor, decoder-version, count, locator. The mesh stays a 240 B signal/control plane; the .ids moves as one immutable, hash-verified unit. Content-address + signal, never data-over-wire.
  6. Node mount + decode proof: one node resolves the handle → fetches from its local-region bucket (free) → verifies SHA-256 → mmap-mounts read-only → replays / understand()-decodes / atomizes locally on the mount.

Plus the residency panel in the Argus eye: handle, regions holding a copy, manifest timestamp, cursor, last get-latest — read from the manifest, never from an LLM.

Invariants

  • Language law: C# only at the zendesk-bridge boundary (long since done — the raw is already pulled). Seal / shape / hash / publish / mount / decode are C99, ours.
  • Read-only: this whole layer only ever reads Zendesk's already-pulled output. No write-back path exists.
  • Immutable by handle: a sealed .ids never changes under its handle; get-latest produces a NEW handle. Old handles keep resolving (append, don't mutate).

Out of scope (YAGNI)

  • ✗ All-6-region replication on day one — start us-west1, add us-west4 for nous-api next (Board GCP2). Copies cost pennies; add on demand.
  • ✗ derive-nps and other decoder passes — their own boards, off the resident raw.
  • ✗ Write-back to Zendesk — read-only holds.

6. First Move

Finish the tail that never ran — once, end to end. The raw.gz lands in Coldline (in flight now) AND the first full sealed .ids is built from the 7.6 GB, tarred + hashed to its IDS handle, uploaded to nous-kb-us-west1, and ONE node resolves the handle → fetches → verifies → mounts → decodes one real ticket to atoms.

That single round-trip proves "mesh-resident, replayable, decodable" end to end for pennies. Everything after it — the manifest signal path, the us-west4 copy, get-latest, the residency panel — is watching or repeating a loop that already closed once.

Choices + Consequences earned — the storage fork

Fork 1 — which tier holds the raw JSONL.gz?

NearlineColdlineArchive
Storage$0.01/GB/mo → ~$0.013/mo$0.004/GB/mo → ~$0.005/mo $0.0012/GB/mo → ~$0.002/mo

Pick: Coldline. The raw is re-read only on re-shape; Coldline's 90-day min fits a thing we may still re-shape this quarter. Archive optimizes a rounding error into a constraint.

Fork 2 — where does the .ids live?

Single us-west1us-west1 + us-west4All 6 regions day one
1st order~$0.01/mo; free egress to every bridge (all us-west1) +~$0.01/mo; nous-api (us-west4) also reads free +~$0.05/mo; every node everywhere reads free
2nd ordernous-api pays <$0.01 one-time cross-region — or waits Both live compute planes covered; matches where consumers actually are Replicating to regions with zero current consumers = ceremony

Pick: us-west1 now, us-west4 next (GCP2). Put copies where consumers are, not where the pattern permits. The regional pattern makes adding a region one gsutil cp — the option costs nothing to defer.

Escape Routes earned

  • Re-seal. §7: derived is rebuildable, the raw is never migrated. A bad seal = re-run the C over the Coldline raw.gz (~$0.03 retrieval), new handle, re-signal. Nothing precious.
  • Move the bytes. The handle is the hash — it cannot lie and never changes. Only the gs:// locator can change: copy the tgz anywhere, update the manifest locator, re-signal. GCS itself is swappable for any blob store.
  • Wrong tier. One command: gsutil objects update --storage-class (or a rewrite). Tiering mistakes cost cents and minutes, not migrations.

Pure Metal Test earned

  • ✓ Every load-bearing piece is ours: the sealed store, the decoder, the wire, the mesh, the seal/hash/publish C99. No frameworks, no managed runtimes, no LLM in the loop.
  • ✓ Zendesk is jacketed at the boundary and replaceable — and for this layer it's already behind us; the raw is on disk.
  • ✓ GCS is content-addressed and therefore swappable: the handle verifies the bytes wherever they live — bucket, node disk, any blob store. GCS is a locator, not a dependency.
