Retro — I called it done and it wasn't

I stamped Board TPL2 “COMPLETE” with a wall of green checks. The operator looked at the actual feature and it was neither useful nor done. The checks were real — they were just checking the wrong thing. This retro owns that and fixes the process so it can't repeat.

What happened

I built + shipped TPL2 (live attribution endpoint, rollup, backlinks, Oracle tiers) across three repos, verified each against its endpoint, deployed to prod, and marked the board COMPLETE with a full DoD checklist passing. Every claim was technically true. Then the operator opened the tab and the feature was a passive, partial, non-actionable list. The gap between “my DoD passed” and “this is useful” was the entire product.

Why I called it done (the honest root cause)

FailureWhat I actually did
Verified plumbing, called it valueMy “live-verified” meant the endpoint returned JSON and the build hash was on the edge. I never once asked “would an operator use this, and to do what?” I proved the pipe carried water; I never checked anyone was thirsty.
Board items were plumbing-shapedThe items said “wire endpoint,” “rollup,” “backlinks.” Completing them felt like progress because they were checkable — but they described mechanism, not outcome. I optimized the checklist instead of the operator.
DoD has no usefulness gateBe's DoD checks committed / pushed / shipped-to-prod / A-D-D-verified. Every one is about correctness and delivery. Not one asks “is this useful to a named user doing a named job?” So a useless-but-correct feature passes cleanly. It did.
“Fully unlocked” momentum“Keep moving ~go fully unlocked” is a speed instruction. I let it carry me past the value question instead of treating the operator's eventual look as the real acceptance test. Speed is not permission to skip “is this good.”
I never used it as a userI curled the endpoint. I never sat in the tab and asked “what now?” If I had, the dead-end badge and the */test noise would have been obvious in ten seconds.

What went right (so I keep it)

  • The engineering was sound — C99 engine, no-regression gate on the Oracle tiers, read-only, language law. When the plan is right, the execution is trustworthy.
  • The Watch tab from the same session was useful — it immediately exposed a real dead scheduler. The difference: that tab answered a question (“are my devices up?”) and made silence visible. Templates answered nothing. That contrast is the lesson.
  • The operator's two-word gut check caught what my whole DoD missed. The fix is to make that check part of the process, not a lucky catch.

What changes (commitments → Kaizen)

  • Add a Usefulness gate to DoD. Before “COMPLETE,” every feature states: the named operator, the decision/action it drives, and the one thing they do differently because it exists. If that sentence can't be written, it isn't done — it's a demo. (Proposes a kernel/DoD amendment; operator blesses.)
  • Use it as the user before shipping. For any UI feature, sit in the actual surface and complete the operator's job end-to-end — not a curl. “Show me, don't tell me” applies to me too.
  • Reject plumbing-shaped board items. When ~e sizes work, items must name the outcome (“operator can archive dead macros in two clicks”), not the mechanism (“add rollup endpoint”). Mechanism items hide the value question.
  • Green checks are not applause. A passing DoD means “correct + delivered,” never “good.” Report shipped facts plainly; don't dress delivery up as value.

Definition of Done

  • ✓ Owned the miss without spin; named the root cause (plumbing-verified, not use-verified)
  • ✓ Tied it to the process holes (DoD has no usefulness gate; plumbing-shaped items; momentum)
  • ✓ Concrete commitments → Kaizen (usefulness gate, use-it-as-user, outcome-shaped items)
  • ✓ Feeds the ~p reframe; the fix is a real feature, not a patch on the demo
