De-atomize POC — conversation ↔ program ↔ conversation

Built from be/designs/2026-07-09-1755-design-deatomize-poc-fable5.html. Pure C99, read-only, no model, no network. Mounts the sealed raw IDS (all 8.2M facts mmap'd in memory) and proves the decoder runs both directions on real Mendocino Farms threads.

What was built

  • mendo-zendesk-ai/be/tools/deatomize_poc.c (committed 54aca81) — mounts data/full/raw-ids-full (the 431,260-ticket sealed store), and for each real comment runs ORIGINAL → understand() → PROGRAM (.pi) → restate_with(english) → DE-ATOMIZED.
  • Two datamaps: PI_DATAMAP (atoms → .pi operators, the program view) and a new EN_DATAMAP (atoms → plain-English connectives, the VOICE / de-atomize stage).
  • Move 2 — conversation FROM programs: the vocabulary spine that recurs across a cluster of same-situation threads, mined from their real programs.
  • Everything reuse: restate_with, understand, the mmap'd store, the derive.c mount pattern. New code is only the ~140-line harness + the English datamap.

Honest edges

  • De-atomize is lossy on grammar (WordNet Phase-2 not loaded here; hardcoded grammar only): "they'll is able", "working with ibrahim at is". The intent reconstitutes; the surface polish needs the datamap + WordNet enrichment.
  • Move 2's spine carries common-word noise (com, had, well) alongside the real domain spine (lrs, gateway, pair, support, team); a stopword pass + real situation-typing sharpens it. That's next, not POC scope.
