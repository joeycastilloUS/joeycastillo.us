The one that matters: ungrounded ticket-search ✓ verified in source

When an operator asks Argus "how many tickets about refunds this week", the message is keyword-routed to the bridge's /ask, which returns { answer, cited:[caseIds] }. Then:

// ArgusBrain.cs:184
return cited.Count > 0
    ? $"{answer}\n\nCited: {string.Join(", ", cited)}"   // <-- appended with ZERO verification
    : answer;

argus-slack never checks those case IDs against any store. If the model behind /ask invents "Ticket #999999", Argus posts it to the operator as fact. This is the trust-killer — an ops tool that confidently cites tickets that don't exist is worse than one that says "I don't know."

The fix (two layers, both use the sealed IDS you now have)

  • Ground the source: /ask should answer from the sealed IDS (the deterministic store), not a bare model. The store holds every real ticket + comment; grounding = read, not guess.
  • Verify at the edge: argus-slack checks each cited ID exists in the scoped source's IDS before rendering it; unverifiable IDs are dropped or flagged, never shown as fact.

This is exactly the source-scoped design's First Move — grounding is per-source, so it lands with the Zendesk slot.

The architecture answer: one Argus, source-scoped by rights

Your instinct was right — it's a rights thing, not a second app. One Argus shell; each channel/DM is bound to one source slot = { bridge + sealed IDS (grounding) + persona + write lane }, gated by the operator's entitled sources. The Mendo channel talks only Zendesk and grounds on mendo-tickets-ids; a Salesforce-only teammate's Argus never touches Zendesk. One bot, one install; a new source is a mounted slot, not a forked app. Nearly all reuse (§3 bridges, the Slot pattern, Operator-Identity rights, both live IDS buckets). Full design: the source-scoped design.

Why not separate apps: N bots, N installs, N secrets, and every shell fix lands N times. Why not per-message inference: a guessed scope reopens the blur, makes rights unenforceable, and leaves grounding non-deterministic. The channel-scoped rights slot is the only option that gives focus and unity — and the slot boundary is exactly the app-fork seam, so it keeps the "separate app" option open if a source ever outgrows the shell.
