Be ~i inspect — metal.CE Latchkey: Personal URL + Comments for ~fu Published 2026-07-04 18:51 PDT

Does metal.CE have solid Latchkey support with a real personal URL + comments for ~fu?

Read-only inspection of the Latchkey publish chain: be/tools/latchkey_publish.py, be/tools/cap_tokens.py, the Latchkey Publish Protocol, the ~fu spec, the a-d-d pattern, and the C99 skill spec. Three questions, three honest verdicts.

What It Is

The Latchkey capability today: every metal artifact is supposed to be born a Latchkey — at publish time, latchkey_publish.py mints (or idempotently reuses) an unguessable UUIDv4 token, records it in the private Ledger (be/cap-tokens.json), injects a Share button + a live comment Thread into the HTML, hardens it (no-referrer, gate stripped), stages the Door copy at be/s/<token>/index.html, and mirrors it to the public site at /metal/s/<token>/. The token is the whole credential — possession is admission. Comments post to a GCP bridge keyed on doc=<token>, and ~fu is the Attendant: one sweep reads the operator's comments and acts on them.

Five named parts per the pattern (a-d-d/patterns/latchkey.md:19-25): Latchkey (the token), Ledger (private registry), Door (the /s/<uuid>/ serving convention), Thread (comments keyed on the token), Attendant (~fu).

The Personal-URL Gap

What's hardcoded. Three constants pin every operator to Joey's domain:

  • PUBLIC_BASE = "https://joeycastillo.us" — latchkey_publish.py:31. No env override at all (unlike SITE_ROOT).
  • SITE_ROOT = os.environ.get("METAL_SITE_ROOT", "C:/joeycastillo.us") — latchkey_publish.py:30. Overridable, but the cap-URL printed and injected would still say joeycastillo.us even if you redirect the file copy.
  • PUBLIC_PREFIX = "/metal/s" — cap_tokens.py:62.

What personal means here. The convention already defines it: every Pantheon operator owns their FQDN and the matching GitHub org (scottyledell.us → scottyledellUS). Scotty's inspection of Scotty's project should mint https://scottyledell.us/<project>/s/<token>/. Today it would mint https://joeycastillo.us/metal/s/<token>/ — a URL on a site repo Scotty almost certainly cannot write to, so his publish step silently degrades to "staged only" (latchkey_publish.py:153-157 skips when the site root isn't found) and the injected Share button copies a dead link on someone else's domain.

What it would take:

  • Parameterize. Derive PUBLIC_BASE, SITE_ROOT, and PUBLIC_PREFIX from the operator's identity — an operator.fqdn field in be/config.json (or the vault), with CLI/env override. The C99 skill spec already mandates exactly this (skill CLAUDE.md:116-117); the Python is behind its own spec.
  • Per-operator site repo. The operator's FQDN site (<name>US/<name>.us-style, Cloudflare Pages) with a /s/ (or /<project>/s/) Door, robots.txt disallow, no index. Joey's side of this exists and is verified; it's the template.
  • DNS. Already solved by onboarding — the operator must own the FQDN before sovereign storage is provisioned (kernel CLAUDE.md, FQDN↔org convention). No new DNS work; the domain is the operator's by definition.
  • Nothing changes for the Thread. Because doc=token (a globally unique UUID), the comment keying survives any domain move — the same token works on any Door.

The irony is not lost: this very inspection, when published per protocol, will get a cap-URL on joeycastillo.us/metal/s/<token>/ — a live demonstration of the exact gap it documents. The report about the missing personal URL cannot be served from a personal URL.

Comments + ~fu — is the loop real?

Yes, the loop is real and closed — for one operator.

  • Widget (write side): injected into every published artifact; posts POST /comments?doc=<token>&obj=<section> with author + text; loads existing comments; renders a seen by metal receipt; flags ~-prefixed comments as commands (latchkey_publish.py:48-96).
  • ~fu (read side / Attendant): one sweep of GET /comments/recent, act on each unseen operator comment (execute tilde commands scoped to the artifact, answer questions, optionally reply in-thread as author=metal), stamp POST /comments/seen?doc= (commands.md:1862-1875). Deliberately pull-based — no poller, no idle cost (commands.md:1877-1879), which honors the pattern's anti-pattern warning about unwired attendance while dodging its cost.
  • doc=token resolution: the crucial design win. The Thread keys on the Latchkey, never the artifact name (pattern anti-pattern, latchkey.md:63), and get_or_mint records title + rel_path in the Ledger precisely so ~fu can attribute a token back to a human-readable artifact from the Ledger alone (cap_tokens.py:12-14). This works regardless of domain — a personal-URL future needs zero Thread changes.

Why PARTIAL, not SOLID: the backend is a single shared bridge — https://gcp-bridge-692654781984.us-west1.run.app — hardcoded in both the publisher (latchkey_publish.py:29) and the ~fu spec (commands.md:1865). Not per-operator: all operators' comments would land in one store Joey controls, and ~fu's /comments/recent sweep is global — a multi-operator world would leak everyone's threads into everyone's sweep. The pattern itself names this debt: "a GCP bridge-tier dependency that must later retire to the Pantheon mesh" (latchkey.md:45), and the skill boundary agrees the bridge URL must be a parameter (skill CLAUDE.md:116-117, 124-125). For the single-operator present, the loop works end to end; for the personal/per-operator future, the bridge is the second hardcoded Joey-ism.

Bottom Line

Partial — one of three, honestly. The Latchkey engine itself is genuinely solid: idempotent mint, a private Ledger correctly anchored across worktrees and never published, UUID validation, hardening, idempotent injection, a verified Door (robots.txt live), and a protocol that makes every new artifact born a Latchkey. The comments loop is real and closed — widget, bridge, doc=token keying, and a working ~fu Attendant sweep — but it rides a single shared, hardcoded bridge rather than anything per-operator. And the crux — a real personal URL — is flatly missing: PUBLIC_BASE is hardcoded to https://joeycastillo.us, so every operator on the Pantheon publishes into Joey's domain (or fails to publish at all), in direct violation of the skill spec's own "parameters, never hardcoded" rule and the FQDN-is-identity convention the kernel is built on. The fix is well-scoped — derive base/root/prefix from the operator's FQDN, stand up the per-operator Door on their existing domain, parameterize the bridge — and the doc=token design means the Thread survives the move untouched. metal.CE has a solid Latchkey with working comments for ~fu; it does not yet have a personal one.
