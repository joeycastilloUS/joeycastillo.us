Gemini-on-Vertex, grounded on the .ids: Argus answers true

Two moves. Take Argus off the Anthropic API and run its brain on Gemini in Vertex — in your own GCP, over ADC, no key. Then hand that brain a door into the sealed .ids so when you ask it to find a ticket or spot a trend, it reads the real corpus and cites it — instead of making it up.

Grounded on tonight's recon of gcp-bridge (Gemini/Vertex via ADC, already live) + onedine-raw (the C99 IDS query service, already live). The only genuinely new code is a search endpoint that wraps a matcher we already wrote, and a retrieve-then-read loop. Model: Fable 5.

1. Need

You proved it yourself an hour ago: you DM Argus, and Argus-the-voice is a stateless, blindfolded Claude. Ask it "how many detractors today?" and it answers with confidence and invents the number. Two wrongs to fix, and you named both:

  • Wrong brain. Argus calls api.anthropic.com with a key in Secret Manager. You already run Gemini on Vertex inside acmedev-platform-hq — ADC identity, no key, data never leaves the project. Paying a second vendor for a worse-governed path when the good one is already wired is backwards.
  • Blind brain. 431,260 tickets / 977,029 comments are sealed, hash-verified, mesh-resident, and queryable today — and the chat brain cannot see a single one of them. The data is there. The voice just isn't plugged into it.

Why now: the raw landed and the reader (/replay) works; the Gemini jacket is deployed and tested; Argus chat went live tonight. Every part exists in isolation. This design is wiring, not building — the cheapest it will ever be, and the moment it's worth doing is the moment all three are simultaneously live, which is now.

2. Raw user stories operator voice

  • "As the operator, when I ask Argus 'find tickets about a refund that got declined at checkout', I want real Mendo ticket IDs with one-line summaries — not a plausible fiction."
  • "I want Argus running on Gemini in my own Vertex, ADC, no Anthropic key — my data, my project, my bill."
  • "I want to ask 'what are guests complaining about this week' and get patterns grounded in the corpus — ranked, with counts, with example tickets."
  • "When it answers, I want it to have read the real thread (atomized or restated, I don't care which) so the answer traces to a source I can open."
  • "If it can't find anything, I want it to say so — 'no matching tickets' beats a confident hallucination every time."
  • "Let's just see how good we are — point it at the data and find out where it's strong and where it whiffs."

5. Shape two lanes, read-only, language-law-clean

Two independent lanes. Lane A stands alone (kills Anthropic today). Lane B makes the brain useful. Ship A first because it's trivial and unblocks the "no Anthropic" requirement immediately; B is the value.

Lane A — Brain swap (Anthropic → Gemini/Vertex). 
argus-slack  ArgusBrain.Respond(text)
    was:  POST api.anthropic.com/v1/messages   (x-api-key)
    now:  POST {GCP_BRIDGE_URL}/vertex/gemini   (X-Bridge-Token, ADC on the far side)
  — drop ANTHROPIC_API_KEY + ANTHROPIC_MODEL from the service
  — add GCP_BRIDGE_URL + reuse the bridge token already in the mesh
  — persona system-prompt rides along in the messages body, unchanged
Nothing else in argus-slack moves. Approval Cards, the gated Zendesk write — untouched.
Lane B — Ground it in the .ids. 
onedine-raw (C99):   + GET /search?q=&field=&limit=     (wrap rawq matcher)
                     + GET /facets?p=&limit=            (count-by-value)

gcp-bridge (C# jacket):
   POST /ask { question }
     1. extract query terms      (cheap: Gemini one-shot OR literal keyword split)
     2. GET onedine-raw/search   -> candidate case IDs
     3. GET onedine-raw/replay   -> top-K real threads (fields + restated program)
     4. Gemini generateContent   -> answer + cite the case IDs it used
     5. no hits -> "no matching tickets" (never invent)

argus-slack: "find/how many/trend" style DM -> POST /ask ; chit-chat -> /vertex/gemini

Boundaries. Read-only over the IDS — the raw is never written by this path. Language law honored: intelligence (search, facets) stays C99 in onedine-raw; the Gemini/Vertex plumbing is the C# jacket in gcp-bridge; the Slack surface is vanilla. Out of scope (v1): vector/embedding search, Gemini function-calling multi-hop, real BI analytics, any write-back, thread memory (Argus stays single-turn until the grounding proves out — memory is a fast follow).

6. Choices + consequences the forks that matter

Fork 1 — How Gemini reaches the data. (A) retrieve-then-read [PICK]: bridge extracts terms, one deterministic search, reads top-K, Gemini answers. Cheap, predictable, ships this week, easy to judge. (B) function-calling: give Gemini tools and let it drive multi-hop queries. More powerful, but the tool loop is unbuilt in GeminiClient and adds latency + non-determinism. 2nd-order: A gives a clean baseline to measure "how good are we" against — exactly your ask. B is v2 once A shows where the ceiling is. Rule 6: ship A, learn, then decide if B earns its keep.
Fork 2 — Search quality. (A) substring [PICK for v1]: rawq exists, $0, ~200ms over 431K. Misses synonyms ("refund" won't catch "money back"). (B) inverted index / BM25: one more decoder pass, real relevance. (C) embeddings/vector: semantic, catches paraphrase, but new infra + cost. Consequence: substring is honest and instant and tells us the floor. When it whiffs on a real question, that whiff is the spec for B or C — we'll know which by watching it miss, not by guessing now.
Fork 3 — Where the brain lives. Keep ArgusBrain in argus-slack pointing at gcp-bridge [PICK], vs. move all chat into gcp-bridge. Consequence: keeping the thin caller in argus-slack means Vertex + ADC + the data reads all stay behind one door (gcp-bridge), which already holds them. One place for the key-free identity, one place for the corpus. Don't spread it.

7. Escape routes every load-bearing choice reverses

  • Brain swap: a one-line URL revert + re-add the secret puts Anthropic back. The persona and Slack plumbing never changed. Fully reversible.
  • /search + /facets: additive endpoints on a read-only service — if wrong, delete them; nothing else depends on them being right. Fully reversible.
  • /ask: additive route; if the grounded answers are bad, Argus falls back to plain /vertex/gemini chat. Reversible.
  • No irreversible choices. The raw .ids is never mutated by any lane here — the source of record is untouched, so there is nothing to roll back on the data side. ⚠ none.

8. First move smallest proof it's real

Expose GET /search?q= on onedine-raw — wrap the rawq matcher that already exists — deploy, and curl it live:

GET onedine-raw/search?q=refund&limit=10
  -> [ { "s":"case:500...", "p":"Description", "o":"...refund was declined..." }, ... real Mendo IDs ]

Why this first: retrieval-over-HTTP is the keystone — /ask, grounding, trends all stand on it, and nothing proves "the data is reachable by a question" faster. It's ~50 lines around code we've already written and tested as a CLI. The brain swap (Lane A) runs in parallel — independent, trivial, and it satisfies your "no Anthropic" line the same afternoon. When both land: DM Argus "find refund-declined tickets," watch it return real IDs, answered by Gemini, zero Anthropic. That's the whole design in one sentence, proven.
