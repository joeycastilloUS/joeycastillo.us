Raw IDS out of the LLM zone: pure-C pipeline + the Argus eye

v2 — supersedes be/designs/2026-07-09-1443-design-zendesk-raw-ids-import-tool-fable5.html. v1's bones stand (the Argus tab, the /import/* contract); v1's premise was wrong. It said the import "keeps not landing" and "has no home." A deep inspection (~i deep) run an hour later found the fetch live and nearly complete — and found the real rot downstream: the shaper silently broken, the sealer never compiled, every deterministic step trapped in the LLM zone. This design corrects the premise and designs the pipeline out.

Root cause — name it plainly: the entire pull → shape → seal → import → mesh pipeline lives in the LLM zone — ad-hoc python scripts hand-run inside ephemeral Claude sessions, in a sibling repo (mendo-zendesk-ai), with no operational eye. So: the pull dies when the session dies; the shaper silently emits 2 tickets and nobody notices; the sealer never gets compiled; nothing watches; it stalls at "almost" every single time. v1 was half-wrong — the fetch is essentially done. The real rot is that the deterministic steps, which §9 says must be deterministic and ours, are trapped as babysat scripts in the probabilistic zone.

Joey: "we need to figure out where we are at and do this better. let's add to our bridges and pure C code base. take this away from LLM zone. i need way better tools in argus to keep an eye on this and let's hopefully keep this going … we were working on getting the raw data so we can replay and decode/ids as often as we want to."

The move: a deterministic pure-C99 + bridge pipeline that runs headless, survives session death, and is watched + driven from a real Argus operational tool — so the raw lands, stays current, and can be replayed/decoded on demand forever.

2. Raw User Stories

  • As the operator, I want the pull running headless, so that closing a Claude session costs zero — no more session archaeology to find where a pull died.
  • As the operator, I want a shaper that cannot silently do 2 of 408K, so that a partial failure is a loud red number, not a discovery weeks later.
  • As the operator, I want the full raw sealed, so that I can replay and decode into any number of derived IDS as often as I want (§7's whole point).
  • As the operator, I want an Argus eye that shows exactly which stage is stuck — pulled vs shaped vs sealed vs published — so a glance replaces an investigation.
  • As the operator, I want Refresh (get-latest) from the stored cursor, so the raw stays current without a full re-pull.
  • As the operator, I want the sealed raw resident in the mesh under a content-addressed handle, so any node reads one verifiable truth.

5. Shape

Minimum viable structure — the determinism split of §9, applied to data:

  1. Boundary (C#, exists): zendesk-bridge stays the ONLY foreign piece — the fetch jacket. Nothing else foreign, nothing else C#.
  2. Pull loop (C99, mostly built): harden the resumable puller and get its driver OFF the ephemeral session — a headless runner the operator owns (Choice below). Cursor + per-batch provenance stay in the manifest/store, exactly as the python version proved. The python puller retires only after the C99 driver shows cursor parity.
  3. Shape (C99, NEW — replaces the broken python): JSONL → append-only (s,p,o) triples; comments first-class comment:<id> subjects with the public/internal flag. Coverage self-check baked in: counts tickets_in vs tickets_shaped; mismatch = nonzero exit + red in the eye. The 2-of-408K bug becomes structurally impossible to miss.
  4. Seal (C99, FIX): compile raw_import.c; run it on the FULL set → the full sealed nous TripleStore. One writer, append-only, provenance per batch (§7). This step has literally never run at scale — it is a compile and a run away.
  5. Mesh residency (C99, NEW, downstream): seal → SHA-256 IDS handle → manifest signal on the wire (NTRP0001) → resident .ids the prime or any node resolves and mounts read-only.
  6. The eye (vanilla JS tab, NEW): rawids tab in the Zendesk app. Shows LIVE, from the pipeline's own status.json served over /import/status — never from an LLM:
    • Pull: cursor / tickets / comments / eos / pages / rate / ETA.
    • Shape: coverage — tickets shaped vs pulled (this line alone would have caught the 2-ticket bug instantly).
    • Seal: status + IDS handle + store size.
    • Mesh: published? handle resolvable?
    • Controls: Resume pull · Refresh (get-latest) · Re-shape · Re-seal · Replay a ticket.
    The eye is how it "keeps going": a human glances, sees the stalled stage, nudges it.

Invariants

  • Read-only: the pipeline READS Zendesk, WRITES only the local sealed raw. No write-back path exists to disarm.
  • Language law: C# only at the zendesk-bridge boundary; every stage after the boundary is C99, compiled, deterministic, ours.
  • Done = production (Rule 9): "sealed" means the store exists, the handle resolves, and a ticket replays from it — verifiable by a third party, not by a claim in a transcript.

Out of scope (YAGNI)

  • ✗ derive-nps / Focus wiring — a decoder pass off the landed raw, its own board.
  • ✗ Full config-object walk (triggers, automations, macros, SLAs, forms) beyond tickets+comments — append-only means adding it later costs nothing now.
  • ✗ Any write-back to Zendesk — the read-only invariant holds.

6. First Move

The fetch is done; the missing deterministic steps are a compile and a run away. So run them. Compile raw_import.c and a pure-C shaper, run them on the ALREADY-PULLED 7 GB ticket_events.jsonl — 408K+ tickets sitting on disk right now — to produce the first FULL sealed raw IDS, and prove ONE real ticket replays from it: real comments, real thread, decoded from the seal.

That single move converts the pull that's already on disk into the artifact §7 has wanted since 2026-07-04 — and it flushes out every at-scale surprise (memory, malformed events, throughput) before any service or tab exists. The shaper ships with its coverage self-check from line one: shaped == pulled or it halts red. The headless driver, the Argus eye, and the mesh publish follow as the plan's phases — each one watching or feeding a raw that already exists.

Choices + Consequences earned — load-bearing fork

Where do the deterministic steps run?

A — bare headless C99 runner (operator box/VM) B — C99 service behind /import/*, tab-driven (v1's shape) C — folded into zendesk-bridge
1st order Zero new surface; binaries run today next to the 7 GB. Headless + resumable + a status/control contract the eye binds to. One deploy; "add to our bridges" read literally.
2nd order No status surface — the eye has nothing to read; "is it running?" is again a shell archaeology question. The LLM-zone failure with the LLM swapped for a forgotten terminal. The tab reads the pipeline's own status.json; controls (resume, re-shape, re-seal, replay) are HTTP verbs; the contract is location-blind — localhost today, promoted later without a rewrite. Violates language law: the pipeline is OURS and deterministic — putting C99 stages inside a C# service (or rewriting them in C#) plants our code in the jacket. The bridge also holds the vendor secret; fattening it fattens the blast radius.
3rd order Every downstream board inherits "whose terminal is it on?" fragility. The import becomes an operational fact of Argus; refresh is schedulable; the mesh publish is one more verb on the same contract. Vendor-jacket redeploys now risk the data pipeline; two lifecycles, one service.

Pick: B. C fails the language law outright — "add to our bridges" is honored at the boundary (the bridge stays the fetch jacket and takes any new foreign-shaped endpoint), not by moving our deterministic C into the jacket. A is B minus the eye — and the eye is half of Joey's direction. B strictly dominates: a compiled C99 service that IS the headless runner, wearing a thin /import/* status+control contract. First Move runs the bare binaries by hand (A's virtue, once); the service wraps the same binaries. The test — out of the LLM zone, watchable, resumable — only B passes whole.

Escape Routes earned

  • Re-pull. Append-only + per-batch provenance ⇒ a bad pull is a droppable batch; the cursor makes any restart a resume. Worst case is wall-clock at ~10 RPM, zero code.
  • Re-shape / re-seal. Derived is rebuildable by definition (§7); the raw is never migrated. A shaper bug found later = fix the C, re-run over the same JSONL, re-seal, diff. The 7 GB source and the sealed raw both stand.
  • Mesh publish. The handle is the hash of the bytes — it can never lie. Only the locator can change: move the sealed file, re-signal, done. Backing out of mesh residency entirely leaves a fully usable local raw.

Pure Metal Test earned

  • ✓ Every load-bearing piece is ours: nous store, decoder, wire, mesh; the C99 shaper + sealer + driver; the vanilla-JS eye. No frameworks, no managed runtimes, no LLM in the loop.
  • ✓ The one foreign dep — Zendesk — is jacketed at the boundary and replaceable: swap the jacket, everything after the boundary stands.
  • ✓ The one infra dep — where the sealed bytes live — is content-addressed (the handle is the hash) and therefore swappable: box, VM, prime, any node.
