How Operators & the Overlord actually log in

Your open question: "how do the mighty Purcius operators and the overlord actually login — we have the best security code and framework on earth, so which path?" Here is the path, using what Genesis already gives us and nothing borrowed:

  1. Device key, not a password. On first login the browser mints a non-extractable Ed25519 key (Web Crypto, hardware-backed where present). The private half never leaves the device; nothing to phish, nothing to paste.
  2. Warrant from the root. The Purcius key (root for you, operator key for Scotty) signs device_pub ‖ "console" — a Warrant that says "this device speaks for this key." Genesis is the authenticator: it shows a QR / push, you approve on the key-holding device.
  3. Challenge–response. The console hands the device a random nonce; the device signs it; the edge verifies the signature against the Warrant, and the Warrant against the published _key.<fqdn> anchor (DNS/TXT, third-party-verifiable). No secret ever sits on the server.

Best-on-earth, honestly: the crypto is your own audited substrate — ed25519 + X25519 + AES-GCM + Shamir, all KAT-tested in test_substrate.c (RFC 8032/7748, NIST GCM). We are not buying WebAuthn; we are being the authority. The device key is the convenience layer; the Purcius key is the root of trust.

Onboarding — operator and user

Operator (key-holder) — four steps, all on existing code:

  1. Mint — genesis-cli mint 5 3 <fqdn> ./out (or the Genesis app), air-gapped. Emits operator.pub, five 65-byte shares, and the _key.<fqdn> TXT line. Root wiped on return — already built, already KAT-tested.
  2. Publish — the operator plants _key.<fqdn> in their DNS. Third-party-verifiable with dig.
  3. Warrant — the Overlord verifies the anchor from an independent vantage, then signs op_pub ‖ fqdn ‖ role ‖ expires with the Purcius root (genesis_sign, exists today). The Warrant IS the seat.
  4. Enroll a device — the console mints a non-extractable Ed25519 device key (Web Crypto), shows its pubkey as a QR; the Genesis app on the key-holding device scans and signs device_pub ‖ "console" ‖ issued_at with the operator key. Device speaks for operator; operator speaks for the anchor.

User — no key, no ceremony: an invite (below) plus Slack / Discord / Google OIDC. The provider proves who; the invite decides what. Users never touch Genesis.

Invite — a single-use cap-link that becomes identity

Sibling of Latchkey: a Latchkey token grants reading a document; an Invite token grants becoming a member. Same shape — unguessable cap-URL, server-side ledger, fail-closed.

Joey ─▶ mint invite {org: mendo, apps: [argus], bind: michelle@…, expires: 7d}
     ─▶ https://argus.com.ai/join/<token>      (DM'd over the org's Slack)
Michelle clicks ─▶ OIDC login (Slack) ─▶ identity slack:U… bound to org
     ─▶ email appended to argus_crew_v1 ─▶ token BURNED (used_by, used_at)
  • Single-use. Redeemed = burned. A leaked used link is a dead link.
  • Bindable. An invite may pin an email (only Michelle can redeem) or be open-but-expiring for a batch.
  • Carries the grant. Org + apps ride in the server-side record, not the URL. Redeeming writes the allow-list entry — the same argus_crew_v1 mechanism already live.
  • Org membership answer: the invite names the org. One email invited by two orgs = two memberships, deliberately — the workspace that DM'd the link is the org that vouched.

This closes the open question above: Michelle, Elbert, Brendan come off their Slack workspace via one DM'd link each. No admin screen, no CSV, no shared password.

Recovery — lost device ≠ lost identity

Genesis already mints recovery: the root never persists whole — it exists as 5 Shamir shares, any 3 reconstruct, any 2 reveal nothing (information-theoretic, not just hard). Canonical share placement per operator:

#WhereWhy
1Genesis app on the operator's phoneThe everyday share
2Printed, home safeSurvives the phone
3Bank deposit boxSurvives the house
4Trusted person (sealed)Survives the operator's geography
5Operator's private vault repo (<name>US/vault)One share alone leaks zero bits — safe at rest in git

The recover flow — genesis-cli recover then genesis_resplit, both built and self-verifying:

  1. Gather any 3 shares. genesis-cli recover root.key share-a share-b share-c — same 64-byte root, same pubkey.
  2. genesis_resplit — same key, five fresh shares; the old set (including whatever the lost device held) is dead. Redistribute per the table.
  3. Revoke the lost device's Warrant serial; enroll the new device (QR, as in onboarding).

The identity survives untouched: the pubkey never changed, so _key.<fqdn> stays as published and every Warrant remains valid. Only the device layer resets. If enough shares are lost that 3 can't be gathered — that is the one unrecoverable event, by design; the placement table exists to make it require three simultaneous disasters.

A session is never a bearer instrument that outlives the Overlord's word — pause a person at 2:14, they are out by 2:15, and Argus itself contains zero enforcement code to get wrong.

Ages ride inside signed material, so an operator can't quietly extend their own life — changing a date breaks the signature. Grace: the console banners the last 7 days before any refusal, then refuses. The banner is courtesy; the refusal is the policy.
