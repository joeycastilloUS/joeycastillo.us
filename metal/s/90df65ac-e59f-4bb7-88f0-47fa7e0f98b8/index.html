Zendesk → Raw IDS: an import tool in Argus

Land the raw. Watchable, resumable, replayable — from a tab, not a side repo.

1. Summary

This plan operationalizes be/designs/2026-07-09-1443-design-zendesk-raw-ids-import-tool-fable5.html: give the perpetually-stalled full Zendesk import an operational home — a new Raw IDS first tab in the Argus Zendesk app, backed by a small C99 import service exposing a location-blind /import/start | /import/status | /import/replay?ticket= contract. The service drives the live zendesk-bridge (C# jacket) for reads and writes the sealed nous TripleStore per ARCHITECTURE.md §7. The proven v0 tools (C:\mendo-zendesk-ai\be\tools\ — 1,000 tickets → 5,002 triples, round-trip clean) get promoted into argus/src/import/. Why now: every downstream board (NPS, Focus-real-data, audit, conversation-as-program) is blocked on the same missing artifact — the landed raw. Land it once; everything else becomes a decoder pass.

Rule 6: this is a v1 hypothesis. Revision expected — the Unknowns and When To Revisit sections below name where.

2. Intent

The full Zendesk history — tickets WITH comment events — lands in one sealed, append-only Raw IDS with per-batch provenance and a stored cursor, and the operator can start it, watch it (pulled / cursor / triples / elapsed / ETA), resume it after any interruption, and replay one real ticket's actual thread from the raw inside the console — proof by eyes, not by claim. When this plan is done, "do we have the data?" is a glance at the Raw IDS tab, and the import is boring.

3. Scope

In scope

  • Promote the v0 C99 tools (raw_import.c, replay.c, retrieve.c, ticket_json.c, program.c, qstore.c; derive*.c rides along but stays downstream) into argus/src/import/.
  • The C99 import service: /import/start, /import/status, /import/replay?ticket= — localhost first, Cloud Run as a later promotion.
  • The Raw IDS tab — new FIRST tab in the Zendesk app (web/index.html APPS registry): Import all / Refresh / Replay, live progress, standing-raw panel.
  • Bounded proof pull (~30 days, tickets + comment_events + CSAT), then the full start_time=0 corpus pull, then the refresh/get-latest incremental loop.
  • Replay of one real ticket end-to-end from the landed raw, rendered in the tab.

Explicitly OUT of scope — downstream boards off the landed raw

  • ✗ Mesh residency — content-address, manifest signal, bulk channel. A downstream board consumes the raw this tool lands.
  • ✗ derive-nps / Focus demoware-kill — a decoder pass over the landed raw; boards after RAW2.
  • ✗ Full config-object walk (triggers, automations, macros, SLAs, forms, AI config) beyond tickets+comments — §7 wants it eventually; append-only means adding it later costs nothing now.

Plainly: this plan lands the raw and proves it replayable. It does not put the raw in the mesh, does not derive NPS, does not walk config objects. Those chains stalled this import three times; they are cut, not forgotten.

4. Approach

Staged, local-first → Cloud Run promotion. The tab binds ONLY the /import/* contract — the seam. The same C99 binary serves it from localhost in the First Move and from Cloud Run at promotion (own service, one-per-source pattern, mirrors meraki/zendesk-bridge). Local-first proves the pipe watchable and cursor-resumable before any new infrastructure exists; promotion is a deploy, not a rewrite.

Zendesk (foreign, READ ONLY)
   │
   ▼
zendesk-bridge ·· C# jacket (existing Cloud Run) — the ONLY C# in the chain
   │  incremental/ticket_events.json?start_time=<cursor>&include=comment_events  ~10 RPM
   ▼
import service ·· C99, promoted v0 tools          /import/start
   │  one writer · append-only · provenance       /import/status  ◀── polls ── Raw IDS tab
   ▼                                              /import/replay?ticket=
sealed Raw IDS ·· nous triple_store · cursor lives IN the raw
   │
   ▼
replay.c / retrieve.c ·· decode one real thread → trace in the tab
  • C# jacket boundary: the Zendesk fetch stays in zendesk-bridge. Everything after that boundary — service, store, cursor, replay — is C99, ours.
  • Append-only sealed raw: one writer (the service). A changed object appends new triples with a later timestamp; prior state stays replayable. Each batch carries provenance: cursor, wall-clock, endpoint, count. A bad pull is a droppable batch.
  • Cursor in the raw (import:cursor): the store carries its own resume point — a killed session, closed laptop, or service restart resumes, never restarts.
  • get-latest is a view, not a mutation: deterministic read resolving each subject to its most-recent triple. History is never discarded.
  • Replay via the promoted tools: replay.c + retrieve.c decode a ticket's real thread from the raw — the same v0 code that round-tripped clean, now with a console surface.
  • Comments are first-class: comment:<id> subjects beside ticket:<id>, same store shape as v0.

6. Constraints

  • Language law. C# exists only in the zendesk-bridge jacket at the foreign boundary. Import service, store, cursor, replay, tab wiring — C99 / vanilla JS. No new C#.
  • Read-only invariant. The import READS Zendesk and WRITES only the local sealed raw. No write-back path exists — nothing to disarm.
  • One writer, append-only. Only the import service writes the raw. Never mutate, never migrate — a changed object appends. Provenance on every batch.
  • Done = production (Rule 9). RAW1 is Done when the tab is live and a real ticket replays from a real landed raw, third-party verifiable. RAW2/P7 is Done when the deployed contract serves status from a machine that isn't the dev box.
  • Rule 10 — a-d-d proposal, not a write. This plan PROPOSES a small §7 addition — the operational home (Raw IDS tab + import service + /import/* contract) as the named executor of the Raw IDS architecture. It needs operator approval before anything lands in ARCHITECTURE.md. The plan does not write architecture.
  • Rate reality. zendesk-bridge incremental export runs ~10 RPM. The service paces itself; no hammering the jacket, no parallel pullers against one cursor.

7. Unknowns Rule 6 — required

  • Total corpus volume and wall-clock. We have proven 1,000 tickets → 5,002 triples. We do NOT know the full ticket+comment count, so we do not know whether a full start_time=0 pull at ~10 RPM is hours or days. P2's bounded pull gives the first real rate measurement; until then every ETA is a guess.
  • Raw size at full scale. Whether the sealed store fits comfortably where the service runs — fine on the dev box, but on Cloud Run (P7) disk is ephemeral and sized; a multi-GB raw may force a volume/GCS decision we haven't made.
  • Marathon behavior of zendesk-bridge. Rate-limit responses, Cloud Run request timeouts, and jacket restarts over a multi-hour paged pull are unobserved. The cursor makes any of these survivable, but retry/backoff shape is unwritten until we see a real failure.
  • Triple growth with comments at scale. v0's 5 triples/ticket was tickets-shaped. Comments as first-class comment:<id> subjects on long threads may bend triples-per-ticket non-linearly — affects size, seal time, and replay latency estimates.
  • Replay fidelity on the ugly tail. v0 round-tripped clean on 1,000 Mendo tickets; the full corpus has older, weirder threads (encodings, deleted users, malformed events). P4/P5 replays will find what v0 never saw.

8. When To Revisit Rule 6 — required

Concrete signals that trigger a v2 of this plan:

  • The bounded pull (P2) projects the full pull past ~24 hours → re-plan P5: chunked windows, headless scheduled runs, or bring P7 (Cloud Run) forward so the marathon runs off-box from the start.
  • The raw exceeds what the service host holds comfortably (rule of thumb: >50% of available disk, or Cloud Run ephemeral limits at P7) → bring mesh residency forward from its deferred board; the raw's home becomes the next plan.
  • Replay fidelity on real comments drops below the v0 bar (any non-clean round-trip in P4/P5 spot checks) → stop the corpus pull, fix the decoder or the import shape first — a big raw of wrong triples is worse than no raw.
  • The operator wants derive-nps sooner → re-sequence: derive can start against the bounded P2 raw while P5 runs; that's a board reshuffle, not a redesign.
  • zendesk-bridge needs a new endpoint or paging change to serve the pull → that's a jacket change in its own repo with its own board; this plan pauses at the boundary rather than absorbing C# work.
