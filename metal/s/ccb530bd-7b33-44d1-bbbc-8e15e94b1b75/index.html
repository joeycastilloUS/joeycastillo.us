~i — What is the issue: the STAGE3 corpus job

Fable 5 inspection of why stage-mendo-zendesk has failed 3× and never produced the 431K derived corpus. It pulled the actually-deployed source from Cloud Build (not just the local tree) and did the byte math. The earlier "tmpfs exhaustion" call was wrong — corrected here with evidence.

Root cause — ENOENT, not ENOSPC

The deployed run.sh never creates /work/derived-full, so seal_tsv's fopen("/work/derived-full/facts.bin","wb") fails on a missing parent directory. fopen(3) does not create directories. The failure is deterministic, at the very first output write, at 6% memory utilization — not memory.

The fix already exists on disk: local C:\argus\.stage3-job\run.sh (mtime Jul 9 22:47) has mkdir -p /work/derived-full + raw-store rm relief — but that is 38 minutes newer than the last deploy (22:09). The fix was written and never redeployed. The deployed image still runs the old script (its log banner reads (O(1) intern), matching the old source).

The second bug — equally load-bearing

Even with the directory fixed, the deployed stage.c decodes nothing. Its execution log:

stage: tickets=431260 comments=431260 total_atoms=0 avg_atombag=0.0 triples_written=6037640

The deployed stage.c probes kind=ticket first → the full raw has kind=ticket facts → it takes the "synth" path → reads a description field that does not exist in the full raw (the real bodies live under has_comment → comment:<cid> → body, 977,029 comments). Result: every atombag empty, total_atoms=0, a garbage store. (6,037,640 = 14 near-empty triples × 431,260 tickets.)

The corrected stage.c already exists at C:\mendo-zendesk-ai\be\tools\stage.c (probes has_comment FIRST; comment header even says the prior version "had this inverted... producing total_atoms=0"), verified working locally this morning. The job dir still holds the old, inverted one.

Size math — memory was never the wall

Calibrated from a local 985-ticket thread-mode sample + the raw TSV (body avg 609 B). At the real 431,260 tickets / 977,029 comments, thread mode:

  • derived.tsv ≈ 1.2 GB; seal heap (strings ~0.9–1.5 GB transient + facts 263 MB + dict/hix ~130 MB) ≈ 1.6–2 GB; tmpfs at seal (tsv + output + index + tgz) ≈ 3.7 GB.
  • Peak ≈ 6 GB of 32 GiB — ~5× headroom. Confidence: high. The uint32 offset (~0.9 GB) is 4.4× under its cap and dies cleanly if the corpus ever grows ~4×.

Latent bugs surfaced

  • stage.c mode inversion (.stage3-job\stage.c:186) — the ship-stopper; fixed version exists in mendo tools, not yet in the job dir.
  • write_file hides errno (seal_tsv.c:89) — "cannot write" with no strerror(errno) is exactly what turned a 5-second ENOENT into a 32Gi wild-goose chase. One-line fix.
  • No garbage gate — run.sh happily seals + uploads a total_atoms=0 corpus. Stage should exit nonzero when thread-capable raw yields zero atoms.
  • O(n²) ticket seen-set (stage.c:216) — 431K linear scan; caused the 1200→600 t/s decay. Tolerable, but confirm the job task timeout is set for hours before the real thread-mode run.

Recommendation

Redeploy with the fixed stage.c (has_comment-first) and the current run.sh (mkdir + tmpfs relief), plus errno in write_file and a zero-atoms gate. The failure was a missing directory and an inverted mode probe — never memory. 32Gi/8CPU already has 5× headroom.
