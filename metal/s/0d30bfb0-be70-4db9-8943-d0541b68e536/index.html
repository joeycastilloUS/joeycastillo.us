Full raw → cheapest GCS + mesh residency

Plan pair of be/designs/2026-07-09-1639-design-mesh-residency-cheapest-gcp-fable5.html. Six phases, two boards, well under $1/month.

1. Summary

What: execute the tail that never ran — get the full Mendocino Farms Zendesk raw corpus (7.6 GB, 431,260 tickets · 988,345 comments, pull done/eos) into GCP at the cheapest cost, seal it into a full .ids, and make that sealed store mesh-resident so any authorized node replays and decodes-to-atoms on demand.

Why now: the pull completed in the LLM zone on Joey's box; the seal→tar→upload→announce steps only ever ran for the Jul-4 1,000-ticket sample. A laptop disk is the single copy of the corpus, and the raw.gz (862 MB) is uploading to Coldline in the background this session — the rest of the tail should land right behind it.

Operationalizes: be/designs/2026-07-09-1639-design-mesh-residency-cheapest-gcp-fable5.html (residency + cost), which itself sits on be/designs/2026-07-09-1509-design-raw-ids-out-of-llm-zone-v2-fable5.html (the pure-C seal pipeline + Argus eye).

Rule 6: this plan is a v1 hypothesis. The full-scale seal has never run; the mesh publish path may be a gap. Unknowns are named below; v2 lands when P2/P4 meet evidence v1 didn't anticipate.

2. Intent

The full raw lives cheaply and durably in GCP — Coldline for the archive, pennies a month — and the sealed .ids built from it is a content-addressed, mesh-resident unit: one SHA-256 handle, announced to the prime, resolvable by any node, fetched from its local-region bucket for free, verified, mmap-mounted read-only, and replayed / understand()-decoded / atomized on demand. Import once, derive many, replay forever — off Joey's box, for under a dollar a month.

3. Scope

In

  • Raw JSONL.gz (862 MB) → Coldline, us-west1 (gs://acmedev-mendo-tickets-ids/full/) — in flight this session.
  • Full seal: v2 pipeline (compile raw_import.c + C99 shaper) over all 431,260 tickets → full .ids.
  • Tar + SHA-256 → IDS handle + full manifest.json in the existing bucket convention.
  • Upload sealed tgz to nous-kb-us-west1 (Standard, hot).
  • Manifest signal (NTRP0001) to the prime over the wire.
  • Node mount + decode proof: resolve → fetch → verify → mmap-mount → replay + decode ONE real ticket to atoms.
  • Residency panel in the Argus eye: handle, region copies, mounted?, last-decode.

Out — plainly

  • ✗ All-6-region replication day one. Start us-west1; us-west4 is next (Board GCP2). Copies cost pennies; add where consumers are.
  • ✗ derive-nps and other decoder passes — their own boards, run off the resident raw once it's mounted.
  • ✗ Write-back to Zendesk — the read-only invariant holds; no write path exists.

4. Approach

  • Cheapest tiering. Raw.gz on Coldline us-west1 (~$0.005/mo; re-read only on re-shape, ~$0.03 retrieval). Sealed .ids on Standard us-west1 in nous-kb-us-west1 (~$0.01/mo). Total: well under $1/month.
  • Content-address. SHA-256 of the sealed tgz IS the IDS handle — already the bucket's manifest convention. Verification is free; the handle can't lie; GCS is a locator, not a dependency.
  • Regional free egress. Every bridge runs us-west1 — fetches from nous-kb-us-west1 are free. nous-api (us-west4) gets its own regional copy in GCP2; until then a one-time cross-region fetch costs <$0.01.
  • Reuse, don't invent. The bucket exists; the nous-kb-<region> hot/cold pattern exists; nous supplies the sealed store, understand() decoder/atomizer, wire/NTRP0001, mesh, prime. NEW is only the full-scale run of the tail.
  • Language law. C# only at the zendesk-bridge boundary — already behind us; the raw is on disk. Seal / shape / hash / publish / mount / decode are C99, ours.

6. Constraints

  • Cheapest that works. Coldline for the raw, Standard for the .ids, copies only where consumers run. Budget ceiling: $1/month; expected ~$0.02/month.
  • Language law. C# jackets foreign APIs only (zendesk-bridge, already done). Everything this plan builds — seal, hash, publish, mount, decode — is C99.
  • Read-only invariant. This layer only reads the already-pulled raw. No Zendesk write path exists or is created.
  • Done = production (Rule 9). No phase is Done on "uploaded" or "compiled" — GCP1 closes only when a node ACTUALLY mounts and decodes from GCS, verified end to end (P5).
  • Rule 10. This plan proposes; it does not write a-d-d/. It flows WITH a-d-d §7 (import once / derive many / replay forever) and §9 (prime deterministic, reached over wire; mesh is a signal plane, not a data plane). Any spec drift found mid-build stops work and proposes an a-d-d update first (Rule 11).

7. Unknowns

  • Final .ids tgz size at 431K scale. The 1K sample sealed to 227 KB; linear extrapolation says ~100–300 MB, but comment-length distribution at full scale is unmeasured. P2 measures it.
  • Mount path from GCS. Does the prime/node mount directly from GCS, or does it need a local pull to disk before mmap? The design assumes fetch-then-mmap; P5 proves or corrects it.
  • Is the manifest-signal publish path coded in mesh.c today? NTRP0001 exists on the wire, but whether "publish handle+locator to the prime" is implemented or a gap is unverified. P4 finds out.
  • Cross-region cost/latency for us-west4. If nous-api reads us-west1 before the GCP2 copy lands, what does the fetch actually cost and how long does it take at ~200 MB? One measured fetch answers it.

8. When To Revisit

  • .ids far bigger than expected (say >2 GB) → re-run the tiering math; consider Nearline for the tgz or splitting hot/cold by decoder pass.
  • Mesh has no publish path yet (P4 finds a gap) → v2 decides: build the NTRP0001 publish into mesh.c, or fall back to a plain resolver that reads the bucket manifest directly. Either way the handle convention holds.
  • Re-shape frequency turns out high (Coldline retrieval fees stack) → move the raw.gz to Nearline or Standard; one gsutil objects update --storage-class, cents not migrations.
