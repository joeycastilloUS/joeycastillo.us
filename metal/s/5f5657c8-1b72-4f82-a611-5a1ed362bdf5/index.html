Plan v2 — Templates, reframed to a decision

Plan — Awaiting Approval. Supersedes the “see your templates” direction. Grows from the honest inspect + retro (2026-07-09 17:09). The engine stays; the passive list goes.
Supersedes the viewing-verb framing in 2026-07-07-2259-plan-zendesk-templates-in-argus.html and the TPL2 “attributed in N” surface. What ships stays (the C99 engine, the endpoints); what it shows changes from an inventory to a worklist an operator acts on.

Intent

Turn the (good) attribution engine into one decision a named operator makes weekly: where are agents going off-script, and where are we missing a macro? The engine already knows — every agent reply that comes back NOT MATCHED (hand-written when a template existed, or a recurring situation with no template) is a coaching signal and a template-gap signal. That is something neither Zendesk nor the Audit tab surfaces. Kill the passive inventory; ship a worklist with an action on every row.

Scope

In: an Off-Script & Gaps view. Over a real window of resolved recovery-type tickets, classify each agent public reply: matched a template (adherence, fine), near-miss (heavy edit — template drifting), or NOT MATCHED (hand-written). Cluster the NOT-MATCHED replies by similarity to surface recurring hand-written situations = missing macros. Each row is an action: open the ticket, or “draft a macro for this cluster.” Junk templates (*, test, empty) filtered out. Attribution made honest at scale (full window or clearly-labeled sample, clickable tickets).

Out (v1): outcome/recovery correlation (needs NPS-LIVE data — separate); auto-creating macros in Zendesk (read-only holds — we draft, operator creates); per-agent scorecards (adherence-by-person is v2, sensitive); the standalone macro inventory (folds into the Audit tab, which already grades + remediates macros).

Approach

  1. Decide the surface's fate first. The macro inventory (usage/dead/tiers) overlaps the Audit tab. Fold inventory into Audit; the Templates tab becomes the Off-Script & Gaps worklist or is renamed to it. One place per job.
  2. Adherence classify. Reuse the live /tickets/{id}/attribution engine over a window of resolved GR/recovery tickets. Bucket agent replies: matched / near-miss / NOT MATCHED. Filter junk library entries so * and test never appear.
  3. Cluster the gaps. Group NOT-MATCHED replies by mutual similarity (the same atom-diff, reused) → clusters of ≥N similar hand-written replies = a missing macro. Rank by frequency (biggest gap first).
  4. Make every row a decision. Off-script row → open the ticket (real link) + “which template should this have been.” Gap cluster → sample replies + a drafted macro (engine writes a candidate body) the operator can copy into Zendesk. Read-only fire stays operator-side.
  5. Honest counts. Scan the full window or label the sample loudly with what was and wasn't covered (no silent 40/298).

Constraints

  • Usefulness gate (new, from the retro): before anything is “done,” state the operator, the decision, and the one thing they do differently. Phase 1 must pass a live “yes, I'd use this” from the operator, not a passing endpoint.
  • Read-only holds — we draft macros; the operator creates them in Zendesk. No template writes from Argus.
  • Reuse, don't rebuild — the C99 engine, the attribution endpoint, and the Audit remediation lane already exist. This is a surface reframe, not new intelligence.
  • No junk as signal — filter */test/empty library entries; a match on garbage is not a match.
  • Language law — matcher C99, C# jacket at the Zendesk seam, vanilla JS surface.

Unknowns

  • Is “off-script” actually a problem the operator cares about, or is template hygiene (dead/dup cleanup) the real want? Phase 1 tests off-script cheaply; if it lands flat, pivot to the hygiene worklist (also engine-adjacent).
  • What counts as a “recovery-type” ticket to scan? (GR group / CSAT / tag?) — confirm the filter against live data, don't guess.
  • Cluster threshold for “a missing macro” — how many similar hand-written replies = worth a macro. Tune on real data.
  • Does the Templates tab survive as a name, or become “Off-Script” and inventory moves to Audit? Operator's call on IA.
  • Full-window scan cost vs on-demand — a full 298+ scan is many engine spawns; may need a batched/persisted job, not a live rollup.

When To Revisit

  • After Phase 1 — if the operator doesn't act on the off-script list, this whole reframe is wrong; re-plan around hygiene or outcome instead.
  • When NPS-LIVE outcome data lands — add “which templates actually recover guests,” the highest-value view, currently blocked.
  • If drafted macros get copied often — revisit the read-only line (is a gated create-macro worth building?).
