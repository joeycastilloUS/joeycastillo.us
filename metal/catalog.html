The Catalog — patterns + C99 skills, published not exposed

Publish the list, seal the source. metal.de, metal.ce, and every Purcius operator build on the substrate without ever touching kastil-systems.
DESIGN — AWAITING APPROVAL
v2 — revised after a Fable 5 full-court press (5 lenses). The press found the load-bearing fact: the Catalog already exists — a triple store the primes already serve at /catalog/* and over the wire. This design no longer invents a catalog; it adds patterns + skills as new entity types to the one that ships. All five findings are folded in below; their headlines are in the comment thread.

1 · Need

As metal.de, I want to apply a known pattern by name — without cloning kastil-systems.
As a Purcius operator, I want to see the C99 skills I can build on, as a plain list — and know, per entry, how I actually use it.
As kASTIL, I want the source — the formulas — sealed, while the list is public.

Today the patterns and the ~50 C99 skills live in the private a-d-d. Reaching them from an edition means exposing the repo: you'd hand out the sealed core just to read the public edge. Wrong place, wrong access model.

2 · Sourced solution — it already exists

The first draft re-derived a catalog from the Trust pattern and the .pi-reference move. The archaeology pass killed that: the substrate already ships a live catalog, and four mechanisms already are this design.

Already shippedWhat it is
The catalog (prime-server, all 4 primes, :8080)A triple store served at GET /catalog, /catalog?type=, /catalog/{id}, /catalog/tree, /catalog/index. Doctrine: “the store file IS the manifest — sources self-describe as triples.”
The Directory purposeAlready “publish the list, seal the source”: “the Directory syncs to every node… source files never leave origin… you get the gold, not the ore.” One of the four boot stores = what's available.
manifest_export_filteredThe publish step, already a C API: export a subject-prefix (e.g. pattern/) to a .triples manifest — wire-shippable, replayable by a cold node. No new format.
mesh.sealed containerThe seal, already built: null-delimited triples + ed25519 signature by the operator key; bad signature → 0 triples. Same builder, same one-secret root.
The wire skillIts first worked example is a catalog read: “list catalog where domain is geography” over NTRP0001. Editions reading the catalog is not new surface — it's the shipped exchange. Slot's Bus already pins {catalog, wire_handle_catalog_list}.
So patterns and skills are just new entity types in the store that already exists: 
(trust, type, pattern)   (trust, aka, "Hollow Edge/Sealed Core/Anchor Seal")   (trust, status, active)
(wire,  type, skill)     (wire, purpose, comms)   (wire, status, active)
(the-last-test, type, service)   (the-last-test, from, ai-pantheon)   (the-last-test, rent, "$0.05/call")
Query with the endpoints that already ship: GET /catalog?type=pattern, /catalog/trust, /catalog/index. Zero new API. The phantom the draft invented — for-rent.pi — doesn't exist; the real For Rent marker is "from":"ai-pantheon" on skill.json.

3 · Name it true

The instance keeps the name it already has: the Catalog (the store the primes serve). But as a reusable pattern, “Catalog” collides with the live catalog skill and PATTERNS.md's own index — it fails the substrate-collision rule. The publish-projection move (seal the source, sign an outbound public face, discover by well-known query) is a new pattern, #6, proposed name Nameplate (parts: Nameplate · Press · Window) — worth a ~patterns extract, operator to bless the name. Trust governs the boundary but isn't the whole shape (its Anchor Seal is inbound; here the seal is outbound authenticity); Slot's Pin is exactly the one-line DoR/DoD hook; Latchkey is the opposite access model and is explicitly not it.

4 · Shape

Every entry says how you use it (the product, not a brochure)

The ~50 nous skills are compile-time C libraries, not wire services — “consume over the wire” was wrong. Three classes hide in the flat list, each with a real use-path the substrate already defines:

ClassHow you use itCost
pattern (trust, slot, oracle, latchkey, invite)Apply by name in a metal session: ~patterns apply <name> → checklist → implement.$0
library skill (crypt, wire, store, seal, …the ~50)Compiled into primes/relays. You reach its capability through a wire verb the prime exposes (relay TOTP session → NTRP0001). Entry carries served-by: <wire verb | none-yet> — none-yet is roadmap, said honestly.$0 (substrate)
For Rent service (the-last-test, jack-rabbit, …)~verb dispatches → POST /serve/run, Bearer auth (FQDN) → sealed result → MeterEvent bills your FQDN. Discovery: GET /serve/skill already public.metered ($/call)

The Catalog is the price sheet. Each entry carries {name, type, aka, intent, status, class, served-by, rent, auth} — never source. This absorbs the For Rent discovery catalog (closes the open question in tags.json #18): one store, three doors, per-entry.

Visibility is scoped — the list is not a blanket public map

Adversary catch: publishing the security stack's signatures (auth · vault · seal · reveal · wipe) as a blanket-public list is an attack map. Fix: the catalog is scoped — status + a visibility triple gate what each viewer sees. Public discovery returns names + intents; signatures and security-domain internals are operator-scoped (gated by the policy skill against the requester's FQDN). The prime already serves scoped queries; the seal already binds to the operator key. Hollow edge, yes — but a curated edge.

The tilde-contract touch — the whole point

One line in the shared pre-flight + one item in the shared tail reach every gated tilde at once — zero per-tilde edits. Because the DoR pre-flight and the DoD tail are each ONE program, the change lands once. The 13 dor:exempt utilities are excluded by the exemption machinery that already exists — correct: ~h and ~config have nothing to reuse.
  • DoR item 5 (Catalog), class-scoped — “Read the Catalog. Is there a pattern or skill for what this session builds? Apply it, reuse it — never reinvent.” Build tildes (~a ~u ~f ~b ~bt ~demo) stop-and-cite on reinvention; discovery/planning (~i ~d ~laf ~p ~e ~pu ~pt) consult-and-cite; verify/ship/report + utility skip. Unreachable = warn + continue.
  • DoD item 11 (Catalog current) — if the work created or changed anything reusable (a pattern, a skill, a manifest, a public signature), the publish step ran and all three doors reflect it. N/A when nothing reusable changed (most runs). Reuse-verification is the DoR's job — by the DoD it's an autopsy.
  • The uses: field — new .pi frontmatter, written on apply (not hand-declared), resolve-checked at DoD (unknown key fails loud). Free reuse-telemetry for the Learned bands. The shared tail declares its own uses: once (patterns.latchkey, skills.signals) — cards never repeat tail uses.
  • Consumers are wider than three: ~patterns reads it, ~d mines it, and DoR item 5 makes ~a ~u ~f ~i ~bt ~demo ~p ~e ~transfer consumers too — code-gen picks a skill before writing one; ~i names findings in catalog vocabulary; ~e sizes reuse cheaper than invention.

Boundaries (out)

  • OUT: the source. Entries are metadata + signatures, never formulas; a-d-d stays the sealed core.
  • OUT: a package manager. Patterns are applied in-session; library skills run on the substrate via wire verbs; services are called, not vendored.
  • OUT: a second catalog format. It's the existing triple store + manifest_export_filtered + mesh.sealed. No catalog.json schema of our own.

5 · First move

Fix the pre-existing DoD drift (commands.md is 8 items, help-blocks is 10 — re-sync first), then: emit the 5 patterns + ~50 skills as type=pattern/type=skill triples into the existing catalog store (via manifest_export_filtered), serve them on the live GET /catalog?type=pattern, and add DoR item 5. Prove it by querying a prime for ?type=pattern and running one metal session that applies a pattern named from the catalog read, with zero kastil-systems access.

No baby steps: the first move is the production thing at small scale on the store that already ships. If a prime answers /catalog?type=pattern and an edition applies trust by name without cloning the repo, the whole design is proven.

Choices & Consequences load-bearing

ForkPick & why
New catalog, or extend the live one?Extend. The primes already serve /catalog/* over a triple store whose doctrine is “the store IS the manifest.” A second catalog is the reinvention the whole session exists to kill. Patterns/skills = new type triples.
Expose the repo, or project a list?Project (Trust governs). The Directory purpose already does gold-not-ore. Publish a sealed, signed projection; the source never leaves origin.
Blanket-public list, or scoped?Scoped. Security-domain signatures are an attack map; the policy skill gates them by requester FQDN. Public sees names+intents; entitled operators see more.
Brochure, or price sheet?Price sheet. Per-entry class/served-by/rent/auth. Absorbs For Rent discovery (closes tags #18). Discover → apply / wire-call / rent.
Touch every tilde, or the DoR/DoD?DoR/DoD. One line in each shared program reaches every gated tilde; the exemption machinery excludes the 13 utilities. Slot's Pin: one greppable line = live-for-all.

Escape Routes load-bearing

  • Projection behind a publish seam — regenerate from a-d-d anytime; a bad manifest rebuilds from source byte-identical. Nothing lives only at the edge.
  • DoR read is additive — catalog unreachable = warn + continue; the change can never strand a tilde.
  • Seal isolates the face — the sealed container's format can change (triples → wire → future) without touching the sealed source; bad signature → 0 entries, fail closed.
  • uses: is a record, not a wish — append-on-apply + DoD resolve-check; a stale/invented key is a red gate, not silent rot.
