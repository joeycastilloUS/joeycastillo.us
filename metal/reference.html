~chain-reaction — Named Workflow Macros

Commands outside / run once, left to right. Commands inside / / loop until the exit condition is met: board exhausted, tests pass, nothing to fix.

Built-in Chains

GO    ~I ~D ~P ~E /~B ~T/ ~S ~R   — Full lifecycle: inspect to retro
HOTFIX    ~I ~P /~T ~F/ ~S ~R   — Inspect, plan fix, test/fix loop, ship

Custom Chains

~chain-reaction new — interview mode, build a chain interactively.

~<NAME> — shorthand to run any named chain (e.g. ~GO).

Definition of Done (DoD) — 10-Item Mandatory Gate (+4 chain-final)

Every tilde command that produces an artifact must pass all 10 items before the command is considered complete. A-D-D verified is the first gate — spec checked, built matches the architecture.

1.A-D-D verified — the project's architecture spec (a-d-d/) was read and the built work matches it. Drift stops the command.
2.Code committed — All changes in a git commit with a descriptive message. No uncommitted work.
3.Commit hash in telemetry — The be/telemetry.json entry has a non-null commit field. Backfilled in the same commit.
4.Board updated — If a board item was worked, it moved to Done with commit hash: {item} · pts · ~cmd ✅ (abc1234).
5.HTML report saved — Report exists in be/ with correct datetime filename.
6.Index updated — be/index.html has an entry linking to the new report.
7.Pushed to remote — git push succeeded. Work exists on origin, not just locally.
8.No prior work damaged — Done board items untouched unless a new item explicitly targets them.
9.Shipped to production — if the work changed a deployable surface, the live target serves the new bytes, externally verified.
10.Data round-trips — any write path is proven to read back the same value (no write-only illusions).

Chain-Final Rules (#11–#14)

Chain-final commands (~s, ~r) must also pass:

11.Chain complete — every preceding step has a telemetry entry with a non-null commit hash.
12.Site in sync — joeycastillo.us/metal updated to reflect shipped work. Push site repo.
13.~h in sync — Help display matches current commands, aliases, and Ops section.
14.Revision current — All version footers use the post-commit revision number.

Failure Mode

If any DoD item fails, the command is not Done. Fix in-place and re-verify. The gate holds.

Exempt Commands

~h, ~li, ~l, ~cr, ~jc, ~m, ~config, ~tag — utility commands, inline output only, no artifacts.

Definition of Ready (DoR) — 4-Item Session Pre-Flight

Before the first edit of a session, read and verify:

1.Board state — Read be/board.md. Know what's Done, In Progress, Ready. Don't touch Done items.
2.Last telemetry entry — Was the previous chain completed? If last entry is ~b without a following ~s, flag it.
3.Source of truth — If project CLAUDE.md declares canonical files, respect them.
4.Architecture spec — if the project has an a-d-d/ folder, read the spec for the area you'll touch before any work (Rule 10).

4 reads. Takes seconds. Prevents most cross-session work loss.

Chain Reaction Flow

Every tilde command (except ~h) follows this chain:

do the work → render report → save with datetime
→ update Be index → log telemetry
→ commit (backfill hash) → push
→ verify DoD (all 10 pass) → launch demo

Board Convention

Work Items — Triple Format

Every item is a triple: {what, verb, context}

Verb maps to a tilde command:

"add to"~a
"update with"~u
"fix in"~f
"designed for"~d
"plan tests for"~pt
"test across"~t

Sizing

Fibonacci only: 3 or 5 points per item. A 5 is something a college student can do in one night, complete and perfect. Boards total 13–21 points.

Board States

Ready → In Progress → Done

Done format: {item} · pts · ~cmd ✅ (commit-hash)

Versioning

Version is 0. Always. No semantic versions. No major.minor.patch.

Every change is a revision: git rev-list --count HEAD

Display: v0 · rN · YYYY-MM-DD · pts

Total points from be/board.md. Computed by be/pts.py at launch.

HTML Report Template

  • Background: #111111
  • Raised surfaces: #1A1A1A
  • Accent: #6FC3DF (tron blue)
  • Font: Helvetica Neue, Helvetica, Arial, sans-serif
  • Nav: black sticky bar, tron blue bottom border, "Be" brand
  • Max width: 800px
  • No frameworks. No build tools. Vanilla HTML/CSS.
  • DoD checklist before footer: ✓ Item (detail) or ✗ Item (reason)

Telemetry

Every command (except ~h) logs a small per-tilde ledger entry to be/telemetry.json (still tracked in the project repo):

  • triple: {description, verb, project}
  • command: the tilde command
  • date / time / duration_sec: wall-clock timing
  • commit: backfilled after commit

Rolling 100 entries max. Viewer at be/telemetry.html.

Verb mapping: ~a → "added to", ~b → "built in", ~u → "updated in", ~d → "designed for", ~f → "fixed in", ~s → "shipped from", ~p → "planned for", ~e → "estimated for", ~i → "inspected in", ~t → "tested in", ~r → "retro'd in", ~tag → "tagged in".

Runtime telemetry — otel.jsonl + prompts.jsonl (hub-sharded)

The high-volume telemetry trail lives in two runtime-only files: be/otel.jsonl (OpenTelemetry spans) and be/prompts.jsonl (the prompt trail). These are gitignored and never committed to metal — they are runtime artifacts, not source.

  • Per-host sharding — each machine writes to its own path in the hub repo acmedevUS/metal-telemetry under hosts/<hostname>/..., so two machines never touch the same file (no merge conflicts, ever).
  • Synced on the session lifecycle — push-telemetry runs on launch (SessionStart / SessionEnd), and Mercury fires on ~s / ~r, pushing each host's shards up and pulling the merged union down.
