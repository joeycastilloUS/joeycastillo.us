Diff snapshot procedure — one tool, five panels, auto-published

Build a standard procedure: one command captures a five-dimensional AIRBYTE_DB vs AIRBYTE_CLONE snapshot, renders HTML in the site idiom, publishes to joeycastillo.us/airbyte-migration/snapshots/, and emits a click-able URL.

Plan — Awaiting Approval
Intent: Replace the current fragmented diff workflow (six scripts, manual stitching, no publication) with a single command that produces one publish-quality HTML snapshot per invocation. The first invocation backfills the current "post-Phase-2" state. Every subsequent procedure phase gets a before/after pair. The site grows a snapshot history that is the visible audit trail of the clone-rebuild. Source of truth: be/inspects/2026-05-23-1943-inspect-diff-procedure.html.

Phases

1 Core diff collector + 5-panel data model

~45 min. scripts/snapshot.py module that opens one sf_connect session and gathers all five dimensions into a single dict.
  • Counts panel: per-schema row counts AIRBYTE_DB vs AIRBYTE_CLONE via INFORMATION_SCHEMA.TABLES. Mirror quick_count_diff.py logic; produce {schema: {old_rows, new_rows, delta}}.
  • Freshness panel: per-stream gap_days. Mirror audit_clone_freshness.py logic; produce {stream: {prod_max, clone_max, gap_days, status}}.
  • Clustering panel: count + sample of _AIRBYTE_* clustering keys on AIRBYTE_CLONE. Mirror discover_clustering.py.
  • Schema panel: column-set diff per stream (uses parity engine's ColumnSignalDetector approach: data cols common / prod-only / clone-only). Lightweight INFORMATION_SCHEMA.COLUMNS query, no API call.
  • Drift panel: latest L1-L3 verdicts from the Cloud Run /dashboard/data endpoint. Optional --deep adds L4 row-hash sample for top N drifters.
Accept: python scripts/snapshot.py --phase 2 --label post-cleanup --dry-run prints all five panel data structures to stdout. No HTML, no copy.

2 HTML render in site idiom

~30 min. Render the dict into one HTML page that visually belongs next to parity.html.
  • Same nav, CSS palette, hero box, summary-stat row.
  • Five collapsible <section> blocks — one per panel. Each shows N/A with a one-line explanation when the dimension is irrelevant.
  • Hero card surfaces the headline number per panel: row delta, max gap days, clustering count, schema deltas, drift count.
  • Footer: command that generated it, snapshot ID, time, link to the previous and next snapshots if present.
Accept: python scripts/snapshot.py --phase 2 --label post-cleanup writes be/snapshots/2026-05-23-1943-phase2-post-cleanup.html. Opens cleanly in browser. Visually matches parity.html idiom.

3 Publish path + index

~30 min. The --publish flag does the whole site-side dance.
  • Copy local HTML to /c/joeycastillo.us/airbyte-migration/snapshots/<same-filename>.
  • Open or create /c/joeycastillo.us/airbyte-migration/snapshots/index.html. Walk the snapshots directory, build reverse-chronological table. Each row: timestamp + phase + label + headline stat + link.
  • Commit airbyte-parity (the snapshot in be/snapshots/) and joeycastillo.us (the publish copy + updated index) as separate commits with parallel messages.
  • Push both. Wait for CF Pages deploy — print the final URL only after a curl returns HTTP 200 (or after a 30s timeout).
  • Emit to stdout: https://joeycastillo.us/airbyte-migration/snapshots/<filename>.
Accept: python scripts/snapshot.py --phase 2 --label post-cleanup --publish runs end-to-end. URL printed at the end resolves to the new page within 60s.

4 Backfill snapshot for current state + first index page

~15 min. Prove the procedure exists by using it.
  • Run: python scripts/snapshot.py --phase 2 --label post-cleanup --publish.
  • Verify the index page renders with one row. Verify the URL resolves.
  • Paste the URL into the conversation (this turn does that automatically once Phases 1-3 land).
Accept: Snapshot live at joeycastillo.us/airbyte-migration/snapshots/YYYY-MM-DD-HHMM-phase2-post-cleanup.html. Index live at joeycastillo.us/airbyte-migration/snapshots/. Both clickable.

5 Standard procedure documented in both CLAUDE.md files

~10 min. Make the standard discoverable.
  • airbyte-parity/CLAUDE.md: add a Diff Snapshots section noting the tool, the URL pattern, and the one-line invocation.
  • airbyte_migration_tools/CLAUDE.md: add a one-line pointer to the airbyte-parity tool ("for AIRBYTE_DB vs AIRBYTE_CLONE diffs, use airbyte-parity/scripts/snapshot.py").
  • Add a one-line note to each procedure-phase script's docstring suggesting snapshot.py --phase N --label before/--label after as the bracketing call.
Accept: CLAUDE.md greps for "snapshot.py" in both repos return non-empty. Phase scripts mention it in their --help output.

Risks + mitigations

RiskMitigation
L4 in --deep mode times out on huge streamsPer-stream timeout (15s default); skip-with-marker on timeout; never block the snapshot.
Site repo dirty at snapshot timePre-flight check: abort with clear message if git status --porcelain shows changes in joeycastillo.us. Joey resolves, re-runs.
CF Pages deploy lag > 60sPrint "URL pending; check in 1-2 min" instead of polling forever. Local snapshot file is the audit trail regardless.
Two snapshots collide on filename (same minute, same phase, same label)Filename includes HHMM; if a collision happens, the second one's filename gets a -2 suffix.
Snapshot becomes too heavy for fast iterationDefault mode targets <15s. --deep opt-in for slow forensic. Drift panel uses cached /dashboard/data — no extra round-trip to Snowflake.
Snapshots leak schema + row counts publiclyReuse /airbyte-migration/gate.js auth pattern; snapshot pages gated by default same as parity.html.
