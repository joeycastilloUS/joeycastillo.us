Dry-run rehearsal — control + visibility

Three additions to the server-side reclone, all riding the existing cadence-operations schema: a dry mode that exercises the procedure without destruction, gates + abort for mid-flight steering, and heartbeats so long steps are never silent. Hands on the wheel.

Design — Pure Metal

Subject + Intent

Subject: Extend POST /reset/reclone with ?dry=true + ?gate=after-drop|after-create query params, add POST /operations/{opId}/proceed and POST /operations/{opId}/abort control endpoints, and have the long steps (Drop, Create) write a heartbeat row every ~15s with Snowflake query state. Intent: turn the committed-and-blind rehearsal into something the operator can steer — preview without destruction, pause at gate boundaries to think, abort if anything looks wrong, and watch live progress instead of staring at "running" for 12 minutes.

Discovery

  • Board 22 was committed-and-blind. Once POST /reset/reclone fired, the clone was dropping. No abort. No mid-flight gate. Twelve minutes of silent create-clone: running.
  • Driver has these locally, server doesn't. Go.Reclone interactive prompts at each step + destructive-action gate (type "yes"). The server path inherited none of that for autonomous mode — it just barrels through.
  • The cadence-operations doc is the right rail. Adding dryRun, gate, gateState, requestedAbort, lastHeartbeat fields is schema-additive. Same watcher, same launcher patterns, more columns to render.
  • Snowflake exposes query progress. SELECT * FROM TABLE(INFORMATION_SCHEMA.QUERY_HISTORY_BY_SESSION()) (or by query_id captured from SYSTEM$LAST_QUERY_ID) returns EXECUTION_STATUS, BYTES_SCANNED, PERCENTAGE_SCANNED_FROM_CACHE, etc. A side-query every 15s during Drop/Create surfaces "is this healthy?"
  • Driver dry token is "accepted but inert" for reclone (per Go.Reclone.bat). That's a known gap, not a feature.

Naming Pass

?dry=true + ?gate=after-drop + /proceed + /abortwinner · consistent with existing /reset/* shape · reads like the operator's mental model
?mode=dry|gated|live + /control/{proceed,abort}enum is cleaner but combinatorial; gates are independent of dry vs live
A separate /reset/reclone-dry endpointdoubles the surface area for one knob; rejects
Approval token in body (pre-approve all steps)defeats the point -- the gate is for thinking mid-flight, not pre-batching

?dry and ?gate as orthogonal query params. The /proceed + /abort verbs live on the op, where they belong (each op has its own state machine).

Shape

Endpoint / paramWhat it does
POST /reset/reclone?dry=trueOp doc marked dryRun: true. Scheduler pause + resume genuinely fire (realistic rehearsal). Drop + Create steps log "would: DROP DATABASE..." + simulate the historical duration (sleep N seconds) and write a NOOP success. Zero Snowflake destruction.
POST /reset/reclone?gate=after-dropAfter step 1 (Drop) succeeds, set gateState: waiting and hold. The op runs in the background but won't advance until POST /operations/{opId}/proceed.
POST /operations/{opId}/proceedClears the gate — op picks up from where it parked. No-op if not gated.
POST /operations/{opId}/abortSets requestedAbort: true. The op checks this at every gate boundary + every heartbeat tick; if true, it skips remaining destructive steps, runs Resume to leave the scheduler in a clean state, and marks the op aborted.
Heartbeat (in-step, every ~15s)While Drop or Create is running, a side task writes lastHeartbeat: {at, snowflakeQueryId?, executionStatus?, bytesScanned?, elapsedSec, message} to the op doc. Watcher prints one line per heartbeat.

The op doc, with new fields

{ opId, kind, status, startedAt, finishedAt, steps[], cloneCreatedAt?, error?, dryRun: true | false, gate: "after-drop" | "after-create" | null, gateState: "waiting" | "cleared" | null, requestedAbort: false | true, lastHeartbeat: { at, queryId?, executionStatus?, bytesScanned?, elapsedSec, message } | null, }

Choices + Consequences

1. Dry mode -- skip SQL entirely vs run against a sandbox vs noop with delay
a) Skip Drop + Create SQL; log "would: ..."; simulate timing with a sleep (historical avg). Scheduler pause + resume still REAL. b) Run Drop + Create against a throwaway test DB. c) Skip all four steps -- only write logs.
a) Realistic enough to exercise the harness, log shape, watcher, and the operator's drive-the-keyboard motion. Zero Snowflake risk. b) More realistic but requires a sandbox DB + its own teardown; adds infra. c) Doesn't exercise scheduler / async / lock -- loses rehearsal value.
PICK: (a) -- skip SQL, keep scheduler real, simulate timing.
Cheapest realistic. The operator gets to fire the endpoint, watch the trail, hit /proceed at the gate, see /abort interrupt -- without anything to recover. If we later want sandbox-mode, that's an additive escape.
2. Gates -- hardcoded enum vs configurable list
a) Two values: after-drop, after-create (plus implicit none). b) A list of step names: ?gates=drop-clone,create-clone. c) Gate before every step.
a) Reflects the two real human-decision moments in a reclone: "you sure you wanted to drop?" "the clone is fresh -- ready to resume the scheduler?". b) Flexible but no one needs gates anywhere else. c) Annoying; pause-scheduler doesn't need approval.
PICK: (a) -- hardcoded enum.
The two values are exactly the moments a human wants to think. Adding more is a no-op for cost (enum extension) if it ever matters.
3. Abort semantics -- cooperative vs forceful
a) Cooperative: op checks requestedAbort at gate boundaries + heartbeats; if true, skips remaining destructive steps + runs Resume to leave a clean state. b) Forceful: kill the Snowflake query mid-flight via SYSTEM$CANCEL_QUERY.
a) Safe; you can abort BEFORE Drop fires (gate=after-drop is the natural abort point) and DURING Create (between heartbeats). You can't undo a DROP that already started. b) Killing mid-CREATE leaves an unknown partial state in Snowflake -- recovery is harder than the abort is worth.
PICK: (a) -- cooperative abort.
The honest interrupt points are: (i) at the gate, before Drop or Create commits; (ii) during a long step, between heartbeats, before the NEXT step starts. Forceful kill of a multi-minute Snowflake op invites a worse mess than waiting it out.
4. Heartbeat content
a) Pure timer: {at, elapsedSec, message: "still running"}. b) Timer + Snowflake query state via INFORMATION_SCHEMA.QUERY_HISTORY (queryId, executionStatus, bytesScanned). c) Append to a heartbeats[] array.
a) Proves liveness (the cadence service is alive + the step is alive) but not progress. b) Real progress signal -- bytesScanned climbing means Snowflake is doing work; executionStatus going to "BLOCKED" tells you something's wrong. c) Doc grows unbounded over a long Create.
PICK: (b) overwriting lastHeartbeat; not appending.
Real progress, bounded doc size. The historical trail is the per-step started/finished timestamps + the durations they encode; per-tick heartbeat trail isn't worth the doc bloat.
5. Watcher integration
a) Extend Go.Watch to print heartbeats + gate state inline. b) New Go.Watch --live flag for the noisier output. c) New launcher.
a) Operator only learns one verb; quieter modes (skip heartbeats) is a flag. b) Same. c) Two watchers for the same op = confusing.
PICK: (a) -- extend Go.Watch; heartbeats are part of the story.
The watcher already polls the op doc; rendering one more field is trivial. Heartbeats appear inline between step transitions.

Escape Routes

From PICK 1 (skip SQL dry): if dry mode needs more realism, add an opt-in ?dry=sandbox mode that runs Drop+Create on a throwaway DB. The first mode (NOOP) already proves the harness.
From PICK 2 (hardcoded gates): if more gates are needed, the enum extends; the parser change is one line.
From PICK 3 (cooperative abort): if cooperative isn't enough (e.g., the Create is genuinely stuck and never heartbeats), add a force-cancel endpoint that issues SYSTEM$CANCEL_QUERY using the captured queryId. Opt-in, separate verb.
From PICK 4 (overwriting heartbeat): if a historical heartbeat trail becomes useful (debugging a slow step), capture each heartbeat into a sibling cadence-heartbeats/{opId}/{ts} subcollection. Not now.

First Move

The smallest proof Ship ?dry=true alone (PICK 1) — nothing else. The Drop + Create steps log "would: DROP DATABASE..." and sleep the historical avg (Drop ~2min, Create ~13min). Real pause + resume on the scheduler. Op doc has dryRun: true. Fire it once: POST /reset/reclone?dry=true → Go.Watch → trail looks identical to a real run but no clone was harmed. Validate the rehearsal motion end-to-end without risk. Gates, abort, heartbeats follow as additive phases.

What this does NOT do

  • Change the real reclone path -- live mode is unchanged.
  • Add a sandbox DB for dry mode (escape route if it matters).
  • Force-kill Snowflake queries (escape route).
  • Replay or retry-from-step (separate concern; the Driver can do this locally).
  • Remove the Driver -- it stays for operator-day approval flows.

Path to plan

~p phases this: (1) dry mode, (2) gates + /proceed, (3) /abort, (4) heartbeats, (5) Go.Watch render + close. Each is independently shippable; dry mode alone is the rehearsal-saving deliverable.
