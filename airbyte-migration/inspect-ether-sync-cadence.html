~i ETHER sync cadence + drift anatomy

Why does ETHER show as DIFFER, and is it reasonable? Measured the real sync timing in Snowflake. Answer: the drift is pure sync-generation lag -- prod holds newer hourly batches than the frozen clone. Not corruption. And there's a clean signal to window on.

Inspect — Measured

What I measured

Queried _AIRBYTE_EMITTED_AT (Airbyte sync watermark), business cursors (UPDATED_AT), and row counts on 6 ETHER streams across AIRBYTE_DB (prod) and AIRBYTE_CLONE (clone), plus the distinct sync-batch markers on prod ETHER.ORDERS over 30h.

Finding 1 -- ETHER syncs hourly at :00, atomically

Prod ETHER.ORDERS sync batches, newest first -- clean 1.00h cadence, each batch emitted ~3s after the hour:

06:00:02.839 2 rows 05:00 (n/a -- ORDERS quiet) 04:29:13.995 5 rows (one off-cadence batch) 03:00:02.820 15 rows 02:00:03.121 66 rows 01:00:03.010 125 rows 00:00:02.759 260 rows 23:00:02.531 439 rows 22:00:02.353 545 rows ... (peak 16:00 = 2860 rows; overnight troughs ~2-15)

Every ETHER table shares the exact same _AIRBYTE_EMITTED_AT per batch (all 06:00:02.839 on prod, all 02:00:03.121 on clone). ETHER syncs as one atomic batch -- the whole schema moves together. That means we can reason about ETHER drift at the schema level, not per-table.

Prod's last ETHER batch: 06:00:02. Clone's last batch: 02:00:03 (frozen at the reclone). The lag is exactly 4.00h on every stream -- 4 sync generations. Row diffs are tiny and proportional to recent write volume. This is the clock ticking on a frozen snapshot, nothing more.

The 25-27 ETHER DIFFER verdicts are 100% explained by sync lag. The HASH_AGG fingerprint compares whole tables (cadence currently passes cloneTime=null, so no windowing). Prod's 06:00 data != clone's 02:00 data, so every ETHER stream hashes differently. There is no actual data corruption -- the clone simply hasn't received prod's last 4 hourly syncs.

Finding 3 -- two timestamps, two jobs

ColumnWhat it isUse for
_AIRBYTE_EMITTED_ATWhen THIS Airbyte instance synced the row. Different per instance (prod airbyte vs airbyte-philz emit different values for the same logical row).Operational lag -- "how many sync generations behind is the clone." NOT cross-comparable row-by-row.
UPDATED_AT / CREATED_AT (business cursor)From the source system. Same value regardless of which Airbyte synced it.Windowing key -- compare only rows whose source-update is at-or-before the clone's latest source data. Apples to apples.

This distinction is the whole game. emitted_at tells you the clone is 4 syncs behind (the explanation). The business cursor is what you window on to ask "ignoring the syncs the clone hasn't gotten yet, does the data it DOES have match prod?"

Finding 4 -- old-Airbyte format + cursor variety

ETHER tables carry old-Airbyte meta (_AIRBYTE_AB_ID, _AIRBYTE_EMITTED_AT, _AIRBYTE_NORMALIZED_AT, _AIRBYTE_<table>_HASHID) -- not Destinations-V2. Business cursors vary: ORDERS/USERS/SALES_ORDERS have UPDATED_AT; TRANSACTIONS_V3 + TAX_RATES have none (no UPDATED_AT). This is why the earlier blanket cursor-windowing failed -- some streams have NUMBER or no usable timestamp cursor (the 'GRANTS'-era unexpected token + the NUMBER-vs-TIMESTAMP error we hit). Windowing must be cursor-type-aware.

Current vs end state

  • Now: AIRBYTE_CLONE is a frozen Snowflake zero-copy clone (from the reclone). It does not sync; lag grows ~1 generation/hour forever until re-clone.
  • End state (Joey's two-VM design): airbyte-philz feeds AIRBYTE_CLONE on its own schedule. Then lag is bounded by the offset between prod's schedule and airbyte-philz's schedule + sync completion time. The clone's airbyte-philz ETHER schedule should be confirmed via the Airbyte API at NEW_AIRBYTE_URL=http://35.224.144.209:8000 (token auth, creds in airbyte_migration_tools/.env) when sizing the tolerance.

Open questions for ~p

  • What tolerance? How many sync generations / hours may the clone trail before drift is a SIGNAL rather than EXPECTED? (Prod is hourly; a healthy clone should be ≤1-2 syncs behind once airbyte-philz runs.)
  • Window on business cursor where present; what's the fallback for cursor-less streams (TRANSACTIONS_V3, TAX_RATES)? Options: row-count tolerance band, _AIRBYTE_EMITTED_AT-batch alignment, or accept whole-table compare + lag annotation.
  • Per-stream vs per-schema watermark? ETHER syncs atomically, so a single schema-level watermark suffices for ETHER -- but other schemas may differ.
