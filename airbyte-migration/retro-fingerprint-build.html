What went well

  • Design held under contact with reality. Phase 1 proof showed 61.5% MATCH instead of 90%, but the failure was the gate (an assumption), not the primitive. HASH_AGG + column-pruning IS the right approach. Adjusting the gate took one line.
  • Escape routes earned their keep. "EXCEPT magnitude opt-in" was a design escape route; Phase 3 dry-run hung without it and we exercised the route immediately. No re-plan needed.
  • Standard procedure proved itself. Every artifact landed at a click-able URL the moment it was committed. ~i / ~d / ~p / ~e / 5 x ~b / ~r -- 10 artifacts, all published, all linked. The "show me the URL" pattern is now baked in.
  • Helper extracted clean. Lifting the proof script into _fingerprint.py took 30 min and produced 4 reusable functions with clean signatures.
  • publish() bug caught inflight. Phase 4 hit a KeyError on the old 'counts' key in publish()'s commit message. Fixed in seconds, didn't derail the phase.

What was harder than expected

  • The 90% gate was too optimistic for current state. 4 days of frozen-clone drift means 39% of streams legitimately DIFFER. The design needed to interpret that as truth, not failure. Lesson: gates should be calibrated against current world-state, not against the ideal.
  • EXCEPT magnitude is wildly expensive on big tables. Phase 3 dry-run hung at 13+ min with magnitude on (160 EXCEPT queries on multi-billion-row tables). The design's escape route saved us. Lesson: any per-stream operation needs a budget estimate before integrating.
  • Cursor filter doesn't help UPSERT divergence as much as expected. A row whose cursor moves past the filter line on prod will be excluded from prod's view, included on clone (frozen). Two iterations on the proof script to land on the right combination of cursor-filter + cursor-excluded-from-hash.

What to do differently next time

  • Run a small-scale wall-time probe before integrating expensive operations. Estimating "208 streams x N queries x ~2s each" upfront would have predicted the EXCEPT hang.
  • Default-off any operation whose cost is data-dependent. EXCEPT magnitude is now default-off behind a flag. Same pattern should apply to L4/L5 deep diff when we add them.
  • Gate criteria should be derived from observable state, not from textbook ideals. "90% MATCH on current frozen clone" was wishful; "baseline now, track delta on future runs" is reality.

What this enables next

  • Phase 3 (V2 discovery) bracketing. Snapshot before V2 DDL, run DDL, snapshot after. The DIFFER list deltas will tell us exactly which streams were affected.
  • Phase 4 (cron resume) tracking. As syncs catch up, MATCH count should climb toward 100% (with occasional drops as new UPSERTs land). The fingerprint panel makes the trajectory visible.
  • Future forensics. Any time Wesley reports "is X stream in parity?", run a snapshot, look at the DIFFER list, get an answer in 6 minutes.
  • L4 / L5 row-level deep diff can land as an additional opt-in flag against the same helper module. No more re-architecture needed.

Open carries (separate work)

  • Drift panel still N/A. Cloud Run /dashboard/data requires gcloud auth in the snapshot session. Wire it up in a follow-up board.
  • Magnitude per top-N DIFFER streams. Currently all-or-nothing. Could add --magnitude=10 to compute only for top 10 DIFFERs by stream-size. Nice-to-have.
  • Snapshot cron. Right now snapshots are explicit. Could optionally bracket every procedure-phase script automatically. Out of scope for Board 9.
