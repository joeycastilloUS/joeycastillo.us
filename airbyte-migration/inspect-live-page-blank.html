Be · airbyte-parity ~i live page blank

Live op page shows nothing

Operator: "op succeeded but cadence live op shows nothing."

root cause: CORS preflight blocked at IAM

Diagnosis

The HTML file is served from file:// in the browser. Its JavaScript calls fetch("https://cadence-.../operations?limit=1") with an Authorization: Bearer header.

The break: the Authorization header is a "non-simple" header. The browser fires a CORS OPTIONS preflight FIRST -- and that preflight has no Authorization header on it (by design). Cloud Run's IAM layer rejects the unauthenticated OPTIONS request before it ever reaches the cadence app. The preflight fails → the browser refuses to send the real GET → the page stays blank.

Even if I added CORS middleware in the app, Cloud Run's auth layer rejects the OPTIONS request before the app sees it. The only ways to make in-browser fetch work would be (a) make the service unauthenticated (allUsers invoker), or (b) front it with a public proxy. Both worse than the alternative.

Fix

Move the polling out of the browser, into the launcher. PowerShell already has the gcloud token + can fetch /operations cleanly. So: PowerShell loops, fetches the op + gate, renders a static HTML to disk, and the page uses <meta http-equiv="refresh" content="3"> to reload itself. No cross-origin fetches from the browser. No CORS. No token in JS. Works on file://.
PowerShell (Go.Live.ps1) Browser (cadence-live.html) --------------------------- ----------------------------- fetch /operations/{opId} fetch /escalate render HTML -> file write file <meta refresh content="3"> sleep 3s (browser reloads file from disk) loop renders the new HTML ... op terminal? omit meta tag. final state stays put.

Side benefits: no browser-cached token at risk, polling cadence happens in one place (the launcher) with proper auth + error handling + a visible polling status in the console window.

Path to ship
