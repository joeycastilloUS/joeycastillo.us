Plan — Awaiting Approval
Intent: the next time a reclone fires, anyone with HTTP can see every step in real time — no "what does your screen say?", no inferring from schedulerState. POST /reset/reclone → opId → /operations/{opId} → watcher follows the steps.

Phases

1 Operations schema + read endpoints

Firestore cadence-operations doc shape (opId, kind, status, startedAt, finishedAt, steps[], cloneCreatedAt?, error?). GET /operations?limit=N + GET /operations/{opId}. Both header-guarded? Read endpoints can be authenticated-invoker only; no Confirm header for reads.
Accept: Writing a hand-crafted op doc to Firestore and hitting GET /operations/{opId} returns it cleanly. Empty list when no ops.

2 Reclone harness -- no-SQL (First Move)

POST /reset/reclone guarded by X-Reset-Confirm. Spawns a background task (Task.Run) that records the four step doc updates: pause-scheduler (real Cloud Scheduler pause), drop-clone (NOOP, marks succeeded), create-clone (NOOP, marks succeeded, sets cloneCreatedAt = now), resume-scheduler (real Cloud Scheduler resume). Returns 202 + {opId}.
Accept: POST returns opId immediately; polling /operations/{opId} every 5s sees the four steps flip to succeeded over a few seconds; scheduler genuinely pauses + resumes around the NOOP steps. Op status transitions running -> succeeded.

3 Drop AIRBYTE_CLONE step (real SQL)

Replace the drop-clone NOOP with the real DROP DATABASE AIRBYTE_CLONE via the existing SnowflakeConn. Lift the exact SQL from Cadence.Driver/Steps/DropCloneStep.cs. Honor the 3600s CommandTimeout (Board 13 fix). On failure, write step.status=failed + step.error; op.status=failed; do NOT proceed to Create (clone would already be dropped + recreating without it would be a different bug).
Accept: POST /reset/reclone on a sandbox actually drops AIRBYTE_CLONE; op doc shows drop-clone succeeded with timing; an injected failure (e.g. permission) records step.error and halts the op.

4 Create AIRBYTE_CLONE FROM AIRBYTE_DB step

Replace the create-clone NOOP with the real CREATE DATABASE AIRBYTE_CLONE CLONE AIRBYTE_DB. Lift from Cadence.Driver/Steps/CloneFromProdStep.cs (no COPY GRANTS — Board 11 fix). 3600s timeout. On success, write step.cloneCreatedAt = the time the CREATE returned. Then proceed to resume.
Accept: POST /reset/reclone end-to-end on the live service: pause -> drop -> create -> resume, all four steps logged with timing, AIRBYTE_CLONE present with cloneCreatedAt matching the create's finishedAt; a follow-up /tick measures the fresh clone (signal=0 or near).

5 Server-driven watcher + close

Replace the watch_gate.py hack with a real watcher (Go.Watch or rolled into Go.Cutover) that polls /operations/latest + /escalate. Knows the precise step instead of inferring from schedulerState. Optionally Go.Reclone --server hits /reset/reclone instead of the Driver. Build report + close Board 22.
Accept: A rehearsal run end-to-end with NO console-Q&A: I hit /reset/reclone, the watcher prints "drop-clone: running", "drop-clone: succeeded", "create-clone: running", ..., "resume: succeeded", then signal collapses to 0, gate GREEN. The next time a destructive op runs, anyone can see every step.
