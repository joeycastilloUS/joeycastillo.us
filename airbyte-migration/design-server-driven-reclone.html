Server-driven reclone + operations log

Lift the destructive reclone into the cadence service with a Firestore ops log, so the rehearsal + cutover chain runs and is observed entirely from cadence. No console-Q&A. No "what does your screen say?" The next time a reclone fires, anyone with HTTP can watch every step live.

Design — Pure Metal
What prompted this. The 2026-05-29 rehearsal worked — Go.Reclone succeeded, gate went GREEN — but the gate-watcher saw scheduler=Paused for 37 min and couldn't tell whether the Driver was mid-clone, waiting at an approval gate, or stuck. I asked the operator what his console showed. That is the bug. Cadence already has the Snowflake key and Firestore; lifting the reclone into the service makes the whole chain server-observable.

Subject + Intent

Subject: A server-side reclone — POST /reset/reclone (header-guarded) spawns an async task that runs Pause → Drop → Create → Resume in the cadence service, writing per-step progress to a cadence-operations Firestore collection. Intent: end the console-Q&A dependency. Anyone with HTTP can fire and watch the entire chain — the watcher reads /operations instead of inferring state from schedulerState. The Driver stays for interactive local runs; autonomous runs go through the endpoint.

Discovery

  • cadence already owns the load-bearing pieces: the Snowflake connection (uses it for /tick), Firestore writes, the Cloud Scheduler pause/resume (/reset/pause + /reset/resume).
  • The reclone's destructive SQL (DROP DATABASE / CREATE DATABASE FROM CLONE) lives in Cadence.Driver Steps — runs on the operator's workstation. That makes progress local-only.
  • The observability gap surfaced today: 37 min of scheduler=Paused told the watcher nothing about which step — was Drop running, Create running, or just waiting for an approval? A server-side op log answers it directly.
  • Honest scope: the Driver is still valuable for INTERACTIVE local runs (real cutover, an operator wants per-step approval). The server-side path is for autonomous chains + remote observation. They coexist.
  • Auth: cadence-svc already loads the Snowflake key from Secret Manager — no user creds needed for the autonomous path. The Driver's gcloud auth print-access-token dance becomes optional.

Choices + Consequences

1. Async vs sync execution
a) Async: return opId immediately, run in background, poll /operations/{opId}. b) Sync: hold the HTTP request open until done.
a) Standard for long-running ops; Cloud Run kills idle requests at the timeout, but background tasks survive within the instance lifetime; risk = instance recycle mid-op. b) Simple but Cloud Run's request timeout (we configured 540s) is shorter than a worst-case clone; would 504 mid-flight even on success.
PICK: (a) async + log every step.
Long-running by nature. The op log is the recovery mechanism: if the instance recycles, the log shows the last successful step and someone can resume manually.
2. Where the Drop/Create SQL lives
a) Copy the SQL strings from Cadence.Driver/Steps/ into a new Cadence.Service.Operations.ReconeOp. b) Extract to a shared library both reference.
a) Pragmatic; 2 short SQL strings; duplication risk is small + caught by integration test. b) Cleanest long-term but pulls scope (a new Cadence.Shared project).
PICK: (a) for now; refactor to shared if a third caller appears.
DRY is a guideline, not a religion. Two copies of DROP DATABASE AIRBYTE_CLONE won't drift faster than the migration finishes.
3. Driver's future
a) Driver stays as a local interactive UI for forensic + cutover-day approval runs. b) Driver becomes a thin client of /reset/reclone. c) Retire the Driver entirely.
a) Two paths to the same effect; the Driver's per-step interactive gates remain useful at real cutover. b) One path, but loses the local-only forensic mode (no internet). c) Removes a working tool that just helped this rehearsal.
PICK: (a) Driver stays; server path is additive.
"Roll your own. The best code is no code." — but only when it's no code worth deleting. The Driver works. Leave it; add a server path beside it.
4. Progress granularity
a) Per-major-step (pause/drop/create/resume). b) Per-SQL-statement + heartbeat. c) Just start/end markers.
a) Four log writes per op, enough to drive any watcher; matches the Driver's existing step model. b) Richer but writes/second on a long Create; over-engineering. c) Tells you nothing while it's running — back to the watcher's blind spot.
PICK: (a) per major step.
Four data points (paused, dropped, created, resumed) gives the watcher everything it needs to localize where things are or where they failed.
5. Auth model for /reset/reclone
a) X-Reset-Confirm header guard, IAM-authenticated invoker only. b) Add a one-shot token in the body. c) Allow unauthenticated (allUsers).
a) Matches /reset/wipe + /reset/baseline + /reset/retention — consistent. b) Extra friction without a clear threat model. c) Anyone can drop your clone — obvious no.
PICK: (a) X-Reset-Confirm + run.invoker IAM.
Same guard as the existing destructive endpoints; same surface area for the operator to learn.

Escape Routes

From PICK 1 (async + instance recycle): if Cloud Run recycles mid-op, the operations doc has the last successful step + the running step's startedAt. Operator resumes manually from there (Driver) or via a future /reset/reclone/resume?from=step. Reversible — no work lost from the log.
From PICK 2 (copied SQL): if Drop SQL drifts between Driver and service, an integration test runs the same SQL against a fresh sandbox and catches it. Refactor to shared lib if a third caller appears.
From PICK 3 (Driver stays): if the duplication becomes maintenance pain, Driver gets refactored into a thin client of the endpoint. Reversible.
From PICK 4 (granularity): if four steps isn't enough during the Create (the slow one), add an in-step lastHeartbeatAt field. Cheap upgrade.

First Move

The smallest proof Ship /reset/reclone + /operations as a harness with no SQL — the background task does only Pause → Resume, writing the four step records (pause-scheduler / drop-clone-NOOP / create-clone-NOOP / resume-scheduler) to cadence-operations/{opId}. Poll /operations/{opId} from the watcher and see the steps appear in real time. Once the harness is proven (~30 lines), the actual DROP + CREATE SQL drops in as two more await calls. The watcher rewrite is independent and immediate.

What this does NOT do

  • Remove the Driver. It stays as the local interactive path; cutover-day operator approvals will likely still use it.
  • Make the reclone faster. The Snowflake clone takes what it takes.
  • Change auth for the existing endpoints. Same header guard, same IAM.
  • Generalize to N operation kinds. One kind now (reclone); the schema admits more but the code only handles reclone.
