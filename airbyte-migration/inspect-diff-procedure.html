Diff visibility across the AIRBYTE_DB ↔ AIRBYTE_CLONE migration procedure

What we already have, what's missing, and where the gap blocks a click-through publish-to-site standard.

Inspect — Answer Below
Subject: Joey wants to see the diff between AIRBYTE_DB and AIRBYTE_CLONE at each step of the migration procedure, rendered as HTML, auto-published to joeycastillo.us/airbyte-migration/, with a click-able link surfaced back into the conversation each time. Today this exists piecewise — the diff dimensions are owned by 5+ separate scripts and one production dashboard, but nothing produces a single per-phase snapshot and nothing auto-publishes. This is the standard we should make standard.

What It Is

The clone-rebuild procedure (be/plans/2026-05-20-1236-plan-clone-rebuild-and-parity-baseline.html) walks AIRBYTE_CLONE through six phases. Each phase transforms the clone in a different dimension. The "diff" between AIRBYTE_DB and AIRBYTE_CLONE is therefore not one number — it's a tuple of five measurements, each with its own tool today.

Stack

  • Five diff scripts in airbyte_migration_tools/scripts/ + airbyte-parity/scripts/: quick_count_diff.py, audit_clone_freshness.py, discover_clustering.py, verify.py, verify_batch.py. Each owns one dimension. Each emits stdout markdown or JSON.
  • Parity engine — C# service on Cloud Run (airbyte-parity-d5vc4tye3a-uc.a.run.app). 5 layers: L1 (counts via INFORMATION_SCHEMA), L2 (column sums), L3 (column intersection + min/max), L4 (PK-based row-hash diff, 1000-row cap), L5 (surgical row inspect). /dashboard/data emits per-stream verdicts as JSON.
  • Site — joeycastillo.us = Cloudflare Pages, auto-deploy on push to github.com/joeycastilloUS/joeycastillo.us. /airbyte-migration/ already hosts 13 pages (parity.html, stream-comparison.html, append-cutover-plan.html, snowflake-keypair-master.html, etc) all using the same dark + tron-blue idiom.
  • Auth gate on the site — /airbyte-migration/gate.css + gate.js + a Cloudflare Function under functions/airbyte-migration/. Existing pattern: signed-hash gating for sensitive pages.

Patterns Found

1. Diff is dimensional, not scalar

No single number is "the diff." Counts answer "did rows get lost", freshness answers "is clone caught up", clustering answers "did Airbyte's V2 prereqs get satisfied", schema answers "are the columns in alignment", drift answers "is there semantic divergence". A standard snapshot must capture all five — or at least flag which are N/A for the phase.

2. Production parity engine already does steady-state. Snapshots are missing.

The L1-L5 engine runs on cron every tick against the steady-state pair. What it does not do: capture a frozen "before this phase" and "after this phase" view. The dashboard is always now-tense. Procedure work needs then-vs-now.

3. Site publishing is one git push.

Cloudflare Pages auto-deploys the joeycastillo.us repo. To get a new HTML page live at joeycastillo.us/airbyte-migration/foo.html: write /c/joeycastillo.us/airbyte-migration/foo.html, commit, push. CF picks it up in ~30s. There is no deploy.json, no wrangler config to maintain — the push is the deploy.

4. Existing /airbyte-migration pages share a CSS idiom.

Max-width 860px, dark background #111, accent #6FC3DF, summary-box grid with .stat .num for hero numbers, status pills (.status-live, .status-warn, etc), Helvetica Neue. A snapshot template should match so it doesn't feel like a foreign tab.

The gap, named

What is missing: a single tool that, given a phase identifier and a label (before/after or a free-form slug), takes a five-dimensional snapshot of AIRBYTE_DB vs AIRBYTE_CLONE, renders one HTML page in the site idiom, writes it to airbyte-parity/be/snapshots/ for audit and copies it into joeycastillo.us/airbyte-migration/snapshots/ for publication, updates an index page that lists every snapshot in reverse-chronological order, and emits the click-able public URL back to stdout. Today that workflow is six scripts + a manual stitching + a manual copy + a manual commit. No one would do it consistently because it's friction-heavy. The standard procedure exists only on demand if the tool exists in one invocation.

Open Questions (for ~p to resolve)

  • Five dimensions in one snapshot, or one dimension per snapshot? Recommendation: one snapshot = all five (or N/A markers for irrelevant ones). The whole point is one click for one moment in the procedure.
  • Snapshot location on site: /airbyte-migration/snapshots/<file> with an index at /airbyte-migration/snapshots/? Or flat? Recommendation: subdir + index.
  • Which repo owns the snapshot tool: airbyte-parity (the diff-aware repo, already has parity engine + sf_connect helper, and is the "clone perspective" project) or airbyte_migration_tools (the operator's toolkit)? Recommendation: airbyte-parity — it already imports the parity engine concepts.
  • Auto-snapshot per procedure-phase script, or always explicit? Recommendation: explicit invocation. Procedure scripts mention the snapshot command in their output as the next-step nudge. Avoids surprise writes during a tight DDL window.
  • Auth gate on the snapshots? The existing keypair pages are gated. Snapshot pages contain row counts and schema diffs — sensitive enough to also gate, or public enough to leave open? Recommendation: gate them with the same gate.js pattern, since they contain prod schema info.
  • L4/L5 row-level diff inclusion? Cheap dimensions (counts, freshness, clustering, schema) run in ~5s. L4 row-hash diff can be slow on large streams. Recommendation: include L1-L3 always, L4/L5 only with --deep flag.

What ~p must produce

  1. A single tool: airbyte-parity/scripts/snapshot.py --phase <N> --label <slug> [--publish] [--deep]
  2. A template: snapshot HTML that matches the site idiom and renders 5 panels (counts, freshness, clustering, schema, drift)
  3. An index page: joeycastillo.us/airbyte-migration/snapshots/index.html listing every snapshot newest-first
  4. A publish path: write local copy + copy to site repo + auto-update index + git commit + git push — ideally in one command
  5. A first invocation: backfill a "post-Phase-2" snapshot so the standard has its first datapoint visible at the click-able URL right away
