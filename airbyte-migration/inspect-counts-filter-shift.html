Shift the counts filter line back 24h -- no fresh clone needed

The line moves to clone_time - 24h. Anything newer than the line gets dropped from scope. Existing clone is the right reference; no Phase 1 re-do.

Inspect — Two Questions Answered
You asked two things: (1) shift the cutoff to 24h before clone, filter anything newer than the line out of scope; and (2) do we need to re-clone? Short answer: No re-clone. The existing AIRBYTE_CLONE created 2026-05-20 13:19:23 PDT is exactly the right reference. The change is a one-line SQL swap: scope shifts from "rows in the 24h before clone" to "rows older than the 24h before clone". Both prod and clone should be identical in that older slice; any non-zero delta is real divergence, not in-flight noise.

Question 1: Do we need a fresh clone?

No. The existing clone is our reference point. Phase 1 already gave us a Snowflake CLONE database that captured prod's state atomically at 2026-05-20 13:19:23 PDT. That timestamp is the only anchor we need. Shifting the line back 24h is a SQL filter change against the data we already have; the clone itself doesn't change.

Why a fresh clone wouldn't help

  • A new clone would just give us a new anchor at today's timestamp. We'd still need to subtract 24h to define the "stable" window, and we'd still be doing the same WHERE-clause analysis against the new pair.
  • The clone we have is already free of "in-flight" risk: it captured prod atomically. Rows with cursor older than clone_time - 24h were definitely settled when the snapshot was taken.
  • Re-cloning costs nothing meaningful (CLONE is metadata-only in Snowflake) but it would invalidate this snapshot's audit value -- we'd lose the diff line we just established. Better to keep the May 20 anchor and analyze it correctly.

Question 2: What changes about the filter

Current behavior (cursor-windowed, in scope)

-- rows in the 24h window ENDING at clone_time WHERE cursor >= TO_TIMESTAMP('2026-05-19 13:19:23') AND cursor <= TO_TIMESTAMP('2026-05-20 13:19:23')

This was useful (got us from -23M cumulative noise to -5,484 in-window signal) but it includes the very rows most likely to have shifted on prod via UPSERT after clone_time. The negative deltas in that window (ETHER.USERS -6,626, etc) are exactly that artifact.

New behavior (stable line, in scope = older than line)

- WHERE cursor >= TO_TIMESTAMP('2026-05-19 13:19:23') - AND cursor <= TO_TIMESTAMP('2026-05-20 13:19:23') + WHERE cursor <= TO_TIMESTAMP('2026-05-19 13:19:23') -- clone_time minus 24h

One line removed, one line shifted. The line is clone_time - 24h. Everything newer than the line is dropped. Prod and clone should match exactly in the surviving slice -- any non-zero delta is structural divergence, not freshness drift.

Why the new line is the right one

prod cursor timeline (left = older, right = newer):

 ...===========================|=========|=========>
 ........... stable rows ....|in-flight| post-clone
 ..............................|.........|
 ..............................^.........^
 ..............................line......clone_time
 ..............................| SCOPE |
  • Stable rows (cursor <= line): These rows had their last cursor-bearing event more than 24h before clone_time. Vanishingly unlikely to UPSERT a backdated cursor in real life -- so prod and clone agree exactly. Delta = 0 expected.
  • In-flight rows (line < cursor <= clone_time): Captured by both prod and clone at clone_time, but prod has since updated some of them (cursor shifted forward to today). On prod these rows now live with cursor > clone_time and don't appear in either window. On clone they still appear in the original 24h window. This is the source of the -5,484 we saw -- it's real but it's drift caused by UPSERT, not data loss.
  • Post-clone rows (cursor > clone_time): Already excluded; only exist on prod.

Joey's new line shifts the scope left -- away from in-flight territory -- so the comparison stops capturing UPSERT-induced shifts and focuses on rows that should be immutable from both sides' perspective.

Implementation -- two-line change

def collect_counts(cur, clone_time): ... line = clone_time - timedelta(days=1) line_s = line.strftime("%Y-%m-%d %H:%M:%S") # was: WHERE cursor BETWEEN (line) AND clone_time q_prod = f'SELECT COUNT(*) FROM {PROD_DB}."{schema}"."{table}" ' \\ f'WHERE "{cursor}" <= TO_TIMESTAMP(\\'{line_s}\\')' q_clone = f'SELECT COUNT(*) FROM {CLONE_DB}."{schema}"."{table}" ' \\ f'WHERE "{cursor}" <= TO_TIMESTAMP(\\'{line_s}\\')'

Same window helper, just one comparison operator changed (BETWEEN ... AND ... → <=) and a window-end removed. Headline labels in the panel become "Rows older than line" instead of "Rows in window".

Cost + performance

Watch out: the new query has no upper bound on cursor, so Snowflake will scan from beginning-of-time up to the line. For streams that go back years (ETHER.ORDERS 7+ years of UPDATED_AT), that's a full table scan. Today's windowed query was fast because BETWEEN is a tight range. The new query will probably run 2-3x slower per stream.

Mitigation if needed: add a far-back lower bound (e.g., cursor BETWEEN '2024-01-01' AND line). For Philz's data ages that's effectively the whole history minus the in-flight 24h, but Snowflake's pruner will use clustering keys on cursor when they exist. We'll measure after the first run and add a bound only if it hurts.

Open Questions

  • How wide does the "in-flight" buffer want to be? 24h is your call. If we ever see UPSERTs of records older than 24h on certain streams, we'd need to extend the buffer (48h, 7d). Track in the snapshot panel as "buffer used: 24h" so the choice is visible.
  • Should we also surface "in-flight rows on clone" as a sub-metric? Count of rows where cursor BETWEEN line AND clone_time on the clone side -- represents the volume of activity in the 24h buffer that we're deliberately not comparing. Useful context: "X rows were in flight at clone time; this is how busy prod was that day".
  • Do we need to backfill old snapshots with the new filter? The two snapshots already published (post-cleanup, post-cleanup-filtered) used different filter logic. They stay as historical records of their own filter choice; the new one is the first with the shifted line. Consider linking from the snapshots index that explains the filter evolution -- but probably not worth the effort.
  • Edge case: non-timestamp cursors. 7 streams error out today. _AB_SOURCE_FILE_LAST_MODIFIED for COMTRADE is one. Whether to extend cursor parsing or keep them as errors is a separate small follow-up.

Recommendation

  1. Apply the two-line filter change to scripts/snapshot.py collect_counts.
  2. Re-run: python scripts/snapshot.py --phase 2 --label post-cleanup-stable-line --publish.
  3. Check the delta: if < 100 across all schemas, the line is correctly placed and we have our stable baseline. If >> 100, dig into top-drifters to see whether (a) buffer needs to be wider, or (b) clone actually did lose data.
  4. If query runtime > 5 min, add a far-back lower bound (e.g., cursor >= '2024-01-01').
