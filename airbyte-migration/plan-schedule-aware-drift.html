Schedule-aware drift classification

Make cadence separate EXPECTED drift (the clone just hasn't gotten prod's latest hourly syncs yet) from SIGNAL drift (data that should match but doesn't, or a clone that's fallen too far behind). Window the comparison by the business cursor to the clone's last-synced generation; classify and explain the rest.

Approach -- the model

  1. Watermark. For each stream, read clone_wm = MAX(cursor) on the clone and prod_wm = MAX(cursor) on prod. lag = prod_wm - clone_wm. (Use the business cursor -- UPDATED_AT/CREATED_AT -- NOT _AIRBYTE_EMITTED_AT, which isn't cross-comparable. Per the inspect.)
  2. Window. Fingerprint both sides WHERE cursor ≤ clone_wm. This drops the rows the clone hasn't received yet -- the expected-lag rows -- so we compare the same source generation on both sides.
  3. Classify.
    • clean -- windowed fingerprints match. The data the clone has is identical to prod. Healthy.
    • expected-lag -- windowed match, but lag > 0 and within tolerance. Differs from the naive whole-table compare only because prod has newer syncs. Explained, not alarming.
    • signal -- windowed fingerprints still DIFFER (data that should match doesn't), OR lag exceeds tolerance (clone stuck / sync broken). This is what a human should look at.
  4. Explain. Every report line carries the lag: "ETHER 4 syncs / 4.0h behind -- within tolerance -- windowed match: clean."

The tolerance ("period to filter")

This is the time-window Joey described. Two knobs, per-schema (defaults shown):

KnobMeaningETHER default
syncIntervalHoursExpected sync cadence (from the schedule / measured)1.0 (hourly, confirmed)
maxLagGenerationsHow many syncs the clone may trail before SIGNAL2 (i.e. clone may be up to ~2h behind once airbyte-philz runs)

Lag ≤ syncIntervalHours x maxLagGenerations + a grace for sync-completion time = expected. Beyond = signal. While the clone is a frozen Snowflake clone (current state) every stream will exceed tolerance over time -- correct, because a frozen clone IS stale; the report should say "clone frozen N h, re-clone or resume airbyte-philz."

Phases

1 Watermark + lag (read-only, additive)

Compute per-stream clone_wm/prod_wm/lag using the business cursor from stream-catalog (cursorField) or a per-schema override. Emit lag into the /tick snapshot doc alongside the existing drift data. No verdict change yet -- just measure + store.
Accept: /tick writes per-stream lagHours; /firstdiff + report show "N h behind". ETHER shows ~uniform lag. Zero behavior change to existing verdicts.

2 Cursor-windowed fingerprint (type-aware)

Re-enable windowing in the fingerprint: WHERE cursor ≤ clone_wm, but only for timestamp cursors. NUMBER cursors + cursor-less streams fall back to whole-table (flagged "unwindowed"). Fixes the earlier 'GRANTS'/NUMBER-vs-TIMESTAMP errors by detecting cursor type first.
Accept: ETHER windowed fingerprints MATCH (the lag rows removed). Cursor-less streams (TRANSACTIONS_V3, TAX_RATES) cleanly fall back without error. No SQL compilation failures.

3 3-state classifier + tolerance

Combine windowed verdict + lag-vs-tolerance into clean / expected-lag / signal. Tolerance config per schema in appsettings or a small drift-policy.json. Store the verdict in the snapshot doc.
Accept: ETHER (frozen-clone, 4h) classifies expected-lag while within tolerance, flips to signal past it. A deliberately corrupted row (windowed mismatch) classifies signal immediately.

4 Explain it -- reports + readers

Surface the classification + lag narrative in the HTML run report, /firstdiff, and Go.Status. "183/208 raw -> 208/208 windowed; 25 expected-lag (ETHER 4h behind, hourly sync); 0 signal."
Accept: A human reading the report sees "all green except expected sync lag, nothing to chase" without knowing the internals.
