Cutover Procedure

Database rename swap in Snowflake. Zero consumer changes needed.

  1. Pause all connections on old Airbyte servers
  2. Final parity check — row counts match across all schemas
  3. Rename AIRBYTE_DB → AIRBYTE_ARCHIVE
  4. Rename AIRBYTE_CLONE → AIRBYTE_DB
  5. Update V2 destination to point at AIRBYTE_DB
  6. Enable cron schedules on all airbyte-philz connections
  7. Decommission old servers after 48h observation

Downstream consumers (dashboards, dbt, reports) keep working — database name stays AIRBYTE_DB. Archive = instant rollback.

Background

Two production Airbyte V1 instances on aging VMs — 20 + 4 connections, 10 sources, all syncing to Snowflake. Migrated to a single Airbyte 2.0.1 instance on a new VM with token auth, modern connectors, and static IP. Zero production downtime, zero data loss.

Key Decisions

  • Zero-copy Snowflake clone as migration target. All writes hit the clone, never production.
  • SSH-based export from old server (no external API). Captured all 20 connections, state, cursors, credentials in one shot.
  • V2 schema migration: custom batch_apply.py — 505 tables, 1.71B rows, 6 minutes, 2.07 credits on X-Large warehouse.
  • Rate-limit safety: kill switch, 429 monitoring, concurrency cap = 1, prod schedule mapping.
  • Incremental resume: connections replicated with injected cursor states — resume from where production left off.
