How We Got Here

Two production Airbyte instances (V1) run on aging VMs — one with 20 connections, one with 4 — syncing data from 10 sources into Snowflake. The goal: migrate everything to a single Airbyte 2.0.1 instance (V2) on a new VM with token-based auth, modern connectors, and a static IP — without any production downtime or data loss.

Approach

  • Zero-copy Snowflake clone. We cloned AIRBYTE_DB to AIRBYTE_CLONE as the migration target. All dry-run writes hit the clone, never production. Rollback = drop the clone.
  • SSH-based export. The old server had no external API. We built an SSH tunnel to extract all 20 connections, their state, cursor positions, credentials, and catalog configs. The export captured the full migration surface in one shot.
  • Credential pipeline. Secrets were extracted via web_backend API unmask, GCP Secret Manager resolution, and sealed CSV. Every credential was validated independently before wiring into connections.
  • Incremental replication. Connections were replicated INACTIVE on the new server with injected cursor states — so when activated, they resume from where production left off instead of re-syncing from scratch.
  • Rate-limit safety. Kill switch, 429 monitoring, per-connection concurrency cap of 1, and prod sync schedule mapping — all in place before the first external API call.

The V1 → V2 Schema Challenge

Airbyte 2.0 changed its internal schema format. V1 tables use _AIRBYTE_AB_ID, _AIRBYTE_EMITTED_AT, _AIRBYTE_NORMALIZED_AT, and _AIRBYTE_*_HASHID columns. V2 expects _AIRBYTE_RAW_ID, _AIRBYTE_EXTRACTED_AT, _AIRBYTE_META, and _AIRBYTE_GENERATION_ID.

If you just connect V2 to V1 tables, it either fails or does a destructive full refresh — wiping years of incremental history.

Our solution: A custom batch migration tool (batch_apply.py) that runs DDL transformations across all candidate tables in parallel — adding V2 columns, backfilling metadata, and preserving existing data and clustering. We tested this on a Phase 0 harness first (toy scope, failure injection, atomicity proof), then ran the real batch: 505 tables across 6 schemas, 1.71 billion rows, completed in 6 minutes on an X-Large warehouse for 2.07 Snowflake credits. Every table was verified post-batch: row counts unchanged, zero NULLs, signatures match. Full rollback capability preserved via per-table capture files.

We reach 100% data parity when all 11 connections are running on the same cron schedule as production and have completed at least one full sync cycle. At that point, the clone and production are receiving identical data from identical sources on identical schedules.

The current gaps (0.1–35%) are natural drift: production continues to ingest data while the clone was cut at a point in time. Each connection we activate closes its gap. Paytronix is already ahead of production. Once all 11 are live and syncing, the gap closes to zero.

How Cutover Works

Once all connections are verified, the cutover is a database rename swap in Snowflake:

  1. Pause all connections on old Airbyte servers
  2. Final parity check — confirm row counts match across all schemas
  3. Rename AIRBYTE_DB → AIRBYTE_ARCHIVE
  4. Rename AIRBYTE_CLONE → AIRBYTE_DB
  5. Update the V2 Airbyte Snowflake destination to point at AIRBYTE_DB
  6. Enable cron schedules on all connections in airbyte-philz
  7. Decommission old Airbyte servers after 48-hour observation window

Every downstream consumer — dashboards, reports, dbt — keeps working because the database name stays AIRBYTE_DB. The archive sits there as an instant rollback: rename it back if anything breaks. All connectors (including DoorDash and Uber Eats) write to the same schema names as production — no consumer changes needed.
