Phase 4 -- 6 sweep steps with source:driver field

cadence 936c40e (7 files / +310 lines)

The big one. BuildSqlStep parameterized PROD/CLONE -- composes the UNION ALL HASH_AGG SQL via the parent project's BuildBatchedFingerprintSql. ExecuteSweepStep runs the SQL with an inlined reader (the Cadence project's ReadFingerprintMapAsync is private; Driver owns its own Snowflake reader to keep the Cadence public surface clean), accumulates wall time into ctx.SweepWallSec. ComputeVerdictsStep pairs the two fingerprint maps into per-stream Verdicts (MATCH/DIFFER/ERROR/SKIP) and builds the SweepResult tally that mirrors FingerprintPrimitive.RunSweepAsync exactly. WriteFirestoreStep lazy-creates a FirestoreDb and writes the canonical doc shape plus the new source: "driver" field. SweepFlow.cs assembles the canonical 10-step list. --skip-write CLI flag added so forensic runs don't pollute the timeline.

Bug found + fixed inline: BuildSqlStep was passing ctx.CloneCreatedAt for cursor-windowing, which produced SQL of the form WHERE "cursor_col" <= TO_TIMESTAMP('...'). Some streams have NUMBER cursors (snapshot tables with ID-style cursors), and a TIMESTAMP literal fails type-conversion against a NUMBER column. FingerprintPrimitive has no per-stream fallback for that mismatch yet. Cadence Cloud Run sidesteps the issue by passing cloneTime: null (whole-table sweep). Driver now mirrors that exactly.

--auto --skip-write smoke: 9/9 PASS. match=117/208 (56.2%). PROD sweep 0.9-1.5s warm, CLONE sweep 1.7-5.0s. BIT-EXACT match to the last Scheduler-fired sweep before /reset/pause -- the Driver and the service share the same code path and produce identical verdicts.

Decisions Made

ForkPickWhy
Per-phase deploys vs batched (since plan said phase-by-phase)Per-phase, 5 cadence commitsThis is a console app -- no Cloud Build cycles to amortize, so atomic phase commits cost nothing. Each phase ends with a runnable Driver that does strictly more than the previous phase.
RunSweepAsync.ReadFingerprintMapAsync surfaceInline in DriverThe private helper isn't public on FingerprintPrimitive and adding it would couple Cadence's public surface to Driver internals. Inlining a 10-line reader in ExecuteSweepStep is cleaner.
Cursor-windowing for the sweep SQLcloneTime: null (whole-table)Plan's design called for cursor-windowing, but real-world catalog has NUMBER cursors that the timestamp-only TO_TIMESTAMP path can't handle. Matches Cadence Cloud Run service behavior. Future enhancement: per-stream type detection + fallback.
Manual mode scopeReal P/F/N prompt + ManualGuide on each stepPlan said "rare but useful." Implementing it real (not stubbed) costs ~30 lines; SchedulerToggle and FirestoreWipe steps get concrete gcloud fallback strings in their ManualGuide. Future operator on a borked laptop has actionable instructions.
Firestore writes during smoke--skip-write flag, deferred Firestore writesJoey paused snapshot collection mid-session. Auto-running the full SweepFlow would inject a doc with source:driver, which technically isn't "collection" in the autonomous sense but is still a write to the cadence-snapshots collection. Risk-averse: --skip-write flag covers forensic runs that should not pollute the timeline. Joey invokes the full flow with WriteFirestoreStep enabled when he chooses to.
results/ in .gitignoreYes -- forensic runs are operator-localRunResult JSON captures sensitive context (Snowflake query patterns, stream catalog, per-stream verdicts). Better to keep operator-machine-only by default. If durable record is wanted, operator can run with output redirected to a different path.

What this unlocks

  • Operator confidence on cadence the same way Joey trusts mendo-dgx. When the Scheduler-fired tick returns a surprising number, fire the Driver in Interactive and walk each step. The "makes no sense" problem from 2026-05-24 cannot recur.
  • One place to do ops. Reset baseline, pause/resume, wipe -- all available via --flow reset-baseline or as individual steps. Complements Board 10's HTTP endpoints; either path works.
  • Forensic record per investigation. results/runs/run-{ts}.json captures every step + result + ctx slice. Diff two runs to see what changed.
  • Reproducible local smoke after every clone refresh. One dotnet run -- --auto from the operator's machine confirms the full sweep path works against a fresh clone before re-enabling the Scheduler.
  • Pattern reuse. Future cadence ops (Top-N drift forensic, per-stream replay, schema-diff deep dive) become "add a new flow with N steps" rather than "wire a new endpoint."
  • Manual fallback. If Snowflake / Firestore / Scheduler are unreachable, Manual mode shows operator the gcloud / SQL commands to run by hand and records P/F/N.
