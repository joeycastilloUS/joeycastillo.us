Phases

1 Dry mode (First Move)

POST /reset/reclone?dry=true writes dryRun: true on the op doc. Pause + Resume scheduler steps fire for real (rehearsal realism). Drop + Create steps SKIP the SQL, log "would: DROP DATABASE..." + "would: CREATE DATABASE...", and sleep historical averages (Drop ~125s, Create ~760s) so timing looks like a real run. Step records succeed normally.
Accept: POST /reset/reclone?dry=true → op trail looks identical to a real run (per-step started/finished, ~15min total) but AIRBYTE_CLONE is untouched (verified by reading SHOW DATABASES before+after). dryRun: true visible in op doc.

2 Gates + /proceed

?gate=after-drop or ?gate=after-create persists on the op doc. After the named step succeeds, op sets gateState: "waiting" and parks (no advancement). POST /operations/{opId}/proceed sets gateState: "cleared" and the parked task picks up. Op-doc state machine handled in the background task's main loop.
Accept: POST /reset/reclone?dry=true&gate=after-drop → op pauses at "after-drop"; Go.Watch shows gateState: waiting; POST .../proceed → op continues to Create.

3 /abort

POST /operations/{opId}/abort sets requestedAbort: true. The background task checks the flag at: (a) every gate boundary before advancing, (b) every heartbeat tick during a long step, (c) just before each step starts. If set: skip remaining destructive steps, run Resume (clean scheduler state), mark op aborted with a note saying which step the abort took effect at.
Accept: Fire POST /reset/reclone?dry=true&gate=after-drop, then POST .../abort while parked at the gate → Resume runs, op status = aborted, Create + Resume steps skipped/marked aborted with reason.

4 Heartbeats

Inside Drop + Create step execution, a System.Threading.Timer (or background loop) writes lastHeartbeat: {at, queryId?, executionStatus?, bytesScanned?, elapsedSec, message} every ~15s. queryId resolved via SELECT SYSTEM$LAST_QUERY_ID() at step start; details via INFORMATION_SCHEMA.QUERY_HISTORY_BY_SESSION. Timer cancels on step completion. Honest if INFORMATION_SCHEMA returns nothing yet -- log a plain timer pulse.
Accept: Fire a real POST /reset/reclone; Go.Watch prints a heartbeat line every ~15s during Drop + Create with elapsedSec climbing + (when available) Snowflake bytesScanned / executionStatus. Heartbeat clears on step end.

5 Go.Watch render + ship

Go.Watch reads + prints the new fields: dryRun badge in the header; gateState: waiting highlighted yellow with the proceed/abort one-liners; lastHeartbeat printed inline (overwrites prior heartbeat lines or appends?); aborted status colored. Build report + close board.
Accept: A full rehearsal driven via Go.Watch: ?dry=true&gate=after-drop, hit /proceed, hit /abort once mid-Create, watcher prints the whole arc with heartbeats and the final aborted state. No console-Q&A.

Constraints / risks

RiskHandling
Dry mode's sleep duration drifts from real Drop/Create timesPull the durations from the latest succeeded real op (cadence-operations history). Fallback to constants. Updates itself over time.
Gate state requires the op task to poll the docBackground task reads gateState at each gate; lightweight Firestore read every 5s while parked. Bounded -- only when gated.
Heartbeat write rateEvery 15s during long steps = ~50 writes for a 12min Create. Within Firestore's per-doc 1 write/sec cap (Cloud Run side; same doc). Safe.
Abort during a real Drop that has already committedCooperative: we can't un-Drop. The abort takes effect BEFORE the next step. Op record states this clearly in the abort note.
Cloud Run instance recycle while parked at a gateOp doc has all state needed to resume. A future POST /operations/{opId}/resume-from-step?step=N closes this. Out of scope here.
