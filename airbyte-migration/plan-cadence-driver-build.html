Build Cadence.Driver — 5 phases, 15 pts, one new project in the cadence repo

Execute the design at be/designs/2026-05-24-2010-design-cadence-driver.html (and the inspect at be/inspects/2026-05-24-2010-inspect-mendo-dgx-steering-wheel.html). Add a console steering wheel that runs a cadence sweep as a sequence of preview-and-approve steps. Cloud Run keeps doing autonomous Scheduler-fired ticks unchanged; the Driver is the human-in-the-loop investigation mode.

Plan — Awaiting Approval
Intent: Give Joey the same operator confidence with cadence that he has with mendo-dgx. Today's autonomous /tick reports a single number; when it feels wrong, there's no narrative. The Driver runs the same fingerprint sweep as a 10-step flow that previews each action, prompts for approval, executes, captures the result, and recovers from failures. Three modes -- Interactive (default), Auto (matches Scheduler behavior), Manual -- share the same step list. All snapshots tagged source: "driver" so the dashboard timeline differentiates from Scheduler-fired ticks.

Approach

  1. First Move first. Phase 1 stands up the project, a minimal wheel, and one step (Connect). Proves the cross-project reference + auth path work end-to-end on Joey's machine before any of the sweep ceremony.
  2. Wheel before steps. Phase 2 builds out the full SteeringWheel (all six hatches, StepResult capture, failure recovery loop) on top of the minimal proof. After Phase 2, adding new steps becomes mechanical.
  3. Steps in two batches. Phase 3 lands the four "resolve" steps (cheap, read-only, build up StepContext state). Phase 4 lands the six "execute" steps (build/run/compute/write -- the bulk of the actual fingerprint sweep). Same shape, separated for sane commits.
  4. Ops + Auto last. Phase 5 layers the three ops sub-flows (SchedulerToggle, FirestoreWipe, ResetBaseline) on top of the Driver, plus Auto mode + RunResult JSON persistence. Auto mode is the dotnet-run equivalent of what Scheduler does; useful for local smoke after a clone refresh.

Verify dotnet --version >= 8 (already confirmed: 10.0.108 in this session). Branching: same call as Board 10 -- main, no branch ceremony. 2 SteeringWheel full hatches + failure recovery

~30 min. Expand the wheel from a one-step stub to the real loop with all six interactive hatches and the failure-recovery sub-loop. Add ANSI color helpers.
  • Six hatches: [Enter]=Go, [S]=Skip, [B]=Back, [I]=Inspect (stub for now, fleshed out in Phase 3), [F]=Find (search StepContext + last result), [D]=Dump (write current state to results/dumps/dump-{ts}.json), [Q]=Quit
  • Failure recovery: when StepResult.Status == Failed, prompt [R]=Retry / [I]=Inspect / [S]=Skip / [Q]=Quit. Retry calls step.Execute again on the same index.
  • StepResult capture: index, name, status (Pending/Passed/Failed/Skipped), startTime, durationMs, error message, notes, retryCount. No screenshots (cadence is SQL not browser).
  • StepContext: shared state passed between steps. Holds SnowflakeConn, CloneCreatedAt, StreamCursors, BizCols, ProdFingerprints, CloneFingerprints, Verdicts, SweepId. Each step writes its outputs; later steps read.
Accept: ConnectStep still runs in Interactive. [B] from step 1 of 1 = no-op message. [S] marks Skipped and exits flow. [Q] returns exit 0. [D] writes a JSON dump file. Forced failure (set bogus account in config) triggers retry prompt; [R] re-runs and fails again; [Q] exits.

5 Auto mode + ops sub-flows + RunResult persistence

~40 min. Three sub-flows + Auto mode + JSON dump on completion. After this phase, the Driver covers every operation the Cloud Run service does, plus the Board-10 reset endpoints.
  • Auto mode: --auto flag. Same step list, no prompts. Failures still pause for [R/I/S/Q] (matches mendo-dgx). Useful for local smoke after a clone refresh: dotnet run --project Cadence.Driver -- --auto --flow sweep.
  • Manual mode: --manual flag. For each step, show the manual guide (one paragraph of "run this SQL by hand if Snowflake is unreachable"), prompt for [P]ass/[F]ail/[N]otes. Rare-but-useful forensic mode.
  • SchedulerToggleStep: one parameterized step (pause | resume). Reuses CloudSchedulerClient + JobName resolution from Board 10's BuildSchedulerAsync.
  • FirestoreWipeStep: one step. Same wipe logic as /reset/wipe endpoint (500-doc batches). Preview shows current doc count.
  • ResetBaselineFlow: compound flow that strings SchedulerToggle(pause) -> FirestoreWipe -> sweep flow (steps 1-10) -> SchedulerToggle(resume). Invoked via --flow reset-baseline.
  • RunResult JSON: at flow end, serialize the full run (mode + flow + steps + StepResults + final StepContext slice) to results/runs/run-{ts}.json. Mirrors mendo-dgx ResultWriter.
Accept: --auto runs the full 10-step sweep end-to-end with no prompts; writes a Firestore doc with source: "driver"; emits a results/runs/run-*.json file. --flow reset-baseline --interactive walks through pause -> wipe -> sweep -> resume with previews at each ops sub-step. (Smoke note below.)

Smoke strategy

The Driver smokes itself. Unlike Board 10 (which had destructive endpoints requiring Joey to invoke explicitly), the Driver is invoked manually from start. The first time Joey runs cadence-driver --interactive, he walks through every step approving each one. The Connect through Resolves are read-only and verify the auth + catalog path. The Sweep steps run against live Snowflake but only do SELECTs (no writes to AIRBYTE_DB or AIRBYTE_CLONE). The only write is the Firestore snapshot -- a new doc with source: "driver", doesn't disturb the existing trajectory. Reset-baseline sub-flow IS destructive (wipes Firestore + pauses Scheduler) -- but it's gated by the interactive mode's per-step approval, so it only happens if Joey explicitly approves each ops step.

Risks + mitigations

RiskMitigation
Cross-project reference breaks if Cadence project changes its public surfaceDriver only uses currently-public methods from FingerprintPrimitive + SnowflakeConn. If those need to change, the change goes in Cadence first, Driver compiles against new surface.
Snowflake key path differs locally vs cloudDriver uses local file path only (no SecretManager) -- it's an operator-machine tool. Documented in README.
Auto mode re-introduces the "fires autonomously, narrative-less" problemAuto mode is for local smoke, not Scheduler replacement. The README explicitly says "Cloud Run service is the Scheduler-fired path; Driver is for human investigation." Auto exists for "I trust the steps, run it once locally for verification" usage.
Driver and Cloud Run write to same Firestore -- timeline gets noisysource: "driver" field already differentiates. Dashboard can filter by source. If it becomes a real problem, the design's Escape Route 6 fork to a separate collection -- straightforward backfill since the source field already exists.
Reset-baseline sub-flow accidentally fires the wipe in InteractiveEach ops step has its own preview + approval. Wipe step shows current doc count + warns "This deletes N docs from cadence-snapshots." Operator must press Enter to confirm.
SteeringWheel grows beyond ~300 linesMatch mendo-dgx's 395-line ceiling. If the wheel itself exceeds ~400 lines, split: SteeringWheel.Loop.cs + SteeringWheel.Hatches.cs + SteeringWheel.Inspector.cs.
RunResult JSON files accumulate in results/runs/Not in scope for this build, but document a cleanup convention in the README: "delete >30 days old, or move to results/archive/." Not critical -- one run = a few KB.

What this unlocks

  • Operator confidence on cadence the same way Joey already trusts mendo-dgx. When a Scheduler-fired tick returns a surprising number, fire up the Driver in Interactive, walk through the same flow, see each value as it materializes. The "makes no sense" problem from 2026-05-24 doesn't recur.
  • One place to do ops. Reset baseline, pause/resume Scheduler, wipe Firestore -- all available as Driver flows. Complements Board 10's HTTP endpoints; either path works.
  • Forensic record per investigation. results/runs/run-{ts}.json captures every step + result + StepContext slice. Diff two runs to see what changed.
  • Reproducible local smoke after every clone refresh. One dotnet run -- --auto from the operator's machine confirms the full sweep path works against a fresh clone, before re-enabling the Scheduler.
  • Pattern reuse. Future cadence ops (Top-N drift forensic, per-stream replay, schema-diff deep dive) become "add a new flow with N steps" rather than "wire a new endpoint."

Open follow-ups (not in this board)

  • Web UI for the Driver. Console-only matches mendo-dgx. A WebSocket-served Interactive flow could land in a future board if Joey wants browser access.
  • Dashboard filter by source. Once source: "driver" docs exist, the cadence /dashboard can grow a filter chip (driver / scheduler / both). Out of scope here -- Driver lands the field; dashboard work is its own board.
  • Per-stream drill-down step. Once verdicts are in StepContext, a future step could optionally run RowDeltaExplainer-style EXCEPT queries on top-DIFFER streams. Forensic mode. Probably a separate flow (--flow drift-explainer) rather than a step in the base sweep.
  • Save flows declaratively. Today the SweepFlow is a hardcoded List<IStep>. Future improvement: load step list from JSON so operators can compose custom flows without recompile. Out of scope.
