Ether: Switch to CDC

Fix row drift. Capture all UPDATEs. Cut sync volume 95%. One config change.

Switch Ether Postgres from cursor-based replication to CDC (logical replication). Eliminates header/detail row drift, captures all UPDATEs and DELETEs, reduces sync volume from 21.3M rows/hr to ~500. Two hours total including verification.

How CDC Works vs Current

CURSOR-BASED (current) CDC (proposed) ============================ ============================ Job starts Job starts -> SELECT * FROM orders -> Read WAL from last position WHERE updated_at > cursor [t+0m] -> ALL tables, ALL changes, -> SELECT * FROM order_products in transaction order WHERE updated_at > cursor [t+12m] -> SELECT * FROM sales_line_items INSERT order #12345 WHERE updated_at > cursor [t+91m] INSERT line_item for #12345 INSERT tax for #12345 Header at t+0, detail at t+91. = one atomic batch Gap = 91 min. UPDATEs invisible. Gap = 0. All changes captured.

CDC reads the Postgres write-ahead log (WAL) instead of querying tables individually. The WAL records every INSERT, UPDATE, and DELETE across all tables in transaction order. Airbyte reads from a single position — header and detail rows from the same transaction arrive together.

Implementation (2 Hours)

Postgres Side (~15 min)

1
Verify WAL level 
SHOW wal_level;
-- Must be 'logical'. If 'replica', change in config + restart.

RDS/Cloud SQL: parameter group change, auto-restart ~5 min. Self-hosted: edit postgresql.conf, restart.

2
Grant replication privilege 
ALTER USER airbyte_user REPLICATION;
3
Create replication slot + publication 
SELECT pg_create_logical_replication_slot('airbyte_ether', 'pgoutput');
CREATE PUBLICATION airbyte_ether_pub FOR ALL TABLES;
4
Set replica identity for PK-less tables 
-- Only needed for the 11 junction tables without primary keys:
ALTER TABLE table_name REPLICA IDENTITY FULL;

Tables with PKs use DEFAULT identity (no change needed).

Airbyte Side (~5 min)

5
Switch source to CDC mode

Edit the Ether Postgres source. Change replication method from Standard to Logical Replication (CDC).

{
  "replication_method": {
    "method": "CDC",
    "replication_slot": "airbyte_ether",
    "publication": "airbyte_ether_pub",
    "initial_waiting_seconds": 300
  }
}
6
Reset + re-sync

First sync = initial snapshot (~52 min, same as current full sync). Every sync after that: minutes, WAL changes only.

7
Verify

Run 2 hourly syncs. Confirm WAL-only mode, confirm header/detail arrive together, confirm reactions volume dropped from 21.3M to ~500 rows.

Background: Current Issues

Ether runs cursor-based incremental replication — each of 63 tables is queried independently with SELECT * WHERE updated_at > last_cursor. This creates three row-integrity issues, all of which exist in production today (not introduced by migration).

Issue 1: Header/Detail Cursor Drift

Airbyte reads tables sequentially. Header tables and their detail tables are separate queries at different times in the same job. Measured from Job 56 (April 12):

HeaderDetailCursor Gap
ordersorder_products2 min 8 sec
sales_orderssales_line_items_all7 min 53 sec
sales_orderssales_line_items_merchandise1 hr 31 min
sales_orderssales_line_items_bag14 min 45 sec

A new sales order's header is captured, but its line items are 91 minutes behind. Detail rows appear on the next hourly sync. This is the "missing rows" Wesley reported.

Issue 2: ID Cursor Misses UPDATEs

5 detail tables use auto-increment id as cursor instead of updated_at. Only new INSERTs (higher ID) are captured. If an existing row is UPDATED (tax recalculated, attribute changed), the sync never sees it.

TableCursorWhat's Invisible
order_taxesidTax recalculations, rate corrections
order_product_attribute_valuesidAttribute changes on existing items
orders_promotionsidPromotion modifications
cart_productsidCart item quantity changes
cart_product_attribute_valuesidCart customization changes

Issue 3: Full-Refresh Truncate Window

Tables like order_product_taxes (5.75M rows) run full_refresh/overwrite. During sync (~52 min), Snowflake truncates then reloads. Any query joining orders to order_product_taxes during this window returns incomplete results.

Issue 4: reactions Cost Bomb

The reactions table was forced to full_refresh because updated_at has NULL values. Cursor-based incremental can't handle NULLs. Result: 21.3M rows re-synced every hour to catch ~500 new ones. 77% of total Ether volume. ~100 GB/day in Snowflake writes.

CDC doesn't use cursors. It reads the WAL. Every INSERT is captured regardless of NULL column values. The NULL cursor problem becomes irrelevant.

Why This Wasn't Done Before

Production Airbyte (v0.40.28) was configured years ago with cursor-based replication — the default in Airbyte. CDC is opt-in. At 6 streams (44 seconds per sync), cursor drift was sub-second and invisible. At 63 streams (52 minutes per sync), drift grew to 91 minutes and became the "missing rows" problem.

This exists in production today. We didn't create it. We measured it. And CDC fixes it.
