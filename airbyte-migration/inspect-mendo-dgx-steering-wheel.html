What It Is

A 395-line C# class (SteeringWheel.cs) that drives a sequence of IStep objects through three possible modes:

  • Interactive — previews each step, prompts the user for action, then executes. User can approve, skip, go back, inspect the page, search for text, dump HTML, or quit.
  • Manual — shows the step's manual guide, the user does it in the browser themselves, AI records pass/fail/notes. Useful when automation is impossible (e.g. Cloudflare challenges).
  • Auto — runs through every step end-to-end without prompts. Falls back to interactive on failure if it's in interactive mode.

Structure

The IStep interface

public interface IStep { string Name { get; } // "AddItem", "Checkout", ... string Description { get; } // human-readable summary string ManualGuide { get; } // instructions if user does it manually bool RequiresManualIntervention { get; } // hint: this step may need user help Task<StepResult> Execute(IPage page, StepContext context); }

The StepResult record

Captured per step: Index, Name, Status (Pending/Passed/Failed/Skipped/ManualRequired), StartTime, DurationMs, ScreenshotBefore, ScreenshotAfter, Notes, SelectorUsed, Error, RetryCount.

The interactive loop

Stack

  • Language: C# / .NET 8 (same as cadence)
  • Driver UI: Console.WriteLine with ANSI color codes. Pure CLI, no GUI.
  • Step ordering: List<IStep> in the order defined by the Flow class (BasicOrderFlow.GetSteps())
  • State: Held in the SteeringWheel instance and a per-run StepContext passed to each step
  • Discovered selectors: Steps can record selectors they used; future steps can reuse them via StepContext.DiscoveredSelectors
  • Persistence: ResultWriter dumps the full RunResult JSON to results/runs/ when done

Patterns Found

1. Preview always precedes execution

The user sees what's about to happen before it happens. Step name, description, flag if it might need manual help. Then the prompt. This is the single most important property of the pattern.

2. Six escape hatches per step

Skip / Back / Inspect / Find / Dump / Quit. The user is never trapped. Every step is one keystroke away from "no, do something else." Inspect and Find let them explore the current state before deciding. Dump preserves the moment for post-mortem analysis.

3. Three run modes share the same step definitions

The same IStep list runs as Interactive (approve-each), Manual (operator does it, AI records), or Auto (lights-out). The mode is selected at startup; the step authors don't have to know which mode is active. This is why the pattern scales — you write the step ONCE, then choose the trust level at runtime.

4. Failure prompts a recovery action, not a hard stop

If a step fails in Interactive mode, the user sees the error and chooses Retry / Inspect / Skip / Quit. No "restart from beginning." The work already done is preserved.

5. Step results are first-class artifacts

Every step writes a StepResult with status, timing, screenshots, notes. The final RunResult is serialized as JSON and the dashboard renders it. Forensic value: every run is fully reconstructable later.

6. Manual mode is a peer of automated mode

When automation can't do something (Cloudflare challenge, captcha, manual login), the Manual mode prompts the user and records what they did. Not a failure path — a first-class mode. The step's ManualGuide property gives the user the instructions.

7. The DOM Inspector is a sibling tool, not a separate program

Pressing I mid-flow drops the user into RunInspector() which lists visible interactive elements with their selectors, lets them click any one, and returns to the main loop. The user can inspect-click-resume without losing their place.

Open Questions (for ~d to resolve)

  • Where does the steering wheel live? Option A: separate Cadence.Driver/ console app in the cadence repo, references the existing FingerprintPrimitive directly. Option B: an interactive mode on the Cloud Run service exposed via WebSocket/SSE + web UI. Option C: both.
  • Does Scheduler stay auto and Driver is for the operator? Or does the steering wheel REPLACE the Scheduler-fired tick during operator sessions? My instinct: keep Scheduler as Auto, Driver is for human-initiated investigation runs.
  • Inspector tools for cadence equivalent of DOM Inspector? Show current Snowflake session state, current Firestore count, last snapshot, Scheduler state, top DIFFER streams from last sweep. The "what's the current state of the world" call.
  • Step granularity for /tick? mendo-dgx has ~5-8 steps per flow. Cadence's tick has ~10-12 natural steps. Probably right-sized to match.
  • Step preview detail level? mendo-dgx is browser-focused so steps describe page actions. Cadence steps describe SQL queries + data movement. Need a different preview vocabulary (show SQL? show row counts? both?).
  • Recovery: retry-from-step vs full restart? mendo-dgx allows retry-from-step. Same should apply to cadence (e.g. PROD sweep failed; user retries that step; doesn't redo connect + resolve).
