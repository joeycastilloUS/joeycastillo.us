Dry-run rehearsal — control + visibility gaps

Board 22 proved the server-driven reclone works. The next rehearsal needs to be a dry run — trustable preview, mid-flight steering, live progress. Today's three gaps, named.

Inspect · Gap analysis
Operator's exact words: "I want to do a dry run rehearsal but I feel I don't have enough control or visibility to the entire process." The Board 22 rehearsal demonstrated the procedure end-to-end, but it was committed-and-blind: once POST /reset/reclone fired, the clone was dropping and there was no way to pause, abort, or see where inside the 12m 40s create-clone the work was. For a real cutover rehearsal — or a real cutover — that's not enough hand on the wheel.

Gap 1: there is no dry mode

Every POST /reset/reclone is destructive. The Driver has a dry token but it's "accepted but inert" for reclone (per Go.Reclone.bat) — it does NOT prevent the DROP. There's no way to ask "what would happen if I fired this right now?" without firing it. For rehearsal, that means every dry-run is a real-clone.

Today: POST /reset/reclone -> drops + recreates AIRBYTE_CLONE (destructive, always) Want: POST /reset/reclone?dry=true -> pretend-pause / log "would drop" / no SQL / pretend-resume full op doc written, scheduler genuinely paused+resumed for realism, ZERO Snowflake destruction

Gap 2: no mid-flight control

Once an op starts, there's no HTTP way to stop it. ResetLock blocks new ops but doesn't let you interrupt the running one. Specifically missing:

Missing controlWhy it matters for dry-run
Abort an in-flight opIf the dry-run reveals something looks wrong at step 2, you're stuck waiting for steps 3 + 4 before you can act. For a real run, you can't stop a destructive Create mid-flight.
Approval gates between stepsThe Driver does this locally (per-step prompts). Server-side has nothing -- the op runs straight through. Operator can't say "pause after Drop, let me think, then proceed".
Replay / re-fire a single failed stepIf Drop succeeds and Create fails, there's no "retry from Create"; the operator has to manually fix in Snowflake or run the full Driver flow.

Gap 3: no heartbeat visibility within long steps

The Board 22 op log told you which STEP was running but not how far into it:

02:02:48Z create-clone running # ... and then 12 minutes 40 seconds of silence ... 02:15:28Z create-clone succeeded

For 760 seconds the watcher couldn't tell: is the CREATE 10% done? 90%? Has Snowflake hung? Should I worry? Snowflake exposes query progress (rowsProduced, percent for some ops, query state) via INFORMATION_SCHEMA / GET_QUERY_OPERATOR_STATS — today we don't surface any of it. ETA = unknown. Liveness = unknown.

Where these gaps actually hurt. A real cutover rehearsal is "are we ready to do the real thing?" Without dry mode you can't rehearse without destruction. Without abort you can't stop if something surprises you. Without heartbeats you can't tell if a long step is healthy or hung. All three matter more for the REAL cutover than for any rehearsal.

What exists that's reusable

  • cadence-operations doc -- already records per-step status. Adding dryRun, requestedAbort, heartbeat fields is a schema additive.
  • ResetLock single-flight -- already prevents concurrent ops; an abort signal can live alongside it.
  • The watcher pattern -- Go.Watch already polls /operations/{opId}; new fields render with one more line per cell.
  • Snowflake query stats -- the same SnowflakeConn the reclone uses can issue a side query (SELECT SYSTEM$LAST_QUERY_ID() + INFORMATION_SCHEMA.QUERY_HISTORY) to get live progress on the in-flight Create.

What "good" looks like

A rehearsal session should be: (1) hit POST /reset/reclone?dry=true, watch the op trail render without anything actually getting dropped — preview-level confidence. (2) When ready for real, hit POST /reset/reclone with optional ?gate=after-drop so the op pauses for your explicit POST /operations/{opId}/proceed. (3) The watcher prints heartbeats every ~15s during long steps: "create-clone: running 85% (Snowflake QH_xyz, rowsProduced=...)" so you can tell healthy from hung. (4) If anything looks wrong: POST /operations/{opId}/abort halts where possible and records the partial state.

Path to design + plan

~d a design that picks the shapes for: (a) dry-run mode, (b) abort + per-step gates, (c) heartbeat surfacing. ~p phases them. All four ride the existing cadence-operations schema and /reset/reclone endpoint -- additive, no API churn.
