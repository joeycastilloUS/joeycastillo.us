Cadence.Driver — console app steering wheel for human-approved cadence sweeps

Same pattern as mendo-dgx's SteeringWheel, adapted for cadence's Snowflake-fingerprint workflow. Console app that wraps the existing FingerprintPrimitive in step-by-step preview-and-approve. Cloud Run service stays as-is for autonomous Scheduler-fired ticks. The Driver is the human-in-the-loop mode.

Design — Pure Metal

Subject + Intent

Subject: Add Cadence.Driver/ — a separate .NET 8 console app in the acmedevus/cadence repo — that runs a cadence sweep as a sequence of approved steps. Intent: Give Joey the same operator confidence he has with mendo-dgx. When the autonomous Scheduler-fired sweep returns a number that feels wrong, he runs the Driver in Interactive mode, watches each step happen, sees the intermediate data, and either trusts the result or stops mid-flow to investigate. The Cloud Run service keeps doing its autonomous work; the Driver is for human-initiated investigation runs.

Discovery

Per ~i (be/inspects/2026-05-24-2010-inspect-mendo-dgx-steering-wheel.html):

  • mendo-dgx's SteeringWheel pattern is Preview → Prompt → Act → Capture → Recover.
  • Three modes: Interactive (approve each step), Auto (lights out), Manual (operator does it, AI records).
  • Six escape hatches per step: Skip, Back, Inspect, Find, Dump, Quit. Plus failure recovery: Retry, Inspect, Skip, Quit.
  • Same IStep list runs in any of the three modes — the step author never sees the mode selection. This is the property that makes the pattern scale.
  • Step results are first-class artifacts — every run is reconstructable from the saved RunResult JSON.

What cadence has already that this design can reuse:

  • Cadence/Fingerprint/FingerprintPrimitive.cs with public static methods (LoadStreamCursors, ResolveBusinessColumnsAsync, BuildBatchedFingerprintSql, RunSweepAsync). The Driver calls into these directly — no Cloud Run round-trip needed.
  • Cadence/Snowflake/SnowflakeConn.cs + KeyHelper.cs — same Snowflake auth path the service uses.
  • The static infra files (infra/stream-catalog.json) ship with the assembly.
  • Cloud Run service exposes /tick, /health, /dashboard. None of those go away.

Shape

Layout in the cadence repo

cadence/ Cadence/ # existing ASP.NET service (unchanged) Cadence.csproj Program.cs Snowflake/ Fingerprint/ Cadence.Driver/ # NEW console app Cadence.Driver.csproj # references ..\Cadence\Cadence.csproj Program.cs # entry: mode selection, flow load SteeringWheel.cs # 200-300 lines, adapted from mendo-dgx Config.cs # RunMode enum, options Steps/ IStep.cs # interface + StepResult + StepContext ConnectStep.cs ResolveCloneTimeStep.cs ResolveStreamsStep.cs ResolveBizColsStep.cs BuildSqlStep.cs # parameterized for PROD or CLONE ExecuteSweepStep.cs # parameterized for PROD or CLONE ComputeVerdictsStep.cs WriteFirestoreStep.cs Flows/ SweepFlow.cs # the 12-step list for a full /tick equivalent Inspectors/ CadenceInspector.cs # "current state of the world" for the [I] hatch Results/ RunResult.cs # mirror of mendo-dgx RunResult ResultWriter.cs # JSON dumps to results/runs/

Step granularity for a full sweep

#StepWhat the preview shows
1ConnectStepJOEY_BE / ACCOUNTADMIN / DEVELOPER_WH; key path; auth method (JWT keypair via KeyHelper temp file)
2ResolveCloneTimeStepQuery: SHOW DATABASES LIKE 'AIRBYTE_CLONE'. After: created_on, age in minutes
3ResolveStreamsStepinfra/stream-catalog.json path; filter set = 10 active schemas. After: 208 expected, list any unexpected
4ResolveBizColsStep (PROD)One INFORMATION_SCHEMA.COLUMNS query against AIRBYTE_DB. After: total cols, biz cols (after stripping _AIRBYTE_*), per-stream summary
5BuildSqlStep (PROD)UNION ALL of 208 HASH_AGG SELECTs. After: SQL length in chars, first 3 lines
6ExecuteSweepStep (PROD)Show expected wall-clock range. After: actual wall, fingerprint count, sample fps
7BuildSqlStep (CLONE)(same as #5 against AIRBYTE_CLONE)
8ExecuteSweepStep (CLONE)(same as #6)
9ComputeVerdictsStep208 pairs to compare. After: MATCH / DIFFER / ERROR / SKIP per-schema rollup
10WriteFirestoreStepDoc path: cadence-snapshots/sweep-{ts}. Field shape. After: snapshot ID, write status

Optional sub-flows for ops:

  • SchedulerToggleStep — pause/resume cadence-tick-5m (one step, parameterized)
  • FirestoreWipeStep — batch-delete all docs in cadence-snapshots (one step, includes preview of doc count)
  • ResetBaselineFlow — compound: SchedulerToggle(pause) → FirestoreWipe → full Sweep → SchedulerToggle(resume)

Inspector tools (the [I] hatch)

  • Show current Snowflake session: CURRENT_USER, CURRENT_ROLE, CURRENT_WAREHOUSE, account params
  • Show Firestore state: collection name, doc count, latest snapshot summary
  • Show Scheduler state: ENABLED/PAUSED, next run time, last attempt status
  • Show Cloud Run service state: latest revision, ready replicas, latest log entries
  • Show stream catalog: schemas, stream count per schema, cursor field overrides

Mode selection

  • cadence-driver or cadence-driver --interactive (default) — approve each step
  • cadence-driver --auto — run end to end with no prompts (matches what Scheduler does, useful for local smoke)
  • cadence-driver --flow reset-baseline --interactive — named compound flow
  • cadence-driver --skip 1,2,3 — pre-skip steps that you know don't need approval

Choices + Consequences

1. Where does the Driver live -- separate project, separate repo, or interactive mode on the Cloud Run service
a) New Cadence.Driver/ project in cadence repo (proposed). b) Separate acmedevus/cadence-driver repo. c) Interactive mode on the Cloud Run service via WebSocket + web UI.
a) One repo, one solution, shared code references, runs locally on operator's machine. b) Decoupled but duplicate Snowflake config + auth setup. c) Accessible from anywhere but adds WebSocket plumbing complexity and identity-token auth gymnastics for interactive sessions.
PICK: (a) -- separate project, same repo.
Matches mendo-dgx convention exactly (MendoDgx.Driver alongside MendoDgx.Tests). One dotnet build covers both. The Driver can reference Cadence's public classes directly with no service-call overhead.
2. Does the Driver replace the Cloud Run /tick during operator sessions or run alongside
a) Run alongside. Driver and Cloud Run service are independent; Driver is for operator-initiated investigation. b) Driver pauses Scheduler at session start, resumes at session end. c) Driver kills the Cloud Run service for the duration.
a) Two paths writing to same Firestore -- need to differentiate (driver-initiated vs scheduler-initiated). b) Operator gets exclusive control during session but Scheduler misses ticks. c) Way too heavy.
PICK: (a) -- run alongside, tag each snapshot with its source.
Add source: "driver"|"scheduler" field to Firestore docs. Dashboard can filter or color-code. No interference between modes.
3. Inspector tools -- always available or only when explicitly enabled
a) Always available via [I] key. b) Behind a flag for performance reasons. c) Separate command (cadence-driver inspect).
a) Match mendo-dgx exactly; always one keypress away. Inspector calls are cheap (small queries). b) Adds friction; defeats the purpose. c) Loses the flow context.
PICK: (a) -- always available.
The whole point of the steering wheel is to give the operator visibility without leaving the flow. Inspector is part of the flow.
4. Failure recovery -- retry from step vs restart vs ignore-and-continue
a) Retry the failed step in place (default). b) Restart the whole flow. c) Mark failed and continue.
a) Preserves work already done; matches mendo-dgx; user can fix the underlying issue and retry. b) Wastes the Connect + Resolve work. c) Easy to corrupt downstream state.
PICK: (a) -- retry in place.
Matches the established pattern. Driver can offer all four options on failure: Retry / Inspect / Skip / Quit (same as mendo-dgx).
5. Step preview format -- summary text only vs full SQL preview vs both
a) Summary text + minimal data (Joey said "wall time, fingerprint count, sample fps"). b) Full SQL preview by default. c) Both with [V]erbose toggle.
a) Quick scan, low noise. b) 60KB of UNION ALL SQL drowns the prompt. c) Best of both but more state to manage.
PICK: (a) -- summary text. Add [D]ump to save full SQL to disk if needed.
Preview is for the GO/NO-GO decision. Forensic detail goes to file via Dump.
6. Where do snapshots from the Driver get written
a) Same Firestore collection as Cloud Run service. b) Separate cadence-driver-snapshots collection. c) Local results/runs/*.json only, no Firestore.
a) Unified timeline; dashboard shows everything. b) Cleaner separation; two timelines to view. c) Driver-only data isolated from operations record.
PICK: (a) -- same collection, with source: "driver" field.
One timeline. Operator can see "here is the autonomous tick + here is the driver-initiated investigation I did right after, they match." Dashboard filter handles the rest.

Escape Routes

Escape from PICK 1a (project in cadence repo): if the Driver grows beyond ~10 files or develops a UI of its own, split to acmedevus/cadence-driver. The project boundary is clean -- a move is a git filter-branch away.
Escape from PICK 2a (run alongside): if the dual-source Firestore writes cause confusion, add a Driver mode that auto-pauses Scheduler at session start (matches behavior C). Single flag, no code change beyond adding the pause call.
Escape from PICK 3a (always-available inspector): if inspector queries become expensive (e.g. Snowflake STATUS calls add cost), gate behind a flag. Default-on, opt-out.
Escape from PICK 5a (summary preview): if operators want SQL by default, add an env var or config flag CADENCE_DRIVER_VERBOSE_SQL=1 that flips the default.
Escape from PICK 6a (unified collection): if filtering becomes tedious in the dashboard, fork to a separate collection. The source field already exists so backfill is straightforward.

Pure Metal Test

Can this be built with no frameworks, no external runtime, no vendor lock?

.NET 8 console appreplaceable -- any console framework
Cadence.Fingerprint referencereplaceable -- could duplicate the primitive if Cadence project ever split
Snowflake.Data (via Cadence project)load-bearing -- already a Cadence dep, no new dependency
Console.WriteLine + Console.ReadLinestdlib -- works anywhere .NET runs
ANSI color codesstdlib -- not strictly required
System.Text.Json for RunResult serializationstdlib

Load-bearing count: 1. Snowflake.Data, already a Cadence dependency. The Driver introduces zero new external deps -- it's pure C# stdlib + reuses what Cadence already imports. Matches the mendo-dgx pattern (which has zero new deps over MendoDgx.Driver).

First Move

The smallest concrete proof Create Cadence.Driver/ with just three files: Cadence.Driver.csproj, Program.cs (mode selection only), and Steps/ConnectStep.cs. Wire SteeringWheel as a stub that runs one step (Connect) interactively. dotnet run --project Cadence.Driver -- --interactive prompts "Connect to JOEY_BE on DEVELOPER_WH? [Enter]=Go [Q]=Quit", runs the connect on Enter, prints "Connected as JOEY_BE / ACCOUNTADMIN / DEVELOPER_WH" or the error. If that works, the rest is mechanical -- add steps one at a time, each ~30 lines.

What this design does NOT do

  • Replace the Cloud Run service. Cloud Run keeps doing autonomous Scheduler-fired ticks.
  • Add a web UI. Console-only, same as mendo-dgx. A web UI could come later in its own design.
  • Add new Snowflake auth modes. Reuses SnowflakeConn + KeyHelper from the Cadence project.
  • Add new fingerprint logic. Reuses FingerprintPrimitive.
  • Touch airbyte-parity, airbyte_migration_tools, or any other repo.
  • Add a tilde command to Be. The Driver is a normal .NET console app; dotnet run is the invocation.
