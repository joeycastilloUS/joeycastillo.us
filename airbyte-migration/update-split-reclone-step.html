What Was Updated

Previously the Driver had a single RecloneSnowflakeStep that bundled both the DROP DATABASE and the CREATE DATABASE ... CLONE behind one preview, one approval, one StepResult. Per-step observability mattered for two reasons:

  • Forensic clarity -- if the CLONE fails after the DROP succeeds, you want a per-statement timing record showing the DROP completed cleanly. Bundled, you can't tell.
  • Distinct semantic gates -- DROP is the destructive moment. CREATE is the recovery. They deserve separate previews.

CLONE + SHOW DATABASES verify ManualGuide CREATE only + COPY GRANTS note

Why AutoAdvanceRest sits on DropClone

One destructive gate, then walk away. The Drop is the only step in the Driver where the operator definitively gives up the current state. Everything after that is the recovery + rebuild path -- there's no second destructive decision to gate on. So AutoAdvanceRest sets on Drop: confirm the destructive intent, and CreateClone + Wipe + sweep + WriteFirestore + Resume all run unattended. Operator goes to another workstation, comes back to a fresh baseline.
