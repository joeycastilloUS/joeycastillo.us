What it does

Given two Snowflake databases that both receive Airbyte loads, surface every place they diverge — per stream, per column, per row, over time — with enough resolution to act on.

Built for the Philz Coffee migration: prod data lands in AIRBYTE_DB via the legacy Airbyte instance, the new pipeline lands the same data in AIRBYTE_CLONE via airbyte-philz. The service compares both Snowflake databases and tells you exactly where they disagree, in real time.

The Three-Click Drift Drill

Four questions, three clicks each. The whole UX is built around this flow.

1 Click any drifted run row
The dashboard expands an inline L3 column profile panel — per-column row count, null count, hash, min, max for both sides. Disagreeing columns highlighted red.
2 Click "Drill to rows"
Triggers L4 (PK row-hash diff, LIMIT 1000), caches result in Firestore for 1 hour. Returns the drifted PK list with status pills: DRIFT, PROD_ONLY, CLONE_ONLY.
3 Click any drifted PK
Fires L5 surgical fetch from both sides. Side-by-side row diff with divergent fields highlighted server-side. The actual data, not abstractions.

The Five Layers

LayerQuestionMechanismCost
L1Same row count?INFORMATION_SCHEMA.TABLES metadata readfree
L2Same content?HASH_AGG over non-meta columns, optionally time-windowedcredits per scan
L3Which column drifted?Per-column COUNT, NULL, HASH_AGG, MIN, MAXcredits per scan
L4Which rows drifted?FULL OUTER JOIN on PK with row-hash, LIMIT 1000credits + 1h Firestore cache
L5Which fields disagreed?Two single-row SELECT * callsfree single-row

The classifier maps a (L1, L2, L3) tuple into one of: Clean, CountDrift, ValueDrift, StructuralDrift, Unexplained, or Missing. Unexplained means the layers disagree — that's when you click "Drill to rows" to localize.

Snapshot vs. Transactional vs. Audit

Not every stream behaves the same. Each is classified into a parity role:

RoleExampleHow it's compared
transactionalETHER.USERS, SQUARE.ORDERSStandard PK join. Rows mutate over time on a stable identity.
snapshotNETSUITE.CUSTOM_INVENTORYComposite key (ID, day-of-time-column). HASH_AGG within (id, day) collapses within-day duplicates — the 87 NetSuite items captured 3×/day produce one signal per (item, day).
auditNETSUITE.BAK_*Append-only logs — row-level comparison meaningless. L1-L3 still run; L4 and L5 deliberately skip.

Streams classified via infra/parity-roles.json. Defaults to transactional; only streams that need different treatment get an override.

Operational Tools

The same scripts that drove today's NetSuite catch-up are reusable for any other connection that falls behind:

  • scripts/audit_clone_freshness.py — per-stream prod/clone row counts + last-altered timestamp + gap_days. Read-only. Identifies stale schemas in seconds.
  • scripts/refresh_clone_from_prod.py --dry-run — previews CREATE OR REPLACE TABLE ... CLONE statements. Snowflake clone is metadata-only on prod (no read load, no credits charged on the source).
  • scripts/refresh_clone_from_prod.py --connection <name> --apply — executes the refresh. Re-baselines a stale clone in seconds. Then airbyte-philz incremental resumes.
  • scripts/_dump_stream_catalog.py — refreshes the static catalog from Airbyte API; resolves destination namespaces correctly per namespaceDefinition + namespaceFormat.

Try It Yourself

For a stakeholder skim: just open the dashboard. The status grid + drift events tell the story in 30 seconds.

For a hands-on tour:

  1. Open the live dashboard. Watch it auto-refresh. Note the live counts in the cards.
  2. Scroll to "all streams" — 208 rows. Filter chips let you narrow to drifting / snapshot / audit / etc.
  3. Scroll to "recent runs" and pick any row with non-zero drift. Click it.
  4. The row expands inline showing the L3 column profile. Note red-highlighted columns.
  5. Click "Drill to rows ▾". L4 fires; drifted PKs appear with status pills.
  6. Click any drifted PK. L5 fires; side-by-side row diff with divergent fields highlighted.

Three clicks from a drift event to "this column on this row, wrong".
