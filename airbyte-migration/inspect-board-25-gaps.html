Board 25 review — what's still missing

Honest audit of what I shipped vs what was asked. The op-log fields are right; the operator-facing experience is not.

Inspect · incomplete
Operator's exact words: "I want to see the exact sql statement being executed to snowflake before it happens. when it happens and after it happens. and I want you to get and show me the actual snowflake log."

What I delivered captured the SQL/queryId/snowflake fields on the op doc and made the watcher render them. But it doesn't show the SQL on screen before the operator confirms Stage 4, and there's no JWT/auth step in the op trail. Two real gaps.

Gap 1: the SQL is invisible before the destructive confirmation

Go.Rehearse Stage 4 says:

================================================================ STAGE 4 of 5 -- REAL RECLONE -- drops + recreates AIRBYTE_CLONE from prod ================================================================ This is the only destructive step of the rehearsal. Pause scheduler -> DROP AIRBYTE_CLONE -> CREATE FROM AIRBYTE_DB -> Resume. Takes ~12-15 minutes. Confirm by typing yes (anything else cancels). type "yes" to fire, anything else to skip and finish.
What's wrong: the operator is asked to type "yes" without ever seeing the actual SQL strings. The SQL appears in step.sql only after the op starts. By then it's too late to refuse. "Before it happens" was the explicit ask; this is "during it happens."

What it should look like:

Stage 4 will fire these statements against Snowflake (as cadence-svc): 1. DROP DATABASE IF EXISTS AIRBYTE_CLONE (~2 min) 2. CREATE DATABASE AIRBYTE_CLONE CLONE AIRBYTE_DB (~12 min) Both run under JWT auth -- the connection is established with the cadence-svc key pair (a SELECT CURRENT_USER() is fired first to prove the auth works before any destructive statement). type "yes" to fire those exact statements, anything else cancels.

Gap 2: no JWT / Snowflake-auth step in the op trail

Current op steps:

pause-scheduler drop-clone create-clone resume-scheduler

The Snowflake JWT authentication happens silently inside SnowflakeConn.OpenAsync() -- once per ExecAndCapture call. If the JWT is wrong, the FIRST destructive statement is what surfaces the failure, after pause-scheduler has already fired.

What's wrong: auth is invisible AND happens late. The operator wants the JWT step to come first: open the connection, fire a benign SELECT CURRENT_USER(), CURRENT_ROLE(), CURRENT_WAREHOUSE(), capture that query's id + Snowflake QUERY_HISTORY row, mark it succeeded. Then pause. Then drop. Then create. Then resume. If JWT is broken, the op fails at step 0 -- before pause, before anything destructive.

What the op should look like:

snowflake-auth sql: "SELECT CURRENT_USER(), CURRENT_ROLE(), CURRENT_WAREHOUSE()" queryId: 01b8c7c1-... snowflake: status=SUCCESS user=CADENCE_SVC role=... warehouse=... jwt: ok pause-scheduler ... drop-clone sql: "DROP DATABASE IF EXISTS AIRBYTE_CLONE" queryId: 01b8c7d2-... snowflake: status=SUCCESS elapsed=124310ms bytes=... rows=... create-clone sql: "CREATE DATABASE AIRBYTE_CLONE CLONE AIRBYTE_DB" queryId: 01b8c7e9-... snowflake: status=SUCCESS elapsed=760234ms bytes=... rows=... resume-scheduler ...

What IS right in the current delivery

  • step.sql is captured before SQL runs (visible on running transition in the op doc).
  • step.queryId is captured via SYSTEM$LAST_QUERY_ID() right after each statement.
  • step.snowflake pulls the official INFORMATION_SCHEMA.QUERY_HISTORY_BY_SESSION row -- executionStatus, start/end, totalElapsedMs, bytesScanned, rowsProduced, errorCode/Message.
  • History fetch is best-effort -- a logging failure can't block the destructive op.
  • NormalizeFs renders the new Timestamps cleanly through /operations.
  • Go.Watch + Go.Rehearse print the fields when they appear in the op doc.
The skeleton is correct. The two gaps are at the human-facing edges, not in the data plane.
