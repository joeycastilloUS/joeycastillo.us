Approach

  1. Prove first. Phase 1 is a standalone 40-line script that runs the proposed primitive across all 208 streams. No HTML, no integration. If >90% return MATCH, the design holds and we proceed.
  2. Extract. Phase 2 lifts the proven code into a small helper module with clean function boundaries.
  3. Integrate. Phase 3 swaps the helper into snapshot.py, replaces the counts data structure and render function.
  4. Ship. Phase 4 publishes a new Phase 2 snapshot under the new design as proof in production.
  5. Document. Phase 5 updates CLAUDE.md so the next person knows what changed.

2 Extract helper module

~30 min. Lift the proven logic into scripts/_fingerprint.py. Four functions, clean signatures.
def business_columns(cur, db, schema, table, exclude_prefixes=('_AIRBYTE_', '_FIVETRAN_')) def fingerprint(cur, db, schema, table, biz_cols, cursor_filter=None) def except_count(cur, schema, table, biz_cols, cursor_filter=None) def fingerprint_pair(cur, schema, table, clone_time, buffer_hours=24) # -> {'verdict': 'MATCH'|'DIFFER'|'ERROR'|'SKIP', # 'prod_fp': int, 'clone_fp': int, # 'biz_cols': int, 'cursor': str|None, # 'prod_only': int|None, 'clone_only': int|None}
Accept: from _fingerprint import fingerprint_pair; r = fingerprint_pair(cur, 'NETSUITE', 'CUSTOM_ITEM', clone_time) returns the expected dict shape. Unit test: same call from snapshot.py and from the proof script produce identical results.

Risks + mitigations

RiskMitigation
Phase 1 proof shows <90% MATCH -- design wrongHalt at Phase 1 accept. Re-examine: bad cursor on certain streams, table-side schema drift that wasn't anticipated, etc. Possibly add an extra metadata prefix to the exclude list. Don't proceed to Phase 2 with broken signal.
HASH_AGG slow on very large tables (ETHER.ORDERS 4B rows full-refresh, NetSuite custom_* tables)HASH_AGG with column-pruned projection is still a full scan. ETHER.ORDERS may take 30-60s alone. If total proof runtime > 10 min, add an opt-out skip list for the three biggest tables and document the gap. Most streams will be sub-second.
Index rebuild regex breaks because hero stat shape changesUpdate rebuild_site_index in Phase 3 alongside the panel render. Test on a fresh snapshot before pushing.
Non-timestamp cursors error out (7 streams today)Under the new design those tables get fingerprinted without a cursor filter (whole-table compare). They promote from "error" to MATCH or DIFFER. Should improve, not regress.
Stream catalog out of date (new table on prod, not in catalog)Same risk as today; the snapshot tool iterates over what the catalog lists. Could add a sanity check "tables on prod not in catalog" in a later board if it matters; not in scope here.

What this unlocks

  • One-glance parity verdict per table on every snapshot, not a noisy aggregate row delta.
  • Real signal: any DIFFER becomes investigable instead of getting drowned in _AIRBYTE_EMITTED_AT churn.
  • Phase 3 (V2 discovery) bracketing: snapshot before V2 DDL, run DDL, snapshot after. Fingerprints of affected tables should change in predictable ways (new columns → new fingerprint). Diff is interpretable.
  • Phase 4 (cron resume) tracking: as new syncs land, fingerprints will change in business columns too -- this gives us a sharper read on "did the new sync land what we expected" than freshness gaps alone.
  • Cheap to invoke. Snapshot pass remains under 3 minutes; new panel adds maybe 1-2 minutes for 208 streams. Worth it for the signal jump.

Decisions inherited from the design (not re-litigated here)

  • Name: fingerprint. Panel called Fingerprint.
  • Cursor filter default: on (when cursor exists). Buffer: 24h before clone_time.
  • Magnitude qualifier: EXCEPT COUNT only on DIFFER.
  • Column pruning: Python-side prefix check on INFORMATION_SCHEMA result.
  • Re-clone: no.
  • Counts panel: replaced, not added alongside.
