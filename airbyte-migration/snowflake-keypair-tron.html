// Trace contract

TRON is only safe with full traceability. Every action AI takes during the chain emits one structured event to a single append-only sink. The contract:

Sink

where every event lands

Append-only NDJSON file at be/otel.jsonl. Each line a single JSON object. Pre-existing in airbyte-parity (used by Be telemetry). Same file, additional events with "channel": "tron".

Event shape

required fields per event
{ "channel": "tron", "phase": "B", "step": 6, "action": "gcloud_secrets_create", "args_redacted": { "secret_name": "acmedev-tools-svc-private-key", "project": "acmedev-platform-hq", "data_file": "<file>:hash=sha256:a3f9...", "passphrase": "<redacted>" }, "actor": "claude-opus-4-7", "started_ts": "2026-05-08T03:14:22.117Z", "ended_ts": "2026-05-08T03:14:23.842Z", "duration_ms": 1725, "result": "ok", "diff": { "before": "secret_does_not_exist", "after": "secret_exists_v1" }, "next_action_planned": "gcloud_secrets_add_iam_policy_binding" }

Secret values are never in the trace -- only sha256 hash + redaction marker.

Action coverage

what must emit an event
  • Every gcloud invocation (secrets, run services, builds, IAM)
  • Every snowsql / Snowflake REST call
  • Every filesystem write (key file, .env, source code edit)
  • Every git operation (add, commit, push)
  • Every dotnet build/test invocation
  • Every HTTP request to a Cloud Run service URL (smoke tests)
  • Every read-only verification check (BEFORE the next action that depends on it)

If AI cannot emit the event for an action, AI must not perform the action.

Audit query

how a human reads it after-the-fact
jq -c 'select(.channel == "tron")' be/otel.jsonl | jq -s ' group_by(.phase) | map({ phase: .[0].phase, actions: length, duration_total_s: (map(.duration_ms) | add) / 1000, failures: map(select(.result != "ok")) | length, last_action: (sort_by(.ended_ts) | last | .action) })'

One command, one summary per phase, one line of context per action. If the summary shows failures > 0 in any phase, audit drills into raw events.

// Pre-conditions

Every gate below must be green before TRON arms. AI verifies each one and refuses to enter TRON if any fails.

1   Procedure proven arming

  • Required: the manual or AI-assisted playbook has run end-to-end at least once on a non-production target (e.g., a test Snowflake user, or a sandbox Cloud Run project).
  • Verify: be/otel.jsonl contains a prior trace marked "channel": "manual" or "channel": "assisted" with all 10 phases result: "ok".
  • If unsatisfied: run AI-assisted mode first. Refuse TRON.

2   Sink writable + tail visible arming

  • Required: be/otel.jsonl exists, is append-only, and a separate terminal can tail -f it.
  • Verify: AI writes a test event with "action": "tron_preflight" and confirms it appears in the tail.
  • If unsatisfied: create the file + ensure tail is open. Refuse TRON.

3   Kill switch armed arming

  • Required: file ~/.tron-kill does not exist before start; creating it during execution causes AI to halt cleanly after the current phase. AI checks the file's existence between every phase.
  • Verify: AI starts a 30-second dry-run of the kill check (creating + reading + deleting ~/.tron-kill) and emits a trace event for each.
  • If unsatisfied: dry-run failed. Refuse TRON.

4   Wesley async-authorized arming

  • Required: Wesley has approved the master plan AND has pre-authorized TRON to run his side of the chain unattended (Phase B SQL block, Phase E spot check, Phase F spot check, Phase G spot check, Phase I cleanup).
  • Verify: a signed approval message in Slack OR a token file at ~/.tron-wesley-ok with timestamp ≤ 24h old.
  • If unsatisfied: route to AI-assisted mode where each Wesley step is human-mediated. Refuse TRON.

5   Resource bounds set arming

  • Required: max-action-count, max-wall-clock, max-cost. AI declares these in the first trace event of the run.
  • Defaults: 200 actions, 4 hours, 0 incremental Cloud Run revs above the plan's 6.
  • If exceeded mid-run: AI halts and emits "result": "halt_resource_bound". No new actions until human reviews.

6   Rollback windows registered arming

  • Required: for each phase that introduces persistent state (B, C, E, F, G, I), AI emits a "rollback registered" event before the phase begins, listing the exact reverse-command set.
  • Effect: if any phase fails or kill-switch fires, AI runs the registered rollback for the most recent persistent-state phase and halts.
  • If unsatisfied: rollback registration failed. Refuse TRON.

// Activation

Once all 6 pre-conditions are green, the operator types one command:

~tron snowflake-keypair-migration

AI replies with a final pre-flight summary: trace sink confirmed, kill switch armed, Wesley authorized, resource bounds, rollback windows registered, expected wall-clock, expected total events. Operator types arm. AI begins.

From this point forward, no further human input is required for normal completion. The operator may close the laptop, walk away, sleep. The chain runs.

// Kill switch

Halt the chain at any time:
touch ~/.tron-kill

AI checks for this file between every phase boundary. On detection, AI:

  1. Emits a "channel": "tron", "action": "kill_detected" event
  2. Completes any partial action that's already in-flight (does not abort mid-write)
  3. Runs the registered rollback for the most recent persistent-state phase
  4. Emits a final "action": "tron_halted", "result": "killed_by_operator" event
  5. Stops

Maximum halt latency: the duration of one in-flight action (typically 1-30 seconds; never more than 2 minutes for a Cloud Run deploy).

To resume: delete ~/.tron-kill and re-arm with ~tron snowflake-keypair-migration --resume. AI starts from the next phase after the last completed one.

// Post-hoc audit

After completion (success or halt), the operator reads:

tail -200 be/otel.jsonl | jq -c 'select(.channel == "tron")' \ | jq -s 'group_by(.phase) | map({ phase: .[0].phase, actions: length, duration_s: (map(.duration_ms) | add / 1000), failures: map(select(.result != "ok")) | length, first_action_at: (sort_by(.started_ts) | first | .started_ts), last_action_at: (sort_by(.ended_ts) | last | .ended_ts) })'

One row per phase. Failures > 0 anywhere = drill into raw events:

jq -c 'select(.channel == "tron" and .phase == "F" and .result != "ok")' be/otel.jsonl

For audit-grade verification:

  • Cross-check Snowflake QUERY_HISTORY: every query AI ran via snowsql appears in QUERY_HISTORY with matching session timestamp + USER_NAME + ROLE_NAME.
  • Cross-check Cloud Run revision history: every gcloud run services update in the trace corresponds to one revision bump in gcloud run revisions list.
  • Cross-check git: every commit in the trace is in git log with the expected hash + message.
  • Cross-check GCP secrets: every secrets create event has a corresponding entry in gcloud secrets list (or, if Phase I ran, a corresponding delete event 24h+ later).

If any cross-check disagrees with the trace, the trace is the canonical record -- the discrepancy is logged as an audit finding for human review. The trace cannot be retroactively edited (append-only file with restrictive permissions).

// What TRON never does

  • Run without all 6 pre-conditions green. Refuses to arm.
  • Skip a registered rollback when halting after persistent state was created.
  • Write a secret value into the trace. Hashes + redaction markers only.
  • Continue past a phase where verification failed.
  • Re-attempt a halted action automatically (idempotent retries during the same in-flight action are fine; cross-phase retry requires --resume from a human).
  • Force-push, drop a Snowflake user, delete a GCP secret, or remove a Cloud Run env var that wasn't on the registered action list.
  • Send messages to Wesley or any human (the trace is the message).
