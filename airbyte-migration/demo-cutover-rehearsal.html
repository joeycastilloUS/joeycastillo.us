Cutover Rehearsal

The cutover procedure, run live against the current frozen-clone state with the full operator toolset. Read-only + disposition steps were exercised for real (then restored); the destructive + external steps are documented and gated.

What this rehearsal proved The loop works end to end on live data: see it (Go.Status / Go.Replay) → triage (Go.Signal) → dig (Go.Escalate → verified /parity/l3) → decide on the record (Go.Risk) → gate (green only at 0 un-dispositioned + 0 blockers). Two live signals showed two different truths, and the tooling told them apart.

The procedure

0 Pre-flight

Go.Status # scheduler Enabled? latest sweep? verdict counts?
Confirm cadence is ticking (5-min) and you're reading a fresh sweep. Rehearsal saw Enabled, clean=113 benign=93 signal=2.

1 See the timeline

Go.Replay # the cascade movie -- how drift evolved (356 ticks) Go.Replay <stream> # one stream's verdict lane + flips Go.Replay at <time> # freeze-frame at any past tick
Orient before you act: what's been drifting, since when, and whether it's trending worse or holding.

2 Triage the worklist

Go.Signal # ONLY genuine signals + the cutover gate line
Everything Clean/Cosmetic/Benign is suppressed. Each signal is tagged [needs triage] or [BLOCKER].

3 Dig each signal

Go.Escalate <stream> # verified /parity/l3: which columns drifted + row delta
This is the honest deep-diff (Board 20). It tells lag apart from divergence. Read the row delta sign: clone behind (negative) leans lag; clone with extra/changed rows leans divergence.

4 Decide, on the record

Go.Risk <stream> explain "<why it's noise>" # lag / expected -> Benign Go.Risk <stream> accept "<why we live with it>" # real, accepted -> Benign Go.Risk <stream> block "<why it must be fixed>" # real, blocker -> stays RED Go.Risk # review the register
Every disposition needs a comment. Loop steps 3-4 until every signal is dispositioned.

5 Feed the clone (external + destructive)

Go.Reclone # refresh AIRBYTE_CLONE from prod (DESTRUCTIVE: drop + recreate) # -- OR -- point airbyte-philz (new Airbyte) at the clone and let it catch up
Gated. Go.Reclone drops and recreates AIRBYTE_CLONE. The real cutover feeds the clone from new Airbyte (airbyte-philz) — external infra. NOT fired in this rehearsal. When fired, stale-lag signals (like OFFLINE_STORES) collapse to clean; genuine divergences (like SPREE) either resolve or stay as real blockers.

6 Confirm green

Go.Signal # cutover gate: GREEN -- 0 un-dispositioned, 0 blockers Go.Replay # the recovery reel -- signals collapsing back to clean
Green is honest: every red is fixed, explained, or accepted with a comment. Replay the recovery as proof on film.

7 Cut over (external)

Switch consumers from AIRBYTE_DB to AIRBYTE_CLONE (promote / repoint). Outside cadence. Keep cadence running after — Go.Signal should stay clean.

Rehearsal disposition cycle (exercised, then restored)

To prove step 4 + the gate end to end, both signals were dispositioned live and the gate observed, then cleared:

Left clean. The dispositions above were a rehearsal — both were cleared. SPREE_CMS_SECTIONS and OFFLINE_STORES are un-dispositioned now, so the real calls are yours at real cutover time.

Where the gate stands now

RED — 2 un-dispositioned, 0 blockers. To reach green at real cutover: feed the clone (step 5), which should collapse the stale-lag signals; then disposition or fix whatever genuinely remains. The procedure above is the path.

Gaps noticed during the rehearsal

  • Go.Replay <stream> lanes collide on one temp file — opening two stream lanes back-to-back, the second overwrites the first before the browser reads it. View lanes one at a time, or a future tweak gives each stream its own temp file.
  • No single "rehearse" command yet — this was a hand-run sequence. Could be wrapped as Go.Cutover (read-only walkthrough) if rehearsing becomes routine.
