Cadence.Driver CLI — human-friendly verb surface

Today's CLI is a tangle of flags inherited from the build-up phases. Operator just called them shit. Re-shape around verb launchers that match the existing Go.X.bat family, drop the negative flags, collapse three competing "trust mode" flags into one explicit token.

Design — Pure Metal

Subject + Intent

Subject: Replace the current Go.Driver --flow X --skip-write --unattended -i surface with verb-first launchers (Go.Sweep, Go.Reclone, Go.Rebaseline, Go.Status, Go.Diff, etc.) so the operator types verbs, not switches. Eliminate negative flags (--skip-write); collapse the three trust-mode flags (-i, -a, -u) into one positional token. Intent: The Driver should feel like Go.Sign, Go.Be, Go.Status — verbs you remember, no man page required.

Discovery

Today's surface, dumped from Go.Driver --help:

-i, --interactive approve each step (default) -a, --auto run end-to-end without prompts (failures pause) -m, --manual operator runs each step by hand, AI records P/F/N -u, --unattended activate AutoAdvanceRest (after Drop, rest auto-runs) --skip-write omit WriteFirestoreStep (no doc written) -f, --flow <name> sweep (default) | reclone | reset-baseline -h, --help this help

And the actual invocations the operator has had to type this session:

Go.Driver --interactive --flow reset-baseline Go.Driver --auto --skip-write --flow sweep Go.Driver --interactive --flow reclone Go.Driver --interactive --flow reset-baseline --unattended Go.Driver --flow sweep --skip-write

Pain points found by inspection:

  • Verbs hidden behind a generic flag. The verb of the command is reset-baseline or reclone, but the operator types --flow X. The flag adds zero meaning — it's pure boilerplate before the actual verb.
  • Three competing flags for one axis. -i, -a, -u, -m all sit on the "how much do I babysit" axis. Four flags, but the values are mutually exclusive. One axis, one slot.
  • Negative flag. --skip-write reads as "don't do X" — the operator has to mentally invert. Worse, the flag is named after the implementation (WriteFirestoreStep) not the operator intent ("don't pollute the timeline").
  • Flag combos that should be one-liners. The most common forensic command is Go.Driver --interactive --flow sweep --skip-write. Five tokens for "dry sweep, ask me." Should be one or two.
  • Inconsistent style across the launcher family. Go.Status takes a positional integer. Go.Diff takes -List and positional paths. Go.Driver takes --flag + value. Three different conventions.
  • No discoverability via tab completion. Operator types Go., hits Tab — sees Go.Driver, Go.Status, Go.Diff. The verbs the Driver can actually do (sweep, reclone, rebaseline) aren't there. They're hidden inside the --flow flag's value space.

Naming Pass

Five candidates for the shape of the new surface:

Go.Driver --flow X (current)keep flags · ergonomic for scripts, awful for humans · what we have now
Go.Driver sweep dry asksubcommand + positional · git-style · one launcher, many verbs · flat but reads OK
Go.Sweep, Go.Reclone, Go.Rebaseline, ...winner · matches Go.Status / Go.Diff / Go.Sign family · tab-discoverable · one verb = one launcher
Go.Driver ~s, ~r, ~btilde-mirror · namespace collision with Be tildes (~s = ship, ~r = retro) · rejected
Go do=sweep mode=dry trust=askproperty-bag · cute · nobody types like this · rejected

The winning surface (per-verb launchers) maps to the existing Go.X.bat family Joey already has muscle memory for. Go.Sign, Go.Be, Go.Status, Go.Diff are all already in this convention. Adding Go.Sweep, Go.Reclone, Go.Rebaseline is a continuation, not an invention. Each launcher is a thin .bat wrapper around cadence-driver.exe.

Modifier tokens (positional, after the verb)

Each launcher accepts up to two positional modifier tokens. No --flags in the common cases.

TokenEffectDefault when absent
askPrompt before each step (current --interactive).default
goNo prompts, run end-to-end. Failures still pause (matches mendo-dgx). Equivalent to current --auto.off
walkHybrid: prompt up to and including the load-bearing gate step (e.g. DropClone in Rebaseline), then unattended. Equivalent to current --unattended.off
manualOperator does each step by hand, Driver records P/F/N. Equivalent to current --manual.off
dryDon't write to Firestore. Replaces --skip-write with positive name.off (writes are live by default)
liveExplicit live mode -- does nothing today since live is default; reserved for future when default flips.n/a

Rule: tokens are positional, order-independent, lowercase. Operator types verbs, not flags.

Plus a top-level discovery: typing Go. + Tab in PowerShell already lists all .bat launchers in C:\cadence. That becomes the man page.

Choices + Consequences

1. One launcher with subcommands vs per-verb launchers
a) Per-verb launchers (Go.Sweep, Go.Reclone, ...). b) One Go.Driver with subcommand: Go.Driver sweep. c) Both: per-verb wrappers + Go.Driver low-level.
a) Maximum discoverability via tab completion; one .bat per verb is light. b) One launcher, less file noise, but loses tab discoverability. c) Best of both with mild redundancy.
PICK: (c) -- per-verb launchers PLUS Go.Driver retained.
Go.X family is already the muscle memory; new verbs match. Go.Driver stays as escape hatch for unusual flags or scripts. Per-verb launchers are thin shells that just shell out to Go.Driver with the right flags filled in -- one source of truth, multiple ergonomic faces.
2. Positional tokens vs flags for modifiers
a) Positional bare words (Go.Sweep dry go). b) Flags (Go.Sweep --dry --go). c) Single mode token (Go.Sweep ask-dry).
a) Reads like English; no flag prefix overhead. b) Standard but verbose for common cases. c) Combinatorial explosion of mode names.
PICK: (a) -- positional words, lowercase, order-independent.
"Go.Sweep dry" is one less character class to type than "Go.Sweep --dry" and reads cleaner. Order independence (dry go == go dry) removes a thing to remember. Validation rejects unknown words with a clear list of what was expected.
3. Negative flag (--skip-write) vs positive (dry)
a) Rename --skip-write to dry (positive token). b) Invert default (writes off by default) + add live token. c) Keep --skip-write (status quo).
a) Positive name; same default behavior (writes live). b) Safe default but autonomous cadence + Driver default split confuses ops. c) Operator complains every time.
PICK: (a) -- positive name, default unchanged.
"dry" reads cleaner than "skip-write" and isn't named after the implementation. Default stays live so the Driver and the autonomous Cloud Run tick produce the same shape of data. Operator opts into dry runs explicitly for forensic captures (which is what they were already doing). No behavior change, just a name change.
4. Trust mode -- collapse -i / -a / -u / -m
a) Single positional token (ask | go | walk | manual). b) Keep 4 flags. c) Pick one default + drop the rest.
a) One axis, one slot, mutually exclusive enforced. b) Operator picks the right flag from a menu. c) Loses real flexibility (manual mode is rare but useful).
PICK: (a) -- single token.
The 4 current flags ARE one axis. The new surface enforces that. "ask" is the safest default (prompts), "go" is "trust me, run it", "walk" is the hybrid (prompt up to gate, then unattended), "manual" is the forensic outlier. No combinations allowed (the wheel ignores any conflicting second mode token).
5. Where do the per-verb launchers live -- C:\cadence or PATH-installed
a) C:\cadence\*.bat (current Go.Status, Go.Diff pattern). b) %USERPROFILE%\bin\*.bat with PATH entry. c) Both.
a) Operator must cd to C:\cadence (or have it on PATH). b) Type from anywhere; one-time PATH setup. c) Mild duplication.
PICK: (a) -- C:\cadence, no PATH change.
Joey already runs Go.Driver / Go.Status / Go.Diff from C:\cadence; new launchers join the same family. PATH installation can be a future enhancement (single line in a Go.Be.Setup or similar). Don't add PATH ceremony in this design.
6. What about Go.Pause / Go.Resume / Go.Wipe -- ship them now or defer
a) Ship the verb set Joey actively uses (Sweep, Reclone, Rebaseline). Defer Pause/Resume/Wipe to follow-up. b) Ship the full set. c) Defer everything; just rename existing.
a) Smallest scope; covers the painful cases. b) Complete but more files to ship. c) Doesn't address the actual complaint.
PICK: (a) -- ship Sweep/Reclone/Rebaseline first.
Those three are the verbs that map to multi-step flows where the flag-flag-flag pattern was worst. Pause/Resume are single HTTP calls; the operator has them via /reset/pause already. Wipe is destructive and rare; should stay behind the Driver. Ship the three that fix the noise.

Escape Routes

Escape from PICK 1 (per-verb + retained Go.Driver): if the launcher count grows beyond ~6-7, consolidate. The Go.Driver subcommand form (b) can absorb them later. Each per-verb launcher is ~10 lines of .bat; deleting one is trivial.
Escape from PICK 2 (positional tokens): if operators want to script the Driver and prefer explicit flags, the underlying cadence-driver.exe still accepts --interactive / --auto / --skip-write. Per-verb launchers translate positional words to flags; scripts can bypass and use Go.Driver directly.
Escape from PICK 3 (dry default unchanged): if "dry should be default" becomes obvious in practice, flip the default and have the existing live launchers default to dry. One config switch.
Escape from PICK 4 (single mode token): if a true combination needs to exist (e.g. "go + per-step delay"), add a separate positional token rather than re-introducing two trust-mode tokens at once. Keeps the mutual exclusion intact.
Escape from PICK 5 (C:\cadence only): a future Go.Be.Setup launcher can add C:\cadence to PATH on one command. Out of scope for this design.
Escape from PICK 6 (ship 3 not 6): if Pause/Resume/Wipe earn their place, add them in a follow-up. Each one is a few lines of .bat.

Pure Metal Test

Can this be built with no frameworks, no external runtime, no vendor lock?

.bat launchersreplaceable -- pure Windows batch, no deps
Argument parsing inside .battrivial -- positional %1 %2 %3 with case-folding
Underlying cadence-driver.exe (already exists)load-bearing -- still does the actual work
PowerShell wrappers for Go.Status / Go.Diffalready in use; not new dep
Token-to-flag translation~5 lines of batch logic per launcher

Load-bearing count: 1. The Driver itself is the underlying tool; launchers are thin shells. Each launcher: 10-15 lines of .bat. Total new code: ~80 lines across 3-4 files.

First Move

The smallest concrete proof Ship Go.Sweep.bat alone. ~15 lines: parse positional words (ask|go|walk|manual, dry|live), translate to --interactive/--auto/--unattended/--manual + --skip-write, invoke Go.Driver --flow sweep .... Test all four common invocations: Go.Sweep, Go.Sweep dry, Go.Sweep go, Go.Sweep go dry. If the translation pattern works for Sweep, the same skeleton clones to Reclone and Rebaseline. No code in cadence-driver.exe changes for the First Move.

What this design does NOT do

  • Modify cadence-driver.exe's underlying flag parser. The .NET CLI surface stays as-is; launchers translate positional words to existing flags. (Optional future polish.)
  • Add PATH installation. Launchers live in C:\cadence; operator runs them from there or invokes via full path.
  • Change defaults (writes are still live, prompts are still default). Just renames and restructures.
  • Ship Pause/Resume/Wipe verbs in this round. Those are easy follow-ons if wanted.
  • Add tab-completion meta-data. PowerShell discovery by typing Go. + Tab already works for .bat files.
  • Touch Go.Status or Go.Diff -- they're already in the right shape.

Path to plan + build

~p formalizes this into 3-4 phases (Go.Sweep First Move; Go.Reclone + Go.Rebaseline; help-output unification; optional cadence-driver.exe flag deprecation). ~e sizes into one board (probably 9-12 pts since each launcher is small). ~b chain. No Cloud Build cycles -- pure local launcher work.
