~f Cadence.Driver uses gcloud user creds, not ADC

First run of Go.Driver --flow reset-baseline died at Step 1 with an opaque gRPC invalid_grant. Root cause: Application Default Credentials were stale. Fix: bypass ADC entirely; have the Driver inherit the operator's regular gcloud auth login identity via a token subprocess.

Shipped -- Re-run Cleared

What Was Broken

On the operator's first interactive run, Step 1 (SchedulerToggle Pause) failed:

== Step 1 SchedulerToggle(Pause) Pause Cloud Scheduler job (no more autonomous /tick fires until Resume) Will PAUSE acmedev-platform-hq/us-central1/cadence-tick-5m [Enter]=Go [FAIL] Status(StatusCode="Internal", Detail="Error starting gRPC call. TokenResponseException: Error:"invalid_grant", Description:"Bad Request", ...") [R]=Retry [I]=Inspect [S]=Skip [Q]=Quit:

Neither retry nor inspect would have helped -- the credential was unusable.

Root Cause

Both Google Cloud SDKs used in the Driver -- CloudSchedulerClient (Board 11 ops sub-flows) and FirestoreDb (WriteFirestoreStep + FirestoreWipeStep + Inspector) -- defaulted to Application Default Credentials when no Credential was supplied to their builders / static factories.

ADC is a separate credential store from regular gcloud auth login. It lives at %APPDATA%\gcloud\application_default_credentials.json (Windows) and is populated by gcloud auth application-default login, a completely different browser flow. The operator had run regular gcloud auth login (active account: joey.castillo.acmedev@gmail.com) but had not run the application-default variant. Either ADC was never set up on this machine, or an old token had its refresh grant revoked.

The invalid_grant error bubbled up through gRPC as a generic "Internal" status, hiding the actual diagnosis.

Design call: the operator was already authenticated via the regular gcloud account that owns acmedev-platform-hq. Requiring a second login dance solely because the SDK defaults to ADC is needless friction. The Driver should inherit whatever gcloud is configured to use.

What Was Fixed

New helper class shells out to gcloud auth print-access-token and wraps the result in a GoogleCredential the SDKs accept:

// cadence/Cadence.Driver/Auth/GcloudAuth.cs public static class GcloudAuth { public static GoogleCredential GetCredential(string? account = null) { var token = PrintAccessToken(account); return GoogleCredential.FromAccessToken(token); } private static string PrintAccessToken(string? account) { var fileName = OperatingSystem.IsWindows() ? "gcloud.cmd" : "gcloud"; var args = new List<string> { "auth", "print-access-token" }; if (!string.IsNullOrEmpty(account)) args.Add($"--account={account}"); // ... process start, read stdout, surface stderr on failure ... return token.Trim(); } }

Every GCP client construction now uses the builder + Credential pattern:

FileBeforeAfter
WriteFirestoreStep.csFirestoreDb.Create(proj)new FirestoreDbBuilder { ProjectId=proj, Credential=GcloudAuth.GetCredential(...) }.Build()
FirestoreWipeStep.cs (preview + execute)FirestoreDb.Create(proj)same builder pattern
SchedulerToggleStep.csawait CloudSchedulerClient.CreateAsync()await new CloudSchedulerClientBuilder { Credential=GcloudAuth.GetCredential(...) }.BuildAsync()
CadenceInspector.cs (Firestore + Scheduler options)same default-ADC callsboth replaced with builder + Credential

Inspector hint strings updated from gcloud auth application-default login to gcloud auth login -- Driver uses gcloud user creds, NOT ADC so the diagnostic matches reality if ever the active account is wrong or revoked.

appsettings.json gained Cadence:GcloudAccount (empty by default = use active account). Set to a specific email to pin: useful if Joey ever has multiple gcloud accounts active and needs to force one identity.
