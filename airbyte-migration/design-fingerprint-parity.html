Be · airbyte-parity ~design: Fingerprint parity

Fingerprint -- noise-free parity check via column-pruned HASH_AGG

One scalar per table, projected past Airbyte's metadata columns, optionally cursor-windowed. Same scalar on prod and clone = identical business state. No Time Travel needed.

Design — Pure Metal

Subject + Intent

Subject: Replace the snapshot tool's counts panel with a parity check that survives Airbyte's metadata churn, optionally scopes to the settled past, and gives a one-glance verdict per table. Intent: A diff that's right by default. Column-pruned HASH_AGG as the primitive, cursor-windowed scoping as the modifier, EXCEPT COUNT(*) as the magnitude qualifier when the verdict comes back DIFFER. Existing clone stands; Time Travel is dead on this account; this is the highest-signal cheapest path that exists.

Discovery

Per ~i (be/inspects/2026-05-23-2358-inspect-snowflake-parity-features.html):

  • Account: ZN77432 AWS us-west-2, Standard edition (1-day retention cap inferred).
  • Time Travel reach: under 24 hours in practice; clone is 3 days old; can't reach back to clone_time. Time Travel is off the table.
  • HASH_AGG(col, col, ...) works perfectly: prod and clone return identical scalars when we project past _AIRBYTE_*/_FIVETRAN_* metadata. Tested on NETSUITE.CUSTOM_ITEM: 8393363139480235559 on both sides.
  • EXCEPT on the same projection returns 0 rows in both directions for the same table -- confirms HASH_AGG isn't lying.
  • INFORMATION_SCHEMA.COLUMNS gives us the column list for each table; the metadata filter is a name-prefix exclusion. No manual maintenance.
  • The clone we have is already a valid prod snapshot. Re-cloning gives a fresher anchor but unlocks no new capability since Time Travel can't extend retroactively.

User leaning: filter out at least the last 24h before clone time, so in-flight UPSERTs don't pollute the diff. That composes cleanly with column-pruning -- two filters applied to the same projection.

Naming Pass

parity_checkprecise · generic · could be anything
fingerprint_difftechnically accurate · the act · two words
verdictshort · binary · presumes the result, not the method
parity_sealsealed = match · metal-adjacent · new term to learn
fingerprintwinner · one noun · describes the primitive AND the panel · same word Snowflake uses internally

A HASH_AGG output is a fingerprint of a multiset. The act is "fingerprint a table"; the verdict is "fingerprints match or they don't"; the panel is "fingerprints". One word, no metaphor stretching, no fork in vocabulary. The cursor-windowed mode and the column-pruning are how we fingerprint, not separate names. The panel replaces the current "Counts" panel and is called Fingerprint.

Shape

The primitive

SELECT HASH_AGG({biz_col_list}) AS fp FROM {db}."{schema}"."{table}" [WHERE {cursor} <= TO_TIMESTAMP('{line_ts}')] -- optional, applied if stream has cursor

Run on PROD and CLONE side. Compare scalars. Equal ⇒ MATCH. Unequal ⇒ DIFFER (and run EXCEPT for magnitude).

The column list

SELECT COLUMN_NAME FROM {db}.INFORMATION_SCHEMA.COLUMNS WHERE TABLE_SCHEMA = '{schema}' AND TABLE_NAME = '{table}' ORDER BY ORDINAL_POSITION; # python: [c for c in cols if not c.startswith('_AIRBYTE') and not c.startswith('_FIVETRAN')]

Pull from PROD side only (treat as authoritative schema). Strip metadata. Quote each column. Result: ordered projection that survives schema evolution as long as Airbyte sticks to its prefix convention.

The magnitude qualifier (only when DIFFER)

SELECT COUNT(*) FROM ( SELECT {biz_col_list} FROM AIRBYTE_DB."{schema}"."{table}" [WHERE {cursor} <= TO_TIMESTAMP('{line_ts}')] EXCEPT SELECT {biz_col_list} FROM AIRBYTE_CLONE."{schema}"."{table}" [WHERE {cursor} <= TO_TIMESTAMP('{line_ts}')] )

Only invoked when fingerprints differ. Runs twice (prod-clone and clone-prod) for symmetric "rows on one side only" counts. Slow on big tables; that's why it's not the primary check.

The panel rendering

Per-schema rollup + per-table verdict table. Headline stats:

  • MATCH count (green, large): how many tables passed
  • DIFFER count (red, large): how many failed
  • SKIP count (gray): no cursor + opted out, or column-list mismatch
  • ERROR count (orange): query fail

Per-table row when DIFFER: prod fp, clone fp, prod-only rows, clone-only rows, cursor used (or 'all-time'). Per-table row when MATCH: just verdict + cursor used (or 'all-time'). Hero stat: "X / Y tables MATCH at cursor <= line".

Filter scoping decision tree

  1. If table has cursor field (from stream-catalog.json or parity-roles.json): apply WHERE cursor <= clone_time - 24h.
  2. If table is snapshot mode (no cursor): apply no filter; fingerprint the whole table. Snapshot streams replace on every sync, so business rows should match clone's last pre-clone-time load.
  3. If cursor is non-timestamp (bigint ID, string): apply no filter; fingerprint whole table. Mark "untrimmed" in the verdict row so the human knows.

Choices + Consequences

1. HASH_AGG vs HASH_AGG + EXCEPT magnitude
a) Fingerprint only (MATCH/DIFFER, no count). b) Fingerprint then EXCEPT only when DIFFER. c) Always run both.
a) Cheapest. Can't answer "how many rows differ". User has to drill in manually. b) Best of both -- cheap default, expensive only when needed, max signal. c) Wasteful on MATCH cases (most tables, hopefully).
PICK: (b) -- run EXCEPT only when fingerprints differ.
Most streams should MATCH after column-pruning + cursor-windowing. EXCEPT is expensive (full-table set diff); only worth it when we already know there's something to count.
2. Cursor-window filter: always, never, or per-stream
a) Always apply the 24h-before-clone filter when a cursor exists. b) Never apply -- compare all-time. c) Per-stream opt-in.
a) Default protection from in-flight UPSERT noise; some streams may not need it but the cost is minimal. b) Catches structural divergence anywhere in history; UPSERT noise comes back. c) Adds config surface; another knob to misconfigure.
PICK: (a) -- apply when cursor exists, skip when it doesn't.
Joey's stated preference, and the most defensible default. UPSERT noise is real and recurring; users can re-run with --all-time flag if they want.
3. Column-pruning: prefix regex vs explicit list vs in-DB function
a) Python-side prefix check on INFORMATION_SCHEMA.COLUMNS results. b) Maintain explicit per-table whitelist in infra/business-columns.json. c) SQL-side filter via WHERE COLUMN_NAME NOT LIKE '_AIRBYTE%' ESCAPE... (tested, escape syntax fragile on Snowflake).
a) Zero maintenance; self-updating; same logic the C# parity engine uses. b) Defensive against unexpected metadata column names but requires manual upkeep on every schema change. c) Escape-character gymnastics; ruled out by test.
PICK: (a) -- Python prefix check, configurable prefix list.
Mirrors the production parity engine's MetaColumnFilter. Default prefixes: _AIRBYTE_, _FIVETRAN_. If a stream sneaks in a different metadata column, we add the prefix in code (one place) and every stream benefits.
4. Re-clone now to reset the anchor?
a) Yes -- DROP + CREATE CLONE today, get a fresh anchor matching current prod state. b) No -- keep the May 20 clone, it's still a valid frozen snapshot.
a) Fresh diff baseline; loses 4 days of existing snapshot audit trail; doesn't unlock Time Travel (still 1-day retention cap); requires Phase 0 freeze choreography. b) Existing snapshots remain meaningful; clone is increasingly stale relative to live prod (~5,500 rows of UPSERT drift per day); we proceed with current data immediately.
PICK: (b) -- don't re-clone yet.
A re-clone doesn't fix any of the noise problems and trades audit continuity for a slightly fresher reference. The fingerprint design works on the existing clone. If parity sweep reveals real divergence we can't explain, then a re-clone becomes a forensic step rather than a precaution.
5. Tool surface: replace counts panel, or add new fingerprint panel
a) Replace counts panel in snapshot.py. Old data goes away. b) Add fingerprint as panel #6 alongside the existing 5.
a) Single source of truth, no confusion about which number to trust. b) Both views available; comparing the two helps debug. Larger HTML, more wall-time per snapshot.
PICK: (a) -- replace.
The current counts panel is noise-laden by design. Keeping it as a side-by-side comparison normalizes the noisy view. Replace cleanly; the old snapshots remain on the site as historical record of the noisy era.

Escape Routes

Escape from PICK 1b (run EXCEPT only on DIFFER): if EXCEPT runtime per DIFFER table becomes unacceptable on large tables (ETHER.ORDERS 4B rows), gate EXCEPT behind a --magnitude flag and default to fingerprint-only.
Escape from PICK 2a (cursor-window always): add --all-time flag that drops the WHERE clause. Logged in the panel header so the user sees which mode produced the report.
Escape from PICK 3a (prefix check): if a new metadata column slips in (e.g., _DBT_* from a downstream transform), add the prefix to the exclusion list in one place. The Python helper handles it for all 208 streams.
Escape from PICK 4b (don't re-clone): if the parity sweep shows mass DIFFER, re-clone (one SQL statement, no real cost) and re-run. The cost of being wrong here is one phase 1 re-do.
Escape from PICK 5a (replace panel): historical snapshots stay on the site -- the noisy counts panel is preserved in those pages. New snapshots only show the fingerprint panel.

Load-bearing count: 1. The Snowflake HASH_AGG primitive. Acceptable -- AIRBYTE_DB and AIRBYTE_CLONE are on Snowflake; we're not introducing a foreign dependency, we're using the platform's native fingerprint function. If we ever move off Snowflake, every other parity tool also needs rewrites; this one isn't an outlier.

First Move

The smallest concrete proof Write a 40-line standalone Python script that does only step 1 of the new design: for every active stream, pull column list, project past _AIRBYTE_*, run HASH_AGG on both DBs, print SCHEMA.TABLE MATCH or SCHEMA.TABLE DIFFER (prod_fp clone_fp). No HTML, no publish, no panel integration. If > 90% of streams return MATCH on the existing clone, the design is correct and we can fold it into snapshot.py with confidence. Wall time estimate: 2-4 minutes for 208 streams.

What this design does NOT do

  • Change the existing clone. The May 20 anchor stands.
  • Touch the parity engine or its Cloud Run dashboard.
  • Modify airbyte_migration_tools scripts.
  • Require a Snowflake edition upgrade.
  • Add new external dependencies.
  • Introduce a config file (column prefixes default in code).
