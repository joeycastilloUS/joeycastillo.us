Build cadence reset surface — 5 endpoints in 5 phases

Execute the design at be/designs/2026-05-24-1659-design-cadence-reset.html. Bring reset capability into cadence so no out-of-band Python is ever needed again. ~13 pts. One board.

Plan — Awaiting Approval
Intent: Add five endpoints to the cadence Cloud Run service: /reset/pause, /reset/resume, /reset/wipe, /reset/baseline, /reset/status. All under X-Reset-Confirm: yes header guard. Compound /reset/baseline for the "just re-cloned" workflow. Dashboard gets a Reset button. Service account gets cloudscheduler.admin. No new infrastructure -- just C# code + IAM bump + a redeploy.

Approach

  1. SDK + IAM first. Phase 1 adds the Scheduler SDK and grants the SA cloudscheduler.admin. Without these the rest can't work.
  2. /reset/wipe alone. Phase 2 is the First Move proof -- replicate today's Python wipe behavior in C#. Deploy + smoke test.
  3. Pause + resume. Phase 3 adds Scheduler control. Verify by pausing, watching scheduler.state, resuming, watching next-run.
  4. Compound baseline. Phase 4 chains pause -> wipe -> tick -> resume in one endpoint. The canonical "I just re-cloned" call.
  5. Dashboard button. Phase 5 surfaces the Reset action in the live /dashboard. Confirm modal. Calls /reset/baseline via fetch with the operator's identity token.

Pre-flight

Before Phase 1: verify cadence Cloud Run service is healthy. Verify Scheduler is paused (current state from today's reset). Confirm the wipe-via-Python pattern matches what we want C# to do (deletion behavior identical). Branching: do this build on a branch reset-surface, merge to main after Phase 5 smoke passes -- or YOLO to main since cadence is single-operator. Recommend main, no branch ceremony.

Phases

1 SDK package + IAM bump

~20 min. Add Google.Cloud.Scheduler.V1 to csproj. Grant cadence-svc cloudscheduler.admin on the project. Update appsettings.json with Reset config block.
# csproj <PackageReference Include="Google.Cloud.Scheduler.V1" Version="3.7.0" /> # IAM SA="cadence-svc@acmedev-platform-hq.iam.gserviceaccount.com" gcloud projects add-iam-policy-binding acmedev-platform-hq \ --member="serviceAccount:$SA" --role="roles/cloudscheduler.admin"
Accept: Local build clean. gcloud projects get-iam-policy shows the new role on cadence-svc.

2 /reset/wipe endpoint (First Move)

~30 min. Implement the wipe using FirestoreDb.WriteBatch + Delete. Add header-guard helper. Deploy. Smoke against current state.
// Program.cs sketch app.MapPost("/reset/wipe", async (HttpContext ctx, FirestoreDb fs, IConfiguration cfg) => { if (ctx.Request.Headers["X-Reset-Confirm"] != "yes") return Results.BadRequest(new { error = "missing X-Reset-Confirm: yes header" }); var coll = cfg["Reset:SnapshotsCollection"] ?? "cadence-snapshots"; var docs = await fs.Collection(coll).GetSnapshotAsync(); var batch = fs.StartBatch(); int count = 0; foreach (var d in docs.Documents) { batch.Delete(d.Reference); if (++count % 500 == 0) { await batch.CommitAsync(); batch = fs.StartBatch(); } } if (count % 500 != 0) await batch.CommitAsync(); return Results.Ok(new { deleted = count }); });
Accept: Deploy. POST /reset/wipe with header -> { deleted: N }. Without header -> 400. Re-call returns { deleted: 0 } (idempotent).

3 /reset/pause + /reset/resume

~30 min. Add CloudSchedulerClient. Implement both endpoints. Idempotent via wasAlreadyPaused detection.
var scheduler = await CloudSchedulerClient.CreateAsync(); var jobName = JobName.FromProjectLocationJob( cfg["Reset:SchedulerProject"]!, cfg["Reset:SchedulerLocation"]!, cfg["Reset:SchedulerJobName"]!); await scheduler.PauseJobAsync(jobName); // or ResumeJobAsync
Accept: Pause -> gcloud scheduler jobs describe shows PAUSED. Resume -> ENABLED with next run within 5 min.

4 /reset/baseline (compound) + /reset/status

~30 min. Compound endpoint: call pause + wipe + RunSweepAsync (existing) + resume in sequence. Status endpoint reads scheduler state + firestore count + last sweep doc.
app.MapPost("/reset/baseline", async (... services ...) => { // 1. pause await scheduler.PauseJobAsync(jobName); // 2. wipe var deleted = await WipeSnapshotsAsync(fs); // 3. baseline tick var sweep = await FingerprintPrimitive.RunSweepAsync(snow, catalogPath, null, null); var snapshotId = $"baseline-{DateTimeOffset.UtcNow:yyyyMMddTHHmmss}"; await fs.Collection("cadence-snapshots").Document(snapshotId).SetAsync(...); // 4. resume await scheduler.ResumeJobAsync(jobName); return Results.Ok(new { wiped = deleted, baseline = sweep, resumedAt = DateTimeOffset.UtcNow }); });
Accept: One POST -> pause logged, wipe deleted X, baseline tick recorded as Firestore doc, resume completed. Total time: 60-180s depending on cold/warm.

5 Dashboard Reset button + CLAUDE.md update

~30 min. Add "Reset" button group to /dashboard. Confirm modal calls /reset/baseline with header + identity token. Update CLAUDE.md in airbyte-parity to mention /reset/* endpoints.
<!-- in /dashboard HTML --> <button onclick="if(confirm('Wipe N snapshots + re-baseline? This pauses the Scheduler briefly.')) { const t = ...identity token...; fetch('/reset/baseline', { method: 'POST', headers: { 'X-Reset-Confirm': 'yes', 'Authorization': 'Bearer '+t } }) .then(r => r.json()).then(j => alert('Done. Baseline: ' + j.baseline.match + '/' + j.baseline.total)); }">Reset baseline</button>
Accept: Dashboard renders Reset button. Click triggers confirm modal. Confirmed click runs full baseline; alert shows result. CLAUDE.md grep "reset" returns matches in cadence section.

Risks + mitigations

RiskMitigation
cloudscheduler.admin grant is too broadBind at project level for now. Could narrow to per-job resource binding later via conditions on the IAM binding.
Reset button accidentally clickedHeader guard + browser confirm modal. Two friction points. If still happens, add a 5-second hold-to-confirm.
/reset/baseline runs while Scheduler tick is in-flightPause is the first step. Tick is short (~30s warm). Worst case: an in-flight sweep completes and writes one extra doc which the wipe sees and deletes anyway.
SDK version conflict with Google.Cloud.FirestoreBoth Google Cloud SDKs share the same core (Google.Api.Gax etc). 3.7.0 of Scheduler is compatible with 3.7.0 of Firestore.
Header guard not enforced in some edge caseHelper function used in every endpoint. Unit-test the helper. Defensive default-deny.
Re-running /reset/baseline mid-sweep stacks operationsAdd a simple in-process lock (SemaphoreSlim). One reset at a time per instance. Cloud Run can have multiple instances; second instance gets 409 Conflict.

What this unlocks

  • One-click reset after every re-clone. Joey re-clones, hits the dashboard Reset button, baseline is fresh in 30-180 seconds.
  • No more Python wipe scripts. No more bash loops with curl parens-in-URL gotchas. The capability is part of the service.
  • Audit trail. Reset operations log to Cloud Run logs with the operator identity. Who reset when is queryable.
  • Foundation for future ops. /reset/* surface is the right place to add things like /reset/retention (delete snapshots older than N days), /reset/clear-warnings (clear ERROR-state streams), etc.

Open follow-ups (not in this board)

  • Public-access dashboard. Currently /dashboard requires identity token. IAP or gate.js wrap could land in a future board.
  • Per-stream subcollection writes. Currently snapshot doc has tallies; per-stream verdicts in subcollection would enable drill-down. Out of scope here.
  • Cost telemetry endpoint. /reset/status returns snapshot count + state; could add /cost/today with QUERY_HISTORY-derived $. Separate board.
