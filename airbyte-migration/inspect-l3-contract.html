The contract (from source)

AirbyteParity/Endpoints/ParityEndpoints.cs + Engine/Models/ParityResult.cs. Prod/clone DB pair comes from the server's ParityOptions — the caller only describes the stream.

POST /parity/l3 (also /parity/l1 for row-count delta; same body) body ParityRequest { schema: "ECOMM", table: "SPREE_CMS_SECTIONS", nonMetaColumns: ["ID","TITLE","BODY", ...], // required -- the business columns primaryKeyColumns: ["ID"] // optional } // sinceUtc / limit ignored for L3 200 L3Result { stream: { ... }, columns: [ ColumnProfilePair { column: "BODY", prod: { column, rowCount, nullCount, hash, min, max }, cloneSide: { column, rowCount, nullCount, hash, min, max }, rowCountMatches, nullCountMatches, hashMatches, minMatches, maxMatches }, ... ], verdict: "Match" | "Diff" | "Missing" | "Inconclusive" }

The per-column match flags are the localization: a Signal stream's drifted column is the one with hashMatches=false (or min/max/null mismatch). That is the "which column, how bad" answer Story D needs.
