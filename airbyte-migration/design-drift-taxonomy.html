Cadence drift taxonomy — make "signal" mean something

Every diff cadence sees must land in one of four buckets so an alert is only an alert when it should be. Three of them are explainable: timing lag, the old-vs-new-Airbyte version gap, and acknowledged differences. Only the fourth is a real problem. Adopt the verdict vocabulary the parity engine already shipped.

Design — Pure Metal

Subject + Intent

Subject: A classification model that turns every cadence per-stream diff into one of four verdicts -- clean cosmetic benign signal -- backed by three explanation sources (sync-lag windowing, old/new-Airbyte version awareness, operator acknowledgements). Intent: "183/208" today means nothing -- it's mostly sync lag. After this, a stream is only red when it's genuinely wrong; everything else carries a one-line reason. And the names match the parity engine (Board 6) so the two systems speak one language.

Discovery

  • From ~i ETHER sync cadence: ETHER drift = uniform 4h sync-generation lag. Windowing on the business cursor collapses it to a match. (Source 1: timing.)
  • New input from operator: the clone will be rebuilt on the NEW Airbyte (airbyte-philz, Destinations-V2). Prod is OLD Airbyte (_AIRBYTE_AB_ID, _AIRBYTE_EMITTED_AT, basic-normalization). New emits _airbyte_raw_id, _airbyte_extracted_at, _airbyte_meta, typing-and-deduping. So the clone can legitimately differ in metadata shape, normalization, and possibly column typing -- not corruption, not lag. (Source 2: version/migration.)
  • The parity engine (airbyte-parity Cloud Run, Board 6) already shipped a verdict set: Clean / Cosmetic / Benign / Signal with a DriftClassifier. Board 1 (carry-set) defines an ack workflow with a Firestore acknowledgements collection. (Source 3 + reuse.)
  • cadence is the cheap 5-min HASH_AGG heartbeat. It should classify what it can cheaply and escalate ambiguity to signal -- not reimplement the L3-L5 engine.

Choices + Consequences

1. Verdict vocabulary -- adopt engine's vs cadence-native
a) Adopt Clean/Cosmetic/Benign/Signal. b) New cadence words (clean/lag/version/signal).
a) One language across both systems; dashboard pills + DriftClassifier already exist to mirror. b) More literal to cadence's sources but forks the vocabulary; operator learns two dialects.
PICK: (a) -- adopt the engine's four.
Consistency beats literal. The engine already classifies into these; cadence becomes the cheap front-line screen using the same labels. Lag maps to Benign, version-format maps to Cosmetic, accepted diffs map to Benign-via-ack.
2. How does cadence recognize a version/Cosmetic difference cheaply?
a) Business column-SET comparison (names/types from INFORMATION_SCHEMA) + meta-prefix detection. b) Full L3 column-signal analysis. c) Don't -- call everything unwindowed a Signal.
a) Cheap (one INFORMATION_SCHEMA read, already done for biz cols), catches schema-shape drift from the Airbyte upgrade. b) Accurate but heavy -- that's the parity engine's job. c) Floods Signal with migration noise; defeats the purpose.
PICK: (a) -- column-set + meta-prefix screen.
cadence already resolves business columns per stream. Comparing the SETS (prod vs clone) is nearly free and directly flags "new airbyte changed the shape." Value-level version diffs that pass the column-set check fall to the ack path or Signal.
3. Accepting a known version/value difference -- ack store
a) New cadence-acks Firestore collection (stream + reason + ts + by). b) Reuse the parity engine's acknowledgements collection. c) Static drift-policy.json in the repo.
a) cadence owns its acks, simple, matches Board 1 shape. b) Shared truth but couples the two services' data. c) Versioned + reviewable but redeploy to change; no runtime ack.
PICK: (a) for runtime acks + (c) for durable policy.
A cadence-acks Firestore collection for operator "I accept this diff" (runtime, mirrors Board 1). A committed drift-policy.json for the durable knobs (tolerances, per-schema sync intervals, known-version-difference rules). Two layers: policy is code-reviewed; acks are operational.
4. Where does the classifier run -- cadence or defer to the parity engine
a) cadence classifies cheaply (windowing + column-set + ack); escalates true ambiguity to Signal. b) cadence calls the parity engine's /parity/l3 for every diff. c) Merge the two services.
a) cadence stays a fast heartbeat; Signal volume is low + meaningful; deep analysis stays in the engine on demand. b) Couples the 5-min loop to the heavy engine; latency + cost. c) Big rewrite, out of scope.
PICK: (a) -- cheap classify, escalate to Signal.
cadence answers "is this worth a human?" cheaply. When the answer is yes (Signal), the human (or the parity engine's L3-L5) investigates. Keeps the heartbeat cheap and the signal rare.
5. Cursor-less / NUMBER-cursor streams (TRANSACTIONS_V3, TAX_RATES)
a) Whole-table compare + row-count tolerance band, flagged "unwindowed". b) Force a synthetic cursor. c) Skip them.
a) Honest -- can't window without a timestamp cursor; a small row-count delta within lag tolerance reads Benign. b) Fragile. c) Blind spots.
PICK: (a) -- whole-table + row-count band, marked unwindowed.
Per the ~i, some streams have no timestamp cursor. For those, compare whole-table and treat a row-count delta proportional to lag as Benign; a structural/large delta is Signal. Mark them so the operator knows windowing wasn't applied.

Escape Routes

From PICK 1 (engine vocabulary): if cadence needs a distinction the engine's four don't capture, add a sub-reason field rather than a fifth verdict -- keeps the top-level language stable.
From PICK 2 (column-set screen): if shape-only screening misclassifies, escalate that stream to the parity engine's L3 column-signal for the real call; cadence caches the result.
From PICK 3 (two-layer acks): if the split is confusing, collapse to policy-only (no runtime acks) -- slower to accept a diff but fully versioned.
From PICK 4 (cheap classify): if Signal volume is too noisy, wire the optional /parity/l3 escalation for borderline cases.

First Move

The smallest proof Add the column-set comparison to the existing sweep: for each stream, diff the business column NAMES prod vs clone. Emit a per-stream shapeMatch: true|false into the /tick doc. On ETHER today (old airbyte both sides, frozen clone) shapeMatch is true everywhere -- proving the screen is correct and quiet. The day the clone flips to new airbyte, shapeMatch goes false on format-changed streams and we have caught the version gap with one cheap field, before building the full classifier.

What this does NOT do

  • Reimplement the parity engine's L3-L5 (column-signal, row-delta-explainer). cadence screens; the engine explains on demand.
  • Change the Airbyte sync schedules or the migration plan.
  • Auto-accept version differences -- a human acks them (or a policy rule does, explicitly).
  • Replace the raw match% -- it stays, with the verdict breakdown beside it.
