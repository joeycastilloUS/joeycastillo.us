Go.Replay — play the movie of the drift

cadence already keeps every 5-minute tick as an immutable snapshot. That pile of snapshots is a film reel nobody has watched. Go.Replay is the projector: step the timeline and watch the clone fall out of sync stream by stream — and, once it is fed, watch the signals collapse back to clean. Pure read over data that already exists.

Design — Pure Metal

Subject + Intent

Subject: A time machine over the verdict timeline. cadence persists every /tick as a snapshot carrying verdict counts + a per-stream drifts array. Go.Replay reconstructs the evolution from that series — aggregate (the whole-system movie), per-stream (one stream's verdict journey), and freeze-frame (the full picture at any past moment). Intent: Go.Signal answers "what is wrong NOW." Go.FirstDiff answers "when did each break." Replay answers the question between them — "show me the whole story unfold." It is the generalization of "show me from moment one to moment two, what is the diff."

Discovery

  • The buffer already exists: 127 snapshots in cadence-snapshots, 5 min apart (~10.5h and growing; retention not fired). Each doc: snapshotId, time, clean/cosmetic/benign/signal counts, and a drifts array (non-Clean streams w/ verdict + reason + lag; driftSchemaVersion=2).
  • The walk primitive already exists: GET /firstdiff walks the same series server-side to find each stream's first divergence. Replay generalizes that walk from "first frame of trouble" to "the whole film."
  • Go.Diff already diffs the last 2 sweeps; Go.Dash already renders live HTML from /reset/status. Replay is the same shape, extended over time.
  • The story in the reel: clone frozen, prod marching — streams fall out one by one (the cascade). When airbyte-philz feeds the clone, the same reel runs in reverse: signals collapse to clean. That recovery is the cutover proof, on film.
  • Sharp edge: a /reset/baseline (re-clone) is a cut in the film — fingerprints reset, the timeline restarts. Replay must mark the cut, not interpolate a lie across it.

Naming Pass

Go.Timelinedescriptive · but a timeline is a static thing; replay is an action you perform on it
Go.RewindVCR verb · but biased to going back; the reel also plays forward (the recovery)
Go.Scrubprecise for freeze-frame · too narrow -- it is only one of the three modes
Go.Replaywinner · the operator's own word · VCR verb that covers play / scrub / rewind · action over the reel

Keep the operator's word. "Replay" is the action (play the recorded reel); the three modes (aggregate / stream / freeze-frame) are arguments to it, not separate launchers — one verb, like Go.Ack's list/record/clear.

Data flow

cadence-snapshots (Firestore, already written every tick) | v GET /replay [?stream= ?from= ?to=] server-side projection (mirrors /firstdiff) | aggregate: [{snapshotId,time,clean,cosmetic,benign,signal}, ...] | stream: [{time, verdict, reason, lagHours}, ...] (Clean inferred by absence) v Go.Replay.ps1 -> inline-SVG HTML time-lapse + terminal one-liner -> pops in browser

Out of scope: A-vs-B arbitrary two-frame diff (Go.Diff already covers last-2; generalizing is a later nicety), forecasting, and re-running fingerprints over history (replay shows recorded frames, it does not recompute them).

Choices + Consequences

1. Where does the timeline walk run -- server endpoint vs PowerShell-direct-Firestore
a) New GET /replay projects the series server-side. b) Go.Replay reads cadence-snapshots from PowerShell directly.
a) Reuses the service's Firestore access + the snapshot schema lives in one place; mirrors the proven /firstdiff pattern; the launcher stays a thin renderer. b) A second place that must know the schema + its own Firestore creds; drifts from how every other Go.* works (they hit HTTP).
PICK: (a) -- GET /replay server projection.
/firstdiff already walks this exact series server-side. Replay is the same walk with a wider lens. One schema-owner, one auth path, thin client — consistent with the whole toolset.
2. Output -- HTML time-lapse vs terminal sparkline
a) HTML primary (the movie) + a terse terminal one-liner. b) Terminal-only ASCII sparkline. c) HTML-only.
a) The 127-point curve needs pixels to read; HTML pops in the browser like every Be artifact + Go.Dash; the one-liner gives instant in-terminal context. b) Instant but can't show a 4-series 127-point film legibly. c) Loses the quick glance.
PICK: (a) -- HTML movie + terminal one-liner.
Mirror Go.Dash exactly: generate HTML from a live endpoint, pop it. Print one line (window, ticks, first-signal time) so the terminal still tells you something without opening a browser.
3. The chart -- pure SVG/CSS vs a charting library
a) Inline SVG polyline/area + CSS, generated in the string builder. b) Chart.js / D3 from a CDN.
a) Pure metal, no framework, no build, no network dep, ages forever -- the Be rule. b) Prettier faster but a framework + CDN dependency in a vanilla-HTML house.
PICK: (a) -- inline SVG, no framework.
"No frameworks. Vanilla HTML/CSS/JS." A stacked-area of four verdict series is a handful of SVG <polygon> points computed in PowerShell. It is the discipline, and it never breaks.
4. One mode or three
a) Ship all three (aggregate / stream / freeze-frame) sharing one endpoint + params. b) Aggregate only for v1.
a) The stream lane is the one Joey will actually stare at (watch SPREE_CMS_SECTIONS break); aggregate alone is pretty but answers less. One endpoint with ?stream= + ?at= keeps it cheap. b) Faster to ship, but leaves the most useful view for "later" (= never).
PICK: (a) -- all three, one endpoint, thin renderer per mode.
The modes are arguments, not new services. The marginal cost over aggregate is small and the stream lane is the payoff. Phase them, but ship them.
5. Baseline cuts -- heuristic detect now vs explicit baselineEpoch field
a) Heuristic: detect a tick where clean jumps to ~total (a re-clone) and draw a cut marker. b) Add a baselineEpoch to each snapshot and segment on it.
a) Works on the snapshots we already have, zero schema change, good enough to stop the film lying across a re-clone. b) Correct and explicit but needs a /tick write change + only helps future snapshots.
PICK: (a) for v1, (b) as the follow-up.
The reel we want to watch right now (frozen clone) has no cut yet. Heuristic marking covers the first re-clone; the explicit field is the durable fix once we are writing new snapshots through a baseline anyway.

Escape Routes

From PICK 1 (/replay endpoint): if the projection gets heavy, the ?from/&to window (already in the contract) bounds it; retention already caps the buffer. Reversible to server-side downsampling.
From PICK 3 (inline SVG): if SVG generation fights back, fall back to a per-tick verdict-count table -- still a replay, just less cinematic. No rewrite.
From PICK 4 (stream lane on old ticks): pre-driftSchemaVersion=2 snapshots lack per-stream reasons -- clamp the lane window to "since v2" and label the boundary rather than guessing.
From PICK 5 (baseline heuristic): if the heuristic false-detects a cut, switch to the explicit baselineEpoch field (the planned follow-up). Fully reversible.

First Move

The smallest proof Add GET /replay returning only the aggregate series — [{snapshotId, time, clean, cosmetic, benign, signal}, ...] ordered by time. Hit it and confirm it returns the full 127-point curve from frozen-clone start to now. That one array is the movie's data; every render mode is downstream of it. If the array comes back ordered and complete, the read-projection is real and the rest is presentation.

What this does NOT do

  • Recompute fingerprints over history — it replays recorded snapshots, it does not re-sweep the past.
  • Change capture or the 5-minute cadence — read-only over what /tick already writes.
  • Replace Go.Diff (last-2) or Go.FirstDiff (first divergence) — those stay as quick shortcuts; replay is the general timeline lens.
  • Forecast — it shows the past, not the future.
