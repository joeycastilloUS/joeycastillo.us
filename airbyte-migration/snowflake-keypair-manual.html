Snowflake Key-Pair Migration — Manual Playbook

Human runs every command. No AI. Audit-ready.

Use this mode when: AI is unavailable, an audit observer is watching, you're running this procedure for the first time, or the change-management policy requires it. Same end state as the AI-assisted and TRON modes; just slower and fully manual. Refer to the master plan for strategy and rollback details.

Phase A — Generate keys

1 Create the keys directory Joey

mkdir -p ~/.snowflake-keys chmod 700 ~/.snowflake-keys cd ~/.snowflake-keys
Expect: directory exists, permissions 700.

2 Generate ACMEDEV_TOOLS_SVC keypair Joey

# Pick a strong passphrase, save it to your password manager read -s -p "ACMEDEV_TOOLS_SVC passphrase: " P1; echo # Encrypted private key openssl genrsa 2048 \ | openssl pkcs8 -topk8 -v2 aes256 -inform PEM -passout pass:"$P1" \ -out acme_tools_svc.p8 # Public key (PEM format -- this is what Wesley pastes into ALTER USER) openssl rsa -in acme_tools_svc.p8 -passin pass:"$P1" -pubout \ -out acme_tools_svc.pub cat acme_tools_svc.pub
Expect: -----BEGIN PUBLIC KEY----- ... -----END PUBLIC KEY----- printed. Save the passphrase ($P1) to your password manager labeled "ACMEDEV_TOOLS_SVC keypair passphrase."
If openssl errors on aes256: use aes-256-cbc in older builds, or omit -v2 aes256 for default cipher.

3 Generate CRUNCHTIME_USER refresh keypair Joey

read -s -p "CRUNCHTIME_USER passphrase: " P2; echo openssl genrsa 2048 \ | openssl pkcs8 -topk8 -v2 aes256 -inform PEM -passout pass:"$P2" \ -out crunchtime_user.p8 openssl rsa -in crunchtime_user.p8 -passin pass:"$P2" -pubout \ -out crunchtime_user.pub cat crunchtime_user.pub
Expect: a second PEM block printed. Save P2 to password manager.

4 Verify both files exist and are readable Joey

ls -la ~/.snowflake-keys/
Expect: 4 files (acme_tools_svc.p8 + .pub, crunchtime_user.p8 + .pub), all -rw-------. Mark Phase A complete.

7 Smoke test from Joey's laptop Joey

snowsql -a zn77432 -u ACMEDEV_TOOLS_SVC \ --private-key-path ~/.snowflake-keys/acme_tools_svc.p8 \ --private-key-passphrase "$P1" \ -q "SELECT current_user(), current_role()"
Expect: ACMEDEV_TOOLS_SVC | ACMEDEV_TOOLS_READ_ROLE. Mark Phase B complete.
If "JWT token is invalid": the RSA_PUBLIC_KEY on Snowflake doesn't match this private key. Re-run cat acme_tools_svc.pub, send to Wesley, ask him to re-run ALTER USER ACMEDEV_TOOLS_SVC SET RSA_PUBLIC_KEY = '...'.

Phase D — .NET driver supports key-pair

10 Edit SnowflakeOptions.cs Joey

Add 3 properties: PrivateKeyFile, PrivateKeyPassphrase, Authenticator. All optional. Default Authenticator empty (uses password path).

11 Edit SnowflakeClient.cs connection-string assembly Joey

If Authenticator == "SNOWFLAKE_JWT" and PrivateKeyFile is non-empty, build:

connStr = $"account={Account};user={User};role={Role};warehouse={Warehouse};" + $"authenticator=SNOWFLAKE_JWT;private_key_file={PrivateKeyFile};" + $"private_key_file_pwd={PrivateKeyPassphrase}";

Else fall back to existing password assembly. Keep both paths.

12 Add 4 tests in SnowflakeClientAuthTests.cs Joey

  • Password path unchanged (regression test)
  • JWT-with-passphrase path produces correct connection string
  • JWT-without-passphrase path produces correct connection string
  • Missing key file with JWT authenticator throws clear error
cd /c/airbyte-parity dotnet test AirbyteParity.Tests --filter "FullyQualifiedName~SnowflakeClientAuth"
Expect: 4/4 new tests pass + existing 277 pass = 281/281. Mark Phase D complete.
If existing tests break: the password assembly path was disturbed. Diff your changes; the password path must be untouched if Authenticator is unset.
