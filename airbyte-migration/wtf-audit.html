6. Anti-patterns — the named rules

AP-1: launcher constellation instead of operator dashboard
What happened: shipped 14+ separate Go.* launchers (Go.Status, Go.Signal, Go.Risk, Go.Escalate, Go.Replay, Go.Watch, Go.Live, Go.Dash, Go.Cutover, Go.Rehearse, Go.Diff, Go.FirstDiff, Go.Reclone, Go.Rebaseline). Operator must mentally assemble the picture across CLI invocations.
Example: cadence/Go.Dash.ps1 shows signal-first state; cadence/Go.Replay.ps1 shows timeline; cadence/Go.Live.ps1 shows one op. Same operator, three browsers tabs, none auto-refreshing.
Rule: Default surface for any "I want to see X" ask = section on the unified dashboard. New launcher only when no existing page is the right home AND the operator explicitly wants a CLI.
AP-2: per-board narrow gap fix without auditing whole operator journey
What happened: each board (23-27) solved a real gap (timestamps, SQL fields, auth step, CORS) but no board stepped back to ask "is the dashboard whole?"
Example: Board 27 fixed Go.Live's blank page. Go.Live now renders ONE op. The operator's actual ask -- a living dashboard -- was never the unit of work.
Rule: Every 3 boards, run a ~i operator-journey: walk through what the operator actually sees end to end before opening the next feature board. If the journey is broken, the next board is "fix the journey," not the next backlog item.
AP-3: verified my work in op doc JSON instead of on the operator's screen
What happened: declared "shipped" after reading step.sql: "would: DROP..." in Firestore. Operator's screen was blank.
Example: Board 25 verification was "dry op doc contains the field." Operator saw nothing rendered until Board 27 fixed the CORS rendering -- and even then, only for one op.
Rule: "Shipped" requires a rendered HTML screenshot OR a live browser page showing the new value. JSON in op doc is necessary but not sufficient. If I can't paste the operator-visible result, it's not done.
AP-4: claimed shipped before operator validated end-to-end
What happened: marked boards "✅ DONE" minutes after deploy, before the operator ran them.
Example: Board 24 item 1 (?dry=true) -- shipped + closed + retro'd; operator's first run came days later. Board 26 -- shipped after 3 iterative sub-bug fixes from my side, none with operator at the keyboard.
Rule: Boards stay In Progress until the operator runs the new value end-to-end and types "ok." I can deploy + dry-verify, but the board doesn't close until operator confirms.
AP-5: added new tools instead of fusing existing into one view
What happened: every "I want to see X" produced a new Go.X.ps1. The operator's mental model is one living page; my reflex was a new launcher.
Example: operator asked for timeline → I built Go.Replay (separate from Go.Live, separate from Go.Dash). Asked for "see cadence as we go" → built Go.Watch + Go.Live (still both fragmented).
Rule: Default action for any new operator-visibility ask = "add a section to Go.Dash (the unified dashboard)" not "Go.X new launcher." Launchers are the back door, not the front door.
