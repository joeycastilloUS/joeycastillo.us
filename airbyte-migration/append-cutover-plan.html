Both streams are full_refresh/append — the source dumps all rows every sync, and Airbyte appends them to the destination table instead of replacing. The table grows by N rows per sync, forever.

Why These Are Misconfigured

ETHER.data_warehouse.tax_rates

What it is: A reference table of tax rates. 534 rows represent the full dataset.

What happened: Configured as full_refresh/append on an hourly cron. Every hour, all 534 rows are appended. Over time, this created 3.36 million rows — the same 534 tax rates duplicated over 6,300 times.

The gap: Production has 3,363,325 accumulated rows. The clone (currently INACTIVE for this stream on airbyte-philz) has 144,714 fewer. This gap is not missing data — it is fewer hours of duplication.

Should be: full_refresh/overwrite — 534 rows, refreshed hourly, table stays at 534.

NETSUITE.custom_inventory

What it is: A custom inventory reference table. Approximately 523 unique items per sync, ~698 rows with variants.

What happened: Configured as full_refresh/append on a manual schedule. Each manual trigger appends all rows again. The table has accumulated 30,014 rows from repeated appends.

The gap: Production has 30,014 rows. The clone has 698 (one sync worth). The 29,316 row gap is entirely duplicate accumulation on prod that the clone hasn't replicated.

Should be: full_refresh/overwrite — ~698 rows, refreshed on demand, table stays at ~698.

Cutover Risk Analysis

Append streams are the highest risk category for a clone-rename cutover. Here is why:

The Problem with Append + Rename Swap

For incremental or overwrite streams, the cutover is clean: pause prod, verify parity, rename databases, resume. The data is the same shape on both sides.

For append streams, the data diverges by design. The clone has fewer appended copies than production. After the rename swap:

  • tax_rates would go from 3.36M rows to 3.22M rows — a visible drop of 144K rows
  • custom_inventory would go from 30K rows to 698 rows — a 97.7% drop

If any downstream query uses COUNT(*), GROUP BY _AIRBYTE_EMITTED_AT, or relies on the append timeline for auditing, it would break or produce different results after cutover.

This is not just lag — it is structural divergence. You cannot close the gap by running more syncs on the clone, because each sync adds a new batch of duplicates on both sides. The gap persists (and may grow) until the mode is changed.

Recommended Plan: Convert to Overwrite

The cleanest path is to fix the misconfiguration on both servers before the rename swap. This eliminates the divergence problem entirely.

Step 1: Verify No Downstream Dependency on Append History

Check whether any dbt model, dashboard, or report queries the _AIRBYTE_EMITTED_AT timeline on these two tables. Specifically look for:

  • SELECT ... FROM data_warehouse.tax_rates WHERE _AIRBYTE_EMITTED_AT > ...
  • GROUP BY DATE(_AIRBYTE_EMITTED_AT) on either table
  • Any audit or versioning logic that assumes one row per sync cycle

Expected result: No dependencies. These are reference tables — consumers query the latest values, not the append history. But verify before changing.

Step 2: Change Both Streams to Overwrite on airbyte-philz

Use the Airbyte API to update the connection catalog:

  • ETHER.data_warehouse.tax_rates → full_refresh | overwrite
  • NETSUITE.custom_inventory → full_refresh | overwrite

This is a catalog-level change via POST /v1/connections/update — change destination_sync_mode from append to overwrite in the stream config.

Step 3: Run One Sync Each

Trigger a manual sync for both connections after the mode change. On the first overwrite sync:

  • tax_rates: 3.36M rows → 534 rows
  • custom_inventory: 30K rows → ~698 rows

The tables shrink to their true size. All V2 metadata columns (_AIRBYTE_RAW_ID, _AIRBYTE_EXTRACTED_AT, _AIRBYTE_META) are populated fresh.

Step 4: Proceed with Standard Rename Cutover

With both streams on overwrite, the clone and production will have identical row counts for these tables after one sync each. The rename swap proceeds with zero append-related risk.

These two tables are now identical to every other full_refresh/overwrite stream in the system — no special handling needed.

Step 5: Update Production Connections Post-Cutover

After the rename swap completes and the new AIRBYTE_DB is live, update the production connection configs on airbyte-philz to match the overwrite mode. This ensures the fix persists going forward.

Do not skip this. If you only change the clone-side config, the old append mode will be inherited when the connection configs are migrated.

Alternative: Backfill from Production

If for any reason the append history must be preserved (audit requirement, compliance, etc.), the alternative is to backfill the missing rows from production into the clone before cutover:

-- Backfill tax_rates gap
INSERT INTO AIRBYTE_CLONE.ETHER.DATA_WAREHOUSE__TAX_RATES
SELECT * FROM AIRBYTE_DB.ETHER.DATA_WAREHOUSE__TAX_RATES
WHERE _AIRBYTE_EMITTED_AT > (
  SELECT MAX(_AIRBYTE_EMITTED_AT)
  FROM AIRBYTE_CLONE.ETHER.DATA_WAREHOUSE__TAX_RATES
);

-- Backfill custom_inventory gap
INSERT INTO AIRBYTE_CLONE.NETSUITE.CUSTOM_INVENTORY
SELECT * FROM AIRBYTE_DB.NETSUITE.CUSTOM_INVENTORY
WHERE _AIRBYTE_EMITTED_AT > (
  SELECT MAX(_AIRBYTE_EMITTED_AT)
  FROM AIRBYTE_CLONE.NETSUITE.CUSTOM_INVENTORY
);

Downside: This closes the gap but preserves the misconfiguration. The tables will continue to grow post-cutover. You will eventually need to convert to overwrite anyway — better to do it now while the migration provides a clean break point.
