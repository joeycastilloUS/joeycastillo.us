Approach

Per-service cutover with the password kept in env as a one-revision fallback. JWT path lights up first; only after 24 h of clean cron runs is the password env removed. Reversal is < 60s per service (re-add password env, redeploy prior revision). Snowflake side: ALTER USER … SET RSA_PUBLIC_KEY = … never removes the password — both auth methods coexist on Snowflake until Phase I.

B — Snowflake ALTER USER

3 pts · Wesley · on Snowflake (~10 min)

Wesley creates one new user and sets RSA public keys on three existing ones:

CREATE USER SNOW_SYNC_SVC PASSWORD = NULL DEFAULT_ROLE = SNOW_SYNC_ROLE; ALTER USER SNOW_SYNC_SVC SET RSA_PUBLIC_KEY = '<P4 pub>'; ALTER USER PARITY_SVC SET RSA_PUBLIC_KEY = '<P1 pub>'; ALTER USER CRUNCHTIME_USER SET RSA_PUBLIC_KEY = '<P2 pub>'; ALTER USER JOEY SET RSA_PUBLIC_KEY = '<P3 pub>';

Existing passwords stay in place — both auth methods coexist until Phase I.

Rollback: ALTER USER … UNSET RSA_PUBLIC_KEY; drop SNOW_SYNC_SVC.

Constraints

  • Per-service cutover order is independent except snow-sync's plaintext password should be rotated by Wesley before or with Phase F — treat the literal-env value as compromised.
  • No coordinated deploy window required; cron-driven services validate themselves on next tick.
  • Secret values never leave Joey's machine in plaintext; .p8 files are encrypted at rest with passphrases stored separately in Secret Manager.
  • Phases B, E, F, G need Wesley reachable for ~5 min each. Phases A, C, D, H, I, J are Joey-only.

What this plan does NOT do

  • Doesn't touch any non-Snowflake auth (CRUNCHTIME_SFTP_*, CRUNCHTIME_API_*, etc).
  • Doesn't change Snowflake roles, warehouses, or grants — only the auth method on existing users.
  • Doesn't migrate any third party (Airbyte, Fivetran, etc).
  • Doesn't add new monitoring — existing airbyte-parity dashboard already detects auth-induced drift.

Acceptance for the whole migration

  1. All 5 must-migrate services run a successful Snowflake query via JWT for 72 consecutive hours.
  2. All 4 Snowflake users show HAS_PASSWORD = false.
  3. Zero Cloud Run services have any SNOWFLAKE_PASSWORD env or secret binding.
  4. airbyte-parity test count holds at ≥ 281 (277 today + 4 new auth tests).
  5. Runbook merged + linked from CLAUDE.md.
