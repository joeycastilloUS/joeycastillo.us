Snowflake Key-Pair Migration — Master Plan

Consolidate snow-sync + airbyte-parity onto one Snowflake user with key-pair auth. Three execution playbooks; same end state.

Plan — Awaiting Approval
The Ask Approve the consolidation. One new Snowflake user (JOEY_BE) replaces PARITY_SVC for airbyte-parity and joey for snow-sync. Two roles split read from write. Wesley owns one SQL block (~5 min). Joey executes through whichever of three pre-built playbooks (manual / AI-assisted / TRON) matches the moment. Reversible per-phase.
Why now: 2FA enforcement on joey + PARITY_SVC is incoming. snow-sync still has its Snowflake password as a literal Cloud Run env var (visible to anyone with describe). This plan replaces the prior 2026-05-07 single-user-per-service approach with consolidation that matches the org's existing CRUNCHTIME_USER pattern and halves the ongoing rotation surface.

Approach

Per-service cutover with the password kept in env as a one-revision fallback (where applicable -- snow-sync removes it immediately because the literal value was already exposed). JWT path lights up first; password env removed only after 24 h of clean cron runs. Reversal is < 60s per service. Snowflake side keeps the old user's password set until Phase I cleanup -- both auth methods coexist.
