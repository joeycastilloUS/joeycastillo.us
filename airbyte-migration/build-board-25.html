~b Board 25 -- SQL visibility on reclone ops

Every destructive SQL statement now travels with its full receipt: the exact SQL before it runs, Snowflake's QUERY_ID when it runs, and Snowflake's official QUERY_HISTORY record after. Same fields render in dry mode with a "would:" prefix and no execution record. Verified dry; real path is the operator's trigger.

Shipped · Deployed · Verified (dry)
The fields, on the op doc steps. For drop-clone + create-clone:
  • sql — the exact text. Dry runs prefix with "would: ".
  • queryId — Snowflake's id (SELECT SYSTEM$LAST_QUERY_ID()). Null on dry.
  • snowflake — the QUERY_HISTORY_BY_SESSION row: executionStatus, startTime, endTime, totalElapsedMs, bytesScanned, rowsProduced, optional errorCode/errorMessage. Empty on dry.

1. ~u Server-side capture

ExecAndCapture(sql) replaces the old ExecSql -- runs the statement, captures SYSTEM$LAST_QUERY_ID(), then pulls the matching row from INFORMATION_SCHEMA.QUERY_HISTORY_BY_SESSION(RESULT_LIMIT => 100) with a defensive 30s timeout. History fetch is best-effort: if it fails (timeout, transient permissions), the op succeeds with step.snowflake.fetchError noted; the destructive op is never blocked by a logging failure.

SQL is written to the step doc before StartStep so it's visible the moment status flips to running.
