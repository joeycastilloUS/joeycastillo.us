Unified live operator dashboard

One page. Auto-refreshes. Re-clone → see we're in sync, see state every 5 min, click signals for explanation, dig in. Closes Board 28's WTF audit.

Design · Pure Metal

Subject + Intent

Subject: Replace Go.Dash with a single self-refreshing HTML page that consolidates the four operator views (current state, timeline with reclone cut markers, signal worklist with click-through, recent reclone ops). PowerShell polls server-side every 30s and writes the page to disk; browser auto-refreshes via <meta>. Intent: kill the launcher constellation (AP-1). The operator's mental model is one living page; the data plane already exists; this fuses it.

Discovery

  • Data sources all already exist: GET /reset/status (scheduler + last sweep + verdict counts), GET /escalate (gate + signals), GET /replay (timeline series + baseline cuts), GET /operations?limit=5 (recent reclones), GET /firstdiff (per-stream first divergence).
  • Rendering pattern proven by Board 27: PowerShell polls + writes static HTML + browser meta-refresh. Zero CORS, zero token in JS, works on file://.
  • "Click things and get explanation" resolves via HTML <details> + anchored sections. No JS fetch needed -- the same poll cycle pre-renders the drill content inline.
  • Reclone-as-cut markers on the timeline: /replay already has a baselineCuts heuristic; it didn't fire on the 2026-06-03 reclone (clean jumped 56%->100% but pre-clone fraction had been low for days). Improve by also pulling /operations?kind=reclone and overlaying those actual reclone events directly as cuts. Truth beats heuristic.

Choices + Consequences

1. Rendering -- server-poll + meta-refresh vs SPA fetch in browser vs Cloud Run static file
a) PS polls, writes HTML, browser meta-refresh. b) HTML with JS fetch (CORS issue per Board 27). c) Cadence service renders + serves HTML route.
a) Proven (Board 27); no CORS, no token in JS. b) Blocked at IAM (Board 27 inspect). c) Would require deploy churn for every page change + a new route + auth.
PICK: (a) -- same pattern as Go.Live, scales to many sections.
One architectural pattern across Go.Live + Go.Dash. Familiar. No CORS. Works offline-ish (page stays readable if cadence goes down).
2. Reclone-as-cut markers -- heuristic only vs actual /operations vs both
a) Keep the /replay clean-fraction heuristic. b) Pull /operations?limit=20 and overlay successful real reclones as cuts. c) Both -- heuristic finds unannounced cuts, real ops mark the known ones.
a) Misses 2026-06-03 21:23Z. b) Always right -- the op IS the cut. c) Defensive but the heuristic adds value only when ops collection is empty (won't happen now).
PICK: (b) -- real ops as the truth.
The op doc IS the reclone. Heuristic was a workaround for not having that data; now we have it.
3. Click-to-drill on signals -- expand inline vs new HTML file vs link to Go.Live
a) HTML <details> expand inline; drill content pre-rendered each poll. b) Each signal generates $env:TEMP\stream-XYZ.html; anchor link. c) Link to per-stream Go.Replay HTML.
a) All in one page; one refresh; readable offline. b) More files but cleaner per-stream context. c) Reuses Go.Replay but loses inline context.
PICK: (a) inline <details>. Drill content fits in ~10 lines per signal.
Operator click → instant expand, no navigation. Polled content stays fresh. Browser bookmarkable. AP-5 rule satisfied.
4. L3 deep-diff freshness on the worklist
a) Cache L3 per signal for 5 min (live Snowflake scans are expensive). b) Fire L3 every poll (~30s). c) Show only on first click ("fetch deep diff" button).
a) Pragmatic; signal state doesn't change minute-to-minute; one L3 per signal per 5min. b) Snowflake cost + slow page. c) Loses pre-rendered insight.
PICK: (a) 5-min cache per signal, stored alongside the page state.
L3 is the expensive primitive; cache it. Operator sees deep diff inline without firing the call manually.
5. Refresh cadence -- 30s vs 5 min vs match cadence ticks
a) 30s meta-refresh. b) 5 min (match cadence ticks). c) 60s.
a) Page feels alive; ops + gate flip is visible quickly; small cost. b) Operator stares at stale page between ticks. c) Compromise.
PICK: (a) 30s.
During a reclone, op state changes every minute or so; 30s catches transitions. Outside reclones, tick interval is 5 min but the operator can still see gate flips quickly. Cost is negligible (Firestore reads are cheap).

Escape Routes

From PICK 1 (PS poll): if the PS-poll model strains (operator wants the page on a phone, an auditor's machine without gcloud), expose a read-only HTTP endpoint on cadence that renders the same page server-side. The page contract is identical; only the host shifts.
From PICK 3 (inline details): if a signal has too much drill content to fit inline, link out to a per-stream file (escape to PICK 3b).
From PICK 4 (L3 cache): if L3 caching gets stale-feeling, add a manual "refresh deep diff for this signal" button that triggers a separate PS-side fetch.

First Move

Smallest proof: replace Go.Dash.ps1 with the new poll-loop + meta-refresh page that renders ONLY the hero (gate + sweep counts + scheduler) + the recent-ops list. Fire Go.Dash; page opens; auto-refreshes every 30s; operator sees the gate flip as it changes; clicks a recent op link → opens the existing Go.Live page. That's enough to prove the architecture and replace the broken existing Go.Dash. Timeline + worklist + drill add as phases 2-4.

What this does NOT do

  • Add new launchers (AP-1 rule).
  • Let the operator type Go.Risk into the browser (commands shown to copy; future board if needed).
  • Replace Go.Live (still the right home for ONE op deep view, linked from the dashboard).
  • Replace Go.Replay (still the right home for full timeline + per-stream lane drilldown; linked from the dashboard).
  • Remove existing launchers (Go.Signal, Go.Risk, Go.Escalate stay as the action surface; the dashboard is the read surface).
