What Was Built (5 phases batched)

Phase 1 -- SDK + IAM + Reset config block

Google.Cloud.Scheduler.V1 3.6.0 NuGet added to Cadence.csproj (plan called 3.7.0, but latest on nuget is 3.6.0). Reset config block added to appsettings.json with SchedulerProject/Location/JobName/SnapshotsCollection. IAM: cadence-svc@acmedev-platform-hq granted roles/cloudscheduler.admin via gcloud projects add-iam-policy-binding.
Accept: Local dotnet build clean (0 warnings, 0 errors after version bump). gcloud get-iam-policy shows the new role on cadence-svc.

Phase 2 -- POST /reset/wipe

FirestoreDb.StartBatch + Delete loop over cadence-snapshots with 500-doc commit batches. X-Reset-Confirm: yes header guard via shared RequireConfirm helper. SemaphoreSlim wraps the wipe to prevent concurrent destructive operations.
Accept: POST to live endpoint without header returns HTTP 400 with explicit error message. Wipe smoke deferred per handoff (live trajectory is the user-visible artifact).

Phase 3 -- POST /reset/pause + POST /reset/resume

CloudSchedulerClient.PauseJobAsync / ResumeJobAsync via BuildSchedulerAsync helper that resolves the JobName from config. Both endpoints idempotent: report wasAlreadyPaused / wasAlreadyEnabled for visibility. Resume returns the next scheduled run time.
Accept: Both endpoints reachable, both reject HTTP 400 without confirm header. Pause/resume smoke deferred (would interrupt unattended cadence).

Phase 4 -- POST /reset/baseline (compound) + GET /reset/status

Compound endpoint chains PauseJobAsync -> wipe loop -> RunSweepAsync -> ResumeJobAsync. The baseline snapshot is tagged isBaseline: true in Firestore. SemaphoreSlim.WaitAsync(0) returns HTTP 409 Conflict if another reset is in progress. Status endpoint reads scheduler job state + counts snapshots + returns latest sweep doc.
Accept: /reset/status verified live -- HTTP 200, scheduler.State=Enabled, snapshotCount=492, lastSweep at 19:50:04 (117/208, 56.2%). /reset/baseline POST without header -> HTTP 400. Compound smoke deferred until next re-clone.

Phase 5 -- Dashboard Reset button + airbyte-parity CLAUDE.md

MinimalDashboardHtml in Program.cs grew a "Reset surface" button bar with 5 buttons: Status, Pause Scheduler, Resume Scheduler, Wipe Snapshots, Re-baseline. doReset() JS helper injects the X-Reset-Confirm header for POSTs and prompts via confirm() for destructive actions. The Wipe + Re-baseline buttons get a danger style (red border, red text). CLAUDE.md in airbyte-parity grew a new "Cadence reset surface (Board 10, shipped 2026-05-26)" section documenting all 5 endpoints, the header guard, and the post-re-clone workflow.
Accept: Dashboard renders the new button bar inline. CLAUDE.md grep "reset" returns the new section.

Decisions Made

ForkPickWhy
5 separate deploys (plan) vs 1 batched deploy1 batchedPlan's per-phase deploys existed for smoke gating. With destructive smoke deferred, there's nothing to verify between phases that one final deploy doesn't cover. Collapsed ~3.5h of ceremony to ~1.5h.
Scheduler SDK version3.6.0 (not plan's 3.7.0)3.7.0 doesn't exist on nuget.org. Nearest version is 3.6.0; works fine with Firestore 3.7.0 since both share Google.Api.Gax core.
SemaphoreSlim per-process vs distributed lockPer-processCloud Run max-instances=2 in cloudbuild.yaml. A second instance hitting reset gets HTTP 409 Conflict. Operator-driven workflow doesn't need distributed; if it ever does, the escape route from the design is upgrade to Firestore-backed lock.
Header guard placementShared RequireConfirm4 mutating endpoints need it; one helper means no copy-paste drift. Read-only /reset/status skips the helper since there's nothing to confirm.
Dashboard button stylingInline in MinimalDashboardHtmlSame vanilla pattern as the rest of the dashboard. Five buttons, browser confirm(), fetch with header injection. Wipe + Re-baseline marked danger (red).
