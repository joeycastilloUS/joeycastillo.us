Cadence — C# Cloud Run microservice on philzcoffee for fresh-clone + 5-min fingerprint drift tracking

Fresh clone, baseline snapshot, then a 5-minute heartbeat showing how parity decays over time. C# service in the same Snowflake account region (us-west-2). Honest cost model: $250-2,600/month depending on optimizations -- the spread matters more than the headline number.

Plan — Awaiting Approval
Intent: Stand up a cadence microservice in the philzcoffee GCP project that (1) does a fresh CREATE OR REPLACE DATABASE AIRBYTE_CLONE CLONE AIRBYTE_DB, then post-clone cleanup, then (2) runs the column-pruned HASH_AGG fingerprint sweep across 208 active streams every 5 minutes. Each sweep gets stored as a snapshot and compared against the baseline (post-clone state, where MATCH should be ~100%) and against the previous sweep (delta-per-tick). A web endpoint surfaces the live drift trajectory. The point: see parity decay in real time. As clone gets stale, watch the MATCH count drop. Replaces the manual snapshot rhythm with a continuous signal.

Cost model (the headline)

Naive: 5-min cadence, no optimization

~$2,600 / mo
288 sweeps/day × ~5 min DEVELOPER_WH = 24 credits/day. At ~$3/credit Standard ed: ~$72/day × 30 = $2,160 + Cloud Run/Scheduler ~$15. Steep.

Batched: UNION ALL fingerprints

~$430 / mo
Single query per sweep returning all 208 HASH_AGGs. ~45s wall-clock; 0.0125 cred/sweep × 288 = 3.6 cred/day × ~$3 = ~$11/day = ~$330 + Cloud Run/Scheduler ~$15.

Batched + 15-min cadence

~$110 / mo
96 sweeps/day. Same per-sweep cost. Or stay at 5-min during the V2 discovery window and back off after.

Cost levers (in order of impact)

  1. Batch fingerprints into one query. ~6x reduction. Switch from 416 sequential HASH_AGG calls to one CTE: WITH t1 AS (SELECT 'X' name, HASH_AGG(...) fp FROM AIRBYTE_DB.X.A), t2 AS (...) SELECT * FROM t1 UNION ALL t2 UNION ALL .... Same per side, then a second query for clone.
  2. Stretch the cadence. 5 min → 15 min = 3x reduction. 5 min → hourly = 12x reduction. Trade resolution for cost.
  3. Skip MATCH streams after they've been MATCH N sweeps in a row. Only fingerprint streams that have changed recently. Could cut compute by 50-70% during steady-state. Adds complexity though.
  4. Pause auto-suspend tighter. DEVELOPER_WH auto-suspend is 60s. Each 5-min sweep would resume the WH and run 45s; idle 4 min before next sweep. That's 5 min of paid time per 5-min interval -- 100% paid utilization. If we batched + auto-suspended at 30s, we'd reclaim ~3.5 min idle credits per cycle.
  5. Conditional run. Skip the sweep if Cloud Run cold-start detects no schema changes in INFORMATION_SCHEMA.TABLES.LAST_ALTERED. Wouldn't help much given Airbyte writes constantly, but a slow-running stream might benefit.

Approach

  1. Plan-time confirmations. Verify edition (Standard vs Enterprise) for accurate credit pricing. Verify nothing else lives on philzcoffee that conflicts with a Cloud Run service named cadence.
  2. Setup once. Joey or the service runs CREATE OR REPLACE DATABASE AIRBYTE_CLONE CLONE AIRBYTE_DB followed by Phase 2 cleanup (drop _AIRBYTE_* clustering keys, fix orphan generation rows). This becomes the baseline anchor.
  3. Service scaffolds. .NET 8 ASP.NET Web API. Reuse the Snowflake.Data connection pattern from AirbyteParity. JOEY_BE keypair from Secret Manager.
  4. Fingerprint port. C# version of _fingerprint.py with the UNION ALL batching applied -- not just a literal port; the C# version is the fast version.
  5. Cron + storage. Cloud Scheduler hits POST /tick every 5 min. Result stored to Firestore (one doc per sweep, sub-collection per stream).
  6. Compare logic. Each sweep computes delta-vs-baseline (cumulative drift) and delta-vs-previous (per-tick churn). Both stored alongside the raw fingerprints.
  7. Dashboard. One HTML page at /dashboard. Live MATCH count graph (last 24h, 7d). Per-stream drift timeline. Cost-per-day tally.
  8. Cost telemetry. Each sweep records QUERY_HISTORY credits via SELECT SUM(CREDITS_USED_CLOUD_SERVICES + CREDITS_USED_COMPUTE) FROM SNOWFLAKE.ACCOUNT_USAGE.QUERY_HISTORY WHERE QUERY_TAG = ''. Dashboard shows running daily total.

3 Fingerprint primitive in C# (UNION ALL batched)

~3 hr. Port the Python helper. Apply the batching optimization. This is the speed unlock that drops per-sweep cost from $0.25 to $0.038.
// One query, all 208 fingerprints per side. Pseudo: // WITH fps AS ( // SELECT 'ECOMM.SPREE_USERS' AS stream, HASH_AGG("ID", "EMAIL", ...) AS fp // FROM AIRBYTE_DB.ECOMM.SPREE_USERS WHERE "UPDATED_AT" <= '...' // UNION ALL // SELECT 'ECOMM.SPREE_ORDERS', HASH_AGG(...) FROM AIRBYTE_DB.ECOMM.SPREE_ORDERS WHERE ... // UNION ALL ... (206 more) ... // ) // SELECT * FROM fps;
  • Build per-stream SELECT <literal> AS stream, HASH_AGG(<cols>) AS fp FROM ... SQL strings from stream-catalog.json + INFORMATION_SCHEMA column lookups (cached per service-start).
  • UNION ALL them. Run once on PROD, once on CLONE. Diff in memory.
  • Non-timestamp cursors: fallback to whole-table per-stream (same logic as Python helper).
  • Cache column lists per service-instance lifetime (refresh on schema drift; rare).
Accept: POST /tick returns within 60 seconds with payload { matchCount, differCount, errorCount, totalStreams, snapshotId, elapsedMs }. End-to-end credit consumed per tick <= 0.02 (measured via QUERY_HISTORY).

4 5-min cron + Firestore storage + per-tick deltas

~3 hr. Cloud Scheduler → HTTPS POST /tick. Each tick stores: snapshot doc + per-stream verdict subcollection + delta-vs-baseline + delta-vs-previous.
  • Scheduler job cadence-tick-5m on */5 * * * *. Uses Cloud Run service identity for auth.
  • Firestore collection cadence-snapshots/{snapshotId} doc with headline tallies + timestamp. Subcollection streams/{schema}.{table} with per-stream verdict.
  • Delta computation: at write time, read previous snapshot doc + baseline doc, compute deltas, store inline. Avoids client-side join.
  • Retention policy: keep all snapshots indefinitely (Firestore is cheap; ~210 docs × 288/day × 365 = 22M docs/year ≈ few GB).
Accept: Scheduler fires every 5 min for 1 hour. Firestore shows 12+ cadence-snapshots docs. Each doc has baselineMatchCount, previousMatchCount, currentMatchCount, creditsUsed.

Risks + mitigations

RiskMitigation
Cost exceeds budget. Naive 5-min cadence is ~$2,600/mo on Standard.UNION ALL batching is mandatory. Default cadence configurable; recommend 15-min after V2 discovery completes. Daily cost tile gives early warning.
UNION ALL query exceeds Snowflake statement-size limit on 208 streamsSnowflake's limit is generous (megabytes). 208 small SELECTs UNION ALL'd is well under. If we hit it, split into 2 queries of 104 each.
Schema drift between service-start and a tick (table column added on prod after column-list cache)Cache column lists with 1-hour TTL. On HASH_AGG error, invalidate cache and retry once.
Per-tick sweep overruns 5-min window during V2 discovery (lots of new columns)Tick is async-fire-and-forget from the scheduler perspective. Internal lock so two ticks can't overlap. If one runs long, the next one skips with a logged note.
QUERY_TAG cost attribution misses Cloud Services creditsQUERY_HISTORY returns both CREDITS_USED_COMPUTE and CREDITS_USED_CLOUD_SERVICES. Sum both.
Snowflake auto-suspend resumes WH on every tick = full 60s charged per tick even if query takes 45sDocument this. The cost model assumes worst-case 5 min paid per 5-min cycle.
Fresh clone interrupts an airbyte-philz cron run mid-flightVerify cron is FROZEN before re-clone. If unfrozen, freeze first.
Service crashes mid-tick; partial Firestore writesUse Firestore transactions. Sweep writes the headline doc LAST, after all per-stream docs are committed. Dashboard reads use headline doc as source of truth.

What this unlocks

  • Live drift visibility. Watch MATCH count drop over hours/days post-clone. Predict when re-clone is needed based on drift rate.
  • Detect prod anomalies fast. If MATCH count suddenly DROPS by 20 streams in a single tick, that's a prod-side mass update worth investigating. Currently invisible.
  • Cost-aware operation. Daily $ tile means we'll know before the bill arrives, not after.
  • Procedure-phase signal upgrade. Phase 3 V2 discovery DDL produces an immediate visible spike in DIFFER count. No need for explicit before/after snapshots.
  • Foundation for L4/L5. Same Firestore + dashboard pattern extends to row-level deep diffs as opt-in.
