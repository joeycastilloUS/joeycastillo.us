Snowflake language + features for parity checking on this account

What this Snowflake account (ZN77432, AWS us-west-2) actually gives us for comparing AIRBYTE_DB vs AIRBYTE_CLONE -- and the one trick that collapses the "noise" entirely.

Inspect — Hands-On Findings
Headline: The "noise" in every diff so far is the _AIRBYTE_* metadata columns (4 per table on average: AB_ID, EMITTED_AT, NORMALIZED_AT, HASHID). When we strip them and compare only business columns, prod and clone match exactly. HASH_AGG returns the same fingerprint and EXCEPT returns 0 rows in both directions. That's the design unlock -- everything else is detail. Time Travel is unusable on this account (1-day retention cap, clone is 3 days old), so the path forward is column-pruned HASH_AGG + cursor-windowed scoping, not historical replay.

What It Is

Joey asked what Snowflake language and features we can lean on for a noise-free parity check. This inspect is the hands-on inventory: which features exist, which work on this account (Standard edition signs), which we can use today, and which are fool's gold. Every claim below was tested against the live AIRBYTE_DB + AIRBYTE_CLONE tonight, not paraphrased from docs.

Stack -- the live test results

Test 1: Time Travel reach

SELECT COUNT(*) FROM AIRBYTE_DB.NETSUITE.CUSTOM_ITEM AT(OFFSET => -3600); -- 1 hour ago -- ⇒ 7636 (works) SELECT COUNT(*) FROM AIRBYTE_DB.NETSUITE.CUSTOM_ITEM AT(OFFSET => -82800); -- 23 hours ago -- ⇒ ERROR 000707: Time travel data is not available... beyond the allowed

Even within the supposed "1 day" retention, the actual reach is much shorter -- the table only has time-travel data for a few hours back. Going to clone_time (2026-05-20, 3 days ago) is impossible. Time Travel is dead for our use case.

Test 2: EXCEPT on raw tables (the noise)

SELECT COUNT(*) FROM ( SELECT * FROM AIRBYTE_DB.NETSUITE.CUSTOM_ITEM EXCEPT SELECT * FROM AIRBYTE_CLONE.NETSUITE.CUSTOM_ITEM ); -- ⇒ 7636 -- (CUSTOM_ITEM has 7636 rows total. ALL of them appear different.)

100% "difference" rate. Cause: _AIRBYTE_EMITTED_AT and _AIRBYTE_NORMALIZED_AT get fresh timestamps on every sync run. Clone is frozen at the last pre-clone-time sync; prod has run many times since. Same business rows, different metadata. HASH_AGG(*) confirms: -1971404395514912172 on prod vs -4684837344629546606 on clone.

Test 3: EXCEPT on business columns only (the unlock)

-- INFORMATION_SCHEMA confirms 8 cols total: 4 business, 4 _AIRBYTE_* SELECT COUNT(*) FROM ( SELECT col1, col2, col3, col4 FROM AIRBYTE_DB.NETSUITE.CUSTOM_ITEM EXCEPT SELECT col1, col2, col3, col4 FROM AIRBYTE_CLONE.NETSUITE.CUSTOM_ITEM ); -- ⇒ 0 (and 0 in the reverse direction) SELECT HASH_AGG(col1, col2, col3, col4) FROM AIRBYTE_DB.NETSUITE.CUSTOM_ITEM; -- ⇒ 8393363139480235559 SELECT HASH_AGG(col1, col2, col3, col4) FROM AIRBYTE_CLONE.NETSUITE.CUSTOM_ITEM; -- ⇒ 8393363139480235559 IDENTICAL
This is the design. Drop 4 metadata columns, run HASH_AGG on the rest, compare. Same fingerprint = the business data is bit-for-bit identical. One scalar value per table. ~2 seconds per medium table. Decisive verdict per stream: MATCH or DIFFER.

Patterns Found

1. The noise is structural, not stochastic

Every Airbyte sync rewrites _AIRBYTE_EMITTED_AT. The clone preserves whatever timestamp was current at clone time; prod marches forward. The "difference" is therefore deterministic: number of rows differing = number of rows in the table. That's why EXCEPT * said 7636/7636. Once we know this, the fix is obvious -- project away the metadata.

2. Column-set is stable per stream, derivable from INFORMATION_SCHEMA

We don't need to maintain a manual list of "business columns per table". INFORMATION_SCHEMA.COLUMNS gives us every column, we filter by name prefix, the result is the business projection. Same query works for any table on either DB. Self-updating as schemas evolve.

3. HASH_AGG is order-independent

Unlike HASH per row + then SUM, HASH_AGG is designed for "fingerprint a set of rows regardless of order". That matters because clone and prod may have rows physically stored in different orders (Snowflake doesn't guarantee scan order). HASH_AGG of the same multiset returns the same scalar.

4. The "in-flight" noise is separate from metadata noise

Even with column-pruning, rows whose cursor field has been UPDATEd on prod since clone time will still differ -- because prod's value for some business column changed. That's real divergence, not metadata churn. The cursor-windowed filter (only count rows with cursor <= clone_time - 24h) excludes those, leaving only structurally settled rows. Two separate filters, both useful, both compose cleanly.

5. Per-table parity is one HASH_AGG away

Once we have the column list, the parity check is one round-trip per side per table. 208 streams × 2 queries = 416 cheap queries. Snowflake caches column-pruned scans well. Estimate: 1-3 minutes total wall time for a full sweep, vs. the current per-row COUNT pattern that takes ~45s for counts alone.

Open Questions (for ~d to resolve)

  • Re-clone or not? Re-cloning gives a fresh anchor but doesn't unlock Time Travel (we're capped at 1 day retention by edition). The current clone (May 20) is still a valid prod snapshot for comparison. Lean: don't re-clone unless we have another reason to.
  • Cursor-windowed filter on top of column-pruning? Compose both: WHERE cursor <= clone_time - 24h applied to the projected SELECT. Probably yes for incremental streams, N/A for full-refresh / snapshot tables.
  • Verdict granularity: per-table MATCH/DIFFER Boolean, or also count of differing rows? Both are cheap; HASH_AGG gives Boolean, EXCEPT COUNT gives the magnitude. Probably show both in the panel.
  • What about non-timestamp cursors? The 7 streams that errored on the current cursor-windowed counts have bigint or string cursors. For HASH_AGG-only parity the cursor doesn't matter (we just hash the whole table); the cursor filter only matters when we want to time-bound. So those 7 streams probably "promote" to clean MATCH/DIFFER under the new design.
  • Pre-existing parity engine overlap. The C# parity engine already has MetaColumnFilter and column-set diffing for steady-state production. The snapshot tool is for procedure-bracketing. They can share the "business columns per table" logic but stay separate tools.
  • Auth fingerprint for HASH_AGG runtime. JOEY_BE + ACCOUNTADMIN + DEVELOPER_WH -- same path as today's snapshot. No change needed.
