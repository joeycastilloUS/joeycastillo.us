Be · airbyte-parity ~design: cadence reset surface

cadence reset surface — pause + wipe + baseline endpoints inside the C# service

Bring the reset capability into the same service that creates the data. No more out-of-band Python wipes against Firestore. Five endpoints, all idempotent, all OIDC + confirmation-header guarded.

Design — Pure Metal

Subject + Intent

Subject: Add a self-contained reset surface to cadence so the service can pause its own Scheduler, wipe its own Firestore snapshots, and re-baseline -- all from C# code in the same process that produces those snapshots in the first place. Intent: One place to look when something needs to be reset. No Python scripts. No "remember the gcloud incantation". The cadence service owns its lifecycle.

Discovery

Today's reset took three out-of-band steps:

  1. gcloud scheduler jobs pause cadence-tick-5m ...
  2. A Python loop calling Firestore REST API batch-commit delete (after pip install + auth dance + parens-in-URL workaround)
  3. Manual curl POST /tick with OIDC token to seed the baseline

Friction at every step. Worse: the reset logic lives in this conversation transcript rather than in the service. Next time, the next operator (or me-in-a-new-session) has to rediscover the choreography. Reset should be a button.

Available primitives in C# / .NET that make this trivial:

  • Google.Cloud.Firestore — already a package reference. WriteBatch.Delete() + CommitAsync() covers the wipe.
  • Google.Cloud.Scheduler.V1 — new package. CloudSchedulerClient.PauseJobAsync / ResumeJobAsync.
  • Existing FingerprintPrimitive.RunSweepAsync reused for the baseline tick.
  • ASP.NET endpoint mapping — trivial additions to Program.cs.

Naming Pass

/admin/*conventional · generic · doesn't say what it does
/control/*vague · control is a verb without object
/lifecycle/*accurate · long · bureaucratic
/ops/*short · ops-y · mostly empty meaning
/reset/*winner · describes the intent · matches what Joey said · verb the operator wants to do

Pick the noun that matches the operator's mental model. Joey said "I want to reset" -- so the surface is /reset/*. Sub-paths: /reset/pause, /reset/resume, /reset/wipe, /reset/baseline, /reset/status.

Shape

Five endpoints

EndpointMethodEffectIdempotent?
/reset/pausePOSTPause Cloud Scheduler job cadence-tick-5m. Returns {paused: true, wasAlreadyPaused: bool}.Yes
/reset/resumePOSTResume Scheduler. Returns {resumed: true, nextRun: ts}.Yes
/reset/wipePOSTBatch-delete all docs in cadence-snapshots collection. Returns {deleted: N}.Yes (re-run on empty = no-op)
/reset/baselinePOSTCompound: pause -> wipe -> run one /tick as baseline -> resume. Returns the baseline sweep result.Yes (the canonical "I just re-cloned" operation)
/reset/statusGETReturns {schedulerState, snapshotCount, lastSweepAt, lastSweepMatch, cloneCreatedAt}. Read-only health-check for the operator.Yes

Auth + safety

  • All endpoints require X-Reset-Confirm: yes request header. Prevents accidental browser/curl invocation.
  • Same OIDC identity-token gate as /tick (Cloud Run --no-allow-unauthenticated).
  • Compound /reset/baseline can be invoked via dashboard "Reset" button (which uses fetch + the user's identity token).

Service account IAM (new)

cadence-svc already has datastore.user (covers Firestore delete) and secretmanager.secretAccessor. New role needed:

  • roles/cloudscheduler.admin on the project (to call PauseJob/ResumeJob on its own Scheduler job). Could be narrowed to roles/cloudscheduler.jobRunner if that exists -- doesn't, the smallest is admin.

Configuration surface

Scheduler job name + location is fixed today. Make it configurable so swapping cadence or moving regions doesn't break reset:

{ "Reset": { "SchedulerProject": "acmedev-platform-hq", "SchedulerLocation": "us-central1", "SchedulerJobName": "cadence-tick-5m", "SnapshotsCollection": "cadence-snapshots" } }

All overridable via CADENCE_RESET__SCHEDULER_* env vars (the standard ASP.NET nested-config pattern).

Choices + Consequences

1. Where lives the wipe code -- C# in cadence vs sidecar Python vs Cloud Function
a) C# in cadence (proposed). b) Separate Python sidecar. c) Cloud Function triggered by Firestore TTL or HTTP.
a) One process, one repo, one auth identity, reuses existing IAM. b) Two languages, two CI pipelines, drift risk. c) New deploy target, separate IAM, more moving parts.
PICK: (a) -- C# in cadence.
Joey's explicit request. Co-location with the data writer. Reuses everything already in place.
2. Compound /reset/baseline vs separate ops only
a) Both -- granular ops + a compound for the common case. b) Granular only -- caller orchestrates. c) Compound only -- one button.
a) Operator picks the right tool; common case is one call. b) More flexible but every reset needs 3-4 calls. c) Less flexible; what if you only want to pause?
PICK: (a) -- both.
The granular ops are nearly free once we have the underlying libraries wired. The compound covers the "I just re-cloned" workflow in one click.
3. Confirmation guard -- header / query-string / request body
a) X-Reset-Confirm: yes header. b) ?confirm=yes query string. c) {"confirm": true} JSON body.
a) Hard to send by accident; explicit. b) Easy to accidentally copy-paste into a browser URL bar. c) Body parsing for POSTs that don't otherwise need a body.
PICK: (a) -- header.
Headers are invisible in browser URL bars, browser history, and shell history (mostly). Hardest to invoke accidentally. Matches the existing OIDC pattern.
4. Scheduler pause via gcloud subprocess vs Google.Cloud.Scheduler.V1 SDK
a) Add SDK package, call PauseJobAsync. b) Process.Start("gcloud", "scheduler jobs pause ...").
a) Strongly-typed, in-process, no subprocess overhead. New package dependency (~few MB). b) No new dep but spawns processes, no gcloud in the Cloud Run container by default.
PICK: (a) -- SDK.
gcloud isn't in the container. We'd have to install it. SDK is the right primitive in a C# service.
5. /reset/baseline ordering -- pause-wipe-tick-resume vs wipe-tick-pause-resume etc.
a) pause -> wipe -> tick -> resume. b) wipe -> pause-during-tick -> resume after.
a) No risk of a Scheduler tick racing during wipe; baseline tick is deterministic. b) Shorter pause window but tick could race with wipe.
PICK: (a) -- pause first.
Race-free. The "duration paused" is ~10s + the sweep time (~120s cold-start on fresh clone). Worst case we miss one scheduled tick. Acceptable.
6. Dashboard "Reset" button -- present or hidden behind a toggle
a) Always visible. b) Hidden behind ?admin=1 query string. c) Separate /admin page.
a) Discoverable; one accidental click away from disaster (mitigated by confirm modal). b) Hidden but anyone reading the HTML can find it. c) Cleaner but more friction.
PICK: (a) + confirm modal.
Already gated by OIDC at network layer. The header-confirmation is the real guard. UI confirm-modal adds the "are you sure" friction at the right moment.

Escape Routes

Escape from PICK 1a (wipe in C#): the Python REST loop still works as the emergency fallback if the C# endpoint is broken. Document it once in CLAUDE.md and forget.
Escape from PICK 2a (compound): if /reset/baseline behaves unpredictably, document deprecating it in favor of the granular ops. Caller orchestrates explicitly.
Escape from PICK 3a (header guard): if accidental invocation happens despite the header, add a second-factor (e.g. ?confirm=YYYY-MM-DD with today's date) without breaking the existing header contract.
Escape from PICK 4a (SDK): if the SDK is unmaintained or has a critical bug, replace with HTTP call to Scheduler REST API. Both speak the same protocol.
Escape from PICK 5a (pause-first): if the pause-during-tick window becomes a problem (e.g. we miss too many ticks), add a "skip-pause" flag to /reset/baseline for the fast-path.
Escape from PICK 6a (visible button): hide behind ?admin=1 if the visible button gets clicked accidentally. Trivial CSS change.

Pure Metal Test

Can this be built with no frameworks, no external runtime, no vendor lock?

ASP.NET Web API endpointsreplaceable -- any HTTP server framework
Google.Cloud.Firestore SDKload-bearing -- Firestore is the storage we picked
Google.Cloud.Scheduler.V1 SDKload-bearing -- Scheduler is what we're pausing
OIDC identity token authreplaceable -- standard JWT, any provider
Header-based confirmationtrivial -- one if-statement

Load-bearing count: 2. Both Google Cloud SDKs. Acceptable -- we're building a Cloud Run service that operates Firestore + Cloud Scheduler. Those ARE the platform.

First Move

The smallest concrete proof Add /reset/wipe endpoint only, using the already-injected FirestoreDb from Phase 4. ~30 lines in Program.cs + a header-guard helper. Deploy and call it to confirm a real wipe replicates today's Python behavior. If that works, the other four endpoints follow the same pattern with the Scheduler SDK and existing FingerprintPrimitive.

What this design does NOT do

  • Touch the parity engine on airbyte-parity Cloud Run. Cadence's reset only affects cadence's own state.
  • Re-clone Snowflake databases. The DROP/CREATE OR REPLACE is Joey's surgical operation; cadence doesn't own it.
  • Replace the snapshot tool in airbyte-parity. That tool is for manual procedure-bracketing; cadence is for continuous heartbeat.
  • Add a UI framework. Vanilla HTML + fetch() for the dashboard button. Same pattern as today.

Path to plan + build

~p formalizes this into a build plan (probably 4-5 phases / 13-15 pts: SDK install + IAM bump, wipe endpoint, pause/resume endpoints, baseline compound + status, dashboard button). After approval, ~b chain executes it.
